# 运维日志监控：部署与运行机制详解

> 本文面向**运维人员**，完整讲解如何部署 SuperBizAgent 项目，以及系统是如何实现"用 AI 自动监控服务器日志、分析告警、给出处理方案"这一核心能力的。

---

## 一、系统能做什么（运维场景说明）

运维人员触发 `/api/ai_ops` 接口后，系统会**全自动**完成以下任务：

```
1. 调用 Prometheus 获取当前所有活跃告警（哪些服务出问题了）
2. 根据告警名称，到知识库中检索对应的处理方案（公司内部运维手册）
3. 获取当前时间，调用腾讯云日志服务 MCP 工具查询相关日志
4. 对每条告警做根因分析 + 日志取证 + 方案执行
5. 自动生成一份结构化的《告警运维分析报告》
```

**不需要人工手动查日志、手动翻文档**，AI 全程自动驾驶。

## 三、部署步骤

### 3.1 第一步：部署 Milvus 向量数据库（Docker）

项目提供了完整的 `docker-compose.yml`，包含 Milvus 及其依赖（etcd + MinIO）和管理界面（Attu）。

```bash
# 进入 docker 目录
cd manifest/docker

# 启动所有服务（后台运行）
docker-compose up -d
```

启动后各服务端口：

| 服务 | 端口 | 说明 |
|---|---|---|
| Milvus | 19530 | 向量数据库主服务（Go SDK 连接此端口） |
| Milvus HealthCheck | 9091 | 健康检查 |
| MinIO 控制台 | 9001 | 对象存储管理界面 |
| Attu | 8000 | Milvus 可视化管理界面（浏览器访问） |

**验证 Milvus 是否正常：**

```bash
curl http://localhost:9091/healthz
# 返回 OK 即正常
```

---

### 3.2 第二步：配置腾讯云日志 MCP URL

日志监控的核心依赖是**腾讯云日志服务（CLS）的 MCP 接入**。运维人员需要：

**① 申请 MCP URL**

- 登录腾讯云控制台
- 进入"日志服务（CLS）" → "MCP 接入"
- 申请 SSE 类型的 MCP URL，格式为：`https://mcp-api.tencent-cloud.com/sse/XXXX`
- 参考官方文档：`https://cloud.tencent.com/document/product/614/118699`

**② 填入代码**

打开 `internal/ai/tools/query_log.go`，将申请到的 URL 填入：

```go
// 第 23 行
mcp_url := "https://mcp-api.tencent-cloud.com/sse/你的MCP_URL"
```

> ⚠️ **注意**：如果 `mcp_url` 为空字符串，程序启动时会直接 `panic`，提示："要使用GetLogMcpTool，请先按照教程申请mcp url"

**③ 配置日志主题**

在系统 Prompt 中需要指定日志主题参数（已内置于 `chat_v1_ai_ops.go` 和 `chat_pipeline/prompt.go`）：

```
日志主题地域：ap-guangzhou
日志主题id：869830db-a055-4479-963b-3c898d27e755
```

如果你的日志主题不同，需要修改这两处的 Prompt 配置，改为你自己的地域和主题 ID。

### 3.4 第四步：导入运维知识库文档

这是让 AI 知道"怎么处理告警"的关键步骤。

**文档格式要求**：Markdown（`.md`）文件，按 `#` 一级标题分节，每节描述一类告警的处理步骤。

**示例文档结构（`docs/告警处理手册.md`）：**

```markdown
# ServiceDown 服务下线告警
当出现 ServiceDown 告警时，处理步骤如下：
1. 首先查询最近 10 分钟的应用日志，关键词过滤 "panic" "fatal"
2. 检查数据库连接是否正常...
3. 重启服务命令：systemctl restart xxx

# HighCPUUsage 高CPU告警
当 CPU 使用率超过 90% 时：
1. 查询日志中是否有死循环迹象...
```

**批量导入命令（把所有 md 文件入库）：**

```bash
# 将文档放到 knowledge_cmd/docs/ 目录下
cp your-docs/*.md internal/ai/cmd/knowledge_cmd/docs/

# 运行批量入库脚本
go run internal/ai/cmd/knowledge_cmd/main.go
```

**或者通过 HTTP 接口上传单个文件：**

```bash
curl -X POST http://localhost:6872/api/upload \
  -F "file=@告警处理手册.md"
```

> 上传后会自动触发入库流水线：读文件 → 按标题切分 → Embedding → 写入 Milvus

### 4.2 Plan-Execute-Replan Agent 工作流

系统使用 **Plan-Execute-Replan** 架构，三个大模型角色分工协作：

```
┌─────────────────────────────────────────────────────────────────┐
│  第一阶段：Planner（DeepSeek-V3-Think，深度思考模型）             │
│                                                                 │
│  输入：固定的运维分析任务描述（Prompt）                            │
│  输出：结构化的执行计划，例如：                                    │
│    步骤1: 调用 query_prometheus_alerts 获取所有活跃告警           │
│    步骤2: 对每个告警调用 query_internal_docs 查处理方案           │
│    步骤3: 调用 get_current_time + 日志工具查询相关日志            │
│    步骤4: 汇总分析，生成报告                                      │
└──────────────────────┬──────────────────────────────────────────┘
                       │ 执行计划
┌──────────────────────▼──────────────────────────────────────────┐
│  第二阶段：Executor（DeepSeek-V3-Quick，快速工具调用模型）         │
│                                                                 │
│  逐步执行计划中的每个步骤，按需调用工具：                           │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ 工具1: query_prometheus_alerts  → 查 Prometheus 告警      │  │
│  │ 工具2: query_internal_docs      → 查 Milvus 知识库        │  │
│  │ 工具3: get_current_time         → 获取当前时间            │  │
│  │ 工具4: MCP Log 工具系列          → 查腾讯云日志            │  │
│  │ 工具5: mysql_crud               → 查数据库（可选）         │  │
│  └──────────────────────────────────────────────────────────┘  │
│  MaxIterations: 999999（不限制执行次数，直到任务完成）             │
└──────────────────────┬──────────────────────────────────────────┘
                       │ 执行结果反馈
┌──────────────────────▼──────────────────────────────────────────┐
│  第三阶段：Replanner（DeepSeek-V3-Think，深度思考模型）            │
│                                                                 │
│  判断：执行结果是否满足目标？                                       │
│  - 满足 → 输出最终报告，流程结束                                   │
│  - 不满足 → 重新制定计划，回到 Executor 继续执行                   │
│  最大迭代轮次：20（planexecute.Config.MaxIterations）             │
└─────────────────────────────────────────────────────────────────┘
```

### 4.3 日志查询工具的工作原理（核心）

**腾讯云 CLS + MCP 协议**是实现"实时查日志"的技术核心。

#### 连接建立过程

```go
// query_log.go

// 1. 创建 SSE 类型的 MCP 客户端，连接腾讯云 CLS MCP 服务
cli, err := client.NewSSEMCPClient(mcp_url)

// 2. 启动连接
cli.Start(ctx)

// 3. MCP 协议握手初始化
initRequest.Params.ProtocolVersion = mcp.LATEST_PROTOCOL_VERSION
cli.Initialize(ctx, initRequest)

// 4. 获取 MCP 服务暴露的所有工具（腾讯云自动注册的日志查询工具列表）
mcpTools, err := e_mcp.GetTools(ctx, &e_mcp.Config{Cli: cli})
```

#### MCP 工具能做什么

连接成功后，腾讯云 CLS MCP 会自动暴露一组日志查询工具，常见能力包括：

- **按时间段查询日志**：给定开始/结束时间和关键词，返回匹配的日志条目
- **按日志主题查询**：指定 `地域` + `日志主题ID` 精确定位到某个服务的日志
- **日志聚合统计**：统计某段时间内的错误次数、QPS 等指标

#### Agent 如何触发日志查询

Executor 在执行"查日志"步骤时，流程如下：

```
Executor 推理：需要查询 ServiceDown 告警的相关日志
  ↓
选择 MCP Log 工具（由腾讯云 CLS 注册）
  ↓
构造参数：
  - region: "ap-guangzhou"（来自 Prompt 中的配置）
  - topicId: "869830db-a055-4479-963b-3c898d27e755"
  - startTime: (通过 get_current_time 工具先获取当前时间，再减去用户指定时长)
  - endTime: 当前时间
  - query: "ServiceDown OR panic OR fatal"（基于告警名和知识库内容自动构造）
  ↓
MCP 工具执行 → 腾讯云 CLS API → 返回匹配日志
  ↓
Executor 拿到日志内容 → 继续推理分析根因
```

## 五、Prometheus 告警监控的工作原理

`query_metrics_alerts.go` 中实现了对 Prometheus Alerts API 的查询：

```go
// 查询地址（本机 Prometheus）
baseURL := "http://127.0.0.1:9090"
apiURL := fmt.Sprintf("%s/api/v1/alerts", baseURL)
```

**前提条件**：服务器上需要已运行 Prometheus，并配置好告警规则（Alert Rules）。Agent 查询到的字段：

| 字段 | 说明 |
|---|---|
| `alertname` | 告警名称（如 `ServiceDown`、`HighCPUUsage`） |
| `description` | 告警描述（来自 Prometheus Alert Rule 的 `annotations.description`） |
| `state` | 状态：`firing`（触发中）/ `pending`（待触发） |
| `activeAt` | 告警激活时间（RFC3339 格式） |
| `duration` | 自动计算的持续时长（如 `2h30m15s`） |

> ⚠️ **注意**：代码中 `queryPrometheusAlerts` 函数第 54 行有一个 `return PrometheusAlertsResult{}, nil`，这意味着当前 Prometheus 查询功能**返回空结果**（仅做演示框架），实际使用需要删除这行早返回，让后续的 HTTP 请求真正执行。

## 八、当前已知的限制和注意事项

- **Prometheus 查询**：当前代码有提前返回（空结果），需要修改才能真正查询本机 Prometheus（删除 `query_metrics_alerts.go` 第 54 行的 `return`）
- **MCP URL**：必须先申请才能使用，否则服务启动就会 panic
- **日志主题 ID**：Prompt 中硬编码了 `ap-guangzhou` 和具体的主题 ID，需要改为自己的
- **知识库必须先入库**：如果 Milvus 中没有文档，AI 查不到处理方案，会输出"未找到相关文档"
- **模型 API Key 安全**：不要把 API Key 硬编码到代码里，使用配置文件或环境变量管理
