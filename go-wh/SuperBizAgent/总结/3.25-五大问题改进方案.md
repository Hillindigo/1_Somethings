# 五大问题改进方案

---

## 快速概览

| # | 问题 | 改动位置 | 改动量 | 收益 |
|---|---|---|---|---|
| 1 | 每次请求重建 Graph | controller + pipeline | 中 | 性能大幅提升 |
| 2 | 会话记忆重启丢失 | utility/mem | 中 | 数据持久化 |
| 3 | RAG TopK=1 且无阈值 | retriever.go | 小 | 召回质量提升 |
| 4 | 日志主题 ID 硬编码 | prompt.go + config | 小 | 多环境可配置 |
| 5 | AIOps 无超时控制 | controller + context | 小 | 防止接口永久挂起 |

---

## 一、每次请求重建 Graph（性能问题）

### 简单说

`BuildChatAgent` 被放在每次请求的处理函数里，导致每次用户发消息都要重新初始化 Milvus 连接、创建 Embedding 客户端、编译 Graph。这些操作很重，只需要做一次。

**改法**：把 `BuildChatAgent` 的调用移到服务启动时（`main.go`），把返回的 `runner` 存到 `ControllerV1` 结构体里，请求处理时直接用。

---

### 详细改法

#### 当前代码结构（问题所在）

```go
// internal/controller/chat/chat_v1_chat.go
func (c *ControllerV1) Chat(ctx context.Context, req *v1.ChatReq) (res *v1.ChatRes, err error) {
    runner, err := chat_pipeline.BuildChatAgent(ctx)  // ← 每次请求都执行这里
    // ...
    out, err := runner.Invoke(ctx, userMessage, ...)
}

// internal/controller/chat/chat_v1_chat_stream.go
func (c *ControllerV1) ChatStream(...) {
    runner, err := chat_pipeline.BuildChatAgent(ctx)  // ← 又执行一次
    sr, err := runner.Stream(ctx, userMessage, ...)
}
```

每次请求都会触发：
- `milvus.NewRetriever` → 建立 Milvus 连接
- `ark.NewEmbedder` → 初始化豆包 Embedding 客户端
- `openai.NewChatModel` → 初始化 DeepSeek 客户端
- `react.NewAgent` → 初始化 ReAct Agent
- `g.Compile(ctx, ...)` → 编译整张有向图

#### 改法步骤

**第一步：修改 `ControllerV1` 结构体，新增 `runner` 字段**

```go
// internal/controller/chat/chat_new.go

type ControllerV1 struct {
    service *sse.Service
    // 新增：预编译好的 runner，全局复用
    runner  compose.Runnable[*chat_pipeline.UserMessage, *schema.Message]
}

func NewV1() chat.IChatV1 {
    ctx := context.Background()
    runner, err := chat_pipeline.BuildChatAgent(ctx)  // 启动时初始化一次
    if err != nil {
        panic(fmt.Sprintf("初始化 ChatAgent 失败: %v", err))
    }
    return &ControllerV1{
        service: sse.New(),
        runner:  runner,
    }
}
```

**第二步：Chat / ChatStream 直接用 `c.runner`，不再重建**

```go
// chat_v1_chat.go
func (c *ControllerV1) Chat(...) {
    // 删除：runner, err := chat_pipeline.BuildChatAgent(ctx)
    out, err := c.runner.Invoke(ctx, userMessage, compose.WithCallbacks(...))
}

// chat_v1_chat_stream.go
func (c *ControllerV1) ChatStream(...) {
    // 删除：runner, err := chat_pipeline.BuildChatAgent(ctx)
    sr, err := c.runner.Stream(ctx, userMessage, compose.WithCallbacks(...))
}
```

**第三步：线程安全确认**

Eino 编译后的 `Runnable` 是无状态的（请求上下文通过参数传入），天然支持并发调用，不需要加锁。

#### 改动涉及的文件

```
internal/controller/chat/chat_new.go         ← 结构体新增字段 + 启动时初始化
internal/controller/chat/chat_v1_chat.go     ← 删除 BuildChatAgent 调用
internal/controller/chat/chat_v1_chat_stream.go  ← 删除 BuildChatAgent 调用
```

## 二、会话记忆重启丢失（持久化问题）

### 简单说

当前用 Go 的 `map[string]*SimpleMemory` 存历史消息，进程重启全部消失。改成 Redis 存储，重启后对话历史仍在。

**改法**：把 `SimpleMemory` 的 `SetMessages` / `GetMessages` 操作替换为 Redis 的 `RPUSH` / `LRANGE`，JSON 序列化消息后存入，`MaxWindowSize` 通过 `LTRIM` 控制。

---

### 详细改法

#### 当前代码结构

```go
// utility/mem/mem.go

// 内存存储，key=会话ID，value=消息列表
var SimpleMemoryMap = make(map[string]*SimpleMemory)

type SimpleMemory struct {
    ID            string
    Messages      []*schema.Message
    MaxWindowSize int   // 最多保留 6 条
    mu            sync.Mutex
}

func (c *SimpleMemory) SetMessages(msg *schema.Message) {
    // append 到内存切片，超出 MaxWindowSize 则丢弃头部
}

func (c *SimpleMemory) GetMessages() []*schema.Message {
    // 直接返回内存切片
}
```

#### 改法步骤

**第一步：新建 Redis 版本的 Memory 实现**

在 `utility/mem/` 下新建 `redis_mem.go`，实现与当前 `SimpleMemory` 相同的接口：

```go
// utility/mem/redis_mem.go

type RedisMemory struct {
    ID            string
    MaxWindowSize int
    rdb           *redis.Client  // go-redis 客户端
}

// key 格式：chat:memory:{会话ID}
func (r *RedisMemory) key() string {
    return fmt.Sprintf("chat:memory:%s", r.ID)
}

func (r *RedisMemory) SetMessages(ctx context.Context, msg *schema.Message) error {
    data, _ := json.Marshal(msg)
    // RPUSH：追加到列表尾部
    r.rdb.RPush(ctx, r.key(), data)
    // LTRIM：只保留最后 MaxWindowSize 条（从右边算）
    r.rdb.LTrim(ctx, r.key(), int64(-r.MaxWindowSize), -1)
    // 设置过期时间（比如 7 天），避免 Redis 内存无限增长
    r.rdb.Expire(ctx, r.key(), 7*24*time.Hour)
    return nil
}

func (r *RedisMemory) GetMessages(ctx context.Context) ([]*schema.Message, error) {
    // LRANGE 取全部
    vals, _ := r.rdb.LRange(ctx, r.key(), 0, -1).Result()
    msgs := make([]*schema.Message, 0, len(vals))
    for _, v := range vals {
        var msg schema.Message
        json.Unmarshal([]byte(v), &msg)
        msgs = append(msgs, &msg)
    }
    return msgs, nil
}
```

**第二步：Redis 连接统一在 `utility/client` 中初始化**

```go
// utility/client/redis.go

func NewRedisClient(ctx context.Context) (*redis.Client, error) {
    addr, _ := g.Cfg().Get(ctx, "redis.addr")
    rdb := redis.NewClient(&redis.Options{
        Addr: addr.String(),  // 例如 "localhost:6379"
    })
    if err := rdb.Ping(ctx).Err(); err != nil {
        return nil, err
    }
    return rdb, nil
}
```

**第三步：配置文件新增 Redis 配置**

```yaml
# config/config.yaml
redis:
  addr: "localhost:6379"
  password: ""
  db: 0
```

**第四步：调用方改用 `RedisMemory`**

```go
// chat_v1_chat.go

// 原来
History: mem.GetSimpleMemory(id).GetMessages()
mem.GetSimpleMemory(id).SetMessages(schema.UserMessage(msg))

// 改为
redisMem := mem.GetRedisMemory(id, rdb)
History: redisMem.GetMessages(ctx)
redisMem.SetMessages(ctx, schema.UserMessage(msg))
```

#### 注意事项

- `schema.Message` 中的 `ToolCalls` 字段是嵌套结构，JSON 序列化时需要确认完整性
- Redis 存的是 JSON 字符串列表，调试时可直接 `LRANGE chat:memory:xxx 0 -1` 查看
- 如果 Redis 不可用，可以降级回内存版本，不要直接 panic

#### 改动涉及的文件

```
utility/mem/redis_mem.go              ← 新建，Redis 版本实现
utility/client/redis.go               ← 新建，Redis 连接初始化
internal/controller/chat/chat_v1_chat.go         ← 切换调用
internal/controller/chat/chat_v1_chat_stream.go  ← 切换调用
config/config.yaml                               ← 新增 redis 配置项
```

## 三、RAG TopK=1 且无相似度阈值（召回质量问题）

### 简单说

只取最相似的 1 条文档，信息量太少，容易答不完整。更严重的是没有相似度下限，就算文档和问题完全不相关，那 1 条也会被强行注入 Prompt，给 AI 带来噪声干扰。

**改法**：`TopK` 改为 3，同时加一个相似度分数过滤，低于阈值的文档不注入。

---

### 详细改法

#### 当前代码

```go
// internal/ai/retriever/retriever.go

r, err := milvus.NewRetriever(ctx, &milvus.RetrieverConfig{
    Client:      cli,
    Collection:  common.MilvusCollectionName,
    VectorField: "vector",
    OutputFields: []string{"id", "content", "metadata"},
    TopK:      1,       // ← 问题1：只取1条
    Embedding: eb,
    // 没有 ScoreThreshold ← 问题2：无阈值过滤
})
```

#### 改法步骤

**第一步：TopK 改为 3，加 ScoreThreshold**

```go
r, err := milvus.NewRetriever(ctx, &milvus.RetrieverConfig{
    Client:         cli,
    Collection:     common.MilvusCollectionName,
    VectorField:    "vector",
    OutputFields:   []string{"id", "content", "metadata"},
    TopK:           3,      // 取最相似的 3 条
    Embedding:      eb,
    // ScoreThreshold: 0.7,  // 相似度低于 0.7 的不返回（需确认 eino-ext milvus retriever 是否支持此字段）
})
```

> **注意**：`ScoreThreshold` 是否支持取决于 `eino-ext` 的 `milvus.RetrieverConfig` 结构定义。
> 需要查看源码或文档确认字段名，如果不支持，则在拿到 `[]Document` 后手动过滤：

```go
// 手动过滤低相似度文档（在 prompt.go 或 lambda_func.go 中处理）
filtered := make([]*schema.Document, 0)
for _, doc := range docs {
    score, ok := doc.MetaData["score"].(float64)
    if !ok || score >= 0.7 {  // 没有 score 字段则放行，有则过滤
        filtered = append(filtered, doc)
    }
}
```

**第二步：TopK 参数从配置文件读取（便于调优）**

```yaml
# config/config.yaml
milvus:
  addr: "localhost:19530"
  top_k: 3
  score_threshold: 0.7
```

```go
topK, _ := g.Cfg().Get(ctx, "milvus.top_k")
// RetrieverConfig.TopK = topK.Int()
```

**第三步：理解 TopK 增大的影响**

| TopK | Prompt 长度增加 | 效果 |
|---|---|---|
| 1 | 最小 | 信息可能不足，容易答不完整 |
| 3 | 中等 | **推荐**，覆盖度和 Prompt 长度平衡 |
| 5 | 较大 | 适合文档粒度细、知识点分散的场景 |
| >10 | 大 | 可能超出模型上下文长度，且噪声增加 |

#### 改动涉及的文件

```
internal/ai/retriever/retriever.go   ← TopK 改为 3，加 ScoreThreshold（或后处理过滤）
config/config.yaml                   ← 新增 milvus.top_k 和 milvus.score_threshold
```

## 四、日志主题 ID 硬编码在 Prompt（多环境配置问题）

### 简单说

腾讯云日志的地域和主题 ID 直接写死在系统提示词字符串里，换测试环境、换业务方向，都要改代码重新部署。应该抽到配置文件，代码里读配置拼接。

**改法**：`systemPrompt` 改为模板字符串，地域和主题 ID 从配置文件读取后在服务启动时注入。

---

### 详细改法

#### 当前代码（两处硬编码）

```go
// internal/ai/agent/chat_pipeline/prompt.go（第38行）
var systemPrompt = `
...
- 日志主题地域：ap-guangzhou；日志主题id：869830db-a055-4479-963b-3c898d27e755
...`

// internal/controller/chat/chat_v1_ai_ops.go（第16行）
"5. 涉及到日志的查询,需要先通过日志工具获取相关日志信息，参数必须携带地域和日志主题。"
// AIOps 的 query 里没有显式写日志主题 ID，但 Planner 的 Prompt 同样可能需要
```

#### 改法步骤

**第一步：配置文件新增日志相关配置**

```yaml
# config/config.yaml
cls:
  region: "ap-guangzhou"
  topic_id: "869830db-a055-4479-963b-3c898d27e755"
```

**第二步：`prompt.go` 中 `systemPrompt` 改为函数，动态读配置注入**

```go
// internal/ai/agent/chat_pipeline/prompt.go

// systemPromptTemplate 是带占位符的模板
var systemPromptTemplate = `
# 角色：对话小助手
## 核心能力
...
## 互动指南
- 在回复前，请确保你：
  ...
  • 日志主题地域：{cls_region}；日志主题id：{cls_topic_id}
...`

// BuildSystemPrompt 从配置读取后拼接完整系统提示词
func BuildSystemPrompt(ctx context.Context) (string, error) {
    region, err := g.Cfg().Get(ctx, "cls.region")
    if err != nil {
        return "", err
    }
    topicID, err := g.Cfg().Get(ctx, "cls.topic_id")
    if err != nil {
        return "", err
    }
    prompt := strings.NewReplacer(
        "{cls_region}", region.String(),
        "{cls_topic_id}", topicID.String(),
    ).Replace(systemPromptTemplate)
    return prompt, nil
}
```

**第三步：`newChatTemplate` 改为接收动态 Prompt**

```go
// 原来
func newChatTemplate(ctx context.Context) (ctp prompt.ChatTemplate, err error) {
    config := &ChatTemplateConfig{
        Templates: []schema.MessagesTemplate{
            schema.SystemMessage(systemPrompt),  // 静态字符串
            ...
        },
    }
}

// 改为
func newChatTemplate(ctx context.Context) (ctp prompt.ChatTemplate, err error) {
    builtPrompt, err := BuildSystemPrompt(ctx)
    if err != nil {
        return nil, err
    }
    config := &ChatTemplateConfig{
        Templates: []schema.MessagesTemplate{
            schema.SystemMessage(builtPrompt),  // 动态配置注入
            ...
        },
    }
}
```

**第四步：AIOps 的 query 里同样处理**

```go
// chat_v1_ai_ops.go
func (c *ControllerV1) AIOps(ctx context.Context, ...) {
    region, _ := g.Cfg().Get(ctx, "cls.region")
    topicID, _ := g.Cfg().Get(ctx, "cls.topic_id")
    query := fmt.Sprintf(`
...
"5. 涉及到日志的查询,需要先通过日志工具获取相关日志信息，
    参数必须携带地域(%s)和日志主题(%s)。"
...`, region.String(), topicID.String())
}
```

#### 改动涉及的文件

```
internal/ai/agent/chat_pipeline/prompt.go   ← systemPrompt 改为模板+动态读配置
internal/controller/chat/chat_v1_ai_ops.go  ← query 中的日志参数动态拼接
config/config.yaml                           ← 新增 cls.region / cls.topic_id
```

## 五、AIOps 无超时控制（接口挂起风险）

### 简单说

`plan_execute_replan` 的 Executor `MaxIterations: 999999`，实际上没有上限。如果 AI 在工具调用上陷入循环（比如日志查询一直超时，Agent 一直重试），这个 HTTP 请求会永远不返回，连接资源耗尽。

**改法**：在 `AIOps` 接口入口处用 `context.WithTimeout` 设置整体超时，同时把 Executor 的 `MaxIterations` 改为合理值（如 30）。

---

### 详细改法

#### 当前代码

```go
// chat_v1_ai_ops.go
func (c *ControllerV1) AIOps(ctx context.Context, req *v1.AIOpsReq) (res *v1.AIOpsRes, err error) {
    // ctx 是 HTTP 框架传入的，没有额外超时
    resp, detail, err := plan_execute_replan.BuildPlanAgent(ctx, query)
    // 如果 Agent 卡死，这里永远不返回
}

// executor.go
MaxIterations: 999999,  // 实际无限制
```

#### 改法步骤

**第一步：`AIOps` 入口设置整体超时**

```go
// chat_v1_ai_ops.go
func (c *ControllerV1) AIOps(ctx context.Context, req *v1.AIOpsReq) (res *v1.AIOpsRes, err error) {
    // 设置 5 分钟整体超时（AIOps 分析任务较重，时间可适当放宽）
    ctx, cancel := context.WithTimeout(ctx, 5*time.Minute)
    defer cancel()

    resp, detail, err := plan_execute_replan.BuildPlanAgent(ctx, query)
    if err != nil {
        // context.DeadlineExceeded 说明超时了
        if errors.Is(err, context.DeadlineExceeded) {
            return nil, errors.New("AI 分析超时，请稍后重试")
        }
        return nil, err
    }
    ...
}
```

**第二步：Executor `MaxIterations` 改为合理值**

```go
// executor.go
return planexecute.NewExecutor(ctx, &planexecute.ExecutorConfig{
    Model: execModel,
    ToolsConfig: adk.ToolsConfig{...},
    MaxIterations: 30,  // 原来 999999，改为 30
    // 对于告警分析任务：
    // 告警数 × (查知识库 + 查时间 + 查日志 + 分析) ≈ 4步/告警
    // 10个告警 × 4步 = 40步，30 足够覆盖大多数场景
})
```

**第三步：普通 Chat 接口也加超时（可选）**

```go
// chat_v1_chat.go
func (c *ControllerV1) Chat(...) {
    ctx, cancel := context.WithTimeout(ctx, 2*time.Minute)  // 对话类超时短一些
    defer cancel()
    out, err := c.runner.Invoke(ctx, userMessage, ...)
}
```

**第四步：配置文件化超时时间（便于调整）**

```yaml
# config/config.yaml
timeout:
  ai_ops_seconds: 300    # AIOps 超时 5 分钟
  chat_seconds: 120      # 普通对话超时 2 分钟
```

#### 超时时间选取参考

| 场景 | 建议超时 | 理由 |
|---|---|---|
| 普通对话（/chat） | 60~120 秒 | RAG + ReAct 正常 10~30 秒内完成 |
| 流式对话（/chat_stream） | 120~180 秒 | 流式首包更快，但总时长可能更长 |
| AIOps 分析（/ai_ops） | 300~600 秒 | 涉及多轮工具调用，任务重 |

#### Eino 框架的 context 传播说明

Eino 的 Graph 在执行时会将传入的 `ctx` 透传给每个节点和工具。当 `ctx` 超时后：
- 正在进行的 HTTP 请求（Prometheus、腾讯云日志 API）会被中断
- 正在执行的大模型调用也会被中断
- Graph 的 `Invoke` / `Stream` 方法会返回 `context.DeadlineExceeded` 错误

因此只需要在入口处设置一次超时，整条链路都会自动响应，**不需要在每个工具里单独加超时**。

#### 改动涉及的文件

```
internal/controller/chat/chat_v1_ai_ops.go   ← WithTimeout + 超时错误处理
internal/controller/chat/chat_v1_chat.go     ← WithTimeout（可选）
internal/ai/agent/plan_execute_replan/executor.go  ← MaxIterations 改为 30
config/config.yaml                            ← 新增 timeout 配置项
```

## 综合改动顺序建议

改动之间有依赖关系，建议按如下顺序进行：

```
第一步：问题4（日志主题配置化）—— 纯配置改动，最简单，先热身
第二步：问题5（加超时控制）—— 两处小改，立竿见影
第三步：问题3（TopK 调整）—— 单文件小改，效果可验证
第四步：问题1（Graph 单例化）—— 需要改结构体，依赖前面配置化完成
第五步：问题2（Redis 持久化）—— 改动最大，放最后，可单独分支处理
```
