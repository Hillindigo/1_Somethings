## 一、项目定位（一句话版本）

> "我主要做的是 Go + AI Agent 方向，在 SuperBizAgent 项目里负责了 RAG 对话管道、知识库构建、AIOps 自动化运维三个核心模块的设计与实现，对 AI Agent 的工程落地——特别是 RAG 检索、多模型协作分工、工具安全机制——有比较深入的实践经验。"

---

## 二、项目背景（面向面试官的痛点描述）

企业机房运维团队面对两个核心痛点：

1. **告警响应慢**：Prometheus 触发告警后，运维人员要手动翻处理手册、登录腾讯云控制台查日志、分析根因，整个过程可能耗费几十分钟，依赖个人经验，且容易漏处理。
2. **知识检索难**：企业内部运维知识散落在各类 Markdown 文档中，查阅不方便，新人上手慢。

**我们的解决方案**：通过 AI Agent 平台，将「告警感知 → 知识检索 → 日志分析 → 报告生成」全流程自动化，同时提供智能对话入口让运维人员直接问系统。

## 三、黄金3分钟开场白（完整版）

### 第一分钟：业务背景与痛点

> "我做的这个项目叫 SuperBizAgent，是面向企业机房运维场景的 AI 智能体平台。
> 运维团队的核心痛点是：**告警响应慢、知识检索难**。
> 比如 Prometheus 触发一条告警，运维人员要手动翻处理手册、登录腾讯云控制台查日志、分析根因，整个过程可能要花几十分钟。
> 我们的目标是把这个流程全自动化——告警进来，系统自动查文档、查日志、生成分析报告。"

### 第二分钟：技术架构与个人贡献

> "技术上，我用 Go 语言 + GoFrame v2 搭建 HTTP 服务层，核心 AI 能力基于字节跳动开源的 Eino 框架进行图编排。我负责了三条核心 AI 管道的设计与实现：
>
> **第一条：RAG + ReAct 融合对话管道**
> 用户提问时，系统并行做两件事：从 Milvus 向量数据库检索企业内部知识文档，同时构建对话消息。两路结果汇聚后渲染 Prompt，交给 ReactAgent 进行多轮工具调用推理，工具包括：查 MySQL、查 Prometheus 告警、查腾讯云日志（通过 MCP 协议接入）、查内部知识库。
>
> **第二条：知识库自动构建管道**
> 把 Markdown 文档自动分块、向量化、写入 Milvus，设计了幂等写入机制：写入前先按文件路径删除旧数据再重新写入，同一文档重复上传不产生脏数据。
>
> **第三条：AIOps 自动化运维管道**
> 采用 Plan-Execute-Replan 三阶段架构。规划和重规划用 DeepSeek V3-Think 深度推理模型，执行工具调用用 DeepSeek V3 快速模型，合理分配推理资源；执行偏差时自动触发重规划，最大迭代 20 次防止死循环。"

### 第三分钟：技术亮点与业务成果

> "几个我认为有意思的设计：
>
> 1. **并行管道设计**：RAG 检索和消息构建并行执行，端到端响应速度相比串行提升约 30%；
> 2. **MCP 协议动态扩展工具**：腾讯云 CLS 日志工具通过 MCP SSE 协议接入，不修改核心代码就能扩展新工具，符合开闭原则；
> 3. **数据库操作安全机制**：AI 执行任何 SQL 之前都需要人工 y/n 确认，从架构层面防止 AI 误操作生产数据；
> 4. **滑动窗口会话记忆**：最大保留 6 轮对话，成对丢弃保证 User/Assistant 上下文对齐，避免上下文错位。
>
> 整体上，将告警从触发到分析报告生成全流程自动化，理论上将运维响应时间从数十分钟压缩到分钟级。"

## 四、高频追问 Q&A

### Q1：RAG 检索精准度怎么保证？TopK=1 会不会太少？

> **答**：TopK=1 是有意设计。我们的场景是企业内部运维处理手册，文档结构化程度高，用告警名称做语义检索命中率很高，TopK=1 能减少噪声干扰。
> 如果扩展到更泛化的知识问答，方案是调高 TopK 同时加 reranker 二次排序，这是下一步可迭代的方向。

### Q2：Plan-Execute-Replan 和普通 ReAct 的区别是什么？

> **答**：ReAct 是单步 Think-Act 循环，适合相对简单的工具调用场景；Plan-Execute-Replan 适合 AIOps 这种多步骤、强依赖中间结果的任务。
> 区别在于：PER 增加了全局规划阶段（Planner 制定完整步骤列表）和重规划阶段（Replanner 在执行偏差时自动修正计划），整体鲁棒性显著优于单轮 ReAct。
> 我们还做了模型分工：规划用推理模型保证质量，执行用快速模型降低延迟。

### Q3：会话记忆为什么用内存而不是 Redis？

> **答**：当前是 MVP 阶段，内存实现足够满足需求，且已用 `sync.Mutex` 保证并发安全。
> 风险是服务重启丢失历史。生产化方向是将 SimpleMemory 替换为 Redis 持久化存储，项目里已预留了 `go-redis/v9` 依赖，改造成本很低。

### Q4：MCP 协议是什么？为什么用它？

> **答**：MCP（Model Context Protocol）是一个标准化协议，让 AI 模型能动态发现和调用外部工具，类似 AI 领域的 OpenAPI 规范。
> 我们通过 MCP SSE 客户端接入腾讯云 CLS 日志查询工具。好处是工具扩展无需修改 Agent 核心代码，只要对方实现了 MCP Server 就能即插即用，符合开闭原则。

### Q5：多用户并发 SSE 连接怎么管理？怎么保证消息不串流？

> **答**：用 GoFrame 的 `gmap.StrAnyMap`（并发安全 Map）管理多客户端连接。每个 SSE 连接建立时生成唯一 `client_id`，通过 `context.WithValue` 在请求链路传递，确保 AI 推理结果精准推送给对应客户端，不会串流。

### Q6：Milvus 向量维度 dim=81920 怎么来的？

> **答**：由豆包 ARK Embedding 模型的输出决定，dim=81920 对应 BinaryVector 存储格式。BinaryVector 相比浮点向量存储更紧凑、检索速度更快，适合知识库量级的场景。

### Q7：遇到的最大技术挑战是什么？

> **答**：AIOps 管道的可靠性设计。单轮 ReAct 在多步骤任务中容易执行偏差后陷入循环，我们引入 Replanner 模块评估每轮执行结果，偏差时自动重规划，同时设置最大迭代 20 次上限防止死循环。另外 MySQL 工具的安全机制也是重点——AI 自动化场景下，误操作生产数据的风险需要从架构层面规避，而不是依赖模型自觉。
