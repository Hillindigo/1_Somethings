# 知识库 Agent 运行机制详解

> 本文档基于 `SuperBizAgent` 项目源码深度分析，详细阐述知识库 Agent 的完整运行机制。
> 项目框架：GoFrame + **Eino**（字节跳动 AI 编排框架）
> 向量数据库：**Milvus**
> Embedding 模型：**豆包（Doubao）Ark Embedding**
> 对话大模型：**DeepSeek-V3（OpenAI 兼容接口）**

---

## 一、整体架构概览

知识库 Agent 由两条独立 Pipeline（有向图）组成，分别负责两个阶段：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        知识库 Agent 全貌                              │
│                                                                     │
│  ┌─────────────────────────────────┐                                │
│  │   Pipeline 1：知识入库（索引）     │  ← 文件上传 / 批量导入触发       │
│  │   KnowledgeIndexing Graph       │                                │
│  └─────────────────────────────────┘                                │
│                  ↓ 写入                                              │
│          ┌──────────────┐                                           │
│          │  Milvus 向量库 │                                           │
│          └──────────────┘                                           │
│                  ↑ 查询                                              │
│  ┌─────────────────────────────────┐                                │
│  │   Pipeline 2：知识检索+对话       │  ← 用户提问触发                  │
│  │   ChatAgent Graph（RAG+ReAct）  │                                │
│  └─────────────────────────────────┘                                │
└─────────────────────────────────────────────────────────────────────┘
```

---

## 二、Pipeline 1：知识入库流水线（KnowledgeIndexing）

### 2.1 触发方式

有两种方式触发知识入库：

| 触发方式 | 入口文件 | 说明 |
|---|---|---|
| HTTP 文件上传 | `internal/controller/chat/chat_v1_file_upload.go` | 用户通过 API 上传单个文件 |
| 本地批量导入 | `internal/ai/cmd/knowledge_cmd/main.go` | 遍历 `./docs` 目录下所有 `.md` 文件批量入库 |

### 2.2 流水线节点构成

入口函数：`knowledge_index_pipeline.BuildKnowledgeIndexing(ctx)`

```
START → [FileLoader] → [MarkdownSplitter] → [MilvusIndexer] → END
```

**有向图编译配置：**
- 图名称：`KnowledgeIndexing`
- 触发模式：`compose.AnyPredecessor`（任意前驱节点完成即触发下一节点）

---

### 2.3 各节点详解

#### 节点 1：FileLoader（文件加载器）

- **包**：`github.com/cloudwego/eino-ext/components/document/loader/file`
- **代码**：`knowledge_index_pipeline/loader.go`
- **作用**：读取磁盘上的文件（目前主要是 `.md` 文档），将其转化为 `[]document.Document` 结构
- **输入**：`document.Source{URI: filePath}`（文件路径）
- **输出**：`[]document.Document`（原始文档列表）

> ⚠️ **注意**：目前 `FileLoaderConfig` 是空配置（`&file.FileLoaderConfig{}`），使用默认参数。

---

#### 节点 2：MarkdownSplitter（文档分割器）

- **包**：`github.com/cloudwego/eino-ext/components/document/transformer/splitter/markdown`
- **代码**：`knowledge_index_pipeline/transformer.go`
- **作用**：按 Markdown 标题层级（`#`）进行文档切分，每个标题区块作为一个独立 chunk
- **分割策略**：
  - 按 `#` 一级标题拆分，标题字段映射为 `title`
  - `TrimHeaders: false`：保留标题文本本身
  - 每个 chunk 分配一个新的 UUID（`uuid.New().String()`）作为唯一 ID
- **输入**：`[]document.Document`
- **输出**：切分后的 `[]document.Document`（每个 chunk 一条）

---

#### 节点 3：MilvusIndexer（向量入库器）

- **包**：`github.com/cloudwego/eino-ext/components/indexer/milvus`
- **代码**：`knowledge_index_pipeline/indexer.go` + `internal/ai/indexer/indexer.go`
- **作用**：
  1. 调用豆包 Ark Embedding 模型对每个 chunk 进行向量化
  2. 将向量 + 原始内容 + 元数据一起写入 Milvus Collection
- **Embedding 模型**：`DoubaoEmbedding`（通过 `ark.NewEmbedder` 初始化，读取配置文件中 `doubao_embedding_model` 节）
- **Milvus Collection**：`biz`（`common.MilvusCollectionName`）
- **字段结构**：

| 字段名 | 类型 | 说明 |
|---|---|---|
| `id` | VarChar(255) | 主键，UUID |
| `vector` | BinaryVector(81920 dim) | 文本向量 |
| `content` | VarChar(8192) | 原始文本内容 |
| `metadata` | JSON | 元数据（含 `_source` 文件来源路径） |

---

### 2.4 幂等处理（去重逻辑）

每次入库前会执行**幂等去重**，防止同一文件被重复索引：

```
1. 用 FileLoader 加载文件，取第一个 doc 的 metadata["_source"]（即文件路径）
2. 在 Milvus 中执行 Query：metadata["_source"] == "<文件路径>"
3. 如果查到已有记录 → 提取所有 id → 执行 Delete 删除旧数据
4. 再执行 KnowledgeIndexing Graph 将新数据写入
```

**关键代码（`chat_v1_file_upload.go:buildIntoIndex`）：**
```go
expr := fmt.Sprintf(`metadata["_source"] == "%s"`, docs[0].MetaData["_source"])
queryResult, err := cli.Query(ctx, common.MilvusCollectionName, []string{}, expr, []string{"id"})
// ... 删除旧记录 ...
ids, err := r.Invoke(ctx, document.Source{URI: path}, compose.WithCallbacks(...))
```

---

### 2.5 完整入库流程图

```
用户上传文件 / 批量导入脚本
        ↓
  buildIntoIndex(ctx, filePath)
        ↓
  ┌─────────────────────────┐
  │ 1. FileLoader 预加载     │  → 获取 metadata["_source"]
  └─────────────────────────┘
        ↓
  ┌─────────────────────────────────────┐
  │ 2. Milvus Query：查找同源旧数据       │
  └─────────────────────────────────────┘
        ↓（有旧数据则删除）
  ┌──────────────────────────────────────────────────────┐
  │ 3. KnowledgeIndexing Graph 执行（r.Invoke）           │
  │                                                      │
  │   FileLoader → MarkdownSplitter → MilvusIndexer     │
  │       ↓               ↓                ↓            │
  │    读文件          按标题切分       Embedding+写入     │
  └──────────────────────────────────────────────────────┘
        ↓
  返回入库的 []string（所有 chunk 的 ID 列表）
```

## 三、Pipeline 2：知识检索 + 对话流水线（ChatAgent）

### 3.1 触发方式

| 触发方式 | 入口文件 | 说明 |
|---|---|---|
| HTTP 普通对话 | `internal/controller/chat/chat_v1_chat.go` | 同步返回完整回答 |
| HTTP 流式对话 | `internal/controller/chat/chat_v1_chat_stream.go` | SSE 流式推送（逐 chunk 返回） |

**输入结构 `UserMessage`：**
```go
type UserMessage struct {
    ID      string            // 会话 ID，用于获取历史记忆
    Query   string            // 用户本次问题
    History []*schema.Message // 历史对话消息（从内存读取）
}
```

---

### 3.2 流水线节点构成

入口函数：`chat_pipeline.BuildChatAgent(ctx)`

```
START ──┬──→ [InputToRag]  ──→ [MilvusRetriever] ──→ [ChatTemplate] ──→ [ReactAgent] ──→ END
        │                                              ↑
        └──→ [InputToChat] ─────────────────────────→ ┘
```

**有向图编译配置：**
- 图名称：`ChatAgent`
- 触发模式：`compose.AllPredecessor`（**所有**前驱节点完成后才触发下一节点，保证 RAG 检索和用户输入都准备好后再拼模板）

---

### 3.3 各节点详解

#### 节点 1：InputToRag（输入转 RAG 查询）

- **代码**：`chat_pipeline/lambda_func.go` → `newInputToRagLambda`
- **作用**：从 `UserMessage` 中提取 `Query` 字符串，作为向量检索的查询词
- **输入**：`*UserMessage`
- **输出**：`string`（即用户问题原文）

---

#### 节点 2：MilvusRetriever（向量检索器）

- **包**：`github.com/cloudwego/eino-ext/components/retriever/milvus`
- **代码**：`chat_pipeline/retriever.go` + `internal/ai/retriever/retriever.go`
- **作用**：
  1. 对用户问题调用豆包 Embedding 进行向量化
  2. 在 Milvus `biz` Collection 中做 ANN（近似最近邻）检索
  3. 返回最相关的文档片段
- **检索配置**：
  - `TopK: 1`：只取**最相似的 1 条** chunk
  - 检索字段：`id`、`content`、`metadata`
  - 向量字段：`vector`
- **输出 Key**：`documents`（通过 `compose.WithOutputKey("documents")` 设置，与 Prompt 模板中的 `{documents}` 占位符对应）

---

#### 节点 3：InputToChat（输入转对话上下文）

- **代码**：`chat_pipeline/lambda_func.go` → `newInputToChatLambda`
- **作用**：将 `UserMessage` 中的信息封装为 Prompt 模板所需的 map
- **输入**：`*UserMessage`
- **输出**：`map[string]any`

```go
map[string]any{
    "content": input.Query,                           // 用户问题
    "history": input.History,                         // 历史对话消息
    "date":    time.Now().Format("2006-01-02 15:04:05"), // 当前时间
}
```

---

#### 节点 4：ChatTemplate（提示词模板组装）

- **代码**：`chat_pipeline/prompt.go`
- **作用**：将 RAG 检索结果 + 用户输入 + 历史对话合并为完整的消息列表，传入大模型
- **模板结构**：
  ```
  [SystemMessage]  ← 系统提示词（含 {date} 和 {documents} 占位符）
  [历史消息占位符]  ← history（可选，为空时不插入）
  [UserMessage]    ← {content}（当前用户问题）
  ```
- **系统提示词关键内容**：
  - 角色定义：对话小助手
  - 注入 RAG 检索结果：`相关文档: {documents}`（`==== 文档开始 ====` 与 `==== 文档结束 ====` 包裹）
  - 注入当前时间：`{date}`
  - 输出要求：纯文本，不含 Markdown 语法

> ⚠️ **触发时机**：`AllPredecessor` 模式确保 `InputToChat` 和 `MilvusRetriever` 都完成后，`ChatTemplate` 才开始执行，保证 `{documents}` 和 `{content}` 同时就绪。

---

#### 节点 5：ReactAgent（ReAct 推理+工具调用）

- **包**：`github.com/cloudwego/eino/flow/agent/react`
- **代码**：`chat_pipeline/flow.go`
- **大模型**：DeepSeek-V3 Quick（通过 OpenAI 兼容接口调用）
- **最大步数**：`MaxStep: 25`
- **挂载工具列表**：

| 工具名 | 文件 | 功能 |
|---|---|---|
| `query_internal_docs` | `tools/query_internal_docs.go` | **再次调用 Milvus RAG 检索**（Agent 主动触发的知识库查询） |
| `get_current_time` | `tools/get_current_time.go` | 获取当前时间 |
| `mysql_crud` | `tools/mysql_crud.go` | MySQL 数据库增删改查 |
| `query_metrics_alerts` | `tools/query_metrics_alerts.go` | Prometheus 监控告警查询 |
| MCP Log 工具 | `tools/query_log.go` | 腾讯云日志服务查询（通过 MCP 协议） |

- **工作模式**：ReAct（Reason + Act），大模型根据当前上下文决定是否调用工具、调用哪个工具，循环直到得出最终答案

---

### 3.4 `query_internal_docs` 工具的特殊意义

这是知识库 Agent 最核心的**双重 RAG 机制**：

```
第一次 RAG（自动，Graph 节点触发）：
  用户问题 → MilvusRetriever → 检索相关文档 → 注入 System Prompt

第二次 RAG（按需，Agent 工具调用触发）：
  ReAct Agent 推理 → 判断需要更多知识 → 调用 query_internal_docs 工具
                  → 再次查询 Milvus → 返回检索结果 → 继续推理
```

第一次是**被动注入**（无论问什么都会检索），第二次是**主动按需查询**（Agent 认为需要时才触发）。

---

### 3.5 完整对话流程图

```
用户提问（HTTP /chat 或 /chat/stream）
        ↓
  BuildChatAgent(ctx) 构建图
        ↓
  runner.Invoke(ctx, userMessage)
        ↓
  ┌────────────────────────────────────────────────────────┐
  │  Graph 并行执行两个起始节点：                             │
  │                                                        │
  │  InputToRag ──→ MilvusRetriever                        │
  │  (提取 Query)   (Embedding+向量检索，TopK=1)             │
  │                        ↓ documents                     │
  │  InputToChat ─────────→ ChatTemplate                   │
  │  (封装 content/history/date)  (组装完整 Prompt)         │
  │                               ↓                        │
  │                          ReactAgent                    │
  │                   (DeepSeek-V3 推理循环)                │
  │                    ↙            ↘                      │
  │              需要工具?         直接回答                   │
  │                ↓                                       │
  │           调用工具（最多 25 步）                          │
  │                ↓                                       │
  │           汇总工具结果继续推理                            │
  └────────────────────────────────────────────────────────┘
        ↓
  返回最终 *schema.Message（out.Content）
        ↓
  保存到会话内存（mem.GetSimpleMemory(id).SetMessages）
        ↓
  HTTP 返回给用户 / SSE 流式推送
```

## 四、会话记忆机制

- **存储方式**：内存（`utility/mem`），以会话 `ID` 为 key
- **读取时机**：每次请求时通过 `mem.GetSimpleMemory(id).GetMessages()` 获取历史消息
- **写入时机**：每次对话完成后，将用户消息和 AI 回答分别写入内存
- **消息格式**：
  - 用户消息：`schema.UserMessage(msg)`
  - AI 回答：`schema.SystemMessage(out.Content)`（注意：AI 回答用 SystemMessage 类型存储）
- **流式模式**：通过 `defer` + `strings.Builder` 收集完整回答后再写入记忆

## 五、两条 Pipeline 的关系总结

| 维度 | KnowledgeIndexing | ChatAgent |
|---|---|---|
| 职责 | 知识入库（写） | 知识检索 + 对话（读+推理） |
| 触发 | 文件上传 / 批量脚本 | 用户每次提问 |
| 向量操作 | 写入 Milvus | 查询 Milvus |
| 大模型参与 | 仅 Embedding | Embedding（检索） + DeepSeek（推理） |
| 图触发模式 | AnyPredecessor | AllPredecessor |
| 核心输出 | `[]string`（入库 ID 列表） | `*schema.Message`（AI 回答） |

## 六、关键技术选型说明

### 6.1 为什么用 BinaryVector（二进制向量）

`indexer.go` 中向量字段类型为 `entity.FieldTypeBinaryVector`，维度为 `81920`。
豆包 Ark Embedding 输出为二进制量化向量，存储和检索效率更高，适合大规模文档场景。

### 6.2 TopK=1 的设计考量

`retriever.go` 中 `TopK: 1`，即每次只检索最相关的 **1 条** chunk 注入 Prompt。
- **优点**：减少 Prompt 长度，降低噪声干扰，对于精准问答效果好
- **潜在问题**：如果文档分割粒度过细，单条 chunk 信息不足，可能导致回答不完整
- **补充机制**：Agent 可以通过 `query_internal_docs` 工具**多次主动查询**来弥补

### 6.3 Eino Graph 编排的核心价值

- 节点之间通过**类型安全**的 edge 连接，编译期即可检测类型不匹配
- `AllPredecessor` 模式保证了 RAG 检索和用户输入**同步汇聚**到 ChatTemplate
- `compose.WithCallbacks(log_call_back.LogCallback(nil))` 为每个节点提供统一的执行日志追踪
