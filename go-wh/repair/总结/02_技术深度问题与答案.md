## Go 语言相关

### Q1: GORM 的 Scope 机制是如何实现的？如何保证线程安全？

**答案（PREP 结构）**：

**P（Point - 直接回答）**：
GORM 的 Scope 是通过闭包函数修改 `*gorm.DB` 对象实现的查询修饰符，每次查询都会创建新的 `Statement` 对象来保证线程安全。

**R（Reason - 原理解释）**：
- Scope 本质是 `func(*gorm.DB) *gorm.DB` 类型的函数
- 在查询构建阶段，GORM 会依次执行所有 Scope 函数，修改 SQL 语句
- 每个请求都有独立的 `Statement` 对象，存储查询状态（WHERE、SELECT、JOIN 等）
- `Statement` 对象通过 `db.Statement.Context` 传递请求级别的上下文信息

**E（Example - 项目实例）**：
在 Repair 项目中，我实现了 `TenantScope` 自动注入租户过滤：

```go
// base/db.go
func TenantScope(db *gorm.DB) *gorm.DB {
    tenantID := db.Statement.Context.Value("tenant_id")
    if tenantID != nil && tenantID != "" {
        return db.Where("tenant_id = ?", tenantID)
    }
    return db
}

// 注册为全局 Scope
db.Callback().Query().Before("gorm:query").Register("tenant_scope", func(db *gorm.DB) {
    db.Scopes(TenantScope)
})

// 使用时自动应用
db.Find(&assets) // 自动添加 WHERE tenant_id = ?
```

**P（Point - 总结）**：
Scope 是 GORM 实现横切关注点（多租户、软删除、审计）的核心机制，通过 `Statement` 对象隔离保证并发安全。

### Q2: Goroutine 泄漏的常见原因和排查方法？

**答案**：

**P**：Goroutine 泄漏主要由 Channel 阻塞、Context 未取消、死循环导致，可通过 pprof 工具排查。

**R**：
- Go 没有 GC 回收 Goroutine，需要主动管理生命周期
- 常见泄漏场景：
  1. Channel 发送/接收永久阻塞
  2. Context 未调用 cancel 函数
  3. HTTP 请求未设置超时
  4. 死循环或长时间阻塞的系统调用

**E**：
在通知服务中遇到的泄漏案例：

```go
// ❌ 错误示例：msgChan 永不关闭导致泄漏
func (s *Service) StartConsumer() {
    msgChan := make(chan *Message)
    go s.producer.Subscribe(msgChan) // 持续写入
    
    go func() {
        for msg := range msgChan {  // 如果 producer 停止但未关闭 channel，泄漏
            s.process(msg)
        }
    }()
}

// ✅ 正确做法：使用 Context 控制生命周期
func (s *Service) StartConsumer(ctx context.Context) {
    msgChan := make(chan *Message, 100)
    
    go func() {
        defer close(msgChan)
        s.producer.Subscribe(ctx, msgChan)
    }()
    
    go func() {
        for {
            select {
            case msg, ok := <-msgChan:
                if !ok {
                    return // channel 关闭，退出
                }
                s.process(msg)
            case <-ctx.Done():
                return // 主动退出
            }
        }
    }()
}

// 排查方法
// 1. 使用 pprof 查看 Goroutine 数量
import _ "net/http/pprof"
go func() {
    http.ListenAndServe("localhost:6060", nil)
}()
// 访问 http://localhost:6060/debug/pprof/goroutine?debug=2

// 2. 生产环境监控 Goroutine 数量
go func() {
    ticker := time.NewTicker(10 * time.Second)
    for range ticker.C {
        count := runtime.NumGoroutine()
        if count > 10000 {
            log.Warnf("Goroutine count: %d", count)
        }
    }
}()
```

**P**：核心是 **Context 控制生命周期 + Channel 及时关闭 + pprof 监控**。

### Q3: gRPC 拦截器的执行顺序和最佳实践？

**答案**：

**P**：gRPC 拦截器按注册顺序执行，类似洋葱模型，建议按"认证 → 日志 → 限流 → 业务"顺序注册。

**R**：
- 一元拦截器（UnaryInterceptor）：`handler` 前后各执行一次
- 流式拦截器（StreamInterceptor）：包装 `ServerStream`
- 执行顺序：先注册的先执行 pre-handler，后执行 post-handler

**E**：
Repair 项目的拦截器链：

```go
// base/grpc.go
func NewGRPCServer() *grpc.Server {
    return grpc.NewServer(
        grpc.ChainUnaryInterceptor(
            // 1. 认证拦截器（最先执行，验证失败直接返回）
            AuthInterceptor,
            // 2. 租户提取拦截器（从 Token 提取 tenant_id）
            TenantInterceptor,
            // 3. 日志拦截器（记录请求参数和响应）
            LoggingInterceptor,
            // 4. 监控拦截器（Prometheus 指标）
            MonitoringInterceptor,
            // 5. 限流拦截器（令牌桶）
            RateLimitInterceptor,
            // 6. 恢复拦截器（捕获 panic）
            RecoveryInterceptor,
        ),
    )
}

// 租户拦截器示例
func TenantInterceptor(ctx context.Context, req interface{}, 
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (interface{}, error) {
    
    // Pre-handler：提取租户信息
    md, _ := metadata.FromIncomingContext(ctx)
    tenantID := extractTenantFromMetadata(md)
    ctx = context.WithValue(ctx, "tenant_id", tenantID)
    
    // 调用下一个拦截器/业务逻辑
    resp, err := handler(ctx, req)
    
    // Post-handler：可以修改响应或记录日志
    return resp, err
}
```

**P**：拦截器顺序影响功能正确性，认证和租户提取必须在最前面。

## 数据库与 GORM

### Q4: MySQL 索引失效的常见场景？

**答案**：

**P**：索引失效常见于类型转换、函数计算、OR 条件、LIKE 前缀通配符等场景。

**R**：
MySQL 优化器在以下情况会放弃使用索引：
1. 隐式类型转换（如字符串字段与数字比较）
2. 对索引列使用函数（如 `WHERE YEAR(created_at) = 2024`）
3. OR 条件中有非索引列
4. LIKE 以 `%` 开头（如 `LIKE '%abc'`）
5. 联合索引未遵循最左前缀原则
6. 索引列参与计算（如 `WHERE id + 1 = 100`）

**E**：
Repair 项目中的优化案例：

```sql
-- ❌ 索引失效：对 created_at 使用函数
SELECT * FROM process_instance 
WHERE DATE(created_at) = '2024-01-01';

-- ✅ 优化：使用范围查询
SELECT * FROM process_instance 
WHERE created_at >= '2024-01-01 00:00:00' 
  AND created_at < '2024-01-02 00:00:00';

-- ❌ 索引失效：隐式类型转换（tenant_id 是 VARCHAR）
SELECT * FROM asset_info WHERE tenant_id = 123;

-- ✅ 优化：使用字符串
SELECT * FROM asset_info WHERE tenant_id = '123';

-- ❌ 索引失效：LIKE 前缀通配符
SELECT * FROM asset_info WHERE asset_code LIKE '%ABC%';

-- ✅ 优化：使用全文索引或 ElasticSearch
ALTER TABLE asset_info ADD FULLTEXT INDEX idx_asset_code(asset_code);
SELECT * FROM asset_info WHERE MATCH(asset_code) AGAINST('ABC');
```

**P**：避免索引失效的核心是**不对索引列做任何操作（函数、计算、类型转换）**。

### Q5: 如何排查 MySQL 慢查询？

**答案**：

**P**：通过慢查询日志定位 SQL，使用 EXPLAIN 分析执行计划，针对性优化索引和查询语句。

**R**：
排查步骤：
1. 开启慢查询日志（`slow_query_log = ON`）
2. 分析日志找出慢 SQL
3. 使用 `EXPLAIN` 查看执行计划
4. 根据 type、key、rows、Extra 字段判断问题
5. 优化索引或重写 SQL

**E**：
Repair 项目中的慢查询优化：

```sql
-- 慢查询：查询某租户下所有设备及其组件信息
SELECT a.*, c.component_name, c.component_sn
FROM asset_info a
LEFT JOIN component_info c ON a.id = c.asset_id
WHERE a.tenant_id = 'tenant_123'
ORDER BY a.created_at DESC
LIMIT 20 OFFSET 10000;

-- EXPLAIN 分析结果
+----+-------------+-------+------+---------------+------+---------+------+--------+----------+-----------------------------+
| id | select_type | table | type | possible_keys | key  | key_len | ref  | rows   | filtered | Extra                       |
+----+-------------+-------+------+---------------+------+---------+------+--------+----------+-----------------------------+
|  1 | SIMPLE      | a     | ALL  | NULL          | NULL | NULL    | NULL | 500000 |    10.00 | Using where; Using filesort |
|  1 | SIMPLE      | c     | ref  | asset_id      | asset_id | 303  | a.id |      5 |   100.00 | NULL                        |
+----+-------------+-------+------+---------------+------+---------+------+--------+----------+-----------------------------+

-- 问题分析：
-- 1. type = ALL：全表扫描
-- 2. rows = 500000：扫描 50 万行
-- 3. Using filesort：需要排序，无法使用索引
-- 4. OFFSET 10000：需要先扫描 10000 行再丢弃

-- 优化方案 1：添加联合索引
CREATE INDEX idx_tenant_created ON asset_info(tenant_id, created_at DESC, id);

-- 优化方案 2：使用游标分页
SELECT a.*, c.component_name, c.component_sn
FROM asset_info a
LEFT JOIN component_info c ON a.id = c.asset_id
WHERE a.tenant_id = 'tenant_123' 
  AND a.id > 'last_id'  -- 游标
ORDER BY a.id
LIMIT 20;

-- 优化后 EXPLAIN
+----+-------------+-------+-------+-------------------+-------------------+---------+------+------+----------+-------------+
| id | select_type | table | type  | possible_keys     | key               | key_len | ref  | rows | filtered | Extra       |
+----+-------------+-------+-------+-------------------+-------------------+---------+------+------+----------+-------------+
|  1 | SIMPLE      | a     | range | idx_tenant_created| idx_tenant_created| 303     | NULL |   20 |   100.00 | Using index |
|  1 | SIMPLE      | c     | ref   | asset_id          | asset_id          | 303     | a.id |    5 |   100.00 | NULL        |
+----+-------------+-------+-------+-------------------+-------------------+---------+------+------+----------+-------------+

-- 性能提升：5s → 200ms（25 倍）
```

**P**：慢查询优化的核心是**索引设计 + 避免深分页 + 减少回表**。

### Q6: GORM 的 N+1 查询问题如何解决？

**答案**：

**P**：使用 `Preload` 或 `Joins` 预加载关联数据，避免循环查询。

**R**：
N+1 问题：查询 N 条主记录后，对每条记录再查询一次关联数据，导致 N+1 次查询。

**E**：

```go
// ❌ N+1 查询：查询 100 个工单，每个工单再查询一次任务列表
var instances []ProcessInstance
db.Find(&instances) // 1 次查询

for _, inst := range instances {
    var tasks []InstanceTaskInfo
    db.Where("process_instance_id = ?", inst.ID).Find(&tasks) // N 次查询
    inst.Tasks = tasks
}

// ✅ 优化方案 1：Preload 预加载
var instances []ProcessInstance
db.Preload("Tasks").Find(&instances) // 2 次查询（1 次主表 + 1 次关联表）
// 生成 SQL：
// SELECT * FROM process_instance;
// SELECT * FROM instance_task_info WHERE process_instance_id IN (?, ?, ...);

// ✅ 优化方案 2：Joins 连接查询
var instances []ProcessInstance
db.Joins("LEFT JOIN instance_task_info ON instance_task_info.process_instance_id = process_instance.id").
   Find(&instances) // 1 次查询
// 生成 SQL：
// SELECT * FROM process_instance 
// LEFT JOIN instance_task_info ON ...;

// ✅ 优化方案 3：手动批量查询
var instances []ProcessInstance
db.Find(&instances)

instanceIDs := make([]string, len(instances))
for i, inst := range instances {
    instanceIDs[i] = inst.ID
}

var tasks []InstanceTaskInfo
db.Where("process_instance_id IN ?", instanceIDs).Find(&tasks)

// 手动组装数据
taskMap := make(map[string][]InstanceTaskInfo)
for _, task := range tasks {
    taskMap[task.ProcessInstanceID] = append(taskMap[task.ProcessInstanceID], task)
}
for i := range instances {
    instances[i].Tasks = taskMap[instances[i].ID]
}
```

**P**：选择 Preload 还是 Joins 取决于数据量，1:N 关系用 Preload，1:1 用 Joins。

## 工作流引擎 Camunda

### Q7: Camunda 的任务分配策略有哪些？

**答案**：

**P**：Camunda 支持 Assignee（指定人）、Candidate Users/Groups（候选人/组）、Expression（表达式）三种分配策略。

**R**：
- **Assignee**：任务直接分配给特定用户，该用户独占任务
- **Candidate Users**：任务分配给候选用户列表，任何候选人都可以认领（Claim）
- **Candidate Groups**：任务分配给候选组，组内成员可认领
- **Expression**：使用 JUEL 表达式动态计算分配对象

**E**：
Repair 项目中的动态分配实现：

```xml
<!-- BPMN 流程定义 -->
<userTask id="repair_task" 
          name="维修任务"
          camunda:candidateGroups="${regionService.getEngineerGroup(region)}">
  <extensionElements>
    <camunda:taskListener event="create" 
                          delegateExpression="${taskAssignListener}"/>
  </extensionElements>
</userTask>
```

```go
// 外部任务模式：Go 服务主动拉取任务并分配
func (s *Service) AssignTaskWorker(ctx context.Context) {
    for {
        // 1. 拉取未分配的任务
        tasks := s.camundaClient.FetchAndLockExternalTasks(&camunda.FetchExternalTasksRequest{
            WorkerID: "repair-assign-worker",
            MaxTasks: 10,
            Topics: []camunda.FetchExternalTaskTopic{
                {TopicName: "assign-repair-task"},
            },
        })
        
        for _, task := range tasks {
            // 2. 根据业务规则分配任务
            region := task.Variables["region"].(string)
            engineers := s.getAvailableEngineers(region)
            
            if len(engineers) > 0 {
                // 3. 调用 Camunda API 设置 Assignee
                s.camundaClient.SetAssignee(task.ID, engineers[0].ID)
                
                // 4. 完成外部任务
                s.camundaClient.Complete(task.ID, nil)
            } else {
                // 5. 无可用工程师，延迟重试
                s.camundaClient.HandleFailure(task.ID, &camunda.HandleFailureRequest{
                    ErrorMessage: "无可用工程师",
                    Retries:      3,
                    RetryTimeout: 60000, // 1 分钟后重试
                })
            }
        }
        
        time.Sleep(5 * time.Second)
    }
}

// 业务逻辑：查询可用工程师
func (s *Service) getAvailableEngineers(region string) []User {
    var engineers []User
    s.db.Where("role = ? AND region = ? AND status = ?", 
        "engineer", region, "available").
        Order("workload ASC"). // 按工作量排序
        Limit(1).
        Find(&engineers)
    return engineers
}
```

**P**：复杂分配逻辑建议使用**外部任务模式**，解耦业务逻辑，便于测试和维护。

### Q8: 如何保证 Camunda 引擎与业务数据库的一致性？

**答案**：

**P**：采用**最终一致性 + 补偿机制**，通过定时对账和幂等性设计保证数据一致。

**R**：
- Camunda 和业务数据库是两个独立系统，无法使用 XA 分布式事务
- 强一致性会严重影响性能，且 Camunda 不支持
- 最终一致性通过异步补偿和定时对账实现

**E**：
Repair 项目的一致性保障方案：

```go
// 1. 工单完成流程（最终一致性）
func (s *Service) CompleteRepairTask(ctx context.Context, req *pb.CompleteTaskRequest) error {
    // Step 1: 开启本地事务，更新业务数据
    tx := s.db.WithContext(ctx).Begin()
    defer func() {
        if r := recover(); r != nil {
            tx.Rollback()
        }
    }()
    
    // 更新维修记录状态
    err := tx.Model(&RepairRecord{}).
        Where("id = ?", req.RecordID).
        Update("status", "completed").Error
    if err != nil {
        tx.Rollback()
        return err
    }
    
    // 更新任务状态
    err = tx.Model(&InstanceTaskInfo{}).
        Where("id = ?", req.TaskID).
        Updates(map[string]interface{}{
            "status":        "completed",
            "complete_user": req.UserID,
            "complete_at":   time.Now(),
        }).Error
    if err != nil {
        tx.Rollback()
        return err
    }
    
    tx.Commit()
    
    // Step 2: 调用 Camunda API 完成任务（允许失败）
    err = s.camundaClient.CompleteTask(req.TaskID, map[string]interface{}{
        "result": req.Result,
    })
    if err != nil {
        // 记录失败，等待补偿
        s.db.Create(&FailedCamundaOperation{
            OperationType: "complete_task",
            TaskID:        req.TaskID,
            Payload:       req,
            RetryCount:    0,
        })
        log.Errorf("Camunda complete task failed: %v", err)
        // 不返回错误，业务数据已提交
    }
    
    // Step 3: 异步发送通知（允许失败）
    go s.notifyClient.SendTaskCompleteNotification(req.TaskID)
    
    return nil
}

// 2. 定时补偿任务（每 5 分钟执行）
func (s *Service) CompensateFailedOperations() {
    var failedOps []FailedCamundaOperation
    s.db.Where("retry_count < ?", 3).
        Where("next_retry_at < ?", time.Now()).
        Find(&failedOps)
    
    for _, op := range failedOps {
        switch op.OperationType {
        case "complete_task":
            err := s.camundaClient.CompleteTask(op.TaskID, op.Payload)
            if err == nil {
                s.db.Delete(&op) // 成功，删除记录
            } else {
                // 失败，增加重试次数
                s.db.Model(&op).Updates(map[string]interface{}{
                    "retry_count":   op.RetryCount + 1,
                    "next_retry_at": time.Now().Add(time.Duration(math.Pow(2, float64(op.RetryCount))) * time.Minute),
                })
            }
        }
    }
}

// 3. 定时对账任务（每 5 分钟执行）
func (s *Service) SyncInstanceStatus() {
    // 查询 Camunda 中的活跃实例
    instances, _ := s.camundaClient.GetProcessInstances(&camunda.GetProcessInstancesRequest{
        Active: true,
    })
    
    for _, inst := range instances {
        // 查询本地数据库中的状态
        var localInst ProcessInstance
        s.db.Where("id = ?", inst.ID).First(&localInst)
        
        // 状态不一致，以 Camunda 为准
        if localInst.Status != inst.State {
            s.db.Model(&ProcessInstance{}).
                Where("id = ?", inst.ID).
                Update("status", inst.State)
            
            log.Warnf("Instance %s status synced: %s -> %s", 
                inst.ID, localInst.Status, inst.State)
        }
    }
}

// 4. 幂等性设计
func (s *Service) CompleteTask(taskID string, variables map[string]interface{}) error {
    // 检查任务是否已完成
    var task InstanceTaskInfo
    s.db.Where("id = ?", taskID).First(&task)
    if task.Status == "completed" {
        return nil // 幂等，直接返回成功
    }
    
    // 执行完成逻辑...
}
```

**P**：核心是**幂等性设计 + 异步补偿 + 定时对账**，允许短暂不一致但保证最终一致。

## 分布式系统设计

### Q9: 如何设计一个高可用的通知系统？

**答案**：

**P**：高可用通知系统需要考虑**限流、降级、重试、监控、消息队列**五个方面。

**R**：
- 第三方 API（钉钉、飞书）有频率限制且可能故障
- 大批量工单创建可能触发通知风暴
- 需要保证重要通知不丢失，同时避免重复发送

**E**：
Repair 项目的通知系统设计：

```go
// 1. 限流：令牌桶算法
type RateLimiter struct {
    limiter *rate.Limiter
}

func NewRateLimiter(qps int) *RateLimiter {
    return &RateLimiter{
        limiter: rate.NewLimiter(rate.Limit(qps), qps*2), // 每秒 qps 个令牌，桶容量 2*qps
    }
}

func (r *RateLimiter) Wait(ctx context.Context) error {
    return r.limiter.Wait(ctx)
}

// 2. 降级：API 失败后降级为数据库记录
func (s *Service) SendNotification(msg *NotifyMessage) error {
    // 限流
    s.rateLimiter.Wait(context.Background())
    
    // 尝试发送
    err := s.sendToDingTalk(msg)
    if err != nil {
        // 降级：记录到数据库，人工处理或定时重试
        s.db.Create(&FailedNotification{
            Type:    msg.Type,
            Content: msg.Content,
            UserID:  msg.UserID,
            Status:  "pending",
        })
        log.Errorf("Send notification failed, saved to DB: %v", err)
        return nil // 不返回错误，避免影响主流程
    }
    
    return nil
}

// 3. 重试：指数退避
func (s *Service) sendWithRetry(msg *NotifyMessage) error {
    maxRetries := 3
    for i := 0; i < maxRetries; i++ {
        err := s.dingTalkClient.Send(msg)
        if err == nil {
            return nil
        }
        
        // 指数退避：1s, 2s, 4s
        backoff := time.Duration(math.Pow(2, float64(i))) * time.Second
        log.Warnf("Send failed (retry %d/%d), backoff %v: %v", 
            i+1, maxRetries, backoff, err)
        time.Sleep(backoff)
    }
    
    return errors.New("send failed after retries")
}

    // 发送到 RocketMQ

    }, func(ctx context.Context, result *primitive.SendResult, err error) {
        if err != nil {
            log.Errorf("Publish notification failed: %v", err)
        }
    })
}

func (s *Service) ConsumeNotification() {
    consumer, _ := rocketmq.NewPushConsumer(
        consumer.WithGroupName("repair-notify-group"),
        consumer.WithNsResolver(primitive.NewPassthroughResolver([]string{"localhost:9876"})),
    )
    
    consumer.Subscribe("repair-notify", consumer.MessageSelector{}, 
        func(ctx context.Context, msgs ...*primitive.MessageExt) (consumer.ConsumeResult, error) {
            for _, msg := range msgs {
                var notifyMsg NotifyMessage
                json.Unmarshal(msg.Body, &notifyMsg)
                
                // 限流 + 重试发送
                s.SendNotification(&notifyMsg)
            }
            return consumer.ConsumeSuccess, nil
        })
    
    consumer.Start()
}
```

## 性能优化

### Q10: 如何优化大批量数据导入的性能？

**答案**：

**P**：通过**批量插入、并发处理、异步关联、进度追踪**四个维度优化，可将导入速度从 500 条/秒提升到 5000 条/秒。

**R**：
- 单条插入效率低，网络往返次数多
- 大事务可能导致锁等待和内存溢出
- 关联数据处理（组件、变更历史）拖慢主流程

**E**：
Repair 项目的导入优化方案：

```go
// 1. 批量插入（GORM CreateInBatches）
func (s *Service) ImportAssets(file *multipart.FileHeader) error {
    // 解析 Excel 文件
    rows, err := s.parseExcel(file)
    if err != nil {
        return err
    }
    
    // 数据校验
    validRows := make([]AssetInfo, 0, len(rows))
    for _, row := range rows {
        if err := s.validateRow(row); err == nil {
            validRows = append(validRows, row)
        }
    }
    
    // 批量插入（每批 1000 条）
    batchSize := 1000
    for i := 0; i < len(validRows); i += batchSize {
        end := i + batchSize
        if end > len(validRows) {
            end = len(validRows)
        }
        
        batch := validRows[i:end]
        
        // 使用 GORM 批量插入
        if err := s.db.CreateInBatches(batch, 100).Error; err != nil {
            return err
        }
    }
    
    return nil
}

// 2. 并发处理 + Worker Pool
func (s *Service) ImportAssetsParallel(file *multipart.FileHeader) error {
    rows, _ := s.parseExcel(file)
    
    // 创建任务队列
    taskChan := make(chan []AssetInfo, 10)
    errChan := make(chan error, 10)
    
    // 启动 Worker Pool（5 个并发）
    workerCount := 5
    var wg sync.WaitGroup
    for i := 0; i < workerCount; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for batch := range taskChan {
                if err := s.db.CreateInBatches(batch, 100).Error; err != nil {
                    errChan <- err
                }
            }
        }()
    }
    
    // 分发任务
    batchSize := 1000
    for i := 0; i < len(rows); i += batchSize {
        end := i + batchSize
        if end > len(rows) {
            end = len(rows)
        }
        taskChan <- rows[i:end]
    }
    close(taskChan)
    
    // 等待完成
    wg.Wait()
    close(errChan)
    
    // 检查错误
    for err := range errChan {
        if err != nil {
            return err
        }
    }
    
    return nil
}

// 3. 异步处理关联数据
func (s *Service) ImportAssetsAsync(file *multipart.FileHeader) (string, error) {
    // 创建导入任务记录
    task := &ImportTask{
        ID:     uuid.New().String(),
        Status: "processing",
        Total:  0,
    }
    s.db.Create(task)
    
    // 异步处理
    go func() {
        rows, _ := s.parseExcel(file)
        task.Total = len(rows)
        s.db.Save(task)
        
        // 批量插入主表
        s.db.CreateInBatches(rows, 1000)
        
        // 异步处理组件信息
        go s.processComponents(rows)
        
        // 异步处理变更历史
        go s.processChangeHistory(rows)
        
        // 更新任务状态
        task.Status = "completed"
        task.CompletedAt = time.Now()
        s.db.Save(task)
    }()
    
    return task.ID, nil
}

// 4. 进度追踪（使用 Redis）
func (s *Service) ImportWithProgress(file *multipart.FileHeader) (string, error) {
    taskID := uuid.New().String()
    rows, _ := s.parseExcel(file)
    
    // 初始化进度
    s.redis.HSet(ctx, fmt.Sprintf("import:%s", taskID), map[string]interface{}{
        "total":     len(rows),
        "processed": 0,
        "status":    "processing",
    })
    
    go func() {
        batchSize := 1000
        for i := 0; i < len(rows); i += batchSize {
            end := i + batchSize
            if end > len(rows) {
                end = len(rows)
            }
            
            s.db.CreateInBatches(rows[i:end], 100)
            
            // 更新进度
            s.redis.HSet(ctx, fmt.Sprintf("import:%s", taskID), "processed", end)
        }
        
        s.redis.HSet(ctx, fmt.Sprintf("import:%s", taskID), "status", "completed")
    }()
    
    return taskID, nil
}

    result := s.redis.HGetAll(ctx, fmt.Sprintf("import:%s", taskID)).Val()

    processed, _ := strconv.Atoi(result["processed"])

        Total:      total,
        Processed:  processed,
        Percentage: float64(processed) / float64(total) * 100,
        Status:     result["status"],
    }, nil
}
```

## 系统设计问题

### Q11: 如何设计一个支持多租户的 SaaS 系统？

**答案**：

**P**：多租户系统需要考虑**数据隔离、权限控制、资源隔离、计费**四个维度，可采用共享数据库+租户字段的方案。

**R**：
多租户架构有三种模式：
1. **独立数据库**：每个租户独立数据库（隔离性最强，成本最高）
2. **共享数据库+独立 Schema**：同一数据库，不同 Schema（中等隔离，中等成本）
3. **共享数据库+租户字段**：同一数据库和表，通过 tenant_id 区分（隔离性最弱，成本最低）

**E**：
Repair 项目采用**共享数据库+租户字段**方案：

```go
// 1. 数据模型设计
type BaseModel struct {

    TenantID  string    `gorm:"index;not null"` // 租户 ID（必填）

    CreatedBy string
    UpdatedAt time.Time
    UpdatedBy string
    DeletedAt int64 `gorm:"index"` // 软删除

// 2. GORM 全局 Scope 自动注入租户过滤
func TenantScope(db *gorm.DB) *gorm.DB {
    tenantID := db.Statement.Context.Value("tenant_id")
    if tenantID != nil && tenantID != "" {
        return db.Where("tenant_id = ?", tenantID)
    }
    return db
}

// 注册全局 Scope
db.Callback().Query().Before("gorm:query").Register("tenant_scope", func(db *gorm.DB) {
    if db.Statement.Schema != nil {
        if _, ok := db.Statement.Schema.FieldsByName["TenantID"]; ok {
            db.Scopes(TenantScope)
        }
    }
})

// 3. gRPC 拦截器提取租户信息
func TenantInterceptor(ctx context.Context, req interface{}, 
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (interface{}, error) {
    
    // 从 JWT Token 提取租户 ID
    md, _ := metadata.FromIncomingContext(ctx)
    token := md.Get("authorization")[0]
    claims := parseJWT(token)
    tenantID := claims["tenant_id"]
    
    // 注入到 Context
    ctx = context.WithValue(ctx, "tenant_id", tenantID)
    
    return handler(ctx, req)
}

// 5. 资源隔离（限流、配额）
type TenantQuota struct {
    TenantID       string
    MaxAssets      int // 最大设备数
    MaxUsers       int // 最大用户数
    MaxAPIRequests int // 每分钟最大 API 请求数
}

func (s *Service) CheckQuota(ctx context.Context) error {
    tenantID := ctx.Value("tenant_id").(string)
    
    var quota TenantQuota
    s.db.Where("tenant_id = ?", tenantID).First(&quota)
    
    // 检查设备数量
    var assetCount int64
    s.db.Model(&AssetInfo{}).Where("tenant_id = ?", tenantID).Count(&assetCount)
    if assetCount >= int64(quota.MaxAssets) {
        return errors.New("设备数量已达上限")
    }
    
    // 检查 API 请求频率
    key := fmt.Sprintf("api_rate:%s", tenantID)
    count := s.redis.Incr(ctx, key).Val()
    if count == 1 {
        s.redis.Expire(ctx, key, time.Minute)
    }
    if count > int64(quota.MaxAPIRequests) {
        return errors.New("API 请求频率超限")
    }
    
    return nil
}
```
