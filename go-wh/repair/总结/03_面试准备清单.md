### 技术难题回答模板（STAR 结构）

**示例：Camunda 与业务数据一致性问题**

**Situation（背景）**：
> "在集成 Camunda 工作流引擎时，我们遇到了一个数据一致性问题。Camunda 有自己的数据库，业务数据在另一个数据库，两者无法使用分布式事务保证强一致性。"

**Task（任务）**：
> "我需要设计一个方案，既要保证业务数据的正确性，又要尽量保持 Camunda 引擎状态同步，同时不能影响系统性能。"

**Action（行动）**：
> "我采用了最终一致性 + 补偿机制的方案：
> 1. 业务数据优先：先提交本地事务更新业务数据，再调用 Camunda API
> 2. 失败记录：Camunda 调用失败时记录到失败表，不影响业务流程
> 3. 异步补偿：定时任务每 5 分钟扫描失败记录，使用指数退避重试
> 4. 定时对账：每小时对比 Camunda 和业务数据库状态，发现不一致自动修复
> 5. 幂等设计：所有操作支持幂等，避免重复执行
>
> 具体实现上，我在代码中使用了事务提交后钩子、RocketMQ 消息队列、以及分布式锁来保证补偿任务不重复执行。"

**Result（结果）**：
> "方案上线后，数据一致性达到 99.99%，不一致的情况都能在 5 分钟内自动修复。系统性能没有下降，工单处理延迟保持在 100ms 以内。"

## 💡 高频问题及答案

### Q1: 为什么选择 Camunda 而不是自研工作流？

**答案**：
选择 Camunda 主要基于以下考虑：

1. **成熟度**：Camunda 是业界成熟的 BPMN 2.0 标准实现，经过大量生产验证
2. **可视化**：提供流程设计器，业务人员可以参与流程设计，降低沟通成本
3. **灵活性**：支持复杂的流程逻辑（并行网关、排他网关、子流程等）
4. **开发效率**：自研工作流需要投入大量人力，且难以达到 Camunda 的成熟度
5. **社区支持**：有活跃的社区和丰富的文档

**权衡**：
- 优点：快速上线、功能完善、可维护性强
- 缺点：引入外部依赖、学习成本、需要维护两个数据库

综合考虑，Camunda 的优势远大于自研，是更合理的选择。

### Q2: 如何保证多租户数据安全？

**答案**：
多租户数据安全通过多层防护实现：

**1. 数据库层面**
- 所有表都有 `tenant_id` 字段
- 创建联合索引 `(tenant_id, ...)`，既保证性能又强制隔离

**2. ORM 层面**
- GORM 全局 Scope 自动注入 `WHERE tenant_id = ?`
- 即使代码忘记加条件，也不会泄露其他租户数据

**3. 服务层面**
- gRPC 拦截器从 JWT Token 提取 `tenant_id`
- 注入到 Context，整个请求链路都能访问
- 禁止从请求参数直接获取 `tenant_id`，防止伪造

**4. API 层面**
- 所有写操作强制校验资源归属
- 跨租户操作（如管理员）需要特殊权限

**5. 审计层面**
- 操作日志记录 `tenant_id`
- 定期审计异常访问模式

**代码示例**：
```go
// 拦截器提取租户
func TenantInterceptor(ctx context.Context, req interface{}, 
    info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (interface{}, error) {
    
    md, _ := metadata.FromIncomingContext(ctx)
    token := md.Get("authorization")[0]
    claims := parseJWT(token)
    tenantID := claims["tenant_id"]
    
    ctx = context.WithValue(ctx, "tenant_id", tenantID)
    return handler(ctx, req)
}

// GORM Scope 自动过滤
func TenantScope(db *gorm.DB) *gorm.DB {
    tenantID := db.Statement.Context.Value("tenant_id")
    if tenantID != nil {
        return db.Where("tenant_id = ?", tenantID)
    }
    return db
}
```

### Q3: 系统的性能瓶颈在哪里？如何优化？

**答案**：

**识别的瓶颈**：

1. **数据库慢查询**
   - 问题：深分页查询扫描大量数据
   - 优化：使用游标分页，从 `OFFSET 10000` 改为 `WHERE id > last_id`
   - 效果：查询时间从 5s 降到 200ms

2. **N+1 查询**
   - 问题：循环查询关联数据
   - 优化：使用 GORM Preload 预加载
   - 效果：100 个工单的查询从 101 次降到 2 次

3. **大批量导入**
   - 问题：单条插入效率低
   - 优化：批量插入 + Worker Pool 并发处理
   - 效果：导入速度从 500 条/秒提升到 5000 条/秒

4. **通知风暴**
   - 问题：大批量工单创建触发通知风暴
   - 优化：RocketMQ 削峰 + 令牌桶限流
   - 效果：避免钉钉 API 限流，通知成功率从 60% 提升到 99%

5. **Redis 热 Key**
   - 问题：某些租户的数据访问频繁
   - 优化：本地缓存 + Redis 二级缓存
   - 效果：Redis QPS 降低 50%

**监控指标**：
- API 响应时间 P99 < 500ms
- 数据库慢查询 < 1%
- Redis 命中率 > 95%
- 消息队列积压 < 1000

### Q4: Go 的 Goroutine 和 Java 的线程有什么区别？

**答案**：

**核心区别**：

| 维度 | Goroutine | Java Thread |
|------|-----------|-------------|
| 调度 | 用户态（GMP 模型） | 内核态（操作系统调度） |
| 内存占用 | 2KB 起步，动态增长 | 1MB 固定栈空间 |
| 创建成本 | 极低（微秒级） | 较高（毫秒级） |
| 数量限制 | 百万级 | 千级 |
| 切换成本 | 极低（纳秒级） | 较高（微秒级） |
| 通信方式 | Channel（CSP 模型） | 共享内存 + 锁 |

**GMP 模型**：
- **G（Goroutine）**：用户态轻量级线程
- **M（Machine）**：操作系统线程
- **P（Processor）**：逻辑处理器，持有 G 队列

**优势**：
1. 创建成本低，可以大量创建
2. 自动调度，无需手动管理线程池
3. Channel 通信避免共享内存竞争

**项目实践**：
在 Repair 项目中，我们使用 Goroutine 处理：
- 异步通知发送
- 批量数据导入的并发处理
- Camunda 外部任务轮询

示例：
```go
// 并发处理批量导入
for i := 0; i < workerCount; i++ {
    go func() {
        for batch := range taskChan {
            s.db.CreateInBatches(batch, 100)
        }
    }()
}
```

### Q5: 如何设计一个高并发的工单系统？

**答案**（系统设计题，使用结构化回答）：

**1. 需求澄清**
- 并发量：每秒创建多少工单？查询 QPS？
- 数据规模：总工单量？单租户最大工单量？
- 一致性要求：强一致还是最终一致？
- 可用性要求：允许多长时间的故障？

**2. 架构设计**

```
┌─────────────┐
│   客户端     │
└──────┬──────┘
       │
┌──────▼──────────────────────────┐
│      API Gateway (Nginx)        │
│    - 限流                        │
│    - 认证                        │
└──────┬──────────────────────────┘
       │
┌──────▼──────────────────────────┐
│   工单服务集群 (无状态)          │
│   - 负载均衡                     │
│   - 水平扩展                     │
└──────┬──────────────────────────┘
       │
┌──────▼──────┐  ┌────────────┐
│  MySQL 主从  │  │   Redis    │
│  - 读写分离  │  │  - 缓存    │
│  - 分库分表  │  │  - 限流    │
└──────────────┘  └────────────┘
```

**3. 核心优化**

**数据库层**
- 分库分表：按 `tenant_id` 分库，按时间分表
- 读写分离：查询走从库，写入走主库
- 索引优化：`(tenant_id, status, created_at)`

**缓存层**
- 热点数据缓存：工单详情、用户信息
- 缓存预热：系统启动时加载热点数据
- 缓存更新：写入时异步更新缓存

**应用层**
- 无状态设计：可水平扩展
- 异步处理：工单创建后异步发送通知
- 限流降级：超过阈值返回友好提示

**消息队列**
- 削峰填谷：大批量创建时放入队列
- 解耦服务：工单创建和通知分离

**4. 容量规划**

假设目标：
- 创建 QPS：1000
- 查询 QPS：10000
- 数据量：1 亿工单

**计算**：
- 应用服务器：10000 QPS / 1000 QPS/台 = 10 台
- MySQL：按 1000 万/库分 10 个库
- Redis：按 100GB 内存规划

**5. 监控告警**
- API 响应时间
- 数据库慢查询
- 缓存命中率
- 消息队列积压
