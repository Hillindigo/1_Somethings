### 2. 资产关联关系管理

**挑战**：
部件可以装在主机上，也可以装在其他部件上，形成多层嵌套关系。需要支持递归查询父级信息。

**解决方案**：
- 设计 `related_asset_code`（直接父级）和 `related_parent_asset_code`（最顶层父级）两个字段
- 实现递归查询算法，从部件一直向上查找到主机
- 使用 `visited` map 防止循环引用

**代码示例**：
```go
func (s *componentInfoService) GetRelatedParentInfo(ctx context.Context, fakeSN string) (*ParentInfo, error) {
    visited := make(map[string]struct{})
    nextFakeSN := fakeSN
    
    for nextFakeSN != "" {
        // 防止循环引用
        if _, ok := visited[nextFakeSN]; ok {
            return nil, fmt.Errorf("cycle detected")
        }
        visited[nextFakeSN] = struct{}{}
        
        // 查询当前节点
        components, err := s.assetInfoStorage.GetByFakeSn(ctx, nextFakeSN)
        if err != nil {
            return nil, err
        }
        
        // 如果是部件，继续向上查找
        if len(components) > 0 {
            comp := components[0]
            if comp.RelatedParentSN != "" {
                nextFakeSN = comp.RelatedParentSN
                continue
            }
        }
        
        // 如果是主机，返回结果
        hosts, _ := s.hostInfoStorage.FindByFakeSN(ctx, nextFakeSN)
        if len(hosts) > 0 {
            return &ParentInfo{Related: ..., Parent: ...}, nil
        }
        
        break
    }
    
    return nil, nil
}
```

### 3. 批量导入性能优化

**挑战**：
单次导入 5000+ 条数据，单条插入耗时过长（500 条/秒）。

**解决方案**：
1. **Worker Pool 并发处理**：启动 10 个 Goroutine 并发处理
2. **批量插入**：每批 100 条，减少数据库往返次数
3. **异步处理**：导入任务放入队列，立即返回任务 ID
4. **进度追踪**：使用 `atomic` 包实时更新进度

**性能对比**：
- 优化前：单条插入，500 条/秒
- 优化后：Worker Pool + 批量插入，5000 条/秒
- **性能提升 10 倍**

**代码示例**：
```go
// Worker Pool 实现
workerCount := 10
taskChan := make(chan []mongo.AssetInfo, 100)
var wg sync.WaitGroup

for i := 0; i < workerCount; i++ {
    wg.Add(1)
    go func() {
        defer wg.Done()
        for batch := range taskChan {
            // 批量插入
            if err := s.assetInfoStorage.CreateInBatches(ctx, batch, 100); err != nil {
                klog.Errorf("批量插入失败: %v", err)
            }
            
            // 更新进度
            atomic.AddInt32(&processedCount, int32(len(batch)))
        }
    }()
}

// 分批发送任务
batchSize := 100
for i := 0; i < len(assets); i += batchSize {
    end := i + batchSize
    if end > len(assets) {
        end = len(assets)
    }
    taskChan <- assets[i:end]
}
close(taskChan)
wg.Wait()
```

### 4. 工单回调与资产状态同步

**挑战**：
工作流服务（repair-workflow-service）完成工单后，需要回调维护服务更新资产状态。两个服务的数据库不同，如何保证一致性？

**解决方案**：
1. **gRPC 接口**：工作流服务通过 gRPC 调用维护服务的 `CreateRepairRecord` 接口
2. **异步处理**：维修记录创建后，异步更新资产状态
3. **失败重试**：记录失败的回调请求，定时重试
4. **幂等设计**：同一工单多次回调，只处理一次

**代码示例**：
```go
// 工单回调接口
func (s *repairRecordService) CreateRepairRecord(ctx context.Context, req *apiv1.CreateRepairRecordRequest) (*apiv1.CreateRepairRecordResponse, error) {
    // 1. 创建维修记录
    record := &mongo.RepairRecord{
        InstanceCode: req.InstanceCode,
        AssetCode:    req.AssetCode,
        Items:        convertItems(req.Items),

    }

    go func() {
        for _, item := range req.Items {
            // 换下部件：清除关联关系
            if item.OldSn != "" {
                s.assetInfoStorage.UpdateRelatedSNByAssetCode(ctx, ...)
            }
            
            // 换上部件：建立新关联关系
            if item.NewSn != "" {
                s.assetInfoStorage.UpdateRelatedSNByAssetCode(ctx, ...)
            }
        }
    }()
    
    return &apiv1.CreateRepairRecordResponse{Id: record.ID.Hex()}, nil
}
```

### 5. 项目看板实时统计

**挑战**：
项目看板需要实时统计当前项目的主机/部件/工单数量，按状态分组。数据量大时查询慢。

**解决方案**：
1. **MongoDB 聚合查询**：使用 `$group` 按状态分组统计
2. **合并历史数据**：从 `project_asset_view` 查询已出库的资产
3. **缓存字典项**：状态码转换为中文名称，减少跨服务调用
4. **异步统计**：工单统计异步查询，不阻塞主流程

**代码示例**：
```go
// MongoDB 聚合查询统计资产状态分布
pipeline := mongo.Pipeline{
    {{Key: "$match", Value: bson.M{"project": projectID, "is_deleted": 0}}},
    {{Key: "$group", Value: bson.M{
        "_id": bson.M{
            "asset_type": "$asset_type",
            "status":     "$asset_status",
        },
        "count": bson.M{"$sum": 1},
    }}},
}

cursor, err := collection.Aggregate(ctx, pipeline)
// 结果：[{_id: {asset_type: "host", status: "正常"}, count: 10}, ...]

// 转换为 map[assetType]map[status]count
stats := make(map[string]map[string]int32)
for cursor.Next(ctx) {
    var result struct {
        ID struct {
            AssetType string `bson:"asset_type"`
            Status    string `bson:"status"`
        } `bson:"_id"`
        Count int32 `bson:"count"`
    }
    cursor.Decode(&result)
    
    if _, ok := stats[result.ID.AssetType]; !ok {
        stats[result.ID.AssetType] = make(map[string]int32)
    }
    stats[result.ID.AssetType][result.ID.Status] = result.Count
}
```

### 6. 变更历史字段级别对比

**挑战**：
记录资产变更历史时，需要对比新旧对象的每个字段，生成 Before/After 对比。

**解决方案**：
1. **反射对比**：使用 Go 的 `reflect` 包对比结构体字段
2. **字典项转换**：状态码自动转换为中文名称
3. **忽略无关字段**：忽略 `updated_at`、`updated_by` 等系统字段
4. **格式化输出**：生成易读的变更描述

**代码示例**：
```go
func compareFields(oldObj, newObj interface{}) []*FieldChange {
    changes := make([]*FieldChange, 0)
    
    oldVal := reflect.ValueOf(oldObj).Elem()
    newVal := reflect.ValueOf(newObj).Elem()
    
    for i := 0; i < oldVal.NumField(); i++ {
        fieldName := oldVal.Type().Field(i).Name
        
        // 忽略系统字段
        if fieldName == "UpdatedAt" || fieldName == "UpdatedBy" {
            continue
        }
        
        oldFieldVal := oldVal.Field(i).Interface()
        newFieldVal := newVal.Field(i).Interface()
        
        // 对比字段值
        if !reflect.DeepEqual(oldFieldVal, newFieldVal) {
            changes = append(changes, &FieldChange{
                Field:      fieldName,
                FieldLabel: getFieldLabel(fieldName),
                Before:     formatValue(oldFieldVal),
                After:      formatValue(newFieldVal),
            })
        }
    }
    
    return changes
}
```

### Q2: 你是如何优化批量导入性能的？

**回答（STAR 结构）**：

**Situation（背景）**：
用户需要批量导入 5000+ 条设备数据，但单条插入的方式效率很低，只能达到 500 条/秒，导入一次需要 10 秒以上。

**Task（任务）**：
需要优化导入性能，目标是将导入速度提升到 5000 条/秒以上。

**Action（行动）**：
我采用了以下优化策略：
1. **Worker Pool 并发处理**：启动 10 个 Goroutine 并发处理导入任务，充分利用多核 CPU
2. **批量插入**：使用 MongoDB 的 `InsertMany` 方法，每批插入 100 条，减少数据库往返次数
3. **异步处理**：导入任务放入队列，立即返回任务 ID，用户可以通过任务 ID 查询进度
4. **进度追踪**：使用 `atomic` 包实时更新已处理数量，前端轮询显示进度条

具体实现上，我使用 Channel 作为任务队列，主线程负责解析 Excel 并分批发送到 Channel，Worker Goroutine 从 Channel 取任务并批量插入数据库。

**Result（结果）**：
优化后，导入速度从 500 条/秒提升到 5000 条/秒，**性能提升 10 倍**。5000 条数据的导入时间从 10 秒降到 1 秒，用户体验显著提升。

### Q3: MongoDB 和 MySQL 的选型考虑是什么？

**回答**：

**项目中的实践**：
- 核心业务数据（工单流程）使用 MySQL，保证强一致性
- 资产信息使用 MongoDB，灵活应对字段变更

### Q4: 如何保证工单回调和资产状态的一致性？

**回答**：

**挑战**：
工作流服务和维护服务使用不同的数据库（MySQL 和 MongoDB），无法使用分布式事务保证强一致性。

**解决方案**：
我采用了**最终一致性 + 补偿机制**的方案：

1. **异步处理**：工单完成后，工作流服务通过 gRPC 调用维护服务的 `CreateRepairRecord` 接口，维护服务立即返回成功，然后异步更新资产状态
2. **失败记录**：如果资产状态更新失败，记录到失败表，不影响工单流程
3. **定时补偿**：定时任务每 5 分钟扫描失败记录，使用指数退避重试
4. **幂等设计**：同一工单多次回调，通过 `instance_code` 去重，只处理一次

**权衡**：
- 优点：不阻塞工单流程，系统可用性高
- 缺点：短时间内可能存在数据不一致，但最终会一致

**监控**：
- 监控失败记录数量，超过阈值告警
- 每小时对账，发现不一致自动修复

### Q5: 资产关联关系的递归查询是如何实现的？

**回答**：

**业务场景**：
部件可以装在主机上，也可以装在其他部件上，形成多层嵌套关系。例如：
- GPU 模块装在 GPU 卡上
- GPU 卡装在服务器上

需要支持从任意部件向上递归查询，找到最顶层的主机。

**实现思路**：
1. **数据模型**：每个资产记录两个字段
   - `related_asset_code`：直接父级资产编码
   - `related_parent_asset_code`：最顶层父级资产编码（冗余字段，提升查询性能）

2. **递归查询**：从当前部件开始，沿着 `related_asset_code` 向上查找，直到找到主机或查询失败

3. **防止循环引用**：使用 `visited` map 记录已访问的节点，如果再次访问则报错

**代码示例**：
```go
func (s *componentInfoService) GetRelatedParentInfo(ctx context.Context, fakeSN string) (*ParentInfo, error) {
    visited := make(map[string]struct{})
    nextFakeSN := fakeSN
    
    for nextFakeSN != "" {
        // 防止循环引用
        if _, ok := visited[nextFakeSN]; ok {
            return nil, fmt.Errorf("cycle detected")
        }
        visited[nextFakeSN] = struct{}{}
        
        // 查询当前节点
        components, err := s.assetInfoStorage.GetByFakeSn(ctx, nextFakeSN)
        if len(components) > 0 {
            comp := components[0]
            if comp.RelatedParentSN != "" {
                nextFakeSN = comp.RelatedParentSN
                continue
            }
        }
        
        // 查询主机
        hosts, _ := s.hostInfoStorage.FindByFakeSN(ctx, nextFakeSN)
        if len(hosts) > 0 {
            return &ParentInfo{...}, nil
        }
        
        break
    }
    
    return nil, nil
}
```

**优化**：
- 冗余 `related_parent_asset_code` 字段，大部分场景直接返回，不需要递归查询
- 批量查询接口，一次查询多个部件的父级关系
