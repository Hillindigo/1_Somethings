# 资产数据性能优化：分页查询优化 + 索引设计深度解析

> **对应简历职责**：资产数据的性能优化：分页查询优化、设计索引。
>
> **对应源码文件**：
>
> - `pkg/mongo/asset_info.go` — AssetInfo 数据模型 + 分页查询实现
> - `pkg/service/asset_info.go` — ListAssetInfos 业务逻辑 + 聚合管道排序
> - `pkg/mongo/import_task.go` — ImportTaskItem 模型 + GetLimitPendingConfirmeds
> - `pkg/service/warehouse_asset_view.go` — 仓库维度资产列表查询

---

## 一、背景：为什么需要优化

`asset_info` 集合是核心数据表，字段多（60+ 个字段），查询场景复杂：

- **多条件筛选**：类型、型号、状态、仓库、项目、SN 等任意组合
- **模糊搜索**：7 个字段的 `$regex` 全文搜索
- **多维排序**：按创建时间、注册时间、完整度（含条件排序）
- **软删除过滤**：所有查询都必须带 `is_deleted = 0`
- **高频点查**：按 `factory_sn`、`asset_code` 单条精确查询（导入查重、关联查询）

没有索引时，每个查询都是全集合扫描（Collection Scan），10 万条数据下延迟可达秒级。

## 二、索引设计

> **说明**：项目中索引通过 MongoDB Compass 或运维脚本手动创建，代码层面通过查询字段的使用频率和过滤组合推导出索引设计。以下是基于实际代码查询模式分析出的索引策略。

### 2.1 asset_info 集合的核心索引

#### 索引一：软删除 + 创建时间（列表分页基础索引）

```javascript
db.asset_info.createIndex(
  { is_deleted: 1, create_time: -1 },
  { name: "idx_is_deleted_create_time" }
)
```

**覆盖场景**：`ListWithPagination` 默认排序

```go
// pkg/mongo/asset_info.go:370-407
filter["is_deleted"] = 0
opts := options.Find().SetSkip(skip).SetLimit(pageSize)
opts.SetSort(bson.D{{Key: "create_time", Value: -1}})
```

所有列表查询都带 `is_deleted=0`，把 `is_deleted` 放在索引最左前缀，配合 `create_time` 倒序，使分页查询直接走索引而非全表扫描。

#### 索引二：factory_sn 唯一索引（单点精确查询 + 导入查重）

```javascript
db.asset_info.createIndex(
  { factory_sn: 1 },
  { name: "idx_factory_sn", unique: true, sparse: true }
)
```

**覆盖场景**：

```go

// pkg/mongo/asset_info.go（GetByFactorySNBatch）— 导入优化批量查重

```

`factory_sn` 是业务唯一键（原厂SN），设置唯一索引同时保证数据唯一性和查询性能。`sparse: true` 确保空值字段不占用索引空间（部分资产可能没有原厂SN）。

#### 索引三：asset_code 唯一索引（资产编号精确查询）

```javascript

  { asset_code: 1 },
  { name: "idx_asset_code", unique: true }
```

### 2.2 import_task_items 集合的索引

导入任务的条目表在导入过程中有极高的读写频率（每批 1000 条的查询、更新），索引设计直接影响导入速度。

#### 索引一：task_id + status + type（批次查询索引）

```javascript
db.import_task_items.createIndex(
  { task_id: 1, status: 1, type: 1, create_time: -1 },
  { name: "idx_task_id_status_type_create_time" }
)
```

**覆盖场景**：`GetLimitPendingConfirmeds` — 导入核心循环中最高频的查询

## 三、分页查询优化

### 3.1 基础分页：Offset 分页（ListWithPagination）

```go
// pkg/mongo/asset_info.go:369-408
func (s *assetInfoStorage) ListWithPagination(
    ctx context.Context,
    filter bson.M,
    page, pageSize int64,
    sort bson.D,
) ([]*AssetInfo, int64, error) {

    filter["is_deleted"] = 0

    // 并发执行 Count + Find（实际代码串行，但两者都走索引）
    total, err := s.collection.CountDocuments(ctx, filter)

    skip := (page - 1) * pageSize
    opts := options.Find().SetSkip(skip).SetLimit(pageSize)
    opts.SetSort(bson.D{{Key: "create_time", Value: -1}})

    cursor, err := s.collection.Find(ctx, filter, opts)
    defer cursor.Close(ctx)

    var Assets []*AssetInfo
    cursor.All(ctx, &Assets)

    return Assets, total, nil
}
```

**设计要点**：

- `CountDocuments` 和 `Find` 共用同一个 `filter`，两者都走索引，避免全表扫描
- 默认排序 `create_time DESC` + `_id DESC`，双字段排序防止同一秒插入的数据分页结果不稳定（`_id` 包含时间戳，天然有序）
- `SetSkip` + `SetLimit` 是 MongoDB 的标准偏移分页，对**深翻页**（如第 1000 页）有性能问题（需要跳过前 N 条），但本业务场景用户通常不会翻到很深页面，可接受

### 3.2 复合排序：多字段动态排序

```go
// pkg/service/asset_info.go:307-327
// 根据前端请求动态构建排序
sort := bson.D{{Key: "create_time", Value: -1}, {Key: "_id", Value: -1}}

switch registrationSort {
case "DESC":
    // 注册时间倒序 + 创建时间兜底
    sort = append(bson.D{{Key: "registration_time", Value: -1}}, sort...)
case "ASC":
    sort = append(bson.D{{Key: "registration_time", Value: 1}}, sort...)
}

assets, total, err = s.assetInfoStorage.ListWithPagination(ctx, filter, page, size, sort)
```

`bson.D` 是有序切片，字段顺序决定排序优先级。这里将 `registration_time` 前置，再以 `create_time + _id` 兜底，保证排序结果确定性。

### 3.3 聚合管道实现条件排序（listAssetsWithConditionalCompletenessSort）

这是本项目分页查询中最复杂的优化点。

**业务需求**：按完整度排序时，只对 GPU 模组（`type = EQUIPMENT:TYPE:GPU:MODULE`）参与排序，其他资产类型永远排在末尾。

**问题**：MongoDB 的 `sort` 不支持条件表达式，无法直接表达「只对某些文档用字段 A 排序，其余用常量」。

**解决方案**：用 `$addFields` 动态添加计算字段，再按计算字段排序：

```go
// pkg/service/asset_info.go:457-537
pipeline := bson.A{
    // 1. 先过滤（走索引）
    bson.D{{Key: "$match", Value: filter}},

    // 2. 动态添加 sort_completeness 计算字段
    //    GPU模组用真实 completeness，其余类型用极值（排到最后）
    bson.D{{Key: "$addFields", Value: bson.M{
        "sort_completeness": bson.M{
            "$cond": bson.M{
                "if": bson.M{
                    "$or": bson.A{
                        bson.M{"type": "HOST:TYPE:GPU:GPU"},
                        bson.M{"type": "EQUIPMENT:TYPE:GPU:MODULE"},
                    },
                },
                "then": "$completeness",
                "else": otherTypeSortValue, // DESC时=-9999，ASC时=9999
            },
        },
    }}},

    // 3. 按计算字段排序
    bson.D{{Key: "$sort", Value: bson.D{
        {Key: "sort_completeness", Value: sortDirection}, // -1 or 1
        {Key: "create_time", Value: -1},
        {Key: "_id", Value: -1},
    }}},

    // 4. 分页
    bson.D{{Key: "$skip", Value: (page - 1) * size}},
    bson.D{{Key: "$limit", Value: size}},

    // 5. 去除临时字段，不返回给前端
    bson.D{{Key: "$project", Value: bson.M{"sort_completeness": 0}}},
}

cursor, err := collection.Aggregate(ctx, pipeline)
```

**性能关键**：`$match` 阶段放在管道第一步，利用已有索引过滤数据，后续 `$addFields`、`$sort` 只对过滤后的结果集操作，避免全表计算。

### 3.4 $facet 一次查询同时获取总数和分页数据

在 `ListAssetsAggregate` 中使用 `$facet` 合并 Count 和 Find：

```go
// pkg/service/host_info.go:1832-1838
pipeline = append(pipeline, bson.D{{Key: "$facet", Value: bson.D{
    // 分支1：只计算总数
    {Key: "metadata", Value: bson.A{
        bson.D{{Key: "$count", Value: "total"}},
    }},
    // 分支2：分页数据
    {Key: "data", Value: bson.A{
        bson.D{{Key: "$skip", Value: (page - 1) * size}},
        bson.D{{Key: "$limit", Value: size}},
    }},
}}})
```

**优势**：一次 Aggregate 调用同时返回 `total` 和当前页数据，比 `CountDocuments` + `Find` 两次网络往返少一次 RTT，在 Count 和 Find 共享相同管道前置步骤（`$match`、`$lookup`）时尤为高效。

### 3.5 多条件筛选的 $or 优化

```go
// pkg/service/asset_info.go:219-229 — 全局搜索框
if req.Search != "" {
    searchConditions := []bson.M{
        {"asset_code":                  bson.M{"$regex": req.Search, "$options": "i"}},
        {"factory_sn":                  bson.M{"$regex": req.Search, "$options": "i"}},
        {"fake_sn":                     bson.M{"$regex": req.Search, "$options": "i"}},
        {"pn":                          bson.M{"$regex": req.Search, "$options": "i"}},
        {"related_parent_asset_code":   bson.M{"$regex": req.Search, "$options": "i"}},
        {"related_factory_sn":          bson.M{"$regex": req.Search, "$options": "i"}},
        {"related_parent_fake_sn":      bson.M{"$regex": req.Search, "$options": "i"}},
    }
    filter["$or"] = searchConditions
}
```

**注意**：`$regex` 模糊匹配只有在使用 `^` 前缀锚定时才能走索引，中间匹配（`.*关键词.*`）无法利用 B-Tree 索引，必须扫描文档。这是当前实现的一个性能瓶颈，对应的优化方向是引入 MongoDB Atlas Search 或 Elasticsearch 全文索引。

## 四、数据模型设计对查询性能的影响

### 4.1 软删除字段的设计

```go
// pkg/mongo/asset_info.go:58
IsDeleted int `bson:"is_deleted"` // 0=正常, 1=已删除
```

选用 `int` 而非 `bool`：

- MongoDB 对 `int` 字段的基数（Cardinality）判断更准确
- 未来可扩展为多状态（如 0=正常，1=已删除，2=归档）
- 复合索引 `{is_deleted: 1, create_time: -1}` 中，`is_deleted=0` 是高选择性前缀（约 99% 的数据是未删除的），索引效率高

### 4.2 双时间戳 + _id 兜底排序

```go
// pkg/mongo/asset_info.go:61-62
CreateTime time.Time `bson:"create_time"` // 业务创建时间
UpdateTime time.Time `bson:"update_time"` // 最后更新时间
```

默认排序使用 `create_time DESC + _id DESC`：

- `create_time` 是业务时间，可能受 NTP 校时影响出现重复值
- `_id`（ObjectID）包含 4 字节时间戳 + 5 字节随机数 + 3 字节计数器，同秒内唯一且有序
- 两者组合保证分页结果绝对稳定，不会出现「同一条数据在第 N 页和第 N+1 页都出现」的问题

### 4.3 $in 批量查询替代循环单条查询

多处代码将 N 次单条查询优化为 1 次 `$in` 批量查询：

```go
// 优化前（伪代码）：N 次 FindByFactorySN
for _, sn := range factorySNs {
    asset, _ := FindByFactorySN(ctx, sn)  // N 次数据库查询
}

// 优化后：1 次 $in 查询
// pkg/mongo/asset_info.go（GetByFactorySNBatch）
cursor, err := s.collection.Find(ctx, bson.M{
    "factory_sn": bson.M{"$in": factorySNs},  // 1 次查询
    "is_deleted": 0,
})
```

同样的模式在以下方法中使用：

- `FindByAssetCodes` — 批量按资产编号查询
- `FindByAssetCodesOrSns` — 批量按编号或SN查询
- `BatchCheckAssetCodesExist` — 批量检查资产编号是否存在（用于导入预校验）

## 五、import_task_items 的查询优化

导入过程中，`GetLimitPendingConfirmeds` 是最高频的查询，优化方式体现在三个层面：

### 5.1 精确的复合过滤条件

```go
// pkg/mongo/import_task.go:279-287
filter := bson.M{
    "task_id": objectId,  // ObjectID 精确匹配（高选择性）
    "status":  "pending", // 枚举字段过滤
    "type":    itemType,  // "new" 或 "duplicate"
}
```

三字段组合过滤，每次只取满足条件的数据，不把无关条目加载到内存。

### 5.2 Limit 控制单次查询量

```go
// pkg/mongo/import_task.go:296-298
if limit > 0 {
    opts.SetLimit(limit) // = 1000
}
```

每次最多取 1000 条，配合 `SetSkip` 不用（不用 Skip，直接靠状态机控制已处理数据），避免大结果集传输。

### 5.3 状态机代替 Skip 实现无偏移分页

这是项目中最巧妙的分页优化设计：

```text
传统分页方式（有 Skip 偏移）：
  第1批：Skip=0,    Limit=1000  → 扫描第 1-1000 条
  第2批：Skip=1000, Limit=1000  → 扫描第 1-2000 条（重复扫描前1000条！）
  第N批：Skip=N*1000 → 随着 N 增大，扫描范围线性增长，越来越慢

本项目方式（状态机代替 Skip）：
  第1批：filter={status:"pending"} → 取1000条 → 立刻标记为 processing
  第2批：filter={status:"pending"} → 取1000条（已处理的不在结果中）
  每次都从满足条件的结果集顶部取，无需 Skip，始终是 O(1) 偏移
```

对应代码：

```go
// pkg/service/import_worker_start_import_patterns.go
// Step1: 取 pending 状态数据（无 Skip，直接取前 1000）
items = GetLimitPendingConfirmeds(opCtx, taskId, "pending", itemType, 1000, sort)

// Step2: 立即标记为 processing（下次查询不再返回这批数据）
UpdateImportTaskItemStatus(opCtx, itemIds, "processing")

// Step3: 处理完成后标记为 completed
UpdateImportTaskItemStatus(opCtx, itemIds, "completed")
```

## 六、已知性能瓶颈与优化方向

| 当前问题 | 原因 | 优化方向 |
| --- | --- | --- |
| `$regex` 模糊搜索慢 | 中间匹配无法走 B-Tree 索引，全文档扫描 | 引入 MongoDB Atlas Search 或 Elasticsearch |
| `ListWithPagination` 深翻页慢 | `SetSkip(N)` 需要扫描前 N 条 | 改用 Cursor-based pagination（记录上次最后一条的 `_id`） |
| `CountDocuments + Find` 两次查询 | 串行执行，两次网络往返 | 改为 `$facet` 一次聚合查询（已在 ListAssetsAggregate 中实现） |
| 多表联查（asset + project + warehouse）每行单独查字典 | 循环内查缓存，缓存未命中时走数据库 | 预加载全量字典缓存（已部分实现 GetCache） |

## 七、面试高频追问 Q&A

**Q1：你说设计了索引，具体用了哪些？为什么这么设计？**

> `factory_sn` 和 `asset_code` 是业务唯一键，设置唯一索引同时保证数据完整性和精确查询性能；`is_deleted + create_time` 复合索引覆盖所有分页列表查询的过滤+排序；`project` 和 `warehouse_id` 索引覆盖两个核心查询维度；`import_task_items` 上建 `task_id + status + type + create_time` 复合索引是因为导入循环里每秒可能调用十几次这个查询，无索引时直接拖慢整体导入速度。

**Q2：MongoDB 的 `$regex` 为什么不走索引？怎么优化？**

> `$regex` 只有在模式以 `^` 开头（前缀匹配）时才能利用 B-Tree 索引做范围扫描；中间匹配（如 `.*关键词.*`）需要逐文档检查，相当于全表扫描。优化方向：① 对高频搜索字段建 MongoDB Text Index（支持全文搜索但不支持中文分词）；② 引入 Elasticsearch 做专业全文搜索；③ 业务上限制搜索条件（如要求前缀匹配）。

**Q3：你用的 Skip+Limit 分页，深翻页会不会很慢？**

> 会。`SetSkip(N)` 在 MongoDB 中即使走了索引，也要在索引上跳过前 N 个条目，N 越大越慢。本项目业务上用户很少翻到深页（资产管理场景一般用搜索定位，不会手动翻页），所以当前可接受。严格的深翻页优化是用 Cursor-based pagination：记录上一页最后一条记录的 `_id`，下一页用 `_id > lastId` 代替 Skip，始终是 O(1)。

**Q4：import_task_items 的「状态机代替 Skip」是你自己设计的吗？能说说思路？**

> 核心思路是：把「哪些数据已经处理过」的信息存回数据库（状态字段），而不是靠调用方记录偏移量。每批处理完后立刻将状态改为 `processing`，下次查询的 `filter={status:"pending"}` 自然就不包含已处理数据，每次都从过滤结果的顶部取，等价于无偏移分页。代价是多一次 `UpdateMany` 操作，但这在导入场景下是必须的（状态本身也用于防止重复处理），所以没有额外开销。
