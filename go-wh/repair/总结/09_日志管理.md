# 日志管理分析文档

> 覆盖服务：repair-chassis-service（Java）、repair-maintenance-service（Go）、repair-notify-service（Go）、repair-workflow-service（Go）

---

## 一、整体架构概览

该项目采用**混合技术栈**，日志管理方案按语言分为两套：

| 服务 | 语言/框架 | 日志方案 |
|------|-----------|----------|
| repair-chassis-service | Java / Spring Boot | SLF4J + Logback |
| repair-maintenance-service | Go | xlog（observability/log）+ klog/v2 |
| repair-notify-service | Go | xlog（observability/log）+ 标准 log |
| repair-workflow-service | Go | xlog（observability/log）+ klog/v2 |

## 二、Go 服务日志管理（三个服务共用模式）

### 2.1 核心依赖包

```
git.sy.com/galaxies/observability/log   → 封装 zap 的 xlog 库（内部自研）
k8s.io/klog/v2                          → Kubernetes 风格结构化日志
git.sy.com/galaxies/polar-common-go/base → 公共 base 层，封装日志初始化
```

### 2.2 日志初始化流程

#### 方式一：手动 OpenLogger（repair-maintenance-service）

文件：`repair-maintenance-service/pkg/base/base.go`

```go
logger, err := xlog.OpenLogger(&xlog.Config{
    Fields: xlog.Fields{
        Deployment: c.Deployment,   // 部署环境标识
        Host:       c.HostID,       // 主机ID
        App:        LogApp,         // 应用名：repairnotifyservice
        Component:  LogComponent,   // 组件名：repairnotifyservice
        Type:       xlog.LogTypeGeneral,
    },
    Development: c.DevMode,  // 开发模式：控制台输出更友好
    Debug:       c.Debug,    // 是否开启 Debug 级别
    Sample:      true,        // 启用日志采样
})
xlog.RedirectStdLog(logger)  // 将标准库 log 重定向到 xlog
xlog.ReplaceGlobals(logger)  // 替换全局 logger
commonbase.SetXLogger(logger) // 注入到公共 base 层
```

#### 方式二：通过 commonbase.InitLog（repair-workflow-service）

文件：`repair-workflow-service/pkg/base/base.go`

```go
if err := commonbase.InitLog(cf.GetXlogConfig(conf.LogApp, conf.LogComponent)); err != nil {
    return fmt.Errorf("init log failed: %w", err)
}
```

`GetXlogConfig` 方法由 `polar-common-go/config` 提供，统一封装日志配置构建逻辑。

#### 方式三：main 函数直接初始化（repair-notify-service）

文件：`repair-notify-service/cmd/repair-notify-service/main.go`

```go
err := cbase.InitLog(config.Config.GetXlogConfig(conf.Config, LogApp, LogComponent))
```

日志在 `main.go` 最早完成初始化，早于其他模块（DB、MQ 等）。

### 2.3 日志标识配置

每个 Go 服务在 `pkg/conf/config.go` 中定义两个常量：

| 服务 | LogApp | LogComponent |
|------|--------|--------------|
| repair-maintenance-service | `repairmaintenanceservice` | `repairmaintenanceservice` |
| repair-notify-service | `repairnotifyservice` | `repairnotifyservice` |
| repair-workflow-service | `repairworkflowservice` | `repairworkflowservice` |

这些字段会作为结构化字段注入每条日志，方便日志平台（如 ELK/Loki）过滤。

### 2.4 上下文感知日志（ContextLogger）

所有 Go 服务均提供 `GetCtxLogger()` 函数，返回携带 tracing 信息的上下文日志器：

```go
// repair-workflow-service/pkg/base/base.go
func GetCtxLogger() *xlog.ContextLogger {
    return commonbase.GetTracingLog()
}

// repair-maintenance-service/pkg/base/base.go
func GetCtxLogger() *xlog.ContextLogger {
    return commonbase.GetTracingLog()
}
```

`ContextLogger` 会自动将 gRPC 请求中的 `trace_id`、`span_id` 等追踪信息附加到每条日志，实现**分布式链路日志追踪**。

### 2.5 gRPC 拦截器中的日志中间件

文件：`repair-notify-service/pkg/base/grpc.go`

```go
grpctxlog.UnaryServerInterceptor(contextLogger, grpctxlog.WithDecider(
    func(fullMethodName string, err error) bool {
        // 按接口名过滤，不记录高频低价值的接口日志
        if ok := slices.Contains(conf.LogSkipUnaryAPI, method); ok {
            return false
        }
        return true
    }),
),
grpctxlog.PayloadUnaryServerInterceptor(contextLogger,
    func(_ context.Context, fullMethodName string, _ interface{}) bool {
        // 是否记录请求/响应 payload（可通过配置控制）
        return conf.LogUnaryPayload
    }),
),
```

- **UnaryServerInterceptor**：记录每个 gRPC 请求的耗时、状态码等信息
- **PayloadUnaryServerInterceptor**：可选记录请求/响应 payload
- **LogSkipUnaryAPI**：通过配置跳过高频接口（如健康检查），避免日志噪音
- **StreamServerInterceptor**：同样支持流式 gRPC 日志

### 2.6 数据库日志集成

文件：`repair-maintenance-service/pkg/base/base.go`

```go
confDB := &xgorm.DBConfig{
    // ...
    Debug:  c.Debug,
    Logger: gormxlog.New(logger),  // 将 xlog 注入 GORM，统一日志输出
}
```

通过 `gormxlog.New(logger)` 将 SQL 执行日志桥接到 xlog，实现所有日志统一输出格式。

### 2.7 日志级别使用规范（业务层）

Go 服务业务层主要使用 `klog/v2` 进行日志打印，使用格式约定如下：

#### klog.Infof —— 正常业务流程关键节点

```go
klog.Infof("[StartInstanceFromConfig] 开始创建工单 - ProcessDefinitionCode: %s", ...)
klog.Infof("[sync CreateImportTask] taskChannel is updated")
klog.Infof("===UpsertUserColumnConfig: page_type=%s, user_id=%s", req.PageType, req.UserId)
```

#### klog.Errorf —— 可恢复错误，继续执行

```go
klog.Errorf("[sla] Start error instanceID=%s taskKey=%s: %v", ormInstance.ID, taskInfo.TaskKey, err)
klog.Errorf("[sync updateComponents] propagate from host failed, assetCode=%s, err: %v", ...)
klog.Errorf("===UpsertUserColumnConfig ERROR: %v", err)
```

#### klog.Warningf —— 非致命异常，需关注

```go
klog.Warningf("[sla] fetchNotifyContext: get instance failed workOrder=%s: %v", workOrderID, err)
klog.Warningf("[StartInstanceFromConfig] 未查询到主机信息 - DeviceSN: %s", ...)
```

#### klog.Info（非格式化）—— 简单状态标记

```go
klog.Info("init component channel")
klog.Info("init component channel completed")
```

**日志前缀约定**：所有关键业务日志使用 `[模块名]` 前缀，如 `[sla]`、`[sync]`、`[StartInstanceFromConfig]`，便于快速过滤和定位。

### 2.8 测试环境日志（Mock Logger）

## 三、Java 服务日志管理（repair-chassis-service）

### 3.1 核心依赖

```
SLF4J      → 日志抽象接口（门面模式）
Logback    → 实际日志实现
Lombok @Slf4j → 注解方式注入 logger，无需手动声明
```

### 3.2 Logback 配置

配置文件：`repair-chassis-server/src/main/resources/logback-spring.xml`（本地开发）和 `logback-spring-server.xml`（生产环境）

两个配置文件除日志路径外完全一致：

| 配置 | 本地 | 生产 |
|------|------|------|
| 日志根路径 | `logs/${LOG_HOME}` | `/logs/${LOG_HOME}` |
| `LOG_HOME` 取值 | spring.application.name | spring.application.name |

**日志根级别**：`INFO`

**Appender 配置（5个）**：

| Appender 名称 | 类型 | 过滤级别 | 文件路径 | 单文件上限 | 保留天数 |
|--------------|------|----------|----------|------------|----------|
| console | ConsoleAppender | 无过滤（所有级别） | 控制台 | — | — |
| info | RollingFileAppender | INFO only | `{path}/info.log` | 1 GB | 30 天 |
| debug | RollingFileAppender | DEBUG only | `{path}/debug.log` | 1 GB | 30 天 |
| trace | RollingFileAppender | TRACE only | `{path}/trace.log` | 1 GB | 30 天 |
| error | RollingFileAppender | ERROR only | `{path}/error.log` | 1 GB | 30 天 |

**滚动策略**：`SizeAndTimeBasedRollingPolicy`
- 按**日期+大小**双维度滚动
- 文件命名：`{level}.yyyy-MM-dd.{i}.log.gz`（自动 gzip 压缩归档）
- 归档目录按月分组：`{path}/yyyy-MM/`

**控制台日志格式**（彩色）：
```
2024-01-15 10:30:45.123  INFO 12345 --- [   main] sh.bs.admin.server.AdminApp    : 启动成功
```

**文件日志格式**：
```
2024-01-15 10:30:45.123 [main] INFO  [sh.bs.admin.server.service.impl.ProjectServiceImpl] ProjectServiceImpl.java:45 - 日志内容
```

### 3.3 application.yml 中的日志级别配置

```yaml
logging:
  level:
    sh.bs.admin.server.mapper.*: debug         # Mapper 层开启 debug，可见 SQL
    com.netflix.client: DEBUG
    org.springframework.cloud: DEBUG
    org.springframework.cloud.openfeign: DEBUG
```

Mapper/Feign 等中间件开启 DEBUG，便于开发阶段排查 SQL 和远程调用问题。

### 3.4 Java 日志使用模式

#### Lombok @Slf4j 注解

```java
@Slf4j
@RestControllerAdvice
public class CommonExceptionHandler {
    // 直接使用 log 变量，无需手动声明 Logger
}
```

#### gRPC 拦截器日志

文件：`LogGrpcInterceptor.java`

```java
log.error("Token validation failed - Method: {}, ExceptionType: {}, Message: {}, ...",
    fullMethodName, e.getClass().getSimpleName(), e.getMessage(), ...);
```

> **注意**：该拦截器中还混用了 `System.out.println`（调试遗留代码），正式环境应替换为 log 输出。

#### 全局异常处理日志

```java
// 未知异常
log.error("请求地址:[{}],发生未知异常:[{}]", request.getRequestURI(), e.getMessage());

// 参数校验异常
log.error("请求地址:[{}],发生参数校验异常:[{}]", request.getRequestURI(), fieldErrors.get(0).getDefaultMessage());

// 业务异常
log.error("请求地址:[{}],发生业务异常:[{}]:[{}]", request.getRequestURI(), e.getStatus().getCode(), msg);

// 权限异常
log.error("请求地址:[{}],发生权限异常:[{}]:[{}]", request.getRequestURI(), ...);
```

统一在 `CommonExceptionHandler` 集中处理异常日志，格式固定，包含请求路径、异常类型、详细信息。

## 四、日志管理问题与优化建议

### 4.1 现存问题

| 问题 | 位置 | 说明 |
|------|------|------|
| `System.out.println` 残留 | `LogGrpcInterceptor.java` | 调试代码未清理，生产环境会影响性能且无法过滤 |
| `fmt.Println` 残留 | `repair-notify-service/pkg/base/rocketmq.go` | MQ 初始化失败时使用 `fmt.Println` 而非日志库 |
| `log.Fatal` 残留 | `rocketmq.go:91,95,214` | 使用标准 `log.Fatal` 直接退出，绕过日志系统 |
| base.go LogApp 错误 | `repair-maintenance-service/pkg/base/base.go:28-29` | 常量值写成了 `repairnotifyservice`，应为 `repairmaintenanceservice` |
| klog 与 xlog 混用 | Go 服务业务层 | 初始化用 xlog，业务层用 klog，两套 logger 并存 |

### 4.2 优化建议

1. **统一 Go 服务日志库**：业务层统一使用 `xlog` 或 `klog`，当前混用增加了维护成本
2. **修复 LogApp 常量**：`repair-maintenance-service/pkg/base/base.go` 中 `LogApp = "repairnotifyservice"` 应改为 `"repairmaintenanceservice"`
3. **清理 Java 调试日志**：移除 `LogGrpcInterceptor.java` 中的 `System.out.println`
4. **统一错误日志**：`rocketmq.go` 中 `fmt.Println` 和 `log.Fatal` 替换为 `cbase.GetXLog().Error`
5. **日志采样策略**：当前 `Sample: true` 开启采样，建议文档化采样率配置
6. **gRPC 接口日志过滤**：利用已有的 `LogSkipUnaryAPI` 配置，将健康检查等高频接口加入跳过列表

## 五、各服务日志初始化时序总结

```
repair-workflow-service:
main() → config.LoadDefaultConfig → base.Init → commonbase.InitLog(GetXlogConfig) → 日志就绪

repair-maintenance-service:
main() → config.LoadDefaultConfig → base.Init → xlog.OpenLogger → xlog.ReplaceGlobals → 日志就绪

repair-notify-service:
main() → config.LoadDefaultConfig → cbase.InitLog(GetXlogConfig) → base.InitDB → base.Init → 日志最先就绪

repair-chassis-service:
Spring Boot 启动 → Logback 自动加载 logback-spring.xml → 日志就绪
```
