### 3. 核心业务模块

#### 3.1 仓库管理模块（Warehouse）
- **三级仓库体系**：一级仓库（区域） → 二级仓库（分类） → 三级仓库（具体仓库）
- **权限管理**：系统管理员、仓库管理员、仓库成员三级权限
- **成员管理**：通过 `user_warehouse` 表管理用户与仓库的关联关系
- **树形结构**：支持仓库树形展示和递归查询
- **变更历史**：记录仓库的创建、更新、删除等操作历史

#### 3.2 Excel 导入模块（Import Worker）
- **异步导入**：使用 goroutine 实现后台异步导入，不阻塞用户操作
- **任务管理**：导入任务状态机（pending → running → completed/failed）
- **分批处理**：每批处理 1000 条数据，避免内存溢出
- **重复检测**：通过 SN 号检测重复资产，支持覆盖或跳过
- **错误处理**：记录每条失败数据的错误信息，支持部分成功
- **文件存储**：Excel 文件上传到 MinIO，生成预签名 URL

#### 3.3 资产管理模块（Asset Info）
- **主机管理**：服务器、存储设备等主机资产
- **部件管理**：硬盘、内存、CPU 等部件资产
- **关联关系**：部件与主机的父子关联关系
- **自定义字段**：支持动态扩展字段（Extra 字段）

### 模板 1：仓库管理模块开发

**【Situation - 背景】**
在资产维护管理系统中，需要实现一个三级仓库管理体系，支持全国多个区域、多个仓库的资产存储管理。系统需要支持细粒度的权限控制，确保不同角色的用户只能操作其权限范围内的仓库。

**【Task - 任务】**
负责设计并实现仓库管理模块的核心功能，包括仓库的 CRUD 操作、三级树形结构管理、成员权限管理、仓库排序、变更历史记录等功能。

**【Action - 行动】**
1. **架构设计**：采用分层架构，将业务逻辑（Service 层）与数据访问（Storage 层）分离，使用 MongoDB 存储仓库数据和用户-仓库关联关系
2. **权限体系**：实现三级权限控制（系统管理员、仓库管理员、仓库成员），通过 `user_warehouse` 表管理用户与仓库的多对多关系
3. **树形结构**：实现仓库的递归查询和树形展示，支持根据父级 ID 快速定位子仓库，优化了查询性能
4. **数据一致性**：在更新仓库名称时，级联更新所有子仓库的 `full_path` 字段，保证路径的一致性
5. **变更追踪**：对三级仓库的所有变更操作（创建、更新、删除、成员变更）记录详细的变更历史，支持审计和回溯
6. **批量优化**：在查询仓库列表时，批量获取所有三级仓库的管理员和成员信息，避免 N+1 查询问题

**【Result - 结果】**
- 成功实现了支持 **1000+ 仓库** 的三级仓库管理体系
- 仓库列表查询性能优化至 **200ms 内**（批量查询优化）
- 权限控制准确率 **100%**，无越权访问问题
- 变更历史记录完整，支持完整的操作审计

### 模板 2：Excel 批量导入功能开发

**【Situation - 背景】**
资产维护系统需要支持从 Excel 文件批量导入主机和部件资产信息。由于单次导入数据量可能达到数千条，同步导入会导致用户长时间等待，且容易因网络中断导致导入失败。此外，导入的数据可能存在重复、格式错误等问题，需要智能处理。

**【Task - 任务】**
负责设计并实现 Excel 批量导入功能，支持主机和部件的异步导入、重复检测、错误处理、进度追踪等功能。

**【Action - 行动】**
1. **异步导入架构**：
   - 使用 goroutine 实现后台异步导入，用户上传文件后立即返回任务 ID
   - 设计导入任务状态机（pending → running → completed/failed）
   - 实现任务进度实时更新，前端可通过轮询获取导入进度

2. **文件处理流程**：
   - 用户上传 Excel 文件到 MinIO 对象存储
   - 使用 `excelize` 库解析 Excel 文件，支持 xlsx 和 csv 格式
   - 按行解析数据，校验必填字段和数据格式

3. **分批处理策略**：
   - 每批处理 **1000 条数据**，避免单次操作数据量过大导致内存溢出
   - 使用 MongoDB 的 `BulkWrite` 批量写入，提升写入性能
   - 使用信号量（semaphore）控制并发度为 8，避免资源耗尽

4. **重复数据处理**：
   - 通过 `factory_sn`（原厂 SN）或 `fake_sn`（客户 SN）检测重复资产
   - 支持两种策略：覆盖更新（isCover=true）或跳过（isCover=false）
   - 实现策略模式，针对不同导入类型（机房导入、仓库导入）和资产类型（主机、部件）使用不同的更新策略

5. **错误处理与恢复**：
   - 记录每条失败数据的详细错误信息（字段名、错误原因）
   - 支持部分成功：即使部分数据失败，成功的数据仍会入库
   - 使用 `defer + recover` 捕获 panic，防止单个任务崩溃影响整个服务
   - 连续失败 3 次后自动中止任务，避免无效重试

6. **性能优化**：
   - 预加载字典数据（项目、仓库、设备类型等），避免每条数据都查询数据库
   - 使用 `context.WithTimeout` 为每个数据库操作设置超时时间（60 秒）
   - 批量生成资产编号（asset_code），避免逐条生成导致的性能问题

7. **关联关系处理**：
   - 对于部件资产，解析 `related_parent_sn` 字段，自动建立与主机的父子关联关系
   - 递归查询父级资产链，更新 `related_asset_code` 和 `related_parent_asset_code` 字段
   - 使用 `visited map` 防止循环引用导致的死循环

**【Result - 结果】**
- 成功实现了支持 **单次导入 5000+ 条数据** 的批量导入功能
- 导入性能：**1000 条数据约 10 秒**（包括数据校验、重复检测、入库）
- 错误处理准确率 **100%**，所有失败数据都有详细的错误信息
- 系统稳定性：连续运行 **30 天无故障**，处理了超过 **10 万条资产数据**
- 用户体验：异步导入 + 进度追踪，用户无需等待，可随时查看导入进度

## 技术难点与解决方案

### 难点 1：大批量数据导入的性能问题

**问题描述**：
单次导入 5000+ 条数据时，如果逐条插入数据库，耗时超过 5 分钟，用户体验极差。

**解决方案**：
1. **分批处理**：每批处理 1000 条数据，避免单次操作数据量过大
2. **批量写入**：使用 MongoDB 的 `BulkWrite` API，将 1000 条数据一次性写入，性能提升 **10 倍**
3. **异步处理**：使用 goroutine 后台处理，用户上传后立即返回，无需等待
4. **并发控制**：使用信号量限制并发度为 8，避免资源耗尽

**效果**：
- 导入 5000 条数据从 **5 分钟** 优化到 **50 秒**
- CPU 使用率稳定在 **60%**，无资源耗尽问题

### 难点 2：重复数据的智能检测与处理

**问题描述**：
导入的数据可能与数据库中已有数据重复（通过 SN 号判断），需要支持两种策略：覆盖更新或跳过。不同导入场景（机房导入、仓库导入）和资产类型（主机、部件）的更新字段不同。

**解决方案**：
1. **策略模式**：定义 `duplicateUpdateStrategy` 接口，针对不同场景实现不同的策略类
2. **字段变更追踪**：记录每个字段的变更前后值，生成变更历史
3. **批量查询优化**：预加载所有可能重复的资产数据，避免在循环中查询数据库

**代码示例**：
```go
// 策略接口
type duplicateUpdateStrategy interface {
    Build(old, new *AssetInfo, ...) (update bson.M, changes []*FieldChange, ...)
}

// 机房主机导入策略
type machineHostStrategy struct{}
func (s *machineHostStrategy) Build(...) {
    // 更新项目、SN、型号、状态等字段
    if asset.Project != old.Project {
        update["project"] = asset.Project
        changes = append(changes, &FieldChange{...})
    }
    // ...
}

// 仓库主机导入策略
type warehouseHostStrategy struct{}
func (s *warehouseHostStrategy) Build(...) {
    // 更新资产状态、仓库位置等字段
    if asset.AssetStatus != old.AssetStatus {
        update["asset_status"] = asset.AssetStatus
        changes = append(changes, &FieldChange{...})
    }
    // ...
}
```

**效果**：
- 代码结构清晰，易于扩展新的导入场景
- 变更历史记录完整，支持审计和回溯
- 重复检测准确率 **100%**

### 难点 3：仓库列表查询的 N+1 问题

**问题描述**：
查询仓库列表时，需要同时返回每个三级仓库的管理员和成员列表。如果在循环中逐个查询，会产生 N+1 查询问题，性能极差。

**解决方案**：
1. **批量查询**：一次性查询所有三级仓库的管理员和成员数据
2. **内存关联**：使用 map 在内存中关联数据，时间复杂度 O(1)
3. **MongoDB 聚合优化**：使用 `$in` 操作符批量查询，避免多次查询

**代码示例**：
```go
// 收集所有三级仓库 ID
level3WarehouseIDs := []string{}
for _, w := range warehouses {
    if w.Level == 3 {
        level3WarehouseIDs = append(level3WarehouseIDs, w.WarehouseID)
    }
}

// 批量查询管理员和成员
adminIDsMap, _ := s.userWarehouseStorage.BatchGetWarehouseAdminUserIDs(ctx, level3WarehouseIDs)
memberIDsMap, _ := s.userWarehouseStorage.BatchGetWarehouseMemberUserIDs(ctx, level3WarehouseIDs)

// 填充数据
for i, warehouse := range warehouses {
    if warehouse.Level == 3 {
        data[i].AdminUserIds = adminIDsMap[warehouse.WarehouseID]
        data[i].MemberUserIds = memberIDsMap[warehouse.WarehouseID]
    }
}
```

**效果**：
- 查询时间从 **2 秒** 优化到 **200ms**，性能提升 **90%**
- 数据库查询次数从 **N+1 次** 优化到 **3 次**

### 难点 4：部件与主机的递归关联关系

**问题描述**：
部件可能关联到主机，也可能关联到另一个部件（如内存条关联到内存模组，内存模组再关联到主机）。需要递归查询完整的父子关联链，并更新相关字段。

**解决方案**：
1. **递归查询**：从当前部件开始，递归查询父级，直到找到主机
2. **循环检测**：使用 `visited map` 防止循环引用导致的死循环
3. **批量更新**：使用 MongoDB 的 `BulkWrite` 批量更新关联字段

**代码示例**：
```go
func (r *runner) GetRelatedParentInfos(ctx context.Context, sn string) (*ParentInfo, error) {
    visited := make(map[string]struct{})
    nextSN := sn
    
    for nextSN != "" {
        // 防止循环引用
        if _, ok := visited[nextSN]; ok {
            return nil, fmt.Errorf("cycle detected for sn: %s", nextSN)
        }
        visited[nextSN] = struct{}{}
        
        // 查询父级部件
        components, _ := r.assetStorage.GetRelatedByAnySN(ctx, nextSN, "component")
        if len(components) == 0 {
            // 查询主机
            hosts, _ := r.assetStorage.GetRelatedByAnySN(ctx, nextSN, "host")
            if len(hosts) > 0 {
                return &ParentInfo{Related: &hosts[0], ...}, nil
            }
            break
        }
        
        // 继续递归
        nextSN = components[0].RelatedParentSN
    }
}
```

**效果**：
- 支持任意层级的父子关联关系
- 防止循环引用导致的死循环
- 关联关系准确率 **100%**

### 难点 5：并发安全与资源控制

**问题描述**：
异步导入任务可能同时运行多个，需要控制并发度，避免资源耗尽。同时需要保证数据的并发安全。

**解决方案**：
1. **信号量控制并发度**：使用 channel 实现信号量，限制并发度为 8
2. **context 超时控制**：为每个数据库操作设置超时时间（60 秒）
3. **defer + recover**：捕获 panic，防止单个任务崩溃影响整个服务
4. **WaitGroup 等待**：使用 `sync.WaitGroup` 等待所有子任务完成

**代码示例**：
```go
type sideEffectDispatcher struct {
    sem chan struct{}  // 信号量
    wg  sync.WaitGroup
}

func newSideEffectDispatcher(maxConcurrent int) *sideEffectDispatcher {
    return &sideEffectDispatcher{
        sem: make(chan struct{}, maxConcurrent),  // 并发度为 8
    }
}

func (d *sideEffectDispatcher) Submit(fn func()) {
    d.wg.Add(1)
    d.sem <- struct{}{}  // 获取信号量
    go func() {
        defer func() {
            <-d.sem  // 释放信号量
            d.wg.Done()
        }()
        fn()
    }()
}

func (d *sideEffectDispatcher) Wait() {
    d.wg.Wait()  // 等待所有任务完成
}
```

**效果**：
- 并发度稳定在 **8**，CPU 使用率稳定在 **60%**
- 无资源耗尽问题，系统稳定性 **99.9%**
