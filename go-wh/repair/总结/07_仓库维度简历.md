## 二、主要职责详细描述（按STAR方法）

### 职责一：三级仓库体系的设计与实现

#### **S (Situation) - 业务背景**
- **业务痛点**：公司资产分布在全球多个地区（洲级 → 国家级 → 具体仓库），原有系统缺乏层级化的仓库管理能力，导致资产归属不清、权限管理混乱、跨层级调拨困难。
- **数据规模**：需管理 3 级仓库层级（洲级、国家级、具体仓库），支持数百个三级仓库节点，每个仓库关联数千至上万条资产记录。
- **核心需求**：
  - 支持树形结构的仓库层级管理（最多 3 级）
  - 实现基于角色的仓库权限控制（系统管理员、仓库管理员、普通成员）
  - 支持仓库拖拽排序、跨层级移动、成员管理等复杂操作

#### **T (Task) - 技术任务**
我负责从 0 到 1 设计并实现三级仓库管理模块，包括：
1. **数据模型设计**：设计 MongoDB 仓库表结构，支持父子关系、完整路径、排序序号等字段
2. **核心功能开发**：实现仓库 CRUD、树形结构查询、成员管理、权限校验、拖拽排序等 20+ 个 gRPC 接口
3. **权限体系设计**：设计 `user_warehouse` 关联表，实现细粒度的仓库权限控制

#### **A (Action) - 解决方案与技术难点**

**1. 数据模型设计 - 解决层级关系与路径维护问题**
- **技术选型**：采用 **MongoDB** 存储仓库数据，使用 `parent_id`（存储父级 `warehouse_id`）+ `full_path`（完整路径字符串）的混合方案
  - `parent_id`：存储父级仓库的业务 ID（如 `WH00002`），便于快速查询子节点
  - `full_path`：存储完整路径（如 `亚洲-中国-北京仓库`），便于模糊搜索和路径展示
  - `sort_order`：同层级排序序号，支持拖拽排序
- **难点解决**：跨层级移动时，需级联更新所有子仓库的 `full_path` 和 `level`
  - 实现 `BatchUpdateChildrenPath` 方法，通过路径前缀匹配（`full_path LIKE 'oldPrefix%'`）批量更新子节点
  - 使用 MongoDB 的 `$regex` 查询 + 批量 `UpdateMany` 操作，将单次移动的数据库操作从 O(n) 优化到 O(1)

**2. 权限体系设计 - 解决细粒度权限控制问题**
- **技术方案**：设计独立的 `user_warehouse` 关联表，将用户-仓库-角色关系解耦
  - 字段：`warehouse_id`、`user_id`、`user_type`（1=普通成员，3=管理员）
  - 优势：支持一个用户在不同仓库拥有不同角色，避免在 `warehouse` 表中存储冗余的成员数组
- **权限校验逻辑**：
  - 系统管理员：拥有所有仓库的完整权限
  - 仓库管理员：可管理其所属三级仓库的成员、资产、配置
  - 普通成员：仅可查看仓库信息和资产列表
- **性能优化**：批量查询场景下，使用 `BatchGetWarehouseAdminUserIDs` 一次性获取多个仓库的管理员列表，避免 N+1 查询问题

**3. 树形结构查询优化 - 解决大数据量下的性能问题**
- **技术方案**：
  - 一次性查询所有未删除仓库（`is_deleted != 1`），在内存中构建树形结构
  - 使用 `map[warehouse_id]*WarehouseTreeNode` 缓存节点，时间复杂度 O(n)
  - 对于非管理员用户，先查询其所属仓库，再递归向上查询祖先节点，过滤出可见仓库子树
- **性能表现**：300+ 仓库节点的树形结构查询耗时 < 50ms

**4. 拖拽排序功能 - 解决同层级与跨层级移动的复杂逻辑**
- **技术难点**：
  - 同层级拖拽：需调整 `sort_order`，填补空缺序号
  - 跨层级移动：需校验层级限制（最多 3 级）、名称重复、循环依赖，并级联更新子节点
- **解决方案**：
  - 实现 `MoveWarehouse` 方法，区分同层级和跨层级两种场景
  - 跨层级移动时：
    1. 校验目标层级深度（`newLevel + maxDescendantLevel <= 3`）
    2. 检测循环依赖（不能将仓库移动到自己的子节点下）
    3. 批量更新子节点的 `parent_id`、`level`、`full_path`
    4. 重新整理原父级和新父级下的 `sort_order`
  - 同层级移动时：使用 `ReorderSiblings` 方法，按 `sort_order` 重新排序，填补空缺

**5. 变更历史记录 - 解决审计追溯需求**
- **技术方案**：集成 `ChangeHistoryService`，记录仓库的创建、更新、删除、成员变更等操作
- **记录内容**：操作人、操作时间、操作类型、变更前后的字段对比（如成员列表、父级仓库、完整路径）
- **应用场景**：支持仓库管理员查看仓库的完整操作历史，满足合规审计要求

#### **R (Result) - 项目成果**
- **功能完整性**：实现了 20+ 个仓库管理相关的 gRPC 接口，覆盖 CRUD、树形查询、成员管理、权限校验、拖拽排序、变更历史等全流程
- **性能指标**：
  - 树形结构查询：300+ 节点 < 50ms
  - 跨层级移动：包含 10+ 子节点的仓库移动 < 200ms
  - 批量权限校验：100 个仓库的管理员查询 < 100ms
- **业务价值**：
  - 支持全球多地区的仓库层级化管理，资产归属清晰
  - 实现细粒度的权限控制，不同角色的用户只能操作其权限范围内的仓库
  - 提供完整的变更历史记录，满足审计合规要求

### 职责二：Excel 导入模块的性能优化（Worker Pool + 批量插入）

#### **S (Situation) - 业务背景**
- **业务痛点**：资产管理系统需要支持批量导入主机、部件等资产数据，原有的逐行插入方案在导入 1000+ 条数据时耗时超过 5 分钟，严重影响用户体验。
- **数据规模**：单次导入任务可能包含 5000+ 条资产记录，每条记录需生成唯一资产编号、校验数据合法性、插入数据库、创建变更历史等多个步骤。
- **核心需求**：将导入耗时从分钟级优化到秒级，同时保证数据一致性和错误处理能力。

#### **T (Task) - 技术任务**
我负责重构 Excel 导入模块，优化导入性能，包括：
1. **批量生成资产编号**：替换原有的逐条生成方案，实现批量生成唯一编号
2. **批量插入数据库**：使用 MongoDB 的 `BulkCreate` 替代逐条 `Insert`
3. **异步处理副作用**：将变更历史、关联关系等非核心操作异步化，避免阻塞主流程
4. **Worker Pool 并发控制**：使用 Goroutine Pool 控制并发度，避免资源耗尽

#### **A (Action) - 解决方案与技术难点**

**1. 批量生成资产编号 - 解决唯一性与性能问题**
- **原方案问题**：逐条查询数据库获取最大编号，再递增生成新编号，导致 N 次数据库查询
- **优化方案**：实现 `GenerateBatchAssetCodes` 方法
  - 一次性查询当前最大编号（如 `HOST00500`）
  - 在内存中批量生成连续编号（`HOST00501` ~ `HOST05500`）
  - 时间复杂度从 O(n) 降低到 O(1)
- **并发安全**：使用 MongoDB 的原子操作（`FindOneAndUpdate` + `$inc`）保证分布式环境下的编号唯一性

**2. 批量插入数据库 - 解决插入性能瓶颈**
- **原方案问题**：逐条调用 `Create` 方法插入数据，每次插入都需要一次网络往返
- **优化方案**：
  - 使用 MongoDB 的 `InsertMany` 批量插入（每批 1000 条）
  - 实现 `BulkCreate` 方法，支持批量插入 + 错误处理
  - 分批处理：将 5000 条数据拆分为 5 批，每批 1000 条，避免单次请求过大
- **性能提升**：批量插入耗时从 5 分钟降低到 30 秒，提升 **10 倍**

**3. Worker Pool 并发控制 - 解决副作用处理的性能问题**
- **技术方案**：实现 `sideEffectDispatcher` 结构体，基于 Channel + WaitGroup 实现 Goroutine Pool
  ```go
  type sideEffectDispatcher struct {
      sem chan struct{}  // 信号量，控制最大并发数
      wg  sync.WaitGroup // 等待所有任务完成
  }
  ```
- **核心逻辑**：
  - 使用带缓冲的 Channel（`sem`）作为信号量，限制最大并发数为 8
  - 每个任务提交时，先获取信号量（`sem <- struct{}{}`），执行完成后释放（`<-sem`）
  - 主流程调用 `Wait()` 等待所有副作用任务完成
- **应用场景**：
  - 异步创建变更历史记录（`CreateChangeHistory`）
  - 异步更新父级关联信息（`updateParentInfo`）
  - 异步计算资产完整度（`CalCompletenessByAssetCodes`）
- **性能优化**：将副作用操作从主流程中剥离，主流程耗时从 30 秒降低到 **15 秒**

**4. 分批处理策略 - 解决大数据量导入的内存问题**
- **技术方案**：
  - 将导入任务拆分为多个批次（每批 1000 条），逐批处理
  - 每批处理完成后，立即释放内存，避免 OOM
  - 实现 `runPhase` 方法，支持分批查询 `ImportTaskItem` + 批量处理
- **代码示例**：
  ```go
  func (r *startImportRunner) runPhase(itemType string, processFn func([]*mongo.ImportTaskItem) error) error {
      for {
          items, err := r.svc.taskService.GetLimitPendingConfirmeds(ctx, taskID, itemType, "", r.limit, sort)
          if len(items) == 0 { break }
          if err := processFn(items); err != nil { return err }
      }
  }
  ```

**5. 错误处理与回滚机制**
- **技术方案**：
  - 批量插入失败时，记录失败的行号和错误信息，返回给前端
  - 使用 `ImportTaskItem` 表记录每条数据的处理状态（`pending`、`completed`、`failed`）
  - 支持重试机制：用户可选择重新导入失败的数据

#### **R (Result) - 项目成果**
- **性能提升**：
  - 导入 5000 条数据的耗时从 **5 分钟** 降低到 **15 秒**，提升 **20 倍**
  - 批量生成资产编号：从 N 次查询优化到 1 次查询
  - 批量插入数据库：从逐条插入优化到批量插入（每批 1000 条）
- **并发控制**：
  - 使用 Worker Pool 控制副作用操作的并发度（最大 8 个 Goroutine），避免资源耗尽
  - 主流程与副作用操作解耦，提升用户体验
- **业务价值**：
  - 支持用户快速批量导入资产数据，提升运营效率
  - 提供完整的错误处理和重试机制，保证数据一致性
  - 降低服务器资源消耗，支持更高的并发导入任务

### 职责三：工单工作流引擎中仓库维度的应用

#### **S (Situation) - 业务背景**
- **业务场景**：维修工单流程涉及资产的状态变更、仓库流转、质量检测等多个环节，需要在工作流引擎中集成仓库维度的业务逻辑。
- **核心需求**：
  - 工单流程中需要调用维修服务的 gRPC 接口，更新资产的仓库位置、质量状态、资产状态等
  - 支持批量资产入库、批量状态更新等操作
  - 工作流引擎需要支持动态参数传递（如仓库 ID、资产编号等）

#### **T (Task) - 技术任务**
我参与了工单工作流引擎与维修服务的集成开发，负责：
1. **gRPC 回调处理器设计**：实现 `GRPCMaintenanceHandler`，支持工作流引擎调用维修服务的 gRPC 接口
2. **仓库维度参数传递**：设计动态参数传递机制，支持工作流节点中配置仓库 ID、资产状态等参数
3. **批量操作接口开发**：实现批量资产入库、批量状态更新等 gRPC 接口

#### **A (Action) - 解决方案与技术难点**

**1. gRPC 回调处理器设计 - 解决工作流与业务服务的解耦问题**
- **技术方案**：
  - 实现 `GRPCMaintenanceHandler` 结构体，注册多个 gRPC 服务路径（如 `UpdateAssetQualityAndWarehouse`、`BatchAssetsToWarehouse` 等）
  - 工作流引擎在执行任务节点时，根据配置的 URL 路由到对应的 gRPC 方法
  - 使用 `context.WithTimeout` 设置超时时间（30 秒），避免长时间阻塞
- **代码示例**：
  ```go
  func (h *GRPCMaintenanceHandler) Handle(ctx *dax.Context, url string, params map[string]interface{}) error {
      switch {
      case containsIgnoreCase(url, PathUpdateAssetQualityAndWarehouse):
          return h.callUpdateAssetQualityAndWarehouse(ctx, params, taskInfo)
      case containsIgnoreCase(url, PathBatchAssetsToWarehouse):
          return h.callBatchAssetsToWarehouse(ctx, params, taskInfo)
      // ...
      }
  }
  ```

**2. 仓库维度参数传递 - 解决动态参数配置问题**
- **技术难点**：工作流节点的参数可能来自表单输入、上游节点输出、缓存等多个来源
- **解决方案**：
  - 设计统一的参数解析逻辑，支持从 `params` 中提取 `warehouse_id`、`asset_code`、`quality_status` 等参数
  - 对于批量操作，支持解析 JSON 数组（如 `batch_assets_to_warehouse`）
  - 使用 Redis 缓存临时参数（如 `batch_update_status_warehouse`），避免参数在多个节点间传递时丢失
- **代码示例**：
  ```go
  // 从缓存获取仓库 ID
  if cachedWarehouse, err := base.RedisGet(ctx.Context, "batch_update_status_warehouse"); err == nil {
      warehouse = cachedWarehouse
  }
  ```

**3. 批量操作接口开发 - 解决批量资产入库的性能问题**
- **技术方案**：
  - 实现 `BatchAssetsToWarehouse` gRPC 接口，支持批量资产入库
  - 解析工作流传递的批量资产数据（JSON 格式），转换为 gRPC 请求
  - 调用维修服务的批量插入方法，一次性完成多条资产的入库操作
- **应用场景**：
  - 维修完成后，批量将资产入库到指定仓库
  - 资产调拨时，批量更新资产的仓库位置和状态

**4. 工作流实例信息传递 - 解决审计追溯需求**
- **技术方案**：
  - 在 gRPC 请求中携带工作流实例信息（`instance_code`、`instance_name`、`instance_created_by` 等）
  - 维修服务在创建变更历史时，记录工作流实例信息，便于追溯操作来源
- **业务价值**：支持从资产变更历史反向查询到触发变更的工单实例，满足审计需求

#### **R (Result) - 项目成果**
- **功能完整性**：实现了 8 个工作流回调接口，覆盖资产状态更新、仓库入库、批量操作等核心场景
- **集成效果**：
  - 工作流引擎与维修服务无缝集成，支持在工单流程中动态调用仓库相关接口
  - 支持批量操作，单次工单可处理 100+ 条资产的入库或状态更新
- **业务价值**：
  - 实现了维修工单流程的自动化，减少人工操作
  - 提供完整的审计追溯能力，每次资产变更都记录了触发的工单实例信息
