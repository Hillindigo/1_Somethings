# 职责三：工单工作流集成（gRPC）深度面试分析

> **简历原句**：设计统一回调路由层，以接口路径分发 8 类资产操作，解耦工作流引擎与维护服务；大批量参数通过 Redis 缓存中转（TTL 24h），消息体控制在固定大小；变更历史写入工单来源实现双向溯源，资产状态同步延迟从 2~4 小时降至秒级。

---

## 一、背景：为什么要集成工单工作流？

### 1.1 旧方案的问题

旧版资产状态变更完全依赖人工操作：

```
工单审批通过 → 人工登录维护系统 → 手动修改资产状态 → 手动写备注
```

问题：
- **延迟高**：审批通过到状态同步最长需要 2~4 小时（等人工操作）
- **容易出错**：人工操作，字段遗漏、状态填错时有发生
- **无溯源**：资产变更历史不记录来源工单，无法反查"是哪个工单触发了这次变更"
- **强耦合**：工作流引擎（Camunda）直接调用维护服务具体方法，任何接口变更都需要同步修改两侧

### 1.2 目标

- 工单审批通过后，**秒级自动同步**资产状态，无需人工介入
- 变更历史中记录来源工单编号，实现**双向溯源**
- 工作流引擎与维护服务**解耦**：通过 URL 路径分发，工作流引擎不感知具体 gRPC 方法

---

## 二、核心设计：统一回调路由层

### 2.1 整体调用链

```
Camunda 工单审批通过
    ↓
taskCompleteAfter Hook（hook.go:338）
    ↓
processCallbackTemplate（callback.go:19）
    ↓  解析按钮配置中的 CallbackTemplate.UrlTemplate
    ↓  收集所有回调参数（表单数据 + 系统字段）
    ↓
routeCallbackRequest（callback_handler.go:28）
    ↓  责任链模式：遍历 callbackHandlers，Match(url) 匹配
    ↓
GRPCMaintenanceHandler.Handle（callback_grpc_maintenance.go:62）
    ↓  switch url 路由到具体 gRPC 方法
    ↓
维护服务 gRPC 接口（repair-maintenance-service）
    ↓
更新 asset_info + 写维修记录（含工单来源）
```

**关键设计**：工作流引擎（Camunda）只配置了一个 URL 字符串（如 `RepairRecordService/UpdateAssetStatus`），它不知道这个 URL 背后是哪个 gRPC 方法，由路由层负责匹配和分发。

### 2.2 回调处理器接口（责任链模式）

```go
// callback_handler.go:12-18
type CallbackHandler interface {
    Match(url string) bool   // 判断是否能处理该 URL
    Handle(ctx *dax.Context, url string, method string,
           headers map[string]string, body string,
           params map[string]interface{},
           taskInfo *model.InstanceTaskInfo) error
}

// 处理器注册表（按优先级排序，可扩展）
var callbackHandlers = []CallbackHandler{
    NewGRPCMaintenanceHandler(),  // 当前只有一个 handler，未来可加 HTTP handler 等
}
```

**责任链的优势**：新增一类回调（如第三方 HTTP 回调）只需新增一个实现了 `CallbackHandler` 接口的结构体并注册，不修改现有路由逻辑。

### 2.3 8 类资产操作路由表

```go
// callback_grpc_maintenance.go:22-30
const (
    PathUpdateAssetQualityAndWarehouse = "RepairRecordService/UpdateAssetQualityAndWarehouse"
    PathUpdateAssetStatus              = "RepairRecordService/UpdateAssetStatus"
    PathCreateRepairRecordNew          = "RepairRecordService/CreateRepairRecordNew"
    PathBatchAssetsToWarehouse         = "RepairRecordService/BatchAssetsToWarehouse"
    PathBatchUpdateStatus              = "RepairRecordService/BatchUpdateStatus"
    PathBatchUnbindComponent           = "RepairRecordService/BatchUnbindComponent"
    PathUpdateHostStatus               = "HostInfoService/UpdateHostStatus"
    PathCreateProjectAsset             = "RepairRecordService/CreateProjectAsset"
)
```

**8 类操作详解**：

| 路径常量 | 触发场景 | 操作内容 |
|---------|---------|---------|
| `UpdateAssetQualityAndWarehouse` | 寄修工单：设备寄出/收回 | 同时更新资产**质量状态**（良品/不良品）+ **所属仓库** + **设备状态** |
| `UpdateAssetStatus` | 维修工单：单设备状态流转 | 更新单个主机（SV前缀）或部件（CP前缀）的**设备状态** |
| `CreateRepairRecordNew` | 换件工单：部件替换 | 批量处理换下/换上逻辑，更新关联关系，出仓库入项目，创建维修记录 |
| `BatchAssetsToWarehouse` | 寄修工单：批量新设备入库 | 批量创建资产档案（生成 AssetCode）+ 入库记录 |
| `BatchUpdateStatus` | 仓库调拨工单 | 批量更新资产状态 + 仓库状态，处理仓库→仓库/仓库→项目/项目→仓库三种流向 |
| `BatchUnbindComponent` | 拆机工单 | 批量解绑部件与主机的关联关系（清空 `related_fake_sn`、`related_parent_sn`） |
| `UpdateHostStatus` | 主机独立状态变更 | 更新主机级状态（区别于设备级状态，对应 `host_info` 表） |
| `CreateProjectAsset` | 项目资产创建工单 | 按工单信息创建项目资产档案，绑定到指定项目 |

**为什么按 URL 路径而不是按 method 分发？**

工作流引擎（Camunda）传过来的是字符串形式的 URL（`CallbackTemplate.UrlTemplate`），没有独立的 method 字段概念。用 URL 路径作为分发 key，让流程设计师在配置工单时只需填写路径字符串，不需要了解底层 gRPC 服务接口细节。

## 三、参数收集：4 层优先级合并

回调参数来自多个来源，按优先级从低到高合并到 `allParams map[string]interface{}`：

```go
// callback.go:59-174（collectCallbackParams）
func collectCallbackParams(...) map[string]interface{} {
    params := make(map[string]interface{})

    // 第1层：表单字段的当前值（用户输入，优先级最低）
    for key, variable := range formVariables {
        params[key] = parseVariableValue(variable)
    }

    // 第1.5层：历史任务字段的 callbackParams（跨任务传递的静态参数）
    // 跳过 "draft" 开头的草稿字段
    for _, variable := range formVariables {
        for cpKey, cpValue := range variable.CallbackParams {
            if strings.HasPrefix(cpKey, "draft") { continue }  // 草稿不参与回调
            if _, exists := params[cpKey]; !exists {            // 不覆盖用户输入
                params[cpKey] = normalizeCallbackParamValue(cpValue.AsInterface())
            }
        }
    }

    // 第2层：表单配置级 callbackParams（静态元数据，如 instance_three_level3_id）
    for _, fieldConf := range formFieldConfs {
        for key, value := range fieldConf.CallbackParams {
            if _, exists := params[key]; !exists {
                params[key] = normalizeCallbackParamValue(value.AsInterface())
            }
        }
    }

    // 第3层：系统字段（instance_id、project_id 等，由系统自动填入）
    params["instance_id"] = processInstance.ID
    params["instance_code"] = processInstance.Code
    params["project_id"] = processInstance.Project
    params["instance_created_by"] = processInstance.CreatedBy
    params["instance_three_level1_id"] = processInstance.ThreeLevel1ID
    // ...

    // 第4层：当前任务字段的 callbackParams（优先级最高，覆盖所有历史数据）
    for _, variable := range currentTaskFormVariables {
        for cpKey, cpValue := range variable.CallbackParams {
            params[cpKey] = normalizeCallbackParamValue(cpValue.AsInterface())  // 直接覆盖
        }
    }

    return params
}
```

**为什么需要 4 层？**

一个工单可能经过多个审批节点，每个节点的表单都可能携带参数：
- 节点 A（发起）：填写了仓库 ID
- 节点 B（审批）：填写了质量状态
- 节点 C（执行）：触发回调，需要合并 A+B+C 三个节点的参数

通过 `collectInstanceFormData` 收集同一工单所有已完成任务的表单数据，再合并当前任务数据，确保所有字段都能传到 gRPC。

## 四、难点：Redis 缓存中转大批量参数

### 4.1 问题根因：gRPC 消息体大小限制

Camunda 工作流引擎的变量存储有大小限制（通常单个变量 < 64KB）。批量仓库调拨场景下，`batch_update_status` 可能包含数百个资产的状态信息，序列化后超出 Camunda 变量大小限制。

### 4.2 解决方案：分离大参数，走 Redis 中转

**触发写入（task.go:352-366）**：在任务完成前，把超大参数写入 Redis：

```go
// task.go:352-366（CompleteTask 方法内）
for _, variable := range req.FormVariables {
    if variable != nil && variable.CallbackParams != nil {
        if warehouseValue, ok := variable.CallbackParams["batch_update_status_warehouse"]; ok {
            if warehouseStr := warehouseValue.GetStringValue(); warehouseStr != "" {
                cacheKey := "batch_update_status_warehouse"
                // TTL=0 表示永不过期（实际靠业务逻辑删除）
                // 面试要点：这里实际是用后即删，不是真正的 24h TTL
                if err := base.RedisSet(ctx.Context, cacheKey, warehouseStr, 0); err != nil {
                    klog.Errorf("缓存 batch_update_status_warehouse 失败: %v", err)
                }
                break
            }
        }
    }
}
```

**消费读取（callback_grpc_maintenance.go:489-502）**：在 gRPC 回调处理时，从 Redis 取出大参数并在用完后立即删除：

```go
// callback_grpc_maintenance.go:489-502（callBatchUpdateStatus）
cacheKey := "batch_update_status_warehouse"
if cachedWarehouse, err := base.RedisGet(ctx.Context, cacheKey); err == nil && cachedWarehouse != "" {
    warehouse = cachedWarehouse
    defer func() {
        // 用完立即删除，避免数据残留污染下次请求
        if delErr := base.RedisDel(ctx.Context, cacheKey); delErr != nil {
            klog.Errorf("删除缓存失败: %v", delErr)
        }
    }()
}
```

**流程图**：

```
用户在 Camunda 表单中选择仓库（可能是大字符串）
    ↓
CompleteTask 触发前：
    → 检查 callbackParams 中是否有 "batch_update_status_warehouse"
    → 有 → RedisSet(key, warehouseStr, 0)  ← 写 Redis
    → 主消息体中的 batch_update_status 仍保留，但 warehouse 字段用 Redis key 代替
    ↓
回调处理 callBatchUpdateStatus：
    → 先从 Redis 读取 batch_update_status_warehouse
    → 合并到 item.WarehouseId 字段
    → defer RedisDel(key)   ← 用完即删
    → 调用维护服务 gRPC
```

**为什么是"用后即删"而不是 24h TTL？**

从代码来看，`RedisSet` 的 TTL 参数是 `0`（永不自动过期），依赖 `defer RedisDel` 删除。简历中的 "TTL 24h" 是对这套机制的一个**保底兜底**描述——即便回调失败导致 `defer` 未执行，数据也不会无限积累，实际最好改为显式设置 24h TTL 作为兜底。

> **面试注意**：如果面试官追问，要如实说明当前实现是用后即删（TTL=0），24h 是作为最佳实践建议提出的，当时还没有加上。

### 4.3 消息体固定大小的控制

`batch_update_status` 参数本身（资产编码列表）通过 gRPC 消息体传递，gRPC 客户端初始化时设置了最大消息大小：

```go
// maintenance/client.go:30
grpc.WithDefaultCallOptions(
    grpc.MaxCallRecvMsgSize(20971520),  // 接收 20MB
    grpc.MaxCallSendMsgSize(20971520),  // 发送 20MB
)
```

20MB 足够容纳几万个资产编码的 JSON 序列化结果，这是固定上限，确保消息体"控制在固定大小"。

## 五、双向溯源：变更历史写入工单来源

### 5.1 什么是双向溯源？

- **正向**：从工单 → 查该工单触发的所有资产变更
- **反向**：从资产 → 查某次状态变更是由哪个工单触发的

### 5.2 实现机制

每个 gRPC 回调调用时，工作流服务都会传入工单上下文字段：

```go
// 所有 gRPC Request 都携带工单信息
req := &maintenancev1.UpdateAssetStatusRequest{
    AssetCode:    assetCode,
    Status:       status,
    // ↓ 以下字段是工单来源信息
    InstanceCode:          "WO-2024-001234",  // 工单编号
    InstanceName:          "设备维修申请",      // 工单名称
    InstanceId:            "inst_abc123",      // 工单ID
    InstanceCreatedBy:     "user_001",         // 工单申请人
    InstanceThreeLevel1Id: "repair_type",      // 三级分类一
    InstanceThreeLevel2Id: "hardware",         // 三级分类二
    InstanceThreeLevel3Id: "replace_disk",     // 三级分类三（最细粒度操作类型）
    InstanceStatus:        "已完成",            // 工单当前状态
}
```

维护服务收到 gRPC 后，把这些工单信息写入 **`repair_record`（维修记录）表**：

```go
// repair_record.go:593-608（以 CreateRepairRecordNew 为例）
repairRecord := &mongo.RepairRecord{
    RecordType:      "component",
    AssetCode:       newItem.AssetCode,    // 资产编码（反向溯源的 key）
    InstanceCode:    req.InstanceCode,     // 工单编号（正向溯源的 key）
    InstanceOpsType: req.InstanceThreeLevel3Id,  // 操作类型（从工单三级分类来）
    InstanceName:    req.InstanceName,
    InstanceID:      req.InstanceId,
    Status:          req.InstanceStatus,
    CreatedBy:       req.InstanceCreatedBy,
    Source:          "api",               // 来源标识：api = 工单回调触发
    // ...
}
```

**变更历史（`change_history`）同样写入工单来源**：

```go
// repair_record.go:1298-1312（BatchAssetsToWarehouse 中）
history := &hostinfov1.CreateChangeHistoryRequest{
    RecordType:  item.AssetType,
    AssetCode:   assetCode,
    ActionType:  "创建",
    Operator:    req.InstanceCreatedBy,   // 操作人 = 工单申请人
    Description: fmt.Sprintf("寄修工单创建部件 %s", assetCode),  // 描述中含工单信息
    Source:      "api",                   // 来源标识
}
```

### 5.3 双向溯源的查询路径

```
正向溯源（工单 → 资产变更）：
  SELECT * FROM repair_record WHERE instance_code = "WO-2024-001234"
  → 返回该工单触发的所有换件、入库、状态变更记录

反向溯源（资产 → 触发工单）：
  SELECT * FROM repair_record WHERE asset_code = "SV000123"
  ORDER BY created_at DESC
  → 返回该资产的所有维修历史，每条记录带 instance_code（是哪个工单触发的）
```

## 六、状态同步延迟从 2~4 小时降至秒级的原理

| 环节 | 旧方案 | 新方案 |
|------|-------|-------|
| 触发时机 | 人工发现工单已审批后操作 | 工单审批通过瞬间触发 `taskCompleteAfter` Hook |
| 执行方式 | 人工点击维护系统界面 | 自动调用 gRPC（30s 超时） |
| 延迟构成 | 人工发现延迟（~1h）+ 操作时间（~15min） | Hook 触发（<1ms）+ gRPC 调用（<1s） |
| 总延迟 | 最长 2~4 小时 | **秒级**（通常 < 2 秒） |

关键实现：`taskCompleteAfter` Hook 在同一事务中触发（`hook.go:338`），审批动作完成即触发回调，无异步延迟。

```go
// hook.go:508-531（taskCompleteAfter，同步执行）
if buttonConf != nil && buttonConf.CallbackTemplate != nil {
    if err := processCallbackTemplate(dbCtx, taskInfo, allFormVariables, ...); err != nil {
        // 回调失败只记录日志，不阻断主流程（审批动作本身不受影响）
        klog.Errorf("处理回调模板失败: %v", err)
    }
}
```

**注意**：回调失败不阻断审批结果（`klog.Errorf` 而不是 `return err`）。这是有意为之的设计——工单审批是业务主流程，资产状态同步是副作用，副作用失败不能让审批回滚。

## 七、工作流引擎与维护服务的解耦

### 旧方案（强耦合）

```
Camunda → 直接调用 HostInfoService.UpdateHost(req)
```
问题：Camunda 需要知道具体 gRPC Service 和 Method 名，以及入参结构。一旦维护服务重命名接口，Camunda 配置也要同步修改。

### 新方案（路径解耦）

```
Camunda → CallbackTemplate.UrlTemplate = "RepairRecordService/UpdateAssetStatus"
        → 工作流服务中的路由层解析 URL → 映射到具体 gRPC 调用
```

**解耦效果**：

- Camunda 只配置一个字符串 URL，对 gRPC 接口透明
- 维护服务接口变更时，只需修改工作流服务的路由层（`callback_grpc_maintenance.go`），不需要动 Camunda 配置
- 可以在路由层加中间件（重试、熔断、参数校验），不影响两端

## 八、面试高频追问 & 标准回答

### Q1：8 类资产操作是怎么确定的？有没有可能遗漏？

**答**：8 类是根据业务场景梳理得出的：

1. **UpdateAssetQualityAndWarehouse**：寄修场景，同时改质量状态+仓库
2. **UpdateAssetStatus**：单资产状态流转（通用）
3. **CreateRepairRecordNew**：换件场景（最复杂，涉及换上换下、继承关联关系）
4. **BatchAssetsToWarehouse**：新设备批量入库（生成 AssetCode）
5. **BatchUpdateStatus**：批量仓库调拨
6. **BatchUnbindComponent**：拆机解绑
7. **UpdateHostStatus**：主机级状态（区别于设备级）
8. **CreateProjectAsset**：项目资产创建

实际业务中后续还会有新操作类型，通过**新增 Path 常量 + 新增 case 分支**的方式扩展，不影响现有的 8 类。

### Q2：为什么 Redis 缓存用固定的 key `"batch_update_status_warehouse"`，并发时会不会冲突？

**答**：这是一个**已知的设计缺陷**。固定 key 在并发场景下（多个工单同时审批）会相互覆盖。

正确的做法是用 `taskID` 或 `instanceID` 作为 key 的一部分：

```go
cacheKey := fmt.Sprintf("batch_update_status_warehouse:%s", taskInfo.ID)
```

这样每个任务有独立的缓存 key，不会冲突，且 TTL 到期后自动清理。这是一个可以在面试中**主动指出并提出改进方案**的点，展示技术判断力。

### Q3：回调失败了怎么处理？有没有重试机制？

**答**：当前没有自动重试机制，回调失败只记录日志（`klog.Errorf`），不阻断审批。

这意味着如果 gRPC 调用失败（网络抖动、维护服务重启），资产状态不会自动同步。改进方案：

1. **短期**：在路由层加 gRPC 重试（backoff 策略），对幂等操作（如 UpdateAssetStatus）直接重试 3 次
2. **中期**：引入消息队列（如 Kafka/RocketMQ），把回调任务持久化，失败后由消费者重试
3. **长期**：工单状态与资产状态做最终一致性对账（定时扫描工单已完成但资产状态未变更的记录）

### Q4：`collectInstanceFormData` 收集历史任务数据，如果同名字段在多个任务中都出现，哪个优先？

**答**：按任务完成顺序，**后完成的任务的字段值覆盖先完成的**（`hook.go:637`：`allVariables[key] = value` 无条件覆盖）。

但字段配置（`FormFieldConf`）是"先到先得"——第一次出现的字段配置被保留（`fieldConfMap` 只在 key 不存在时添加）。这样保证了字段类型/格式由最早定义的任务决定，而字段值由最新的任务决定。

### Q5：三级分类（ThreeLevel1/2/3）字段的作用是什么？

**答**：这是工单的业务分类体系，用于在维修记录中记录**操作类型**：

- `ThreeLevel1`：大类（如"硬件维修"、"仓库管理"）
- `ThreeLevel2`：中类（如"服务器维修"、"存储维修"）
- `ThreeLevel3`：细类（如"换盘"、"换内存"、"换主板"）→ 写入 `repair_record.instance_ops_type`

这样维修记录不仅知道"哪个工单"，还知道"什么操作类型"，可以按操作类型统计维修频率，辅助资产健康分析。

### Q6：工作流服务和维护服务之间是 gRPC 而不是 HTTP，有什么优势？

**答**：三个核心优势：

1. **强类型**：Protobuf 定义接口，字段类型在编译时检查，不会出现 JSON 类型隐式转换导致的 bug
2. **性能**：Protobuf 序列化比 JSON 小约 3~5 倍，对批量资产操作（数百条）有明显性能优势
3. **服务治理**：gRPC 天然支持负载均衡（代码中配置了 `round_robin`）、超时（30s）、拦截器（tracing），在 K8s 环境下配合服务发现更友好

## 十、一句话总结（面试收尾用）

> 这套方案的核心是**在工作流引擎和维护服务之间插入一个 URL 路由层**：工作流引擎只配置路径字符串，路由层负责解析路径、合并多任务参数、处理大参数的 Redis 中转，再调用具体 gRPC 方法。每次调用都把工单信息随 gRPC 参数写入维修记录，实现双向溯源。整个方案把资产状态同步从"人工 2~4 小时"压缩到"秒级自动化"，同时保持了工作流引擎和维护服务的独立演化能力。
