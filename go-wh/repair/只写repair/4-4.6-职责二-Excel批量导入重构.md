# Excel 批量导入重构深度面试分析

> **简历原句**：Excel 批量导入重构（性能提升 20 倍）：流式逐行解析将内存峰值从 100MB+ 降至 10MB 下；三层去重（文件内/任务内/历史库）覆盖所有重复场景；导入任务状态机支持断点续传与行级错误回溯；协程池异步化副作用操作，导入耗时从 5 分钟降至 15 秒。

---

## 一、背景：为什么要重构？

### 1.1 旧方案的问题

旧版导入是一个"全量读取 + 同步处理"的架构：

```
用户上传 Excel → 服务端读入全部内容（[]byte）→ 解析为 []Row（全部在内存）→
逐行校验 → 逐行写 DB → 逐行写变更历史 → 返回结果
```

这个方案带来三个致命问题：

| 问题 | 表现 | 根因 |
|------|------|------|
| **内存爆炸** | 1 万行 Excel，每行约 10KB，全量加载 ≈ 100MB+ | `xlsx.OpenBinary(fileContent)` 把整个文件解码成对象树，峰值是文件本身的 5~10 倍 |
| **耗时极长** | 1 万行约 5 分钟 | 逐行写 DB（N 次 round-trip）+ 同步写变更历史（N 次 gRPC 调用）完全串行 |
| **无法断点续传** | 处理到一半失败，只能重新上传 | 无任务状态持久化，进度完全丢失 |

### 1.2 重构目标

- 内存峰值 < 10MB
- 耗时 < 15 秒（1 万行）
- 支持断点续传：服务重启不丢失进度
- 行级错误回溯：哪一行出错清晰可查

## 二、难点一：流式解析 —— 内存峰值从 100MB+ 降至 10MB

### 2.1 旧方案：全量加载（一次性 OpenBinary）

```go
// host_info.go: 旧版解析方式（仍存在于部分导入场景）
func parseExcelFile(fileContent []byte, sheetName string) ([]*hostImportRow, error) {
    // 把整个 Excel 文件字节流解码成对象树，内存峰值极高
    wb, err := xlsx.OpenBinary(fileContent)
    // ...
    sheet.ForEachRow(func(row *xlsx.Row) error { ... }) // 全部行已在内存
}
```

**内存模型**：`fileContent`（文件字节）+ `wb`（xlsx 对象树）+ `[]Row`（解析结果）同时存在内存中，峰值是文件大小的 5~10 倍。

### 2.2 新方案：Channel + Goroutine 流式逐行

```go
// pkg/minio/minio.go:112-163
func (m *minioClient) StreamObjectLinesByExcelize(
    ctx context.Context, fileName string,
) (<-chan []string, <-chan error, error) {

    obj, err := m.GetObject(ctx, fileName)   // 从 MinIO 拿到 io.Reader，不下载全部
    rows := make(chan []string, 256)           // 带缓冲 channel，解耦生产/消费速度
    errs := make(chan error, 1)

    go func() {
        defer close(rows)
        defer close(errs)
        defer obj.Close()

        f, _ := excelize.OpenReader(obj)      // excelize 流式模式：不把整个文件装入内存
        defer f.Close()

        rs, _ := f.Rows(sheets[0])            // 返回行迭代器，按需解析
        defer rs.Close()

        for rs.Next() {                        // 逐行迭代，只有当前行在内存
            record, _ := rs.Columns()
            select {
            case rows <- record:               // 发送到 channel，消费者同步处理
            case <-ctx.Done():
                errs <- ctx.Err()
                return
            }
        }
    }()

    return rows, errs, nil
}
```

**内存模型对比**：

| 方案 | 内存构成 | 峰值估算（1万行） |
|------|---------|----------------|
| 旧：全量 OpenBinary | 文件字节 + xlsx 对象树 + 解析结果 | ~100MB |
| 新：流式逐行 | 当前行数据 + channel 缓冲（256行） | < 10MB |

**关键设计**：
- `make(chan []string, 256)`：缓冲 256 行，平衡生产（解析速度）和消费（DB写入速度），避免生产者频繁阻塞
- `ctx.Done()` 监听：任务取消时立刻停止解析，不产生额外内存占用
- `defer obj.Close()`：MinIO 对象流用完即释放，不持有整个文件

### 2.3 消费侧：流式循环处理

```go
// pkg/service/sync.go:376-475（CheckFile 内的 streamLoop）

// 使用 select 同时监听数据通道和错误通道
for linesCh != nil || errCh != nil {
    select {
    case line, ok := <-linesCh:
        if !ok { linesCh = nil; continue }  // 通道关闭，正常结束
        // 逐行转换、校验、打标（new/duplicate）
        assetInfo, errs := TransformLine(task, line, ...)
        CheckRules(ctx, task, assetInfo, ...)
        batch = append(batch, assetInfo.AssetInfo)

        if len(batch) >= BatchSize {         // 攒够 1000 行才写一次 DB
            UpdateTaskData(ctx, task, batch)
            batch = batch[:0]               // 复用 slice，不重新分配
        }
    case e, ok := <-errCh:
        if !ok { errCh = nil; continue }
        // 处理流式解析错误
    }
}
```

`BatchSize = 1000`（`sync.go:35`），每 1000 行触发一次 `InsertMany`，而不是每行一次 `InsertOne`。

## 三、难点二：导入任务状态机 —— 支持断点续传与行级错误回溯

### 3.1 状态机设计

```
用户上传文件
    ↓
CheckFile（文件校验阶段）
    ↓
pending ──────────────────→ running（解析中）
                                ↓
                       ┌───────┴───────┐
                       ↓               ↓
                     failed           wait（等待用户确认重复项）
                                       ↓
                              StartImport（用户触发执行）
                                       ↓
                                   running
                                       ↓
                                   completed
```

状态常量定义（`pkg/mongo/import_task.go:14-38`）：

```go
const (
    TaskStatusPending   = "pending"    // 初始状态：任务已创建，等待校验
    TaskStatusRunning   = "running"    // 运行中：正在解析或正在写入
    TaskStatusWait      = "wait"       // 等待：校验完成，等用户确认重复项
    TaskStatusCompleted = "completed"  // 完成
    TaskStatusFailed    = "failed"     // 失败：格式错误或系统错误

    TaskItemStatusPending    = "pending"    // 行项：待处理
    TaskItemStatusProcessing = "processing" // 行项：处理中（防止并发重复消费）
    TaskItemStatusCompleted  = "completed"  // 行项：已完成
    TaskItemStatusFailed     = "failed"     // 行项：失败（行级错误可追溯）

    TaskItemTypeNew       = "new"       // 新资产
    TaskItemTypeDuplicate = "duplicate" // 重复资产（需用户确认）
)
```

### 3.2 两张表的设计

**import_task 表**（任务级）：

```go

    FailedCount  int64              // 失败数

```

**import_task_item 表**（行级）：

```go
type ImportTaskItem struct {
    ID        primitive.ObjectID
    TaskID    primitive.ObjectID // 关联到 ImportTask
    AssetInfo *AssetInfo         // 完整的行数据快照

    Type      string             // new / duplicate

}
```

**设计意图**：把每一行的数据和状态持久化到 MongoDB，实现真正的断点续传。

### 3.3 断点续传的实现机制

```go
// pkg/service/import_worker_start_import_patterns.go:155-203
func (r *startImportRunner) runPhase(itemType string, handler batchHandler) error {
    for {
        // 每次从 DB 拉取一批 "pending" 状态的行（limit=1000）
        items, _ := r.svc.taskService.GetLimitPendingConfirmeds(
            opCtx, r.task.ID.Hex(),
            mongo.TaskItemStatusPending, itemType, r.limit,
            bson.D{{Key: "create_time", Value: -1}, ...},
        )

        if len(items) == 0 {
            return nil  // 所有行已处理完毕，退出
        }

        // 先把这批行标记为 "processing"，防止并发重复消费
        itemIds := extractIDs(items)
        r.svc.taskService.UpdateImportTaskItemStatus(opCtx, itemIds, mongo.TaskItemStatusProcessing)

        // 执行实际的写入逻辑
        handler(items)

        // 更新任务进度到 DB
        r.task.ImportCount = r.imported
        r.task.UpdateTime = time.Now()
        r.svc.taskService.Update(r.bg, r.task)
    }
}
```

**断点续传的逻辑**：
1. 每次从 DB 查 `status=pending` 的行项，处理完标记为 `completed`
2. 如果服务在处理过程中崩溃，重启后再次执行 `runPhase`
3. `processing` 状态的行会在重启后被重新处理（`processing` 状态不是终态，会被再次查询到）
4. `completed` 行不会被重复处理（过滤条件排除了 `completed`）

**面试重点**：这本质上是一个**幂等消费模式**——把"已消费"这个事实持久化到每一行，而不是依赖内存状态。

### 3.4 行级错误回溯

当某一行写入失败时：


## 四、难点三：三层去重 —— 覆盖所有重复场景

### 4.1 为什么需要三层去重？

单一去重的局限：
- **只查历史库**：无法发现同一文件内自身重复（同一批数据导入两次同样的 SN）
- **只查文件内**：无法发现跨任务的历史重复（两次上传的文件里有相同数据）
- **只查当前任务**：无法断点续传（重新导入会产生重复）

### 4.2 三层去重的架构

```
第一层：文件内去重（CheckFile 阶段，内存 HashMap）
        ↓ 发现重复 → 整行标记为 duplicate，不报错
第二层：任务内去重（GetByFactorySNBatch，查 import_task_item 表）
        ↓ 当前任务已有该 SN → 跳过
第三层：历史库去重（GetByFactorySNBatch，查 asset_info 表）
        ↓ 历史库存在 → 标记为 duplicate type，等用户确认是否覆盖
```

#### 第一层：文件内去重（内存 HashMap）

在 `CheckFile` 阶段，`initCache` 预加载所有必要的元数据查找表到内存，`TransformLine` 逐行校验时用 `errMap`（HashMap）对每行错误去重（同一错误不重复报告）：

```go
// sync.go:843-1092（TransformLine）
errs := make([]string, 0)
errMap := make(map[string]struct{}, 0)  // 文件内行级错误去重
// ...
// 发现错误时加入 errMap 而不是直接 append，
// 避免同一行多个字段错误时重复报告
makeErr(errInfo, errMap)
```

文件内 SN 重复：通过 `cache` 中已见过的 SN set 来检测（`CheckRules` 阶段）。

#### 第二层：任务内去重（查 import_task_item）

同一个任务已经解析过该 SN（断点续传场景下），通过批量查询任务项来去重：

```go
// pkg/mongo/import_task.go:367-392
func (s *importTaskStorage) GetByFactorySNBatch(
    ctx context.Context, taskId string, factorySNs []string,
) ([]*ImportTaskItem, error) {
    filter := bson.M{
        "task_id":               objectId,
        "asset_info.factory_sn": bson.M{"$in": factorySNs}, // 批量查
    }
    // ...
}
```

#### 第三层：历史库去重（查 asset_info，标记为 duplicate）

```go
// import_worker_start_import_patterns.go:903-921（processDuplicateBatch）
// 批量查询历史库中是否已存在这批 FactorySN
exists, _ := r.assetStorage.GetByFactorySNBatch(opCtx, factorySNs)
existMap := make(map[string]*mongo.AssetInfo, len(exists))
for _, a := range exists {
    existMap[a.FactorySN] = a  // O(1) 查找
}

for _, it := range items {
    old := existMap[it.AssetInfo.FactorySN]
    if old == nil {
        r.failed++
        // 标记为 failed（历史库里找不到该 SN，说明数据不一致）
        continue
    }
    // 历史库存在 → 构建差异更新（Strategy Pattern）
    specificUpdate, changes, _, _ := ctx.strategy.Build(old, asset, r.task, ctx, r.userInfo)
}
```

### 4.3 重复资产的处理策略：Strategy Pattern

针对不同场景（机器档案/仓库档案 × 主机/部件），使用**策略模式**构建差异更新：

```go
// import_worker_start_import_patterns.go:503-517
func newDuplicateUpdateStrategy(task *mongo.ImportTask) duplicateUpdateStrategy {
    key := task.ImportType + ":" + task.AssetType
    switch key {
    case MachineHostColumn:      return &machineHostStrategy{}
    case MachineComponentColumn: return &machineComponentStrategy{}
    case WarehouseHostColumn:    return &warehouseHostStrategy{}
    case WarehouseComponentColumn: return &warehouseComponentStrategy{}
    }
}
```

每个策略的 `Build` 方法只更新**发生变化的字段**（字段级 diff），并生成对应的变更历史记录：

```go
// machineHostStrategy.Build（简化示意）
func (s *machineHostStrategy) Build(old, asset *AssetInfo, ...) (bson.M, []*FieldChange, ...) {
    update := bson.M{}
    if asset.Project != old.Project {
        update["project"] = asset.Project
        changes = append(changes, FieldChange{Before: old.Project, After: asset.Project})
    }
    if asset.FakeSN != old.FakeSN {
        update["fake_sn"] = asset.FakeSN
        // ...
    }
    // 只更新真正变了的字段，不做全量覆盖
    return update, changes, ...
}
```

**好处**：
1. 避免无意义的写入（字段没变不写 DB）
2. 变更历史精确记录 before/after，满足审计需求

## 五、难点四：协程池异步化副作用 —— 耗时从 5 分钟降至 15 秒

### 5.1 什么是"副作用操作"？

导入的**主链路**只有：解析行 → 校验 → 写 asset_info → 更新行状态

但每次写入后还需要触发一系列**副作用**：
- 写变更历史（`ChangeHistory`，gRPC 调用）
- 更新 SN 绑定关系（`FakeSnBinding`、`AssetCodeBinding`）
- 计算资产完整度（`CalCompleteness`，CPU 密集）
- 更新父子部件关联关系（`updateParentInfo`）

这些副作用如果同步执行：每条记录增加约 4 次 gRPC 调用 × 5ms = 20ms，1 万行 = 200 秒额外耗时。

### 5.2 协程池设计（sideEffectDispatcher）

```go
// import_worker_start_import_patterns.go:19-45

type sideEffectDispatcher struct {
    sem chan struct{}   // 信号量：控制并发上限
    wg  sync.WaitGroup // 等待所有任务完成
}

// newSideEffectDispatcher 创建一个并发度为 maxConcurrent 的协程池
func newSideEffectDispatcher(maxConcurrent int) *sideEffectDispatcher {
    if maxConcurrent <= 0 {
        maxConcurrent = 8  // 默认 8 个并发
    }
    return &sideEffectDispatcher{sem: make(chan struct{}, maxConcurrent)}
}

// Submit 提交一个任务到协程池
func (d *sideEffectDispatcher) Submit(fn func()) {
    d.wg.Add(1)
    d.sem <- struct{}{}    // 获取信号量（满了就阻塞，实现背压）
    go func() {
        defer func() {
            <-d.sem        // 释放信号量
            d.wg.Done()
        }()
        fn()               // 执行副作用
    }()
}

// Wait 等待所有已提交任务完成（主流程结束时调用）
func (d *sideEffectDispatcher) Wait() {
    d.wg.Wait()
}
```

**这是一个标准的 Semaphore 协程池**：
- `sem chan struct{}` 容量即最大并发数（8）
- `Send` 阻塞→获取槽位，`Recv` 释放槽位
- `sync.WaitGroup` 确保所有任务在 `dispatcher.Wait()` 之前完成

### 5.3 副作用异步化的使用方式

```go
// import_worker_start_import_patterns.go:273-317（processNewBatch）

// 主链路：同步批量写入资产（关键路径，必须等结果）
r.assetStorage.BulkCreate(createCtx, toInsert)
r.imported += int64(len(toInsert))

// 副作用1：异步写变更历史 + SN 绑定（不阻塞主链路）
inserted := toInsert
r.dispatcher.Submit(func() {
    for _, asset := range inserted {
        r.changeHistory.CreateChangeHistory(context.Background(), &CreateChangeHistoryRequest{...})
        r.fakeSnBinding.Create(context.Background(), ...)
        r.assetCodeBinding.Create(context.Background(), ...)
    }
})

// 副作用2：异步更新父子关联 + 计算完整度（CPU 密集，不阻塞主链路）
r.dispatcher.Submit(func() {
    r.updateParentInfo(inserted)           // 更新父级关联
    r.completeness.CalCompletenessByAssetCodes(r.bg, ...)  // 完整度计算
})

// 主链路继续处理下一批...
```

### 5.4 性能提升的量化分析

| 操作 | 旧版（同步） | 新版（异步化后） |
|------|------------|--------------|
| 主链路：写 asset_info（1万行） | N × 1次 InsertOne ≈ 10000ms | BulkCreate 分批 ≈ 500ms |
| 副作用：写变更历史（1万条） | 同步 10000 × 5ms = 50000ms | 异步，不占主链路时间 |
| 副作用：SN 绑定（1万条） | 同步 10000 × 3ms = 30000ms | 异步，不占主链路时间 |
| **合计** | **~300秒（5分钟）** | **~15秒** |

**关键原理**：`InsertMany`/`BulkWrite` 将 N 次 round-trip 压缩为 ceil(N/1000) 次；副作用从串行阻塞改为后台并发，主链路只等最慢的那个 `BulkCreate`，不等所有副作用完成。

### 5.5 主流程结束时等待副作用

```go
// import_worker_start_import_patterns.go:134（Run 方法末尾）
r.dispatcher.Wait()  // 确保所有副作用完成才标记 task = completed

r.task.Status = mongo.TaskStatusCompleted
r.svc.taskService.Update(r.bg, r.task)
```

这保证了：`task.Status = completed` 时，所有变更历史都已经写入，不会出现"任务完成但历史记录还没写"的中间状态。

## 六、整体架构串联：一次完整导入的生命周期

```
用户上传 Excel 到 MinIO
         ↓
CheckFile（sync.go 的 goroutine，通过 taskChannel 接收）
  1. MinIO 流式读取 → StreamObjectLinesByExcelize → Channel
  2. 逐行读取 Channel → TransformLine（字段映射）→ CheckRules（业务校验）
  3. 攒够 1000 行 → UpdateTaskData → InsertMany 到 import_task_item
     - 文件内去重：相同 SN 标记为 duplicate
     - 历史库去重：查 asset_info 已存在的标为 duplicate
  4. 任务状态：running → wait（有重复）/ running → completed（无重复）

用户确认重复项后调用 StartImport
         ↓
StartImport（goroutine 异步执行）
  1. 处理 new 类型行（runPhase("new", processNewBatch)）：
     - 批量生成 AssetCode
     - BulkCreate 写入 asset_info
     - 异步提交：写变更历史、SN绑定、完整度计算
  2. 处理 duplicate 类型行（runPhase("duplicate", processDuplicateBatch)）：
     - 批量查 asset_info 获取旧数据
     - Strategy.Build 计算字段 diff
     - BulkWrite 批量更新
     - 异步提交：写变更历史
  3. dispatcher.Wait() 等待所有副作用完成
  4. 任务状态: running → completed
  5. 清理 import_task_item 中的 completed 行（节省存储）
```

## 七、面试高频追问 & 标准回答

### Q1：Channel 缓冲大小为什么是 256，而不是 1 或 1000？

**答**：这是生产者（流式解析，CPU 密集）和消费者（DB 写入，IO 密集）之间的流量平衡点。

- 缓冲太小（如 1）：生产者每解析一行就要等消费者处理完，解析速度受 DB 延迟拖累，无法充分利用 CPU
- 缓冲太大（如 1000）：内存中积压大量未处理行，内存峰值升高，与流式目标矛盾
- 256 行（约 2~3MB）：在 10MB 内存目标下，给生产者足够的"预读"空间，同时不浪费内存

实际上 256 是经验值，具体数值可以根据行数据大小调整。

### Q2：如果导入过程中服务崩溃，断点续传是怎么工作的？

**答**：两个关键设计：

1. **行级状态持久化**：每行数据存为一条 `import_task_item` 记录，`status` 字段标识该行的处理状态（`pending/processing/completed/failed`）
2. **`runPhase` 的幂等循环**：服务重启后，重新调用 `StartImport`，`runPhase` 从 DB 查询 `status=pending` 的行继续处理

特殊情况：崩溃时处于 `processing` 状态的行。这些行没有被标记为 `completed`，下次 `runPhase` 不会查到它们（只查 `pending`）。这里存在一个小缺陷——崩溃时正在处理的那批行（最多 1000 行）会变成"僵死的 processing"。改进方案是加一个定时修复任务：把超时（如 5 分钟）仍处于 `processing` 的行重置为 `pending`。

### Q3：协程池最大并发 8 是怎么确定的？有什么依据？

**答**：基于两个约束：

1. **下游 gRPC 服务的处理能力**：变更历史服务、完整度服务等下游通常在 Kubernetes 中有固定的 replica 数量，并发太高会导致下游过载
2. **本服务的内存预算**：每个 goroutine 约 2~8KB 栈空间，8 个并发副作用 goroutine 约 64KB，在 Pod 资源限制下可接受

8 是初始设定值，可以通过监控下游 P99 延迟来动态调整。更优雅的方案是暴露为配置参数，通过 ConfigMap 热更新。

### Q4：`processDuplicateBatch` 里的 BulkWrite 是有序还是无序的？

**答**：使用的是默认有序模式（`SetOrdered(true)`）。对于重复资产更新，每条更新都是按 `asset_code` 过滤的独立操作，互不依赖，可以改成无序（`SetOrdered(false)`）以提升并发执行效率。当前是有序模式，保守但正确，后续可以作为性能优化点。

### Q5：三层去重中，"文件内去重"是怎么实现的，有没有可能漏掉？

**答**：文件内去重通过 `CheckFile` 阶段逐行扫描实现。在 `CheckRules` 阶段，每处理一行，会把该行的 `FactorySN` 加入一个内存 Set（`seenSNs`），如果后续行的 `FactorySN` 已在 Set 中，则标记为 `duplicate`。

理论上不会漏掉，但有一个边界：如果 `FactorySN` 为空，则该行不参与 SN 去重（只做其他字段校验）。对于允许为空的场景（如客户档案里有设备尚无 SN），这是合理的豁免。

### Q6：initCache 预加载了大量字典数据，如果字典数据量很大怎么办？

**答**：当前 `initCache` 把项目、仓库、字典等元数据全量加载到内存（`map[string]string`），对于当前规模（几百个项目、几十个字典条目）完全没问题，内存约 1~2MB。

如果数据规模扩大到万级：
1. 改为按需查询：只加载导入文件中出现过的项目/仓库名
2. 引入本地 LRU 缓存：避免每次导入都重新拉取
3. 增量更新：监听字典变更事件，异步刷新缓存
