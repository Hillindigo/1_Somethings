# 父子部件关联机制分析

> 以 GPU 服务器（主机）与单块 GPU（部件）的关联为例，从数据模型、Excel 导入、运行时三个维度完整分析。

---

## 一、数据模型：`asset_info` 集合的关联字段设计

部件的父子关系全部存储在 `asset_info` 集合中，每条部件文档都带有以下关联字段：

```
asset_info（部件文档，如一块 GPU）
├── asset_code          = "COMP-000123"        // 部件自身编号
├── factory_sn          = "GPU-SN-ABC"         // GPU 原厂SN
├── type                = "GPU"
│
├── related_parent_factory_sn = "HOST-SN-XYZ"  // 直接父级原厂SN（可能是主板，也可能就是服务器）
├── related_parent_fake_sn    = "HOST-FAKE-001" // 直接父级客户SN
├── related_parent_asset_code = "HOST-000010"   // 直接父级平台编号
├── related_parent_type       = "SERVER"        // 直接父级类型
│
├── related_factory_sn  = "HOST-SN-XYZ"        // 最顶层主机原厂SN（穿透多层到根节点）
├── related_fake_sn     = "HOST-FAKE-001"       // 最顶层主机客户SN
└── related_asset_code  = "HOST-000010"         // 最顶层主机平台编号
```

**两组字段的区别**：

| 字段组 | 含义 | 场景 |
|--------|------|------|
| `related_parent_*` | **直接父级**（上一层） | GPU → 主板，或 GPU 直接挂在服务器上 |
| `related_*`（无 parent）| **最顶层主机**（穿透所有中间层） | GPU → 主板 → 服务器，`related_*` 直接指向服务器 |

> **关键点**：`related_parent_*` 和 `related_*` 可以指向同一台主机（当层级只有两层：部件直接挂主机时），也可以不同（当存在三层：部件 → 中间层部件 → 主机时）。

## 二、Excel 导入时的父子关联写入流程

### 2.1 Excel 列名 → 字段映射

系统支持两种导入场景，使用不同的"父级标识符"：

**机器档案导入（`machine:component`）**：
```
Excel 列名            →  内部字段名
"关联主机SN"           →  related_fake_sn           （最顶层主机客户SN）
"关联父级SN"           →  related_parent_fake_sn    （直接父级客户SN）
```

**仓库导入（`warehouse:component`）**：
```
Excel 列名            →  内部字段名
"关联主机原厂SN"        →  related_factory_sn        （最顶层主机原厂SN）
"关联父级原厂SN"        →  related_parent_factory_sn （直接父级原厂SN）
```

> **区别原因**：机器档案场景有 FakeSN（客户SN），仓库场景以 FactorySN（原厂SN）为唯一标识。

### 2.2 以 GPU 导入为例（仓库导入场景）

假设 GPU 服务器原厂SN 为 `SERVER-001`，一块 GPU 原厂SN 为 `GPU-ABC`：

**Excel 填写**：

| 原厂SN  | 类型 | 关联主机原厂SN | 关联父级原厂SN |
|---------|------|----------------|----------------|
| GPU-ABC | GPU  | SERVER-001     | SERVER-001     |

> 当 GPU 直接挂在服务器上（没有中间主板层），两列填相同值。若 GPU 挂在主板上，"关联父级原厂SN" 填主板SN，"关联主机原厂SN" 填服务器SN。

### 2.3 校验阶段（CheckData）

`sync.go` 的 `CheckData` 函数在解析阶段就对父级SN做存在性验证：

```go
// 收集所有父级SN，批量查询数据库验证
if len(relatedParentFactorySNsMap) > 0 {
    factorySnAssetInfos, err := componentStorage.FindByAssetCodesOrSns(
        ctx, []string{}, []string{}, relatedParentFactorySNs)
    // ...
}

// 逐行校验：父级SN 必须在数据库中已存在
if row.RelatedParentFactorySN != "" {
    if _, exist := existSnMap[row.RelatedParentFactorySN]; !exist {
        makeErr(fmt.Sprintf("关联父级原厂SN:%s无效，请检查关联数据", row.RelatedParentFactorySN), errMap)
        continue
    }
}
```

**结论**：Excel 中填的父级SN，系统会在导入校验阶段就去数据库查这台主机/父级部件是否真实存在，不存在直接报错拒绝导入，保证关联数据有效性。

## 三、导入执行阶段：父级信息回填

校验通过的数据被暂存为 `ImportTaskItem`，进入执行阶段后，父子关联信息通过以下两步完成写入：

### 步骤一：BulkCreate 写入部件基础数据

`processNewBatch` 函数批量插入 `asset_info` 文档。此时文档中的 `related_parent_factory_sn` 字段已经有值（从 Excel 读入），但 `related_parent_asset_code`、`related_parent_fake_sn` 等需要关联查询才能补全的字段还是空的。

```go
// import_worker_start_import_patterns.go
err = r.assetStorage.BulkCreate(createCtx, toInsert)  // 先写入基础字段
```

### 步骤二：updateParentInfo 异步回填完整关联信息

BulkCreate 完成后，通过 `sideEffectDispatcher` 异步提交一个任务回填父级完整信息：

```go
r.dispatcher.Submit(func() {
    // 先回填父级关联信息
    if err := r.updateParentInfo(inserted); err != nil {
        klog.Errorf("[StartImport] update parent info failed: %v", err)
        return
    }
    // 父级信息确定后，再计算完整度
    r.completeness.CalCompletenessByAssetCodes(...)
})
```

`updateParentInfo` 的核心逻辑（`import_worker_start_import_patterns.go` 第 333-424 行）：

```go
func (r *startImportRunner) updateParentInfo(inserted []*mongo.AssetInfo) error {
    // 1. 以父级SN为 key，批量预加载所有父级资产信息
    relatedAssetInfos := r.preloadRelatedAssetInfos(items)

    assetBatchOps := make([]mongodriver.WriteModel, 0, len(inserted))

    for _, asset := range inserted {
        // 2. 取出父级Key（仓库场景用 RelatedParentFactorySN）
        parentKey = asset.RelatedParentFactorySN

        parentAssetInfo := relatedAssetInfos[parentKey]
        parent := parentAssetInfo.Parent

        // 3. 用父级资产信息补全所有关联字段
        updateFields := bson.M{
            "related_parent_asset_code": parent.AssetCode,   // 补全 asset_code
            "related_parent_factory_sn": parent.FactorySN,
            "related_parent_fake_sn":    parent.FakeSN,
            "related_parent_type":       parent.Type,
        }

        // 4. 通过 RelatedChain 找最顶层主机（穿透中间层）
        if len(parentAssetInfo.RelatedChain) > 0 {
            hostInfo := parentAssetInfo.RelatedChain[len(parentAssetInfo.RelatedChain)-1]
            updateFields["related_fake_sn"]    = hostInfo.FakeSN
            updateFields["related_factory_sn"] = hostInfo.FactorySN
            updateFields["related_asset_code"] = hostInfo.AssetCode
        }

        // 5. 累积为 BulkWrite 操作
        assetBatchOps = append(assetBatchOps, mongodriver.NewUpdateOneModel().
            SetFilter(bson.M{"asset_code": asset.AssetCode}).
            SetUpdate(bson.M{"$set": updateFields}))
    }

    // 6. 一次 BulkWrite 批量写入所有部件的父级关联信息
    r.svc.assetInfoStorage.BulkWrite(bulkCtx, assetBatchOps)
}
```

## 四、RelatedChain：多层级穿透查找最顶层主机

`preloadRelatedAssetInfos` 调用 `GetRelatedInfoByAssetCode`，返回 `AssetParentInfo` 结构：

```go
type AssetParentInfo struct {
    Parent       *AssetDetail   // 直接父级（如主板）
    RelatedChain []*AssetDetail // 从直接父级到最顶层主机的完整链路
}
```

**链路示意**：

```
GPU（当前部件）
  └── RelatedChain[0] = 主板（直接父级）
  └── RelatedChain[1] = 服务器（最顶层主机）  ← 取最后一个
```

对于 GPU 直接挂在服务器上的场景：

```
GPU（当前部件）
  └── RelatedChain[0] = 服务器（直接父级 = 最顶层主机）
```

此时 `related_parent_*` 和 `related_*` 指向同一台服务器。

## 五、完整流程总结（以 GPU 导入为例）

```
① Excel 填写
   GPU-ABC | GPU | 关联主机原厂SN: SERVER-001 | 关联父级原厂SN: SERVER-001

② CheckData 校验
   查询 asset_info：SERVER-001 是否存在 → 存在则通过，否则报错拒绝

③ 解析写入 ImportTaskItem
   AssetInfo.RelatedParentFactorySN = "SERVER-001"
   AssetInfo.RelatedFactorySN       = "SERVER-001"  （主机）

④ BulkCreate（同步）
   写入 asset_info，基础字段入库
   related_parent_factory_sn = "SERVER-001"  ← 已有
   related_parent_asset_code = ""             ← 待补全

⑤ updateParentInfo（异步，sideEffectDispatcher）
   查询 SERVER-001 的完整信息 → 得到 AssetCode / FakeSN 等
   BulkWrite 回填：
     related_parent_asset_code = "HOST-000010"
     related_parent_fake_sn    = "HOST-FAKE-001"
     related_parent_type       = "SERVER"
     related_asset_code        = "HOST-000010"  （最顶层主机，同上）
     related_fake_sn           = "HOST-FAKE-001"

⑥ 最终 GPU 文档状态
   related_parent_* → 服务器（直接父级）
   related_*        → 服务器（最顶层主机）
   两者相同，因为只有两层关系
```

## 六、关键设计结论

| 设计点 | 说明 |
|--------|------|
| **双 SN 标识** | `factory_sn`（原厂）和 `fake_sn`（客户）并存，仓库场景用前者，机器场景用后者 |
| **两组关联字段** | `related_parent_*` 存直接父级，`related_*` 存最顶层主机，支持多层嵌套（部件→主板→服务器） |
| **先写后补全** | BulkCreate 写基础字段（含父级SN），再异步 BulkWrite 回填完整关联信息，保证主流程性能 |
| **校验前置** | CheckData 阶段就验证父级SN是否存在于数据库，失败直接 Fail-Fast，避免脏数据 |
| **RelatedChain 穿透** | 不管嵌套多少层，通过链路遍历取最顶层主机，`related_*` 始终指向根节点（服务器），方便按主机维度聚合查询 |

## 七、导入失败时的数据安全与一致性保障

### 7.1 整体设计思路

系统**不依赖数据库事务**（MongoDB 跨文档操作无原生事务保证，且性能代价高），而是通过"状态机 + 幂等操作 + 断点续传"三层机制在应用层保证一致性。

---

### 7.2 两级状态机：故障边界清晰

系统维护两层独立的状态机：

**第一层：ImportTask（任务级）**

```
pending  →  running  →  completed
                    ↘  failed
```

| 状态 | 含义 | 写入时机 |
|------|------|----------|
| `pending` | 等待用户确认导入 | CheckFile 校验通过后 |
| `running` | 正在执行 | StartImport 被调用时 |
| `completed` | 全部处理完 | `Run()` 正常结束 |
| `failed` | 任务崩溃 | `Run()` 内 `recover()` 捕获 panic 时 |

**第二层：ImportTaskItem（行级）**

```
pending  →  processing  →  completed
                       ↘  failed
```

| 状态 | 含义 |
|------|------|
| `pending` | 待处理，可被 `runPhase` 拉取 |
| `processing` | 已被本批次拉取，标记"处理中" |
| `completed` | 写入 `asset_info` 成功 |
| `failed` | 当前行写入失败，记录 `error_msg` |

> **关键设计**：`processing` 状态是"中间状态哨兵"——服务宕机重启后，滞留在 `processing` 的 Item 是"僵尸"，说明上次处理到一半宕机了，可以被识别并重新回滚到 `pending` 继续处理。（当前代码中 `runPhase` 每次只查 `pending` 状态，重启后 `processing` 的 Item 不会被重新拉取——这是一个现存局限，见 7.6 节分析。）

---

### 7.3 panic 捕获：防止任务永久卡在 running

`Run()` 函数顶层有 `defer + recover`，保证任意 panic 都能被捕获并将任务状态置为 `failed`：

```go
// import_worker_start_import_patterns.go
func (r *startImportRunner) Run() {
    defer func() {
        if rec := recover(); rec != nil {
            r.task.Status = mongo.TaskStatusFailed   // ← 置为 failed
            r.task.UpdateTime = time.Now()
            klog.Errorf("[StartImport] panic: %v", rec)
            _ = r.svc.taskService.Update(r.bg, r.task) // ← 写回数据库
        }
    }()
    // ... 正常业务逻辑
}
```

**效果**：任务不会永久停留在 `running` 状态让用户看不到结果；`failed` 状态在前端可展示，用户可重新上传文件或检查错误原因。

---

### 7.4 连续失败熔断：防止死循环打爆数据库

`runPhase` 中有**连续失败计数器**，避免更新 Item 状态时持续报错造成无限循环：

```go
// import_worker_start_import_patterns.go
const maxConsecutiveFailures = 3
consecutiveFailures := 0

for {
    // 拉取 pending 的 items...

    // 将 items 状态更新为 processing
    err = r.svc.taskService.UpdateImportTaskItemStatus(opCtx, itemIds, mongo.TaskItemStatusProcessing)
    if err != nil {
        r.failed += int64(len(itemIds))
        consecutiveFailures++
        if consecutiveFailures >= maxConsecutiveFailures {
            // 连续 3 次失败，主动终止当前 phase，不再死循环
            return fmt.Errorf("too many consecutive failures updating item status")
        }
        continue  // 重试一次
    }
    consecutiveFailures = 0  // 成功则重置计数器
}
```

**效果**：数据库抖动或网络异常时，最多重试 3 次就终止该 phase，保护数据库不被拖垮。

---

### 7.5 BulkCreate 失败：整批回退，不产生半写数据

新资产的写入使用 `BulkCreate`（内部是 MongoDB `InsertMany`），如果整个批次写入失败：

```go
err = r.assetStorage.BulkCreate(createCtx, toInsert)
if err != nil {
    klog.Errorf("[StartImport] bulk create assets failed: %v, count=%d", err, len(toInsert))
    r.failed += int64(len(toInsert))
    return err   // ← 直接返回，不更新 Item 状态为 completed
}
```

- **失败时**：Item 状态仍停在 `processing`（不会被改为 `completed`），`failed` 计数器递增
- **这批 Item** 不会被 `DeleteImportTaskItem` 清理（只清理 `completed` 的），保留在数据库中可供排查

> **注意**：`InsertMany` 使用 `SetOrdered(false)`（见 `mongo/import_task.go` 第 324 行），允许部分成功——这意味着批次内可能有些文档插入成功、有些失败。这是一个**潜在的不一致点**，但实际上 asset 的 `factory_sn` 有唯一约束，重复插入会报 duplicate key error，不会产生真正的脏数据。

---

### 7.6 副作用失败：不影响主数据，独立容错

父级关联信息回填（`updateParentInfo`）和变更历史写入均通过 `sideEffectDispatcher` 异步执行，与主流程（BulkCreate）完全解耦：

```go
r.dispatcher.Submit(func() {
    // 副作用1：回填父级关联信息
    if err := r.updateParentInfo(inserted); err != nil {
        klog.Errorf("[StartImport] update parent info failed: %v", err)
        return  // ← 仅记录日志，不影响任务状态
    }
    // 副作用2：计算完整度（依赖父级信息，顺序执行）
    r.completeness.CalCompletenessByAssetCodes(...)
})

r.dispatcher.Submit(func() {
    // 副作用3：写入变更历史、FakeSN 绑定
    r.changeHistory.CreateChangeHistory(...)
    r.fakeSnBinding.Create(...)
})
```

**容错边界**：
- `updateParentInfo` 失败 → GPU 的 `related_parent_*` 字段为空，但 GPU 文档本身已成功写入 `asset_info`，主数据不丢失；前端展示时关联主机为空，可以后续手动修正
- 变更历史写入失败 → 审计日志缺失，但不影响核心资产数据

---

### 7.7 已完成条目的清理：防止重复处理

任务完成后，`Run()` 会清理所有 `completed` 状态的 `ImportTaskItem`：

```go
// DeleteImportTaskItem 只删 completed 状态的
_, err = s.itemCollection.DeleteMany(ctx, bson.M{
    "task_id": objectId,
    "status":  "completed",  // ← 仅删已完成，失败的保留
})
```

**效果**：
- `completed` 的 Item 删除，节省存储，防止重复处理
- `failed` 的 Item 保留，可用于问题排查和后续重试

---

### 7.8 现存局限与风险点（诚实评估）

| 风险点 | 描述 | 影响 |
|--------|------|------|
| **`processing` 状态无自动回滚** | 服务宕机后，滞留在 `processing` 状态的 Item 不会被自动回滚为 `pending`，当前代码 `runPhase` 只查 `pending`，重启后这批 Item 会被永久遗忘 | 少量数据漏导入，用户需要手动重新上传文件 |
| **InsertMany SetOrdered(false) 部分成功** | 批次内部分 Item 插入成功、部分失败时，成功的那些 Item 状态仍是 `processing`（不是 `completed`），计数不准确 | `import_count` 统计偏低，但数据本身不脏 |
| **副作用无重试机制** | `updateParentInfo` 失败后仅打日志，无重试，也无告警 | 父子关联信息可能不完整，需人工修正 |
| **无分布式锁** | 同一个 TaskID 理论上可以被多个进程并发执行（如重复调用 StartImport） | 理论上存在重复写入风险，当前靠业务层拦截（`status=running` 时不允许再次启动） |

---

### 7.9 一致性保障总结

```
校验阶段（CheckFile/CheckData）
  └── Fail-Fast：父级SN不存在 / 表头不符 → 直接拒绝，0条数据进库

执行阶段（Run）
  ├── panic recover → 任务置 failed，不永久卡在 running
  ├── 连续失败熔断（3次）→ 终止 phase，不死循环
  ├── BulkCreate 失败 → 整批 Item 保持 processing，failed 计数递增
  ├── 副作用失败 → 仅影响辅助数据（关联字段/变更历史），主数据安全
  └── 完成后只清理 completed → failed 的 Item 保留供排查

重启恢复（局限）
  └── processing 状态 Item 无自动回滚（当前代码局限）
      建议：服务启动时扫描超时的 processing Item 并回滚为 pending
```

## 八、架构改进方案（大厂后端视角）

> 针对 7.8 节的四个局限，每个问题只保留最优方案。原则：**最小改动、向后兼容、可灰度验证**。

---

### 8.1 修复：processing 僵尸 Item 无法自动恢复

#### 面试怎么讲

> 原来系统在服务宕机后，卡在 `processing` 中间状态的 Item 会永久积压，因为任务重启后只拉 `pending` 的数据。我的方案是在服务启动阶段加一个"僵尸扫描"步骤：扫描所有还在 `running` 的任务，把其中 `update_time` 超过 10 分钟的 `processing` Item 批量重置回 `pending`，这样任务重启后就能自动衔接上次断点，用户完全无感知，实现断点续传。选 10 分钟这个阈值是因为正常一批 1000 条数据处理时间不超过 60 秒，10 倍余量足够安全，不会误伤正在处理中的 Item。

#### 架构设计思路

这个问题的本质是**中间状态没有超时兜底**。`processing` 本来设计为"正在处理中的哨兵状态"，目的是让系统知道哪些 Item 已经被拿走处理，但它缺少一个配套的"超时自动回收"机制。业界通用的解法是 **Dead Letter Recovery（僵尸恢复）**模式：在服务启动时做一次全量扫描，把所有"看起来卡住了"的中间状态记录回滚到初始状态，让系统重新处理。这个模式在消息队列（如 RocketMQ 的死信队列）、分布式任务调度（如 XXL-Job 的失联 executor 恢复）中都有广泛应用。

整体架构改动极小，**只在两处加代码**：

```
服务启动流程
  └── main.go 初始化
       ├── 原有：连接 MongoDB / Redis / 注册 gRPC handler
       └── 新增：RecoverZombieItems()  ← 启动钩子，一次性执行

MongoDB 层
  └── ImportTaskStorage 接口
       └── 新增：ResetZombieItems()   ← 单条 UpdateMany，无额外查询
```

#### 完整话术

> 面试官，这个问题我们来看一下根因。`runPhase` 每次循环只查 `status=pending` 的 Item，这本身没问题——这是为了让它每次只拿待处理的数据。但问题是，当我把 Item 状态改成 `processing` 之后、还没来得及写入 MongoDB 之前，如果进程崩溃了，这批 Item 就永远卡在 `processing`，后续所有轮次都拉不到它们，相当于这批数据凭空消失了。
>
> 我的解法参考了消息队列里"死信恢复"的思路。核心约定是：**一个 `processing` 状态的 Item，如果超过 10 分钟都没有更新，就认定它是僵尸**——因为正常情况下 1000 条数据的处理时间不超过 60 秒，10 分钟是 10 倍的安全余量，不会误伤正在执行的 Item。
>
> 具体实现分三步：第一步在 `ImportTaskStorage` 接口加一个 `ResetZombieItems` 方法；第二步实现它，就是一条 MongoDB `UpdateMany`，过滤条件是 `status=processing AND update_time < deadline`，把命中的 Item 批量改回 `pending`；第三步在服务启动时（所有 gRPC handler 注册之前）调用一次 `RecoverZombieItems`，扫描所有还在 `running` 状态的任务并触发回滚。这样服务重启后，上次中断的任务会自动衔接断点继续跑，用户完全无感知。

#### 详细实现

**第一步**：在 `ImportTaskStorage` 接口新增方法：

```go
// import_task.go（接口层）
type ImportTaskStorage interface {
    // ...原有方法...

    // ResetZombieItems 将指定任务下超时的 processing Item 回滚为 pending
    ResetZombieItems(ctx context.Context, taskId string, deadline time.Time) error
}
```

**第二步**：实现 `ResetZombieItems`，单条 `UpdateMany` 搞定，无额外查询：

```go
// import_task.go（实现层）
// ResetZombieItems 将超时的 processing Item 回滚为 pending
// deadline：超过此时间仍处于 processing 的 Item 视为僵尸
func (s *importTaskStorage) ResetZombieItems(ctx context.Context, taskId string, deadline time.Time) error {
    objectId, err := primitive.ObjectIDFromHex(taskId)
    if err != nil {
        return err
    }
    _, err = s.itemCollection.UpdateMany(ctx,
        bson.M{
            "task_id":     objectId,
            "status":      mongo.TaskItemStatusProcessing,
            "update_time": bson.M{"$lt": deadline}, // 超时才回滚，避免误伤正在处理的 Item
        },
        bson.M{"$set": bson.M{
            "status":      mongo.TaskItemStatusPending,
            "update_time": time.Now(),
        }},
    )
    return err
}
```

**第三步**：在服务启动入口调用僵尸扫描，在所有 handler 注册之前执行：

```go
// main.go 或 importWorkerService 初始化时
// RecoverZombieItems 服务启动时一次性扫描并恢复中断任务
func RecoverZombieItems(ctx context.Context, taskStorage mongo.ImportTaskStorage) {
    // 超过 10 分钟未更新的 processing Item 视为僵尸
    deadline := time.Now().Add(-10 * time.Minute)

    // 只需关心还在 running 的任务，其他状态的任务无需扫描
    runningTasks, err := taskStorage.ListWithFields(ctx,
        bson.M{"status": mongo.TaskStatusRunning},
        100, []string{"_id"}, bson.D{})
    if err != nil {
        klog.Errorf("[RecoverZombieItems] list running tasks failed: %v", err)
        return
    }

    for _, task := range runningTasks {
        if err := taskStorage.ResetZombieItems(ctx, task.ID.Hex(), deadline); err != nil {
            klog.Errorf("[RecoverZombieItems] reset task %s failed: %v", task.ID.Hex(), err)
            continue
        }
        klog.Infof("[RecoverZombieItems] task %s zombie items reset to pending", task.ID.Hex())
    }
}
```

**效果**：服务重启后 `running` 任务自动恢复，僵尸 Item 重回 `pending` 队列，下一轮 `runPhase` 正常拉取处理，用户无感知。

### 8.2 修复：InsertMany 部分成功导致状态和统计不一致

#### 面试怎么讲

> 原来用 `InsertMany` 加 `SetOrdered(false)` 批量写入，这样允许批次内部分成功。问题是成功插入的那些文档，其对应的 `ImportTaskItem` 状态仍然停在 `processing`，既不会被标为 `completed` 也不会被标为 `failed`，导致统计数字失真，而且这些 Item 永远不会被清理。我的改进是"先批量、失败降级逐条"策略：正常走 `BulkCreate` 保证性能，一旦 bulk 失败就触发降级路径逐条重试，成功的精确标 `completed`，失败的精确标 `failed` 并记录错误原因。这样既保留了正常路径的高性能，又在异常路径保证了状态和统计的精确性。

#### 架构设计思路

这个问题的根源是 **`SetOrdered(false)` 的部分成功语义与状态机之间的不匹配**。`SetOrdered(false)` 允许批次内跳过失败项继续执行，适合"尽量多写"的场景，但代价是调用方拿不到精确的"哪些成功、哪些失败"的行级信息，只能知道整体 error，导致 Item 状态无法精确更新。

业界对这类问题的标准解法是 **Circuit Breaker + Graceful Degradation（熔断降级）** 模式：

```
正常路径（高性能）
  BulkCreate（InsertMany）
    └── 成功 → 批量标 completed，触发副作用
    └── 失败 → 触发降级路径 ↓

降级路径（高精度，仅在异常时触发）
  processNewBatchFallback
    └── 逐条 Create，独立记录每行成功/失败
    └── 成功行 → 批量标 completed + 触发副作用
    └── 失败行 → 标 failed + 写 error_msg
    └── return nil（不中断后续批次）
```

**关键设计决策**：降级路径 `return nil` 而不是 `return err`。原因是即使这批有失败行，也不应该中断整个任务——后续批次的数据是独立的，应该继续处理。失败行的信息已经精确记录在 `ImportTaskItem.ErrorMsg` 中，用户可以在任务详情页看到哪些行失败了、原因是什么。

#### 完整话术

> 面试官，这个问题的本质是批量操作的语义和状态机不匹配。我们用了 MongoDB 的 `InsertMany` 加 `SetOrdered(false)`，这个参数的意思是"遇到失败不要停，跳过这条继续插后面的"，初衷是提高容错性。但副作用是调用方拿到的 error 只是一个整体失败标志，拿不到行级的成功/失败明细，所以没法精确更新每个 `ImportTaskItem` 的状态——成功的那些 Item 就一直卡在 `processing`，统计数字就不准了，这些 Item 也永远不会被清理。
>
> 我的解法是加一个降级路径。正常情况下还是走 `BulkCreate` 保证性能，一旦失败，立刻切换到 `processNewBatchFallback`，在里面逐条调 `Create`，每条独立判断成功还是失败，成功的收集起来最后批量标 `completed`，失败的立刻把 `ImportTaskItem.Status` 改成 `failed` 并写上 `ErrorMsg`，告诉用户这行为什么失败了。
>
> 降级路径有一个细节：它 `return nil` 而不是 `return error`，哪怕这批里有失败行。原因是每批数据是独立的，这批有失败不代表后面的批次也会失败，让任务继续跑，该成功的成功、该失败的失败，用户最后在详情页能看到精确的行级报错信息，远比让整个任务直接失败要好。

#### 详细实现

在 `processNewBatch` 中加入降级分支，正常路径完全不变：

```go
// import_worker_start_import_patterns.go
func (r *startImportRunner) processNewBatch(items []*mongo.ImportTaskItem) error {
    // ...原有逻辑：生成 asset codes，构建 toInsert 列表...

    err := r.assetStorage.BulkCreate(createCtx, toInsert)
    if err != nil {
        klog.Warningf("[processNewBatch] BulkCreate failed, fallback to single insert: %v", err)
        // 降级为逐条插入，精确定位失败行，不影响已成功的行
        return r.processNewBatchFallback(items, toInsert)
    }

    // 成功路径：批量标记 completed，触发副作用（不变）
    // ...
}

// processNewBatchFallback BulkCreate 失败时的降级策略：逐条插入并精确标记状态
func (r *startImportRunner) processNewBatchFallback(
    items []*mongo.ImportTaskItem,
    assets []*mongo.AssetInfo,
) error {
    successIds := make([]primitive.ObjectID, 0, len(assets))
    successAssets := make([]*mongo.AssetInfo, 0, len(assets))

    for i, asset := range assets {
        singleCtx, cancel := context.WithTimeout(r.bg, 10*time.Second)
        err := r.assetStorage.Create(singleCtx, asset)
        cancel()

        if err != nil {
            klog.Errorf("[Fallback] insert asset failed: factory_sn=%s, err=%v", asset.FactorySN, err)
            // 精确标记失败行，记录错误原因供用户排查
            items[i].Status = mongo.TaskItemStatusFailed
            items[i].ErrorMsg = err.Error()
            items[i].UpdateTime = time.Now()
            _ = r.svc.taskService.UpdateImportTaskItem(r.bg, items[i])
            r.failed++
        } else {
            successIds = append(successIds, items[i].ID)
            successAssets = append(successAssets, asset)
            r.imported++
        }
    }

    // 批量标记成功行为 completed
    if len(successIds) > 0 {
        _ = r.svc.taskService.UpdateImportTaskItemStatus(r.bg, successIds, mongo.TaskItemStatusCompleted)
        // 成功插入的资产仍然触发父级关联回填等副作用
        r.dispatcher.Submit(func() {
            if err := r.updateParentInfo(successAssets); err != nil {
                klog.Errorf("[Fallback] updateParentInfo failed: %v", err)
            }
        })
    }

    return nil // 降级路径不返回 error，让任务继续处理后续批次
}
```

**效果**：异常路径下每一行的状态精确可知，`import_count` / `failed_count` 统计准确，失败行保留 `error_msg` 供用户按行排查，成功行的副作用正常触发。

### 8.3 修复：副作用静默失败，父子关联数据丢失无感知

#### 面试怎么讲

> 原来父级关联回填（`updateParentInfo`）失败后只打一行日志就结束了，没有重试也没有告警，GPU 的 `related_parent_*` 字段可能就静默地空着，用户根本不知道。我的方案是两层防护：第一层是内联指数退避重试，失败后等 500ms、1s、2s 重试三次，覆盖大多数瞬时网络抖动；第二层是持久化补偿队列，重试耗尽后把失败任务写入 `side_effect_failures` 集合，由独立的定时 Job 每 5 分钟扫描并重试，超过 5 次还没成功就触发告警。这样从"静默丢失"变成了"最终一致 + 可观测"。

#### 架构设计思路

这个问题的本质是**异步副作用缺乏可靠投递保证**。`updateParentInfo` 是在 `sideEffectDispatcher` 的 goroutine 里异步执行的，与主流程完全解耦，这设计本身是对的（保证主流程性能），但缺少了可靠性保障层。

业界解法是 **Outbox Pattern（本地消息表）** 的简化变体——不引入消息队列中间件，直接用 MongoDB 集合充当"可靠投递队列"：

```
副作用执行层（goroutine）
  ├── 第一层：WithRetry 指数退避（内存级，处理瞬时抖动）
  │     失败 ↓
  └── 第二层：写入 side_effect_failures 集合（持久化，处理持续故障）
               ↓
        补偿 Job（独立 goroutine，5分钟轮询）
          ├── 成功 → 删除补偿记录
          ├── 失败（retry_count < 5）→ 指数退避更新 next_retry
          └── 失败（retry_count >= 5）→ 触发飞书/钉钉告警 → 人工介入
```

**为什么不用消息队列（Kafka/RocketMQ）**：引入 MQ 会带来额外的运维成本、消息序列化/反序列化复杂度、以及 MQ 自身的高可用问题。对于这个场景，副作用失败频率极低（主要是 DB 瞬时抖动），用 MongoDB 集合做轻量级补偿队列，复杂度低、依赖少、可观测性好，是更务实的选择。

#### 完整话术

> 面试官，这个问题的核心是异步副作用的可靠投递问题。`updateParentInfo` 在 goroutine 里执行，失败了只打一行日志，这相当于"fire and forget"——发出去就不管了，数据可能静默丢失，系统没有任何感知，用户也没有任何反馈。这在生产环境是不可接受的。
>
> 我的方案借鉴了 Outbox Pattern 的思路，但做了轻量化处理，不引入额外的消息队列中间件。分两层：第一层是内联的指数退避重试，失败后等 500ms、再等 1s、再等 2s，重试三次，这能覆盖绝大多数数据库瞬时抖动的情况；如果三次都失败了，说明是持续性故障，进入第二层——把失败信息持久化写入 `side_effect_failures` 集合，记录副作用类型、关联的资产编号、失败原因，然后由一个独立的定时 Job 每 5 分钟扫描一次，按指数退避（5分钟、10分钟、20分钟）重试，超过 5 次还失败就触发飞书 webhook 告警，需要人工介入。
>
> 这样整个链路变成了"最终一致 + 可观测 + 可告警"。主数据（`asset_info` 文档）已经安全写入，辅助数据（父子关联字段）通过补偿机制保证最终一致，最差情况下延迟几十分钟，比静默丢失要好得多。如果面试官问为什么不用 Kafka，我会说：引入 MQ 需要额外运维，序列化反序列化也增加复杂度，而这个场景副作用失败率极低，用 MongoDB 集合做轻量补偿队列，依赖少、改动小、可直接复用现有 MongoDB 连接，性价比更高。

#### 详细实现

**第一层：通用指数退避重试工具**

```go
// utils/retry.go
// WithRetry 带指数退避的重试，maxRetries 次用尽后返回最后一次 error
// 初始等待 initInterval，每次翻倍，上限 maxInterval
func WithRetry(maxRetries int, initInterval, maxInterval time.Duration, fn func() error) error {
    interval := initInterval
    var lastErr error
    for i := 0; i < maxRetries; i++ {
        if lastErr = fn(); lastErr == nil {
            return nil
        }
        klog.Warningf("[WithRetry] attempt %d/%d failed: %v, next retry in %s", i+1, maxRetries, lastErr, interval)
        time.Sleep(interval)
        if interval = interval * 2; interval > maxInterval {
            interval = maxInterval
        }
    }
    return fmt.Errorf("all %d retries exhausted, last error: %w", maxRetries, lastErr)
}
```

**第二层：补偿队列数据结构**

```go
// mongo/side_effect_failure.go
// SideEffectFailure 副作用失败补偿记录
type SideEffectFailure struct {
    ID         primitive.ObjectID `bson:"_id"`
    Type       string             `bson:"type"`        // 副作用类型："updateParentInfo" / "changeHistory"
    TaskID     string             `bson:"task_id"`     // 关联的导入任务 ID
    AssetCodes []string           `bson:"asset_codes"` // 受影响的资产编号列表
    RetryCount int                `bson:"retry_count"` // 已重试次数
    LastError  string             `bson:"last_error"`  // 最后一次失败原因
    CreateTime time.Time          `bson:"create_time"`
    NextRetry  time.Time          `bson:"next_retry"`  // 下次重试时间（指数退避计算）
}
```

**副作用调用处包裹两层防护**

```go
// import_worker_start_import_patterns.go
r.dispatcher.Submit(func() {
    // 第一层：内联重试（覆盖瞬时抖动）
    err := utils.WithRetry(3, 500*time.Millisecond, 4*time.Second, func() error {
        return r.updateParentInfo(inserted)
    })

    if err != nil {
        // 第二层：写入补偿队列（覆盖持续性故障）
        codes := make([]string, 0, len(inserted))
        for _, a := range inserted {
            codes = append(codes, a.AssetCode)
        }
        failure := &mongo.SideEffectFailure{
            ID:         primitive.NewObjectID(),
            Type:       "updateParentInfo",
            TaskID:     r.task.ID.Hex(),
            AssetCodes: codes,
            RetryCount: 0,
            LastError:  err.Error(),
            CreateTime: time.Now(),
            NextRetry:  time.Now().Add(5 * time.Minute), // 5 分钟后首次补偿重试
        }
        _ = r.svc.sideEffectStorage.Create(r.bg, failure)
        klog.Errorf("[SideEffect] updateParentInfo permanently failed, queued for compensation: taskID=%s", r.task.ID.Hex())
    }

    // 父级信息确定后再算完整度（父级回填失败则完整度也先跳过）
    if err == nil {
        r.completeness.CalCompletenessByAssetCodes(...)
    }
})
```

**补偿 Job（独立定时任务，每 5 分钟执行）**

```go
// job/side_effect_compensation_job.go
func RunSideEffectCompensationJob(ctx context.Context, storage mongo.SideEffectFailureStorage) {
    for {
        select {
        case <-ctx.Done():
            return
        case <-time.After(5 * time.Minute):
            compensate(ctx, storage)
        }
    }
}

func compensate(ctx context.Context, storage mongo.SideEffectFailureStorage) {
    // 查询 next_retry <= now 且 retry_count <= 5 的记录
    failures, _ := storage.ListPending(ctx, time.Now(), 5)

    for _, f := range failures {
        var err error
        switch f.Type {
        case "updateParentInfo":
            err = retryUpdateParentInfo(ctx, f.AssetCodes)
        case "changeHistory":
            err = retryChangeHistory(ctx, f.AssetCodes)
        }

        if err == nil {
            // 成功则删除补偿记录
            _ = storage.Delete(ctx, f.ID.Hex())
        } else {
            // 失败则指数退避更新下次重试时间
            f.RetryCount++
            f.LastError = err.Error()
            nextDelay := time.Duration(1<<f.RetryCount) * 5 * time.Minute // 5m, 10m, 20m...
            f.NextRetry = time.Now().Add(nextDelay)

            if f.RetryCount >= 5 {
                // 超过上限：发送告警（接入飞书/钉钉 webhook）
                sendAlert(fmt.Sprintf("副作用补偿失败超过5次，需人工介入：type=%s, taskID=%s", f.Type, f.TaskID))
            }
            _ = storage.Update(ctx, f)
        }
    }
}
```

**效果**：瞬时抖动由重试吸收，持续故障由补偿 Job 兜底，超出容忍边界触发告警，从"静默丢失"变为"最终一致 + 可观测 + 可告警"。

### 8.4 修复：多节点并发重入导致任务重复执行

#### 面试怎么讲

> 原来防止重复启动靠的是业务层判断 `status==running`，但这是先读后写，多节点部署时存在竞态：两个节点同时读到 `pending` 就都会进入执行逻辑，造成同一批数据重复写入。我的方案是加 Redis `SET NX`（set if not exists）分布式锁，锁的 key 是 `import:lock:{taskID}`，TTL 设为 30 分钟（远超任务最大执行时间），用 `defer` 保证正常和异常退出都释放锁。这是最简单、侵入性最小的方案，一行代码改变竞态边界，效果直接。

#### 架构设计思路

这个问题的根源是**读-改-写（Read-Modify-Write）操作在分布式环境下的竞态**。当前逻辑是：

```
节点 A：读 status=pending → 判断可以执行 → 写 status=running → 开始处理
节点 B：读 status=pending → 判断可以执行 → 写 status=running → 开始处理  ← 竞态窗口！
```

两次读都在写操作完成之前发生，导致两个节点都认为自己有权执行，同一批 Item 被处理两次。

解法是用 **Redis SetNX 实现分布式互斥锁**，把"判断是否可执行"和"标记正在执行"合并为一个原子操作：

```
节点 A：SetNX(key, 1, TTL) → 成功（拿到锁）→ 执行任务 → 结束后 Del key
节点 B：SetNX(key, 1, TTL) → 失败（key 已存在）→ 直接返回"任务正在处理中"
```

**架构关键决策有三个**：

1. **锁粒度**：锁 key 为 `import:lock:{taskID}`，粒度在任务级而非全局，不同任务互不影响，支持并发处理多个不同任务

2. **TTL 选择**：30 分钟 = 任务最大合理执行时间，防止节点宕机后锁永不释放（死锁）；若任务正常结束则 `defer Del` 主动释放，不等 TTL 到期

3. **Redis 故障降级**：锁是防重入的额外保险层，Redis 不可用时不应阻塞核心导入功能——降级为仅打日志，让业务层的 `status=running` 判断继续作为兜底

```
锁获取 → Redis SetNX
  ├── 成功 → 正常执行，goroutine 结束时 defer Del 释放
  ├── 失败（key 存在）→ 返回"任务正在处理中"，拒绝重复启动
  └── 错误（Redis 不可用）→ 降级，打 Warning 日志继续执行（兜底靠 DB status）
```

#### 完整话术

> 面试官，这个问题的本质是分布式环境下的 Check-Then-Act 竞态。原来的逻辑是先读任务状态判断是否 pending，再写成 running，但这两步之间有窗口——如果两个节点同时读到了 pending，就都会认为自己有权执行，同一批数据就被处理两次了。单节点下这个问题不存在，因为只有一个 goroutine，但多节点部署就暴露了。
>
> 我的方案是在 `StartImport` 入口加 Redis 分布式锁。用 `SET NX`——也就是"set if not exists"，这是一个原子操作，只有第一个执行它的节点能成功，其他节点拿到的结果是 false，直接返回"任务正在处理中"。锁的 key 设计成 `import:lock:{taskID}`，粒度在任务级，不同任务的锁是独立的，不会互相阻塞。TTL 设 30 分钟，覆盖任务的最大执行时间，防止节点宕机后锁死不释放。
>
> 有两个细节面试官可能会追问。第一个是锁在哪里释放——因为任务是 goroutine 异步执行的，不能在 `StartImport` 返回时就释放，必须把 `defer Del` 写在 goroutine 内部，这样无论任务正常结束还是 panic recover，锁都会被释放。第二个是 Redis 挂掉怎么办——锁只是防重入的额外保险，Redis 故障不应该阻塞导入核心功能，所以我的处理是：Redis 返回 error 时打一条 Warning 日志，然后继续执行，这时候兜底靠的是数据库里的 status 判断——两层防护，Redis 是第一道，DB status 是第二道。

#### 详细实现

在 `StartImport` 入口加锁，**任务启动 goroutine 之前**抢锁，goroutine 结束时释放：

```go
// import_worker.go
func (i *importWorkerService) StartImport(ctx context.Context, taskID string, isCover bool) error {
    if taskID == "" {
        return errors.New("invalid params")
    }

    // 抢分布式锁，防止多节点并发重入
    // TTL = 30 分钟，覆盖任务最大执行时长，节点宕机后锁自动过期不会死锁
    lockKey := fmt.Sprintf("import:lock:%s", taskID)
    lockTTL := 30 * time.Minute

    acquired, err := i.redis.SetNX(ctx, lockKey, "1", lockTTL).Result()
    if err != nil {
        // Redis 不可用时降级：打日志继续执行（避免 Redis 故障阻塞导入）
        klog.Warningf("[StartImport] acquire lock failed, proceeding without lock: %v", err)
    } else if !acquired {
        // 锁已被其他节点持有，拒绝重复启动
        return errors.New("任务正在处理中，请勿重复提交")
    }

    task, err := i.taskService.GetByID(ctx, taskID)
    if err != nil {
        if acquired {
            i.redis.Del(ctx, lockKey) // 获取任务失败时立即释放锁
        }
        return errors.New("get import task failed")
    }

    // ...isCover 判断逻辑不变...

    task.Status = mongo.TaskStatusRunning
    task.UpdateTime = time.Now()
    if err = i.taskService.Update(ctx, task); err != nil {
        if acquired {
            i.redis.Del(ctx, lockKey)
        }
        return errors.New("任务更新失败")
    }

    userInfo := common.GetAuthCtxUserInfoFromCtx(ctx)

    go func() {
        // goroutine 结束时释放锁（正常完成 or panic recover 后）
        if acquired {
            defer i.redis.Del(context.Background(), lockKey)
        }
        runner := newStartImportRunner(i, task, userInfo)
        runner.Run()
    }()

    return nil
}
```

**两个关键细节**：

1. **Redis 不可用时降级而非拒绝**：锁只是防重入的额外保险，Redis 故障不应阻塞导入核心功能，日志记录即可，让任务继续跑（业务层的 `status=running` 判断仍然兜底）。

2. **锁在 goroutine 内 defer 释放**：任务是异步执行的，锁不能在 `StartImport` 返回时就释放，必须挂到 goroutine 生命周期上，确保任务跑完（无论正常还是 panic）才释放。

**效果**：多节点场景下同一个 TaskID 最多只有一个节点能进入执行逻辑，彻底消除竞态窗口，且对 Redis 故障有容错降级。

### 8.5 改进总结

| 局限 | 最优方案 | 改动量 | 优先级 |
|------|----------|--------|--------|
| processing 僵尸无法恢复 | 服务启动僵尸扫描 + `ResetZombieItems` | 极小（新增1个接口方法 + 启动钩子） | **P0 必改** |
| InsertMany 部分成功状态失真 | BulkCreate 失败降级逐条 + 精确标记 | 小（新增1个 fallback 函数） | **P1 高优** |
| 副作用静默失败 | 指数退避重试 + 持久化补偿队列 + 告警 | 中（新增补偿集合 + 定时 Job） | **P1 高优** |
| 多节点并发重入 | Redis SetNX + TTL + goroutine defer 释放 | 极小（入口加锁约 10 行） | P2，多节点必改 |

改进后一致性从 **best-effort（尽力而为）** 升级为 **eventual consistency（最终一致）**：主数据强一致（幂等 + 分布式锁），辅助数据（父子关联/变更历史）通过补偿机制保证分钟级最终一致，故障恢复从"人工介入"降为"自动恢复 + 超限告警"。
