## 难点一：跨层级移动时子节点的级联更新（职责一）

### 一句话引入

> 仓库是三级树形结构（洲→国→仓），把一个含子节点的国家节点移动到另一个洲下面时，它下面所有子仓库的完整路径 `full_path`、层级 `level` 都要跟着变。

### 为什么难

数据模型中每个节点都冗余存储了完整路径，比如"中国/北京仓库"的 `full_path = "亚洲-中国-北京仓库"`。把"中国"移到"欧洲"下时，它的所有子孙节点的 `full_path` 前缀都得从 `亚洲-中国-` 改成 `欧洲-中国-`。如果逐条查出来再逐条更新，N 个子孙就是 N 次写操作，且这个过程不是原子的——中途失败会导致数据一半旧一半新。

另外还有一个容易忽略的问题：**循环依赖检测**——不能把节点 A 移动到 A 自己的子节点 B 下面，否则树会成环。

### 我怎么解决的

**级联更新**用了 MongoDB 的 `$regex` 前缀匹配 + `BulkWrite`：

```go
// 用旧路径前缀匹配所有子孙节点，一次查出来
filter := bson.M{
    "full_path": bson.M{
        "$regex": "^" + oldFullPath + "-",  // 匹配所有以旧路径开头的子孙
    },
}
// 构建批量更新操作，用字符串替换把旧前缀换成新前缀
ops := make([]mongo.WriteModel, 0)
for _, child := range descendants {
    newPath := strings.Replace(child.FullPath, oldFullPath, newFullPath, 1)
    ops = append(ops, mongo.NewUpdateOneModel().
        SetFilter(bson.M{"_id": child.ID}).
        SetUpdate(bson.M{"$set": bson.M{
            "full_path": newPath,
            "level":     newLevel + (child.Level - oldLevel),
        }}),
    )
}
collection.BulkWrite(ctx, ops)  // 1次网络往返完成所有子孙更新
```

**循环依赖检测**：在 Service 层移动前先检查目标节点是否是被移节点的子孙——只需查 `full_path` 是否以被移节点的路径为前缀，O(1) 判断。

### 结果

含 10+ 子节点的跨层级移动从 O(n) 次写操作压缩为 **1 次 BulkWrite**，耗时 < 200ms，且无数据一致性问题。

### 深度分析

**为什么选择 `parent_id + full_path` 混合模型，而不是单一方案？**

树形结构有三种经典存储方案，每种都有致命缺陷：

| 方案 | 查子树 | 获取路径 | 节点移动 | 适用场景 |
|------|--------|---------|---------|---------|
| 邻接表（只存 parent_id） | O(层数) 次递归查询 | O(层数) 次向上回溯 | 只改自身，无需级联 | 层级不深、不需要路径展示 |
| 物化路径（只存 full_path） | 1次 `$regex` 前缀查 | 直接读字段 | 所有子孙都要改路径字符串 | 层级深、查子树频繁 |
| 混合方案（两者都存） | 1次 `$regex` | 直接读字段 | `$regex` 批量改 + BulkWrite | 需要树形展示 + 频繁层级操作 |

本系统是三级固定结构，**树形渲染**（前端展开树）和**跨层级移动**（运营人员拖拽）都是高频操作，混合方案是最优选择：
- `parent_id` 用于构建 parent→children 映射（内存中建图），O(n) 一次遍历装配整棵树
- `full_path` 用于路径展示和批量更新子孙节点

**内存建树的具体实现**：不需要递归查询，一次查出所有节点，在内存中用 map 建图：

```go
// 一次查出所有节点
all, _ := collection.Find(ctx, bson.M{"is_deleted": 0})

// 内存中建图，O(n)
nodeMap := make(map[string]*TreeNode)
for _, w := range all {
    nodeMap[w.WarehouseID] = &TreeNode{Warehouse: w}
}

// 装配父子关系
roots := make([]*TreeNode, 0)
for _, node := range nodeMap {
    if node.Warehouse.ParentID == "" {
        roots = append(roots, node)  // 根节点
    } else {
        parent := nodeMap[node.Warehouse.ParentID]
        parent.Children = append(parent.Children, node)
    }
}
// 结果：只用了 1 次 DB 查询 + O(n) 内存遍历
```

**权限过滤对树形构建的影响**：非管理员用户只能看到自己有权限的仓库，但如果直接过滤掉无权限的节点，树会断裂（孤立子节点看不到）。解决方案：先查出用户有权限的叶节点，再向上补全所有祖先节点（查 full_path 的前缀），使路径连通后再渲染，整个过程只需 2 次 DB 查询。

**循环依赖检测的边界情况**：除了"不能移到自己的子节点下"，还要处理"不能移到自己本身"（父节点 = 当前节点），代码中需要同时判断 `targetID != nodeID` 和 `full_path 不以旧路径为前缀`。

### 面试官追问

**Q：BulkWrite 不是原子的，中途失败怎么办？**

A：MongoDB 的 `BulkWrite` 默认是有序执行，但不是事务——失败前的操作已提交，失败后的操作未提交。对于仓库移动这种低频管理操作，当前用了"补偿重试"：失败后前端提示用户重试，Service 层做幂等性保证（路径已是新值就跳过）。若要严格原子，应开启 MongoDB 多文档事务（`session.StartTransaction()`），但要求副本集部署，当前环境满足条件，这是可以进一步增强的方向。

**Q：full_path 用什么作分隔符，名字里有特殊字符怎么处理？**

A：用 `-` 作分隔符，同时在写入时对仓库名做校验，禁止包含 `-`。更健壮的方案是用 ID 而非名称拼接路径（`WH001-WH002-WH003`），这样名称改动不影响路径，当前是用名称主要是为了前端展示方便，这是已知的设计 trade-off。

## 难点二：三层查重机制中「任务内去重」的必要性（职责二）

### 一句话引入

> Excel 导入时我设计了三层查重：文件内去重、任务内去重、数据库历史去重。其中"任务内去重"是最不直观的一层，但恰恰是唯一能覆盖"同一个导入任务分多次上传相同文件"这个场景的。

### 为什么难

运营人员会这样操作：同一个导入任务，先上传一个含 500 条的文件，发现有问题，再次上传同一份文件（或有重叠的文件）。

- **数据库去重**（查 `asset_info`）发现不了，因为第一批的 500 条还在 `pending` 状态，还没写入 `asset_info`，第二批来查库当然查不到
- **文件内去重**也发现不了，因为它只检查当次上传文件内部的重复，不跨批次

所以需要专门查 `import_task_items` 表，看当前任务内已经有没有相同的 `FactorySN`。

### 我怎么解决的

三层查重依次执行，每层短路：

```
L1 文件内去重
  └── map[FactorySN]rowIndex，O(1)，发现同一文件内重复立即 Fail-Fast

L2 任务内去重（关键层）
  └── 查 import_task_items WHERE task_id=? AND factory_sn IN (本批SN列表)
      一次 $in 查询，发现已存在则分类为 duplicate（待用户决策）

L3 数据库历史去重
  └── 查 asset_info WHERE factory_sn IN (本批SN列表)
      GetByFactorySNBatch，一次 $in 查询替代 N 次单条查询
      命中 → duplicate（已有资产，走覆盖更新路径）
      未命中 → new（新资产，走创建路径）
```

每层用 `$in` 批量查询，整个 CheckData 函数只有 2 次 DB 查询，不管这批数据有多少条。

### 结果

三层查重覆盖了所有真实场景，且没有引入额外的 DB 查询次数（2 次固定开销）。`duplicate` 和 `new` 分类后，导入结果页面可以精确展示"新增 X 条、更新 Y 条、跳过 Z 条"。

### 深度分析

**三层查重的本质：覆盖三个不同的时间窗口**

三层查重不是重复设计，每一层针对的是不同时间窗口内的重复数据：

```
时间线：

[文件上传]──[L1 文件内]──[L2 任务内]──[L3 历史库]──[写入 asset_info]

L1：当次上传文件内部（同一文件内 FactorySN 重复）
     → 数据来源：内存 map，不需要查库

L2：同一导入任务内跨批次（第1次上传 500 条还没入库，第2次上传来了）
     → 数据来源：import_task_items（中间态表，仅活跃任务有数据）
     → 为什么 L3 发现不了：第1批还没写 asset_info，L3 查库查不到

L3：历史数据（已经存在于 asset_info 的存量资产）
     → 数据来源：asset_info（正式数据表）
     → 命中 → duplicate（走覆盖更新分支，不新建）
```

**`$in` 批量查询的性能分析**：

L2 和 L3 都用 `$in` 查询，关键在于索引：

```go
// L2：import_task_items 上有 {task_id:1, factory_sn:1} 复合索引
// 查询：task_id=X AND factory_sn IN [sn1, sn2, ...snN]
// MongoDB 对 $in 的每个值在索引上做独立查找，复杂度 O(k * log m)
// k = $in 值的个数，m = task_items 集合大小（只有活跃任务，通常 < 1 万条）

// L3：asset_info 上 factory_sn 有唯一索引
// 批量查 N 条 SN，1 次查询，不管 N 多大都走索引
```

一批 1000 条数据，CheckData 固定只有 2 次 DB 查询，无论批次大小，DB 查询次数是常数。

**`duplicate` 和 `new` 的后续处理差异**：

```
new     → 直接进入创建队列 → GenerateBatchAssetCodes + InsertMany
duplicate → 进入确认队列 → 前端展示给用户（"这 X 台机器已存在，是否覆盖？"）
            用户确认后 → UpdateMany 覆盖更新（不重新生成编号）
            用户拒绝后 → 标记 skipped，不做任何操作
```

这个设计使运营人员可以安全地重复上传同一份文件——不会导致重复资产，同时给用户完整的决策权。

**Fail-Fast 的粒度设计**：

L1（文件内重复）是硬 Fail-Fast——整批直接拒绝，因为文件本身有错误，没必要继续处理。  
L2/L3（跨批次重复、历史重复）不 Fail-Fast——只是分类为 `duplicate`，继续处理其他行，不因为部分重复就中断整个导入，体验更好。

### 面试官追问

**Q：L2 任务内去重查的是 import_task_items，数据量会很大吗，性能如何？**

A：`import_task_items` 每个任务结束后会删除（`DeleteImportTaskItem`），所以同一时刻这张表里只有活跃任务的数据。加上 `task_id` 的索引，`$in` 查询速度很快。真正的性能瓶颈在 L3 的 `asset_info`，通过 `GetByFactorySNBatch`（`$in` + `factory_sn` 唯一索引）解决。

**Q：L1 Fail-Fast 对用户体验好吗，整个任务都失败了？**

A：L1 是文件内重复，说明上传的文件本身有问题，强制 Fail-Fast 是对的——与其让用户导入一半发现有重复数据，不如一开始就提示"第 23 行和第 47 行 FactorySN 重复"。用户修正文件后重新上传。这是"数据质量优先"的设计选择。

## 难点三：sideEffectDispatcher 的设计权衡（职责二）

### 一句话引入

> 每条资产导入成功后，要异步触发：写变更历史、绑定 SN、更新父级关联、计算完整度。这些副作用不能阻塞主流程，但也不能无限并发——否则会打爆下游 gRPC 服务。

### 为什么难

- **不能同步执行**：5000 条数据，每条都同步调用 gRPC 写变更历史，光这一步就要 5000 次串行 gRPC 调用，几乎翻倍导入耗时
- **不能无限 goroutine**：`go func(){}()` 启动 5000 个 goroutine，会把下游服务的限流打穿，同时本地 goroutine 调度压力也很大
- **必须等全部完成再收尾**：任务状态改为 `completed` 之前，必须确保所有副作用都已执行完，否则用户看到"导入完成"但变更历史还没写

### 我怎么解决的

用 **Channel 信号量 + WaitGroup** 实现固定并发的 Goroutine Pool：

```go
type sideEffectDispatcher struct {
    sem chan struct{}    // 容量固定为 8，相当于信号量
    wg  sync.WaitGroup
}

func (d *sideEffectDispatcher) Submit(fn func()) {
    d.wg.Add(1)
    d.sem <- struct{}{}    // 占一个槽；已有 8 个运行时，这里阻塞等待
    go func() {
        defer func() {
            <-d.sem        // 释放槽
            d.wg.Done()
        }()
        fn()
    }()
}

func (d *sideEffectDispatcher) Wait() {
    d.wg.Wait()            // 主流程在这里等所有副作用完成
}
```

主流程（BulkCreate）完成后立即返回，副作用批量 Submit 进 Pool；最后 `dispatcher.Wait()` 阻塞直到所有 goroutine 都退出，再更新任务状态为 `completed`。

**并发数 8 的确定**：这是经验值，基于下游 gRPC 服务的限流配置（ChangeHistory 服务单实例限流 20 QPS）和服务器 2 核 CPU 综合估算。实际暴露为参数，可以通过压测调整。

### 结果

副作用异步化后，主流程（BulkCreate 阶段）耗时再降约 50%。8 个并发的 Pool 在压测中既保证副作用吞吐，也未触发下游限流。

### 深度分析

**Channel 信号量 vs `sync.Semaphore` vs 第三方 worker pool 的选型对比**

| 方案 | 实现复杂度 | 背压支持 | 错误收集 | 适用场景 |
|------|---------|---------|---------|---------|
| `chan struct{}` + WaitGroup（本方案） | 极低，20 行 | 天然（阻塞 Submit） | 不收集（仅 log） | 副作用类任务，失败不影响主流程 |
| `golang.org/x/sync/semaphore` | 低 | 需手动 | 不收集 | 精确控制并发权重 |
| `errgroup` + semaphore | 中 | 需手动 | 收集第一个错误 | 任务有依赖、需要传播错误 |
| 第三方 worker pool（如 ants） | 低 | 有队列 | 可选 | 高吞吐、任务均匀 |

本方案选择手写 Channel 信号量的核心理由：**副作用失败不应中断主流程**。变更历史写失败、完整度计算失败，主数据（`asset_info` 已写入）不受影响，失败只需记录 log，不需要向上传播错误。`errgroup` 收集错误的语义在这里是多余的负担。

**时序图：主流程与副作用的执行关系**

```
主流程 goroutine：
    ┌─────────────────────────────────────────────────────────────┐
    │ BulkCreate(batch1) → Submit(副作用) × N → BulkCreate(batch2)│
    │   → Submit(副作用) × N → ... → dispatcher.Wait()           │
    └─────────────────────────────────────────────────────────────┘
         ↓ Submit 时                              ↓ Wait() 阻塞直到所有副作用完成

副作用 Pool（最多 8 个并发）：
    ┌────────┐ ┌────────┐ ┌────────┐ ... （最多 8 个同时运行）
    │写变更历史│ │绑定SN  │ │算完整度│
    └────────┘ └────────┘ └────────┘
```

关键时序点：
1. `BulkCreate` 写库成功后立即 Submit 副作用，不等副作用完成
2. 主流程继续处理下一批，副作用和主流程并发执行
3. 所有批次处理完后，`Wait()` 确保无遗漏的副作用
4. 全部完成后才将 `ImportTask.status` 置为 `completed`

**为什么 `d.sem <- struct{}{}` 放在 `go func` 之外（Submit 里阻塞），而不是在 goroutine 内部抢占？**

如果改成这样：
```go
// 错误写法：在 goroutine 内部抢占信号量
go func() {
    d.sem <- struct{}{}  // goroutine 已经被创建，只是阻塞在这里
    defer func() { <-d.sem; d.wg.Done() }()
    fn()
}()
```

goroutine 会被无限创建（每次 Submit 都创建一个），只是大部分阻塞在信号量上。如果提交 5000 个任务，就会有 5000 个 goroutine 同时存在内存中（每个约 2KB 栈），消耗 ~10MB 额外内存。

正确写法把 `d.sem <- struct{}{}` 放在 Submit 里（主 goroutine 阻塞），控制了 goroutine 的创建数量，内存中同时最多只有 8 个副作用 goroutine。

### 面试官追问

**Q：这不就是 worker pool 吗，用 `errgroup` 不行吗？**

A：`errgroup` 限制并发可以用 `errgroup.WithContext` + `semaphore`，能解决问题，但这里选择手动实现有两个原因：① 副作用操作不需要收集错误（失败只打 log，不影响主流程），`errgroup` 的 error 收集语义是多余的；② `Submit` 是批量分散调用（每处理一批 1000 条就 Submit 一次），手写结构体更清晰。如果换成 `errgroup` 也完全可以，本质是一样的。

**Q：Submit 里的 `d.sem <- struct{}{}` 会阻塞主 goroutine 吗？**

A：会，但这是故意的——主流程提交副作用的速度不应该远超 Pool 的消费速度，否则会堆积大量"待执行"的 goroutine 在内存里。Channel 阻塞提供了天然的背压（backpressure）机制。如果不想阻塞主流程，可以改为带缓冲的任务队列，但在当前场景下没必要增加这个复杂度。

## 难点四：Redis 中转大参数——解决 gRPC 消息体膨胀问题（职责三）

### 一句话引入

> 工单节点"批量入库 120 台机器"时，asset_id 列表要从工作流服务传到维护服务。不能直接塞进 gRPC 消息体，因为多个工单节点串联时消息体会持续膨胀，而且 gRPC 默认 4MB 消息体限制随时可能炸。

### 为什么难

工单流程是多节点串联的：节点 A 完成 → 触发回调 → 节点 B 执行 → 再触发回调。如果参数（资产列表）一路跟着消息体传，每经过一个节点就多携带一份，链路越长数据越大。更严重的是，这个参数并不是每个节点都需要的，大部分节点只需要最终节点的参数，中间节点白白携带了数据。

### 我怎么解决的

**存储-Key 分离**模式：

```
写入阶段（工单节点执行完成时，task.go）：
  检测到参数超大（batch_update_status_warehouse 字段）
  → 序列化写入 Redis，Key = "batch_update_status_warehouse"，TTL = 24h
  → gRPC 消息体只传这个 Key（几十字节）

读取阶段（GRPCMaintenanceHandler 处理时，callback_grpc_maintenance.go）：
  从 gRPC 请求中取 Key
  → RedisGet(cacheKey) 读取完整参数
  → 用完立即 RedisDel（用后即删，不等 TTL 过期）
```

```go
// callback_grpc_maintenance.go
cacheKey := "batch_update_status_warehouse"
if cachedWarehouse, err := base.RedisGet(ctx, cacheKey); err == nil && cachedWarehouse != "" {
    warehouse = cachedWarehouse
    defer func() {
        base.RedisDel(ctx, cacheKey)  // 用完即删，节省 Redis 内存
    }()
}
```

TTL 24 小时的理由：工单正常流程几分钟到几小时完成，24 小时足够覆盖所有正常场景，还能覆盖因审批流程暂停等特殊情况。

### 结果

gRPC 消息体始终是固定大小（Key 字符串），与参数规模完全解耦。支持单次工单驱动 100+ 条资产的批量状态变更。

### 深度分析

**为什么这个场景下不能直接用 gRPC 消息体传参？从数量级上理解**

gRPC 默认消息体上限 4MB。一条 asset_id 是 MongoDB ObjectID，十六进制字符串约 24 字节，加上字段名、序列化开销，1 条资产约 50 字节。

```
100 条资产 × 50 字节 = 5KB      → 单次调用可以接受
1000 条资产 × 50 字节 = 50KB    → 接近告警阈值
5000 条资产 × 50 字节 = 250KB   → 明显膨胀
多节点串联 × 3 层     = 750KB   → 不可接受
```

工单流程是多节点串联的，同一份参数在多个节点间随 gRPC 消息体传递，每经过一个节点就被反序列化一次再序列化一次，链路越长开销越大。

**存储-Key 分离的完整数据流**

```
工单节点 A 执行完成（task.go:345-380）
    ↓
检测到 callbackParams 中有 batch_update_status_warehouse 字段（批量资产列表）
    ↓
RedisSet(key="batch_update_status_warehouse", value=JSON(资产列表), ttl=24h)
    ↓
gRPC 消息体中 batch_update_status_warehouse 字段替换为 Redis Key 字符串
    ↓
workflow-service → gRPC → maintenance-service
（消息体：固定 < 1KB）
    ↓
GRPCMaintenanceHandler.callBatchUpdateStatus
    ↓
RedisGet("batch_update_status_warehouse") → 取出完整资产列表
    ↓
执行批量更新业务逻辑
    ↓
defer RedisDel("batch_update_status_warehouse")  → 用完即删
```

**TTL 24 小时的设计边界**

TTL 不能太短也不能太长：

- **太短（如 30 分钟）**：工单可能因为审批流程暂停超过 30 分钟，参数过期后 maintenance-service 读不到 Key，业务失败
- **太长（如 7 天）**：工单异常搁置时 Redis 中残留大量无效数据，占用内存；且如果 Key 没有实例隔离，长 TTL 会增加不同工单互相覆盖的时间窗口
- **24 小时**：覆盖正常工单的最长周期（含人工审批等待），且每天自动清理一次，内存可控

**这个模式的通用性：Claim Check 模式**

这个设计在分布式系统中有一个标准名字：**Claim Check Pattern（行李寄存模式）**，也叫 Reference-based Messaging。核心思路是：消息体只传一个"取货凭证"（Key），真正的大数据放在外部存储，消费方用凭证去取。这个模式在消息队列（Kafka、RabbitMQ）场景中也常用于处理超大消息体。理解这个模式有助于在面试中展示架构设计能力。

### 面试官追问

**Q：Redis Key 没有 instance_code 隔离，不同工单的参数会不会互相覆盖？**

A：这是一个好问题，确实是当前实现的一个局限。当前 Key 是固定的 `"batch_update_status_warehouse"`，如果两个工单同时触发，后写入的会覆盖先写入的。严格来说 Key 应该包含 `instance_code`（工单实例 ID），改为 `"batch_update_status_warehouse:{instance_code}"`，保证每个工单实例的参数隔离。这是一个已知的改进点，当前业务场景下并发工单触发该操作的概率较低，但生产环境高并发时必须修复。

**Q：用后即删（defer RedisDel）是否存在删除失败的情况？失败了怎么处理？**

A：有可能失败，代码里有 `klog.Errorf` 记录错误日志但不阻断流程。兜底是 TTL 24 小时自动过期，所以即使 `RedisDel` 失败，数据最多在 Redis 里存留 24 小时后自动清理，不会造成内存永久泄漏。

## 难点五：ESR 原则背后的 B-Tree 原理（职责四）

### 一句话引入

> ESR 原则不是规定，是 B-Tree 索引的物理结构决定的。等值字段必须放最左，本质是因为只有等值查询才能精确定位 B-Tree 上的一个连续区间；排序字段紧跟等值，是因为该区间内的数据已经按这个字段排好序了，不需要内存 sort；范围字段放最后，是因为范围查询之后索引的有序性已经被截断，后续字段无法再利用。

### 为什么难

不理解这个原理，就不知道为什么 `{is_deleted:1, create_time:-1}` 比 `{create_time:-1, is_deleted:1}` 快，也不知道为什么加了索引但 explain 还是显示内存 sort。

### 我怎么解决的

以 `{is_deleted:1, create_time:-1}` 为例，B-Tree 叶节点排列：

```
[is_deleted=0, create_time=2026-04-07] → doc1
[is_deleted=0, create_time=2026-04-06] → doc2
[is_deleted=0, create_time=2026-04-05] → doc3
[is_deleted=1, create_time=2026-04-07] → doc4
...
```

查询 `{is_deleted:0}.sort({create_time:-1}).limit(20)`：

1. B-Tree 二分查找定位到 `is_deleted=0` 的起点 → O(log N)
2. 顺序读取叶节点，已按 `create_time DESC` 排好 → 读 20 个就停
3. 全程没有内存 sort，`totalDocsExamined = 20`

反过来 `{create_time:-1, is_deleted:1}` 查 `{is_deleted:0}`：

```
[create_time=2026-04-07, is_deleted=0] → doc1
[create_time=2026-04-07, is_deleted=1] → doc4  ← is_deleted=0 和 1 交错
[create_time=2026-04-06, is_deleted=0] → doc2
...
```

`is_deleted` 不是最左字段，无法用 B-Tree 前缀定位，只能全索引扫描再过滤，`totalDocsExamined = 全部文档数`，等于没有索引。

### 实际代码对应

```go
// 本项目 ListAssetInfos 的 filter 和 sort
filter["is_deleted"] = 0                          // 等值（E）
sort := bson.D{
    {Key: "create_time", Value: -1},              // 排序（S）
    {Key: "_id", Value: -1},                      // 兜底排序（S）
}
// 对应索引：{is_deleted:1, create_time:-1}
// ——正是 E → S 的顺序
```

### 深度分析

**explain() 输出的三个关键指标怎么联合判断**

光看一个指标不够，要三个指标联合看：

```
executionStats：
  ├── stage: "COLLSCAN"     → 无索引，必须优化
  │   stage: "IXSCAN"       → 有索引命中
  │
  ├── totalDocsExamined: N  → 实际扫描的文档数
  ├── nReturned: M          → 最终返回的文档数
  │   比值 N/M >> 1          → 索引选择性差，或排序字段未覆盖（产生 in-memory SORT）
  │   比值 N/M ≈ 1           → 索引精准覆盖，性能最优
  │
  └── executionTimeMillis   → 总耗时（含索引查找 + 文档加载 + 排序）
```

**本项目三个接口的 explain 分析结果汇总**

| 接口 | 优化前 stage | 优化前扫描比 | 优化后 stage | 优化后扫描比 | 耗时变化 |
|------|------------|------------|------------|------------|---------|
| `ListAssetInfos`（首页） | COLLSCAN | 10万:20 = 5000:1 | IXSCAN | 20:20 = 1:1 | 2100ms→45ms |
| `GetLimitPendingConfirmeds`（导入循环） | COLLSCAN | 1万:1000 = 10:1 | IXSCAN | 1000:1000 = 1:1 | 800ms→15ms |
| `FindByFactorySN`（精确查找） | COLLSCAN | 10万:1 = 100000:1 | IXSCAN（唯一索引） | 1:1 | 80ms→2ms |

**范围字段放最后的实际意义——以 `project` 查询为例**

本项目有按项目过滤资产的场景：`{project:"P001", is_deleted:0, create_time > "2026-01-01"}`

如果索引设计错误：`{project:1, create_time:-1, is_deleted:1}`

```
执行过程：
1. B-Tree 定位 project="P001" 区间 ✓
2. 在该区间内，数据按 create_time 排序 ✓
3. 过滤 create_time > "2026-01-01"（范围截断，此后 is_deleted 无法利用索引）
4. 对满足范围的文档再过滤 is_deleted=0 → 内存过滤，多扫描大量文档
```

正确设计：`{project:1, is_deleted:1, create_time:-1}`（等值 E → 等值 E → 排序/范围 S/R）

```
执行过程：
1. B-Tree 定位 project="P001" AND is_deleted=0 的区间（两个等值，范围更小）✓
2. 在该区间内，数据按 create_time 排序 ✓
3. 直接按 create_time 范围截取，不需要额外过滤
```

**索引设计的实际代价——写放大问题**

加索引不是免费的。每次 `InsertOne/UpdateOne/DeleteOne`，MongoDB 都需要同步更新所有索引的 B-Tree。本项目 `asset_info` 集合建了 7 个索引，每次写操作会有 7 次 B-Tree 更新。

权衡标准：读多写少（资产列表查询 >> 资产新增），读性能收益远大于写放大代价，加索引合理。对于写多读少的场景（如日志集合），索引数量要严格控制。

### 面试官追问

**Q：`_id` 为什么要作为兜底排序，不加会怎样？**

A：`create_time` 是毫秒级时间戳，但批量导入时 `InsertMany` 一次插入 1000 条，这 1000 条的 `create_time` 可能是同一毫秒。同一毫秒的文档，MongoDB 在没有兜底字段时排序不稳定，翻页时可能出现"幽灵数据"——第 1 页最后一条和第 2 页第一条是同一条记录，或者某条记录在两页都出现或两页都不出现。加 `_id`（ObjectID 天然有全局唯一顺序）作为兜底，保证严格全序，翻页结果稳定可靠。

## 难点六：状态机游标分页——用业务状态替代 skip（职责四）

### 一句话引入

> 导入任务要处理 10 万条 `import_task_items`，每批取 1000 条处理。传统 skip 方案第 100 批要跳过 99000 条，MongoDB 得扫描 9.9 万条才能返回第 100 批的 1000 条，性能随批次线性劣化。我用的方案是：取完一批立即标记为 `processing`，下次查询只查 `status=pending` 的，自然就跳过了已处理的，完全不用 skip。

### 为什么难

这个问题的难点不是技术复杂，而是要意识到：**状态字段本身就是天然的游标**。大多数人看到"分批处理"会直接想到 skip，却忽略了状态机已经隐含了游标位置。

### 我怎么解决的

**传统 skip 方案**（反例，说明问题）：

```
第1批：skip=0,     limit=1000  → 扫描 1000 条
第2批：skip=1000,  limit=1000  → 扫描 2000 条（前1000条被扫描后丢弃）
第N批：skip=N*1000             → 扫描 N*1000 条
```

**状态机游标方案**（实际代码 `import_worker_start_import_patterns.go:155`）：

```go
func (r *startImportRunner) runPhase(itemType string, handler batchHandler) error {
    for {
        // 每次永远 skip=0，查 status=pending 的前 1000 条
        items, _ = GetLimitPendingConfirmeds(taskId, "pending", itemType, 1000, sort)
        if len(items) == 0 {
            return nil  // 全部处理完
        }

        // 立刻标 processing → 相当于把游标向前移动了 1000 步
        UpdateImportTaskItemStatus(itemIds, "processing")

        // 处理
        handler(items)

        // 处理完标 completed
        UpdateImportTaskItemStatus(itemIds, "completed")
    }
}
```

**为什么这等价于游标分页**：每次查询都带 `status=pending`，被标记为 `processing` 的文档自动从下次结果集里消失，不需要记录"上次处理到哪里"。数据库的状态字段就是游标。

**性能对比**（10 万条，每批 1000，共 100 批）：

```
skip 方案：总扫描量 = 1000 + 2000 + ... + 100000 = 约 505 万条
状态机方案：总扫描量 = 1000 × 100 = 10 万条（每批始终只扫描 1000 条）
```

状态机方案总扫描量是 skip 的 **1/50**，且每批耗时恒定 < 80ms，不随批次增长。

### 额外收益

这个设计还附带了两个"免费"的工程收益：

- **并发安全**：多个 goroutine 同时取数，`processing` 状态天然防止重复处理，不需要额外加分布式锁
- **断点续传**：服务宕机重启后，`processing` 状态的数据可以被检测为"僵尸"并重置为 `pending`，自动续传，不丢数据

### 深度分析

**三种服务端批量处理方案的对比**

在实现"服务端大批量处理"时有三种主流方案，各有适用场景：

| 方案 | 游标载体 | 并发安全 | 断点续传 | 适用场景 |
|------|---------|---------|---------|---------|
| skip + limit（传统偏移） | 无，每次计算偏移量 | 需加锁或单线程 | 需记录 offset | 小数据量、无并发要求 |
| 记 `_id` 游标（Keyset） | 上一批最后一条 `_id` | 需加锁防重 | 需持久化游标 | 前端翻页、数据不变 |
| 状态机游标（本方案） | `status` 字段 | 天然（update 原子） | 天然（重置 pending） | 服务端批处理、可并发 |

状态机游标的核心优势是**游标存在数据库里**而不是外部调用方手里，天然具备持久化和并发安全特性，不需要额外的分布式协调。

**状态流转图与错误处理机制**

```
                    ┌─────────────────┐
   初始导入         │                 │
  ─────────────────>│    pending      │<──── 僵尸检测重置
                    │                 │
                    └────────┬────────┘
                             │ GetLimitPendingConfirmeds 取出
                             │ UpdateStatus("processing") 标记
                             ▼
                    ┌─────────────────┐
                    │   processing    │──── 超时未完成 ──▶ 僵尸检测
                    │                 │
                    └────────┬────────┘
                    handler  │ 执行成功         执行失败
                             ├──────────────┬──────────────┐
                             ▼              ▼              ▼
                    ┌──────────────┐  ┌──────────┐  ┌──────────────┐
                    │  completed   │  │ failed   │  │  pending     │
                    │（成功完成）   │  │（永久失败）│  │（可重试）     │
                    └──────────────┘  └──────────┘  └──────────────┘
```

不同错误的处理策略：
- **临时错误**（网络超时、下游 gRPC 抖动）→ 不更新状态，下一次循环重取（天然重试）
- **永久错误**（数据格式错误、业务校验失败）→ 标记 `failed` + 记录 `error_message`，不阻塞后续数据
- **进程崩溃** → `processing` 状态超时后被僵尸检测重置为 `pending`，下次重启续传

**为什么不用消息队列（Kafka/RabbitMQ）来替代这套设计？**

消息队列能解决同样的问题，但引入了额外依赖。对比：

- **引入 MQ**：需要 Kafka/RabbitMQ 部署，消息序列化/反序列化，消费者组管理，消息重放，DLQ（死信队列），运维复杂度大幅上升
- **状态机游标**：复用已有的 MongoDB，无新依赖，代码量 < 100 行，运维成本为零

在数据量 < 100 万条、处理速度要求非极致的场景下，状态机游标是"够用且最简单"的方案。引入 MQ 是过度设计。

**`GetLimitPendingConfirmeds` 的索引设计**

状态机游标的性能依赖于 `status` 字段的索引。`import_task_items` 集合建了复合索引 `{task_id:1, status:1, item_type:1}`：

```go
// 查询：task_id=X AND status="pending" AND item_type="host" LIMIT 1000
// 索引：{task_id:1, status:1, item_type:1}
// 命中三个等值条件（ESR 中的全 E），扫描量 = 返回量 = 1000，比值 1:1
```

这就是为什么每批耗时恒定 < 80ms 而不随批次增长——每次查询的扫描量始终等于返回量，不受已处理数据量的影响。

### 面试官追问

**Q：这和游标分页（记录上一页最后一个 _id）有什么区别？**

A：本质思路一样，都是避免 skip。区别在于游标的载体不同：
- 游标分页：游标是"上一页最后一条记录的 `_id`"，存在调用方（前端/上层服务）手里
- 状态机分页：游标是"文档的 `status` 字段"，存在数据库里

状态机方案更适合**服务端批量处理**场景（无调用方传游标），而且天然解决并发安全问题。游标分页更适合**前端翻页**场景（前端缓存游标）。本项目两种都有用到，各自用在最合适的地方。

**Q：如果 handler 处理失败，`processing` 的数据怎么恢复？**

A：`import_worker.go` 在任务启动时有一个"僵尸检测"步骤：扫描当前任务下所有 `processing` 状态且 `update_time` 超过阈值（比如 10 分钟）的条目，批量重置为 `pending`。这样宕机或处理失败后，下次重启时这些数据会被重新取到并重试，保证不丢数据。
