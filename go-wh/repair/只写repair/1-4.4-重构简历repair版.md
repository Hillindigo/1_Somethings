## 一、项目背景（面试官最想知道：公司为什么要做这个？）

### 业务痛点驱动

该公司为国际化的设备维修服务提供商，在全球多个地区（亚洲、欧洲等）部署了数百个维修仓库，管理数万台服务器主机及配件（部件）的全生命周期流转——从采购入库、项目配发、故障维修、换件更换，到最终报废出库。

**核心痛点有三：**

1. **资产归属混乱**：原有系统无仓库层级概念，所有资产平铺管理，跨地区调拨时无法清晰追踪资产属于哪个地区→哪个国家→哪个具体仓库，导致盘点误差高、责任归属不清。

2. **Excel 人工导入效率极低**：运营人员每次需批量导入数千条资产数据（新采购入库 / 维修后回库），旧系统采用逐行解析 + 逐条插入的方式，5000 条数据需耗时 **5 分钟以上**，且没有重复检测和错误回溯机制，数据质量无法保证。

3. **工单流程与资产状态脱节**：维修工单完成后，部件换上/换下的状态变更依赖人工手动更新，滞后严重，导致仓库库存账实不符，影响二次调拨决策。

**公司的解决思路**：重新建设一套以"资产全生命周期"为核心的维修管理平台，拆分为维护服务（资产/仓库/项目数据）、工作流服务（工单流程）两个微服务，通过 gRPC 打通数据链路，实现仓库资产的实时追踪与批量化运营。

### 职责 1：三级仓库层级体系的从 0 到 1 设计与实现

#### S — 背景与问题

公司仓库分布在全球多个地区，业务上要求支持「洲级 → 国家级 → 具体仓库」三层管理。原系统无层级概念，资产全部平铺存储，跨层级调拨无法追踪归属，权限只能粗放地按"有/无"控制，无法实现仓库粒度的角色管理。随着仓库数量增长到 300+，管理混乱问题愈发突出。

#### T — 我的技术任务

从 0 到 1 独立设计并实现三级仓库管理模块，包含 MongoDB 数据模型设计、20+ 个 gRPC 接口开发、细粒度权限体系、拖拽排序、跨层级移动五大核心能力，需在保证数据一致性的前提下支持树形结构的高效查询与维护。

#### A — 技术方案与难点攻克

**数据模型：`parent_id + full_path` 混合方案**

单纯用 `parent_id` 嵌套存储，查询完整路径需多次递归查库；单纯用物化路径（`full_path`）则更新父节点时子节点全部要改。综合考虑后采用混合方案：
- `parent_id`：存储直接父级业务 ID（如 `WH00002`），快速定位直接子节点，O(1)
- `full_path`：冗余完整路径字符串（如 `亚洲/中国/北京仓库`），路径展示和前缀模糊搜索单次查询即可
- `sort_order`：整型序号，支持同层级拖拽排序，重排只需更新同层兄弟节点

**跨层级移动的级联更新**：移动含子节点的仓库时，所有子孙的 `full_path`、`level` 均需级联更新。若逐条更新则 O(n) 次写操作，性能不可接受。解决方案：用 MongoDB `$regex` 按旧路径前缀匹配所有子孙节点，配合 `BulkWrite` 的 `UpdateMany` 一次批量更新，将 O(n) 次写操作压缩为 1 次，同时在 Service 层做循环依赖检测（不能把节点移动到自己的子节点下）。

**细粒度权限体系**：设计独立的 `user_warehouse` 关联表，字段为 `{warehouse_id, user_id, role}`，将用户-仓库-角色三者解耦。同一用户在不同仓库可拥有不同角色，且仓库主表不存储成员数组，避免成员变更导致仓库文档频繁更新。批量场景使用 `BatchGetWarehouseAdminUserIDs` 一次查询多仓库管理员列表，消除 N+1 查询。

**树形结构内存构建**：一次性查出所有未删除节点，在内存中用 `map[warehouseID]*TreeNode` 建图，O(n) 完成树形装配。非管理员用户先查可见仓库集合，再向上补全祖先节点使路径连通，过滤出可见子树，整个过程只需 2 次 DB 查询。

#### R — 可量化成果

- 实现 20+ 个仓库管理 gRPC 接口，覆盖 CRUD、树形查询、成员管理、权限校验、拖拽排序、变更历史全流程
- 300+ 节点完整树形查询：**仅 2 次 DB 查询，耗时 < 50ms**
- 含 10+ 子节点的跨层级移动：**1 次 BulkWrite 完成，耗时 < 200ms**
- 批量管理员权限查询（100 个仓库）：**< 100ms**

**简历一句话版本（直接贴）**：
> 从 0 到 1 设计实现三级仓库管理模块，采用 `parent_id + full_path` 混合数据模型解决路径展示与层级查询的双重需求；通过 MongoDB `$regex` 前缀匹配 + BulkWrite 将跨层级移动的子节点级联更新从 O(n) 次写操作压缩为 1 次批量操作；设计独立 `user_warehouse` 权限表消除 N+1 查询，300+ 节点树形查询仅需 2 次 DB 交互，耗时 < 50ms。

### 职责 2：Excel 批量导入全链路重构（性能提升 20 倍）

#### S — 背景与问题

运营人员每次批量导入 5000+ 条新采购资产时，旧方案存在四大致命缺陷：
1. **全量加载内存浪费**：Excel 文件上传后全量读入内存再解析，5000 行文件经 excelize 解析后内存峰值约 50MB+，且随文件增大线性增长，存在潜在 OOM 风险
2. **逐条插入性能极差**：每解析一行就执行一次 `InsertOne`，5000 条数据产生 5000 次网络往返，耗时超过 5 分钟
3. **无重复检测，数据质量无法保证**：同一台设备（相同 `FactorySN`）被重复导入多次，仓库库存数量虚高；用户分批上传同一文件时，同一资产被重复创建
4. **无错误回溯和任务状态**：某行校验失败后整个导入中断，已成功的行和失败的行无记录，用户需要从头重传

#### T — 我的技术任务

主导从零重构 Excel 导入全链路，需同时解决性能、数据质量、可观测性三个维度的问题：
- **性能**：流式读取消除 OOM，批量写入替代逐条插入，副作用异步化不阻塞主流程
- **数据质量**：建立三层去重机制，覆盖文件内重复 / 跨批次重复 / 历史数据重复三种场景
- **可观测性**：引入任务状态机支持断点续传、进度展示和精确的行级错误回溯

#### A — 技术方案与难点攻克

**① MinIO 流式解析替代全量加载**

文件存储在 MinIO，通过封装 `StreamObjectLinesByExcelize` 以 Go Channel 流式逐行推送解析结果，主 Goroutine 消费 Channel，内存中同一时刻只保留当前处理批次（1000 行），峰值内存从全量加载的 ~50MB 降至约 8MB。第一行专门做表头校验（列名、列顺序与模板严格对比），发现列缺失/错误立即关闭 Channel 并 Fail-Fast，不浪费后续解析资源。

**② 三层查重机制，覆盖所有重复场景**

每批数据进入 `CheckData` 函数时，依次执行三层查重，每层发现问题立即短路：
- **L1 文件内去重**：用 `map[string]int` 统计本批次 `FactorySN` 出现次数，重复则直接 Fail-Fast 终止，报告具体行号
- **L2 任务内去重**：查询当前 `import_task_items` 集合中已存在的 `FactorySN` 集合，防止用户同一次导入任务中分多次上传同一份文件；此场景数据库去重发现不了（第一批尚未写入 `asset_info`）
- **L3 数据库去重**：批量查询 `asset_info` 集合（`GetByFactorySNBatch`，一次查询 vs 逐条查询），命中则分类为 `duplicate`（等待用户在前端决策是否覆盖更新），未命中则分类为 `new`

**③ ImportTask + ImportTaskItem 两级状态机**

引入任务级别（`ImportTask`）和行级别（`ImportTaskItem`）两张集合：
- `ImportTask`：记录整体进度（`import_count / total_count`），前端轮询展示实时进度条
- `ImportTaskItem` 状态流转：`pending → processing → completed / failed`
  - `error_msg` 字段记录具体失败原因（精确到列名和行号），前端可展示"第 42 行 FactorySN 格式错误"
  - 服务重启后，`processing` 状态的 Item 可被检测为"僵尸"并回滚到 `pending` 断点续传，彻底解决旧版"重启丢进度"问题

**④ GenerateBatchAssetCodes + InsertMany 批量化**

旧方案：为每条新资产单独查一次数据库获取当前最大编号再 +1，5000 条资产产生 5000 次串行查询。

重构方案：
- `GenerateBatchAssetCodes(n int)`：一次查询获取当前编号上界，在内存中连续生成 n 个编号，整个批次只需 1 次 DB 交互
- MongoDB `InsertMany`（即 `BulkCreate`）：每批 1000 条一次性写入，将 N 次网络往返压缩为 ⌈N/1000⌉ 次

**⑤ sideEffectDispatcher：副作用 Goroutine Pool 异步化**

变更历史创建、SN 绑定、父级关联更新、资产完整度计算等副作用操作不影响主库存数据，但逐条同步执行会显著拉长主流程耗时。设计 `sideEffectDispatcher` 结构体，基于带缓冲 Channel（信号量）+ `sync.WaitGroup` 实现固定并发的 Goroutine Pool：

```go
// sideEffectDispatcher 控制副作用并发度，避免打爆下游 gRPC 服务
type sideEffectDispatcher struct {
    sem chan struct{}    // 容量 8，信号量控制最大并发 Goroutine 数
    wg  sync.WaitGroup // 等待所有副作用完成后再标记任务 completed
}

func (d *sideEffectDispatcher) Submit(fn func()) {
    d.wg.Add(1)
    d.sem <- struct{}{}        // 获取信号量，超过 8 个并发时阻塞
    go func() {
        defer func() {
            <-d.sem            // 释放信号量
            d.wg.Done()
        }()
        fn()
    }()
}
```

主流程（BulkCreate）完成后立即返回，副作用异步在 Pool 中执行；`Run()` 末尾调用 `dispatcher.Wait()` 阻塞直到所有副作用完成，再将 `ImportTask` 状态置为 `completed`。并发度上限 8 经过压测确定，既保证副作用吞吐，又不打爆下游变更历史服务的限流阈值。

#### R — 可量化成果

- 导入 5000 条数据耗时：**5 分钟 → 15 秒，性能提升 20 倍**
- 内存峰值：**~50MB（全量加载）→ ~8MB（流式解析，同一时刻只保留 1000 行批次）**
- 资产编号生成：**N 次串行 DB 查询 → 1 次批量查询**
- 副作用异步化后主流程（BulkCreate 阶段）耗时再降约 **50%**
- 完整错误追踪：每行失败原因精确记录至 `ImportTaskItem.error_msg`，前端可展示行级错误；服务重启后支持断点续传

**简历一句话版本（直接贴）**：
> 主导重构 Excel 批量导入全链路，通过 MinIO Channel 流式解析将内存峰值从 ~50MB 降至 ~8MB；设计三层去重机制（文件内/任务内/DB）覆盖所有重复场景；引入 ImportTask 两级状态机实现断点续传与行级错误回溯；GenerateBatchAssetCodes + InsertMany 将编号生成从 N 次串行查询优化为 1 次；sideEffectDispatcher（Channel 信号量 Goroutine Pool）异步化副作用，整体导入耗时从 5 分钟压缩至 15 秒，性能提升 20 倍。

### 职责 3：工单工作流与仓库资产状态的实时联动集成

#### S — 背景与问题

维修工单流程由独立的 `workflow-service` 微服务驱动，工单中涉及部件的换上/换下节点执行完成后，需要将资产的仓库归属、质量状态、父子绑定关系同步回 `maintenance-service`。两个服务数据库完全隔离（各自独立的 MongoDB 实例），旧流程依赖人工在工单完成后手动登记维护系统，平均滞后 2~4 小时，导致仓库库存账实不符，影响二次调拨决策；且无任何操作来源记录，出现库存异常时无法追溯是哪张工单触发的。

#### T — 我的技术任务

在两个独立微服务之间建立 gRPC 回调集成层，要求：

- `workflow-service` 作为调用方，在工单节点执行时通过统一 gRPC 接口触发 `maintenance-service` 中的资产状态变更，调用方无需感知维护服务内部接口细节
- 支持工单节点携带动态参数（资产列表、目标仓库等），且参数规模可能达到 100+ 条资产，不能通过 gRPC 消息体直接膨胀传递
- 实现变更历史与工单实例的双向溯源，既能从工单查到影响了哪些资产，也能从资产变更记录反查触发的工单

#### A — 技术方案与难点攻克

**① GRPCMaintenanceHandler：统一路由分发层**

工作流引擎以 URL 路径（如 `/maintenance/asset/quality-update`）+ 通用参数结构体调用维护服务，由 `GRPCMaintenanceHandler` 解析路径作为路由 Key，将请求分发到对应的具体业务方法：

| 路由路径 | 业务方法 | 场景 |
|---------|---------|------|
| `/asset/quality-update` | `UpdateAssetQualityAndWarehouse` | 换件后更新资产质量和仓库归属 |
| `/asset/batch-in` | `BatchAssetsToWarehouse` | 维修完成批量入库 |
| `/asset/batch-unbind` | `BatchUnbindComponent` | 换下部件批量解绑父机 |
| `/asset/bind` | `BindComponentToMachine` | 换上部件绑定新父机 |
| 以及另外 4 类路由 | ... | 覆盖所有工单节点资产操作 |

这种路由器模式使工作流引擎对维护服务内部接口完全解耦，新增业务类型只需在路由表注册，无需修改工作流引擎代码。

**② Redis 跨节点参数传递，解决大批量参数膨胀问题**

工单中批量操作的资产列表（如"批量入库 120 台机器"）若直接塞入 gRPC 消息体，序列化后体积过大且多节点间透传会持续膨胀。解决方案：

- 上游工单节点执行完成时，将资产 ID 列表序列化写入 Redis（Key 格式：`workflow:params:{instance_code}:{node_code}`，TTL 24 小时）
- 下游 `GRPCMaintenanceHandler` 收到调用时，从 gRPC 请求中取 `redis_key` 字段，再从 Redis 读取完整参数列表
- 此方案将 gRPC 消息体控制在固定大小（仅传 key），参数规模对网络传输零影响

**③ 变更历史与工单实例双向溯源**

gRPC 请求统一携带工作流上下文（`instance_code`、`instance_name`、`node_name`、`operator`），`maintenance-service` 在创建资产变更历史记录时同步写入工单来源字段：

```go
// ChangeHistory 记录中同步写入工单溯源信息
changeHistory := &ChangeHistory{
    AssetID:      asset.ID,
    ChangeType:   ChangeTypeQualityUpdate,
    Operator:     req.Operator,
    // 工单溯源字段
    WorkflowInstanceCode: req.InstanceCode,
    WorkflowInstanceName: req.InstanceName,
    WorkflowNodeName:     req.NodeName,
}
```

最终实现：从资产变更历史点击"来源工单"可跳转工单详情（双向溯源），排查库存异常时平均定位时间从人工比对（数小时）降至秒级。

#### R — 可量化成果

- 实现 8 个工作流回调 gRPC 接口，覆盖资产质量更新、批量入库、批量解绑、父子绑定等全部工单节点操作场景
- 单次工单最多可驱动 **100+ 条资产**的状态批量流转，通过 Redis 中转参数保证 gRPC 消息体不膨胀
- 资产状态同步延迟：**人工录入（2~4 小时）→ gRPC 回调自动同步（< 1 秒）**
- 库存异常排查效率：引入双向溯源后，定位触发工单从**人工比对数小时**降至**秒级查询**

**简历一句话版本（直接贴）**：
> 设计 `GRPCMaintenanceHandler` 回调路由层解耦工作流引擎与维护服务，以 URL 路径分发 8 类资产操作；通过 Redis 跨节点中转大批量参数（TTL 24h），将 gRPC 消息体控制在固定大小；在变更历史中写入工单上下文实现双向溯源，资产状态同步延迟从人工录入的 2~4 小时降至 gRPC 回调的秒级。

### 职责 4：仓库维度的 MongoDB 索引与分页查询性能优化

#### S — 背景与问题

随着单仓库管理资产规模增长至 10000+ 条，仓库资产列表接口（支持仓库、资产类型、质量状态多条件筛选 + 分页）出现明显性能问题：

- **首页加载**：单次列表查询耗时 2s+，运营人员反映页面"打开转圈"
- **深翻页崩溃**：翻到第 50~100 页时偶发超时报错，耗时达 5s+
- **模糊搜索极慢**：在搜索框输入 `FactorySN` 关键字触发 `$or + $regex` 查询，10000 条集合下耗时接近 3s

用 `explain("executionStats")` 分析后发现：**3 类查询均走全表扫描（`COLLSCAN`）**，`asset_info` 集合无任何索引，每次查询都要扫描全部文档。

#### T — 我的技术任务

系统性地排查 `asset_info` 集合的慢查询根因，设计并落地索引策略，同时将分页方案从偏移分页迁移至游标分页，消除深翻页的性能悬崖。具体目标：

- 首页列表查询 < 200ms
- 深翻页（第 100 页）< 100ms，且不随页码增大而劣化
- 模糊搜索结合索引，降低全表扫描比例

#### A — 技术方案与难点攻克

**① `explain()` 根因定位**

对线上高频慢查询执行 `explain("executionStats")`，重点观察三个指标：

- `totalDocsExamined`：全表扫描时等于集合总文档数（10000+），远大于 `nReturned`（20 条）
- `stage: COLLSCAN`：确认无可用索引
- `executionTimeMillis`：首页 2100ms、第 100 页 5400ms（skip 量大时更慢）

**② 复合索引设计（遵循 ESR 原则）**

针对最高频查询模式，按 **ESR 原则**（Equality 等值字段在前 → Sort 排序字段在中 → Range 范围字段在后）设计复合索引：

```js
// 主查询索引：等值过滤（warehouse_id + asset_type）→ 排序（create_time 降序）
db.asset_info.createIndex(
  { warehouse_id: 1, asset_type: 1, create_time: -1 },
  { name: "idx_warehouse_type_ctime", background: true }
)

// 辅助索引：质量状态过滤场景（warehouse_id + quality_status + create_time）
db.asset_info.createIndex(
  { warehouse_id: 1, quality_status: 1, create_time: -1 },
  { name: "idx_warehouse_quality_ctime", background: true }
)
```

建索引后再次执行 `explain()`，`stage` 从 `COLLSCAN` 变为 `IXSCAN`，`totalDocsExamined` 从 10000+ 降至与 `nReturned` 接近。

**③ 游标分页替换 skip+limit，消除深翻页性能悬崖**

`skip(N)` 的本质是让 MongoDB 扫描并丢弃前 N 条文档，第 100 页（每页 20 条）需丢弃 1980 条，第 N 页复杂度 O(N)。

改用 `create_time + _id` 双字段游标分页：

```go
// 游标分页查询：上一页最后一条记录的 (create_time, _id) 作为游标
filter := bson.M{
    "warehouse_id": warehouseID,
    "$or": []bson.M{
        // create_time 严格小于游标时间
        {"create_time": bson.M{"$lt": cursor.CreateTime}},
        // create_time 相同时，用 _id 保证严格全序（避免同秒数据乱序）
        {"create_time": cursor.CreateTime, "_id": bson.M{"$lt": cursor.ID}},
    },
}
// 命中复合索引，每次都从索引精确定位，耗时恒定
```

使用 `create_time + _id` 组合而非单 `_id`，原因：`_id` 是 ObjectID，其时间精度为秒级，同一秒内插入的多条记录 `_id` 排序不等于业务排序；`create_time`（毫秒级时间戳）+ `_id` 双字段组合才能保证严格全序，避免翻页数据重复或遗漏。

**④ 模糊搜索优化：高选择性字段前置 + 避免全字段 $or**

原方案：`$or: [{factory_sn: /keyword/}, {asset_code: /keyword/}, {fake_sn: /keyword/}, {pn: /keyword/}]`，四个字段均为前缀不定的正则，无法命中索引，必须全表扫描。

优化策略：
- 业务分析后，`factory_sn` 是用户最常搜索字段，且数据分布选择性最高（几乎唯一）；对 `factory_sn` 单独建索引 `{factory_sn: 1}`，纯等值或前缀搜索（`/^keyword/`）可命中
- 模糊搜索场景（包含匹配 `/keyword/`）仍走 `COLLSCAN`，但通过先用 `warehouse_id` 等值条件缩小扫描集合范围（利用复合索引），再在结果集内做正则过滤，将实际扫描文档数从 10000+ 压缩至该仓库下的文档数（通常 500~2000 条），耗时大幅下降

#### R — 可量化成果

- 首页列表查询（`warehouse_id` 等值 + 分页 20 条）：**2100ms → 45ms**，降幅 97%
- 深翻页（第 100 页）查询：**5400ms → 恒定 < 80ms**，彻底消除性能悬崖
- `totalDocsExamined / nReturned` 比值：**500:1（全表扫描）→ 约 1.2:1（索引扫描）**
- 模糊搜索耗时（含 `warehouse_id` 前置过滤）：**2800ms → < 300ms**

## 五、面试官会追问的点（提前准备）

### Q1：sideEffectDispatcher 的并发数为什么设置成 8，不是更多？

> **答**：8 是经验值，需结合服务器 CPU 核数和下游服务（ChangeHistory、Completeness 等 gRPC 服务）的限流配置综合考量。并发太高会打爆下游服务或本地 goroutine 调度压力；太低则副作用积压拖慢整体完成时间。实际生产中应通过压测确定最优值，这里的 8 是默认保守值，`newSideEffectDispatcher` 接受参数可动态调整。

### Q2：ImportTaskItem 的状态机设计有什么意义，直接处理完删掉不行吗？

> **答**：不行。保留状态机有三个核心价值：① **断点续传**：服务宕机重启后，`processing` 状态的 Item 可被检测为"僵尸"并回滚到 `pending` 重新处理；② **精确错误回溯**：每行数据的 `error_msg` 字段记录具体失败原因，前端可展示哪行第几列出了什么问题；③ **进度可观测**：前端轮询 `ImportTask.import_count / total_count` 展示实时进度条，比异步黑盒体验好得多。

### Q3：三层查重中，"任务内去重"和"数据库去重"有什么区别，为什么要分开？

> **答**：**数据库去重**是查 `asset_info` 集合，判断这条数据是否已是"存量资产"，命中则走 duplicate（覆盖更新）路径。**任务内去重**是查当前 `import_task_items` 表，防止用户在同一次导入任务中分多次上传同一份文件时，同一个 FactorySN 被重复插入两条 `new` 类型的 Item——这种场景数据库查重发现不了，因为第一批还没有写入 `asset_info`，第二批来的时候查库还是查不到。

### Q4：full_path 字段在跨层级移动时，子节点级联更新会不会有并发问题？

> **答**：当前实现是单线程串行执行（`runPhase` 中批次处理是顺序的），移动操作本身在 Service 层也是同步阻塞的，不存在并发冲突。如果未来要支持高并发的仓库移动，应加分布式锁（Redis SetNX）保护 `warehouse_id` 维度的写操作，并考虑用乐观锁（`version` 字段）检测并发修改。

### Q5：游标分页为什么用 `create_time + _id` 双字段，单用 `_id` 不行吗？

> **答**：不行，原因在于排序语义不一致。`_id` 是 MongoDB ObjectID，其时间精度只到**秒级**，同一秒内批量插入（如 `InsertMany` 一次插入 1000 条）的多条记录，`_id` 的时序不能严格等价于业务上的 `create_time` 顺序。如果单用 `_id` 做游标，同一秒内的数据在翻页时可能出现乱序、重复或跳过。使用 `create_time`（毫秒级时间戳）+ `_id` 组合：先按 `create_time` 降序，相同毫秒内再按 `_id` 降序，才能保证严格的全序关系，翻页不丢数据。**局限**：① 不支持随机跳页；② 需要前端缓存上一页最后一条的 `(create_time, _id)` 作为游标传入下次请求。对于仓库资产列表这种顺序翻页、数据变化不频繁的场景，这些局限完全可以接受。

### Q6：Redis 跨节点参数传递为什么不直接放在 gRPC 消息体里？TTL 设多久合理？

> **答**：主要有两个问题：① **消息体膨胀**：一个批量操作可能涉及 100+ 条资产 ID，序列化后几十 KB，如果有多个工单节点串联传递，每次都要在 gRPC 请求中携带完整列表，链路越长膨胀越严重，还增加了序列化/反序列化开销；② **gRPC 消息大小限制**：默认最大 4MB，大批量场景存在超限风险。通过 Redis 中转只传 Key，消息体始终是固定的几十字节。
> TTL 设为 24 小时的理由：工单流程一般在几分钟到几小时内完成，24 小时足够覆盖所有正常场景；过期后的 Key 被自动清理，不占用内存；若工单因异常搁置超过 24 小时，参数已过期，工单重试时应重新写入新 Key，这也符合幂等性设计要求。

### Q7：ESR 原则是什么？你的索引为什么这样排序字段顺序？

> **答**：ESR 是 MongoDB 官方推荐的复合索引字段排列原则：**E（Equality）等值字段在最前** → **S（Sort）排序字段在中间** → **R（Range）范围字段在最后**。原因：索引树是 B+ Tree，等值过滤先缩小范围，排序字段紧跟等值字段可以利用索引的有序性直接完成排序（避免内存 sort），范围字段放最后是因为一旦有范围条件，索引的连续性就被打破，后续字段无法再有效利用索引。
> 以 `{warehouse_id: 1, asset_type: 1, create_time: -1}` 为例：`warehouse_id` 和 `asset_type` 是等值过滤（E + E），`create_time` 是排序字段（S）。如果把 `create_time` 放在 `asset_type` 前面，查询时就无法同时利用等值过滤和排序，会退化为内存排序，性能大幅下降。
