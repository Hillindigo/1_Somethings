## 一、标准简历模板

### 项目名称
**AIDC 智慧数据中心 IoT 实时数据平台**

### 技术栈
Go 1.22+, gRPC + Protobuf, Gin, Kafka (segmentio/kafka-go), Redis (go-redis), PostgreSQL + GORM, VictoriaMetrics, MinIO, Zstd (klauspost/compress), Casdoor SSO + JWT, Prometheus, Snowflake ID, Docker, Kubernetes + Helm, GitLab CI/CD

### 项目描述
面向数据中心基础设施（DCIM）场景的 IoT 实时数据平台，采用 Go 微服务架构（5 个核心服务），覆盖从 BMS 设备数据采集、预处理、实时消费、指标计算到平台管理的完整数据链路。平台日均处理 [待补充] 万条设备数据，支撑 [待补充] 个数据中心、[待补充] 万级监控点位的实时监控与能耗分析。

### 主要职责

1. **从 0 到 1 架构并交付 IoT 数据预处理网关（aidc-pretreat-service）**，设计大小文件分治策略（<10MB 直发 Kafka，≥10MB 流式 Zstd 压缩 + MinIO 对象存储），基于 `sync.Pool` 实现 Zstd 编码器复用的零拷贝流式压缩管道，构建企业级异步任务队列架构（TaskQueue 10000 容量 + 100 Worker 并发 + 背压控制），集成 Prometheus 全链路埋点（请求数/延迟分布/压缩率/队列深度），支持流式分片、MinIO 异步上传、一次性上传、压缩上传 4 种数据接入模式。

2. **主导设计高性能 IoT 实时数据处理服务（aidc-process-service）**，实现多 Topic Kafka 并行消费框架（每 Topic 独立 goroutine + 独立 Reader），基于工厂模式设计消息解析器（JinhuaParser / RawParser / BMSParser 热插拔），设计 Token Bucket per-topic 差异化限流策略，并发双写 Redis（Hash 结构 + Pipeline 批量 + Lua 脚本原子性时间戳 CAS 解决横向扩容时序冲突）与 VictoriaMetrics（InfluxDB line protocol 分批写入 + DedupCache 时间窗口去重），通过手动 offset 提交保证 at-least-once 消费语义。

3. **架构并交付 DCIM 指标实时计算引擎（aidc-metric-service）**，设计双模式计算架构：流式模式（Kafka calc-task 消费 → 公式求值 → 双写 → 级联触发）+ 定时模式（7 种时间粒度：10s/1min/5min/hour/day/month/交接班），基于 AST 预编译的公式引擎（支持 SUM/AVG/MAX/MIN/COUNT 聚合函数 + 四则运算），Kahn 算法拓扑排序解决计算点 DAG 依赖（最大级联深度 10 层），双层缓存读取（R2 时间点精确值 MGET → R1 Hash HMGET 降级 + 1h 过期检测），DLQ 死信补偿（最多 5 次重试，按错误类型分策略补偿），每 5 分钟 atomic.Value 热加载公式索引实现零停机更新。

4. **重构并独立北向接口数据同步服务（aidc-north-sync）**，从 process-service 解耦为独立微服务，设计三表分离数据模型（north_configure / north_sync_config / sync_task），架构动态多配置调度器（每配置独立 Cron 实例 + map[configName]→cronInstance），实现 ConfigService ↔ Scheduler 联动（CRUD 自动触发调度器启停，通过 SchedulerReloader 接口解耦避免循环依赖），基于版本号增量同步 + 分页拉取（每页 500 条）+ 事务批量写入 PostgreSQL，异步任务管理器（TaskQueue + 状态机 + panic 恢复）。

5. **主导开发 AIDC 平台管理服务（aidc-platform-service）**，注册 12+ gRPC 服务模块（Auth / User / AD User / Menu / Role / Dict / DcomLocation / MetricPoints / DcimMetrics / DataSync / NorthSyncStatistics / StressTest），集成 Casdoor SSO + JWT 认证拦截器（支持 gRPC 全路径 / HTTP 路径 / 路径后缀三种白名单匹配），实现 Redis 分布式锁（SETNX + TTL）保护计算点初始化（防多 Pod 重复执行），封装 VictoriaMetrics 查询构建器（VictoriaQueryBuilder），Snowflake ID 生成器（15 位机器 ID + 7 位序列号，支持 32768 节点 × 128 ID/ms）。

## 三、主要职责 STAR 深度重构

### 职责 1：IoT 数据预处理网关（aidc-pretreat-service）

> **S（背景）**：平台需要接收来自 [待补充] 个机房的 BMS 设备实时数据推送，单次请求数据量从几 KB 到数百 MB 不等，Kafka 默认 1MB 消息限制无法直接承载大数据场景，且高峰期并发请求达 [待补充] QPS，需要同时支持同步快速响应和异步大数据处理两种模式。
>
> **T（任务）**：设计一套能够自动适配不同数据量级的预处理网关，同时保证高并发下的稳定性、吞吐量和可观测性。
>
> **A（方案）**：
> - **架构**了基于 Gin 的 HTTP 数据预处理网关，设计 4 种数据接入模式：流式分片（`OptimizedStreamChunkProcessor`，800KB 阈值精确分片 + 并发信号量控制 + 批量 Kafka 发送）、MinIO 异步上传、一次性上传、压缩上传
> - **研发**了智能数据分流引擎，以 10MB 为阈值自动选择处理策略——小数据直发 Kafka，大数据经 Zstd 流式压缩后存入 MinIO 并向 Kafka 发送引用消息；压缩器使用 `sync.Pool` 复用 `zstd.Encoder` 实例，支持 4 级压缩调优（SpeedFastest → SpeedBestCompression）
> - **设计**了企业级异步任务队列架构（`TaskQueue` + `Worker Pool`）：队列容量 10000、100 个并发 Worker、最大待处理 50000 任务，支持背压控制（队列满时返回 503 拒绝）、优雅关闭（`context.WithCancel` + `sync.WaitGroup`）、健康检查（待处理 < 90% 容量）
> - **集成** Prometheus 全链路监控：`pretreat_requests_total`（请求数 by status/topic）、`pretreat_request_duration_seconds`（延迟直方图）、`pretreat_request_size_bytes`（请求大小分布）、`pretreat_compression_ratio`（压缩率分布）
>
> **R（结果）**：HTTP 请求平均响应时间控制在 [待补充] ms 以内（异步模式立即返回 202），数据预处理吞吐量达 [待补充] MB/s，Zstd 压缩使网络带宽占用降低约 [待补充]%，Worker Pool 利用率稳定在 [待补充]%。

### 职责 2：IoT 实时数据处理服务（aidc-process-service）

> **S（背景）**：平台需实时消费来自多个 Kafka Topic（jinhua、raw、bms-* 等）的设备数据流，日均消息量 [待补充] 万条，且不同 Topic 数据量差异巨大（bms 系列 Topic 数据量是 jinhua 的 [待补充] 倍），横向扩容时多实例并发写入 Redis 导致数据时序冲突（旧数据覆盖新数据）。
>
> **T（任务）**：构建高吞吐、高可靠的实时数据消费处理引擎，将设备数据双写到 Redis（实时查询）和 VictoriaMetrics（历史分析），并解决横向扩容的数据时序一致性问题。
>
> **A（方案）**：
> - **研发**了基于 segmentio/kafka-go 的多 Topic 并行消费框架：每个 Topic 独立 `goroutine` + 独立 `kafka.Reader`，互不阻塞；支持动态 Topic 列表配置和 Handler 注册（`RegisterHandler(topic, handler)`）
> - **设计**了基于 `golang.org/x/time/rate` 的 Token Bucket per-topic 差异化限流策略：`TopicRateLimiter` 为每个 Topic 维护独立限速器（`map[topic]*rate.Limiter`），支持运行时动态调整速率和突发容量
> - **实现**了解析器工厂模式（`ParserFactory`）：根据 Topic 自动匹配解析器（`JinhuaParser` / `RawParser` / `BMSParser`），支持 inline 数据直接解析和 MinIO reference 数据远程拉取两种模式
> - **攻克** Redis 横向扩容时序冲突难题：设计 Lua 脚本原子性时间戳 CAS（`luaCheckAndSetByTimestamp`）——HGET 现有数据 → cjson.decode 提取 dp_timestamp → 比较新旧时间戳 → 仅当新 > 旧时才 HSET，配合 Pipeline 批量执行（每批 1000 条），支持 NOSCRIPT 自动重载；同时提供直接写入模式（`writeDevicesBatchDirect`）作为非扩容场景的高性能替代
> - **优化** VictoriaMetrics 写入：分批写入（maxBatchSize 100 万条）防超时 + `DedupCache` 时间窗口去重（默认间隔可配置，5 分钟清理过期缓存），避免 at-least-once 语义导致的重复写入
> - **采用**手动 offset 提交策略（处理成功后 `CommitMessages`），保证消息不丢失
>
> **R（结果）**：实现 [待补充] msg/s 的稳定消费吞吐量，Redis 实时数据查询延迟 < [待补充] ms，Lua 脚本 CAS 使横向扩容场景下数据时序冲突率降至 0，VMS 历史数据写入成功率 [待补充]%，DedupCache 去重命中率约 [待补充]%。

### 职责 3：DCIM 指标实时计算引擎（aidc-metric-service）

> **S（背景）**：数据中心有 [待补充] 个计算指标（如 PUE、机房功率、温湿度统计），指标之间存在复杂的 DAG 依赖关系（如「楼层总功率」依赖「各机房功率」），且业务方要求新增/修改计算公式后无需重启服务即可生效，计算失败需自动补偿不丢失。
>
> **T（任务）**：构建一套支持实时计算 + 定时调度双模式的流式计算引擎，解决指标依赖排序、公式热更新、计算失败补偿、级联触发、横向扩容幂等性五大核心问题。
>
> **A（方案）**：
> - **架构**了双模式指标计算引擎：
>   - **流式模式**：Kafka `aidc-calc-task` Topic 消费 → `CalcProcessor.Handle()` 10 步流水线处理（反序列化 → 幂等检查 → 分布式锁 → 快照加载 → AST 解析 → delta/rate 前态管理 → 公式求值 → Redis R1 CAS 写入 + R2 时间点精确值写入 → VMS 最佳努力写入 → 级联触发下游）
>   - **定时模式**：自研 `Scheduler` 支持 7 种计算周期（10s / 1min / 5min / hour / day / month / 交接班），每种周期独立 goroutine + `sync.WaitGroup` 优雅关闭
> - **研发**了基于 AST 的公式引擎（`formula.FormulaEngine`）：启动时 `Compile()` 将公式字符串解析为 AST 树（深度限制 20、节点限制 200），运行时 `EvaluateAST()` 递归求值（100ms 超时保护），内置 5 个聚合函数（SUM/AVG/MAX/MIN/COUNT）
> - **实现**了基于 Kahn 算法的拓扑排序（`TopologicalSort`）：将计算点按依赖层级分组，逐层并行计算；内含跨源依赖校验（`validateSourceConsistency`），循环依赖检测（排序后剩余节点 = 有环），软排除无效配置（warning + skip，不阻塞全局）
> - **设计**了双层缓存读取架构：R2 MGET（`r2:{source}:{dpid}:{ts_sec}`，时间点精确值，TTL=900s）→ R1 HMGET 降级（`{source}` Hash，最新值）→ 1h 过期数据 Staleness Check（标记 missing 丢弃），R2 miss 时对 calc 类型依赖执行 30ms 重试（解决级联竞态）
> - **设计**了级联触发机制（`buildCascadeTriggerFunc`）：计算完成后查询 `CalcDependencyIndex.GetDownstream()`，构建下游 calc-task 消息发送回 Kafka，最大级联深度 `MaxCascadeDepth=10`，配合 `CascadeRateLimiter` 防止雪崩
> - **实现**了 DLQ 补偿机制（`DLQHandler`）：按错误类型分策略（`cascade_trigger_failed` → 重触发下游、`config_missing` → 等待热加载后重入队、`formula_error` → P1 告警、`snapshot_expired` → P2 告警），最多 5 次重试，间隔 2s
> - **实现**了公式热加载（`MetricPointLoader`）：通过 `atomic.Value` 存储索引，每 5 分钟从 Redis SCAN `formula:*` 键 → HGETALL 解析 → 拓扑排序 → 原子替换索引，零停机更新
> - **实现**了幂等性保障：Redis `calc:idem:{key}`（TTL=25h 持久幂等）+ `calc:lock:{key}`（TTL=30s 并发互斥），计算完成后立即释放锁
>
> **R（结果）**：流式计算端到端延迟 < [待补充] ms（含 R2/R1 读取 + 公式求值 + 双写），定时计算 [待补充] 个指标全量计算耗时 < [待补充] s，DLQ 补偿使计算成功率从 [待补充]% 提升至 [待补充]%，公式热加载实现零停机更新，级联触发最大实测深度 [待补充] 层。

### 职责 4：北向接口数据同步服务（aidc-north-sync）

> **S（背景）**：平台需要从 [待补充] 个机房的北向 API 定时拉取设备配置数据（位置 / 设备 / 点位三级树形结构），早期同步逻辑内嵌在 process-service 中，仅支持单一配置且修改需重启服务，无法满足多机房、多数据源的动态管理需求。
>
> **T（任务）**：将同步功能独立为微服务，支持多配置管理、运行时动态调度、HTTP API 全生命周期管理，提升运维效率。
>
> **A（方案）**：
> - **主导**将同步模块从 process-service 中拆分为独立微服务（aidc-north-sync），采用 Gin + REST API 架构
> - **设计**了三表分离的数据模型：`north_configure`（北向连接信息：API URL / Token / Source）、`north_sync_config`（同步策略：Cron 表达式 / 自动同步开关 / HTTP 超时重试）、`sync_task`（执行记录：状态机 pending→running→completed/failed + 统计数据）
> - **架构**了动态多配置调度器（`Scheduler`）：每个同步配置拥有独立的 `cron.Cron` 实例（`map[configName]→*schedulerEntry`），支持运行时动态启停；Cron 执行时从 DB 重新读取最新配置（`configDAO.GetByName()`）实现热更新
> - **实现**了 ConfigService ↔ Scheduler 联动机制：配置 CRUD 自动触发调度器启停（创建 → `StartConfig()` / 删除 → `StopConfig()` / 更新 → `reloadScheduler()`），通过 `SchedulerReloader` 接口解耦避免循环依赖
> - **优化**了同步效率：基于版本号（`configVersion`）增量同步，远程版本不变时跳过同步（支持 `force=true` 强制忽略版本对比）；分页拉取（每页 500 条 + 分页终止保护 `pageNo*pageSize >= total`）+ 事务批量写入 PostgreSQL（`Begin → CreateLocations + CreateBmsDevices + CreateBmsPoints → Commit`）
> - **实现**了异步任务管理器（`SyncTaskManager`）：channel 任务队列（容量 10）+ 动态创建 `SyncService` 实例（`createSyncServiceFromRequest`），支持异步提交（`SubmitTask`）和同步等待（`SubmitTaskAndWait`），内含 `panic` 恢复和任务状态持久化
> - **实现**了位置路径自动构建：同步完成后根据 `RawLocationPath` 和 `ParentGUID` 自动计算 `LocationPath`（逗号分隔 ID 路径）、`ParentID`、`FloorID`
>
> **R（结果）**：从单一配置扩展到支持 [待补充] 个机房的独立同步配置，版本增量同步减少 [待补充]% 的无效同步请求，单次全量同步 [待补充] 个位置 + [待补充] 个设备 + [待补充] 个点位，耗时 [待补充] 秒，运维人员可通过 API 实时管理同步任务无需重启服务。

### 职责 5：AIDC 平台管理服务（aidc-platform-service）

> **S（背景）**：平台需要为前端提供统一的 API 入口，涉及用户认证、RBAC 权限控制、设备数据查询、计算点管理等多个业务域，同时部署在 K8s 多 Pod 环境下，计算点初始化需要防止多 Pod 重复执行，VictoriaMetrics 历史查询需要灵活的 PromQL 构建能力。
>
> **T（任务）**：构建一个高可靠的 BFF 层服务，作为前端与底层数据（PostgreSQL / Redis / VictoriaMetrics）之间的桥梁，提供统一认证、权限控制和数据查询能力。
>
> **A（方案）**：
> - **架构**了基于 gRPC 的平台管理服务，注册 12+ 个 gRPC 服务模块：Auth（登录/SSO）、User、AD User（域用户同步）、AuthMenu（菜单管理）、AuthRole（角色管理）、Dict（字典管理）、DcomLocation（位置管理）、MetricPoints（计算点 CRUD + Redis 缓存 + VMS 历史查询）、DcimMetrics（指标服务）、DataSync（数据同步管理）、NorthSyncStatistics（同步统计）、StressTest（压测工具）

## 五、高频面试连环追问准备

### 追问链 1：Kafka 消费与限流

**Q1：你们的 Kafka 消费者使用什么 offset 提交策略？为什么？**
> **PREP 答案**：我们使用 `LastOffset` + 处理成功后手动 `CommitMessages`，保证 at-least-once 语义。原因是 IoT 数据丢失不可接受（影响计算指标准确性），但偶尔重复可以通过 VMS DedupCache 和 Redis HSET 覆盖写来保证幂等。

**Q2：Token Bucket 限流为什么不用滑动窗口？**
> **PREP 答案**：Token Bucket 允许突发流量（burst），适合 IoT 场景中设备定时上报导致的流量尖刺。比如 BMS 系统每 10 秒上报一批数据，这时需要短暂突发处理能力。滑动窗口更适合严格匀速的场景，如 API 网关限流。

**Q3：如果某个 Topic 消费 Lag 持续增长怎么排查？**
> **PREP 答案**：①检查 handler 处理耗时（日志中的 `took` 指标）②检查该 Topic 的限流配置是否过严（`TopicRateLimiter.GetStats()`）③检查下游 Redis/VMS 写入延迟④增加消费者实例（注意 Kafka 分区数限制）⑤排查是否有大消息阻塞（raw topic 可能包含 MinIO reference 需要远程拉取）

### 追问链 2：Redis Lua 脚本与横向扩容

**Q1：Lua 脚本具体怎么实现时间戳 CAS？**
> **PREP 答案**：Lua 脚本是一个原子操作：`HGET` 获取现有 JSON → `cjson.decode` 提取 `dp_timestamp` → 比较新旧时间戳 → 仅当 `new_timestamp > existing_timestamp` 时才 `HSET`，否则返回 0 表示跳过。返回值 1=写入、0=跳过、-1=参数错误。配合 Pipeline 批量执行（每批 1000 条），兼顾时序性和性能。

**Q2：如果 Redis 重启导致 Lua 脚本 SHA 失效怎么办？**
> **PREP 答案**：代码中捕获了 `NOSCRIPT` 错误，清空 `luaScriptSHA` 触发 `SCRIPT LOAD` 重新加载，然后自动重试当前批次。整个过程对上层透明，无需人工干预。

**Q3：为什么不直接用 Redis 的 `WATCH/MULTI/EXEC` 乐观锁？**
> **PREP 答案**：`WATCH/MULTI` 不支持 Pipeline 批量执行（每个 WATCH 需要独立连接），在高吞吐场景（每批 1000 条）下性能极差。Lua 脚本天然原子性，且 `EVALSHA` 支持 Pipeline，是批量 CAS 的最佳方案。

### 追问链 3：指标计算引擎

**Q1：拓扑排序如果检测到循环依赖怎么办？**
> **PREP 答案**：Kahn 算法天然能检测环——如果排序结束后还有 `remaining` 节点未被处理，说明存在循环依赖，返回 `circular dependency detected` 错误。我们的策略是：拓扑排序前先做跨源依赖校验（`validateSourceConsistency`），软排除无效配置（warning + skip 不阻塞全局），在 API 层创建计算点时也做前置校验。

**Q2：公式热加载有没有并发安全问题？**
> **PREP 答案**：有。`MetricPointLoader` 使用 `atomic.Value` 存储索引：读操作直接 `Load()` 无锁；更新操作通过 `Store()` 原子替换——先构建新索引（Load → Parse → TopologicalSort → BuildDependencyIndex），再原子替换，不影响正在进行的计算。这是经典的 COW（Copy-On-Write）模式。

**Q3：DLQ 最多重试 5 次之后呢？**
> **PREP 答案**：超过 5 次后按错误类型分策略：`formula_error`（公式本身有问题）和 `max_cascade_depth_exceeded` 触发 P1 级别告警（`Alerter.FireP1`），通知运维立即介入；`snapshot_expired` 触发 P2 告警（快照过期通常是 Kafka 消费延迟导致，非紧急）；`config_missing` 返回错误等待下次热加载后自动重入队。

**Q4：R2/R1 双层缓存的设计动机是什么？**
> **PREP 答案**：R1（`HSET {source} {dpid} {json}`）存储的是**最新值**，但在级联触发场景中，上游刚写入的值可能被更新的数据覆盖，导致下游计算用到了错误的输入。R2（`SET r2:{source}:{dpid}:{ts_sec}`，TTL=900s）存储的是**某一时间点的精确快照**，确保同一批次触发的级联计算使用一致的数据。R2 hit → 精确值；R2 miss → R1 降级（最新值 + 1h Staleness Check）。

### 追问链 4：预处理网关

**Q1：为什么大文件用流式压缩 + MinIO，而不是统一走 Kafka？**
> **PREP 答案**：Kafka 单条消息大小有限制（默认 1MB，可调至 10MB 但影响 Broker 性能和消费者内存）；MinIO 支持分片上传无大小限制；Zstd 流式压缩（`CompressStream`）边读边压缩边写，避免大文件全量加载到内存导致 OOM。小数据直发 Kafka 保证低延迟，大数据走 MinIO 保证可靠性。

**Q2：异步任务队列满了怎么办？**
> **PREP 答案**：配置了两级保护：① `rejectOnFull=true` 时，队列满立即返回 503 错误给调用方，实现**快速失败**（背压传导）② 健康检查 `Health()` 在待处理 > 90% 容量时返回不健康，K8s 自动摘除流量。这确保服务不会因为队列积压导致 OOM。

### 追问链 5：北向同步服务

**Q1：版本号增量同步，如果远端版本号不变但数据实际有更新？**
> **PREP 答案**：三种保障：① API 支持 `force=true` 参数强制忽略版本对比 ② Cron 调度默认按配置的定时策略全量同步作为兜底 ③ 运维可通过 API 手动触发同步（`SubmitTaskAndWait` 同步等待结果）。

**Q2：为什么从 process-service 拆分为独立服务？**
> **PREP 答案**：三个原因：① process-service 是 IO 密集型（Kafka 消费 + 双写），north-sync 是 CPU/网络密集型（HTTP 拉取 + 数据解析），混合部署资源争抢 ② 同步配置变更需要重启 process-service 影响数据消费 ③ 独立后支持多配置动态管理，每个机房可以有独立的同步策略和调度周期。

### 追问链 6：平台管理服务

**Q1：为什么 Snowflake ID 选择 15 位机器 ID + 7 位序列号？**
> **PREP 答案**：标准 Snowflake 支持 1024 台机器 × 4096 ID/ms；我们的场景是 K8s 部署，Pod 数量可能很多（滚动更新时新旧 Pod 并存）但单 Pod 并发生成 ID 的频率不高，因此增大机器位（32768 节点）减少序列位（128/ms）更合理，避免 Pod 滚动更新时机器 ID 冲突。

**Q2：SETNX 锁如果持有者 Pod 崩溃怎么释放？**
> **PREP 答案**：设置了 TTL，即使持有者崩溃，锁也会在 TTL 过期后自动释放。goroutine 续期是为了正常场景下防止初始化时间超过 TTL。这是「宁可多等一会，不可重复执行」的策略，因为重复初始化会导致 Redis 数据不一致。
