# Kafka 消息分区设计详解

> 本文档梳理整个项目各服务 Kafka 生产者的消息分区策略，包括分区键选择、负载均衡器类型、设计意图及潜在影响分析。

---

## 1. 分区策略总览

| 服务 | Topic | 分区键（Key） | 负载均衡策略 | 核心目的 |
|-----|-------|-------------|------------|---------|
| `aidc-pretreat-service` | K1 原始数据 | **TraceID（随机 UUID）** | Hash 路由 | 同批分片进同一分区；均匀散列 |
| `aidc-metric-service` | `aidc-calc-task`（K2） | **CalcDpid（计算点 ID）** | Hash 路由 | 同计算点消息严格有序消费 |
| `aidc-metric-service` | DLQ 死信队列 | **继承原消息 Key** | Hash 路由 | 保持与原消息相同分区，便于追溯 |
| `aidc-metric-service` | 大消息/通用写入 | 无固定规则 | **LeastBytes（最少字节）** | 动态感知分区积压，自适应均衡 |
| `boss-jh-gateway` | 原始数据推送 | **nil（无 Key）** | **RoundRobin（轮询）** | 最大化吞吐，事务保证一致性 |

## 2. 各服务详细分析

### 2.1 aidc-pretreat-service：以 TraceID 为分区键

**代码位置：**
- `aidc-pretreat-service/internal/kafka/producer.go`
- `aidc-pretreat-service/internal/kafka/producer_chunk.go`

```go
// producer.go - 发送统一消息
kafkaMsg := &sarama.ProducerMessage{
    Topic:     p.topic,
    Key:       sarama.StringEncoder(msg.TraceID),  // UUID 随机字符串
    Value:     sarama.ByteEncoder(data),
    Timestamp: time.Now(),
}

// producer_chunk.go - 发送分片消息（保证同批次分片进同一 Partition）
kafkaMsg := &sarama.ProducerMessage{
    Topic: p.topic,
    Key:   sarama.StringEncoder(traceID), // 使用 TraceID 保证同批次分片进入同一 Partition
    Value: sarama.ByteEncoder(data),
}
```

**TraceID 生成方式：**

```go
// processor.go
traceID := uuid.New().String()  // 每次 HTTP 请求生成一个全局唯一 UUID
```

**设计意图：**

- **分片完整性保证**：单次 HTTP 请求的数据若超过 10MB 阈值，会被拆分成多个分片消息（Chunk）。使用同一 TraceID 作为 Key，通过 Kafka 的 Hash 路由机制，确保所有分片路由到同一 Partition，消费端可按序接收同批次的全部分片
- **自然负载均衡**：TraceID 是随机 UUID，哈希值分布均匀，不同请求自然散列到各分区，避免热分区
- **无跨请求顺序需求**：不同设备上报请求之间无依赖关系，不需要跨 TraceID 保序

**BatchProducer 中的 Key 传递：**

```go
// batch_producer.go - 批量发送时透传调用方的 key
func (bp *BatchProducer) Send(key string, value []byte) error {
    msg := &sarama.ProducerMessage{
        Topic: bp.topic,
        Key:   sarama.StringEncoder(key),  // 调用方传入 key（即 TraceID）
        Value: sarama.ByteEncoder(value),
    }
    return bp.sendMessage(msg)
}
```

### 2.2 aidc-metric-service CalcTriggerEvaluator：以 CalcDpid 为分区键

**代码位置：**
- `aidc-metric-service/pkg/calc/trigger.go`

```go
// sendTask - 发送计算任务消息到 calc-task topic
func (e *CalcTriggerEvaluator) sendTask(ctx context.Context, task *CalcTaskMessage) error {
    data, _ := json.Marshal(task)
    return e.producer.WriteMessages(ctx, kafka.Message{
        Topic: e.calcTopic,
        Key:   []byte(task.CalcDpid), // partition by calc_dpid for ordering（注释说明设计意图）
        Value: data,
    })
}
```

**CalcDpid 的含义：**
- 计算点的唯一 ID，由雪花算法生成，字符串形式，如 `"1234567890123456789"`
- 每个计算点对应一个固定的 CalcDpid，不随时间变化

**设计意图：**

- **同一计算点严格顺序消费**：同一计算点（如某个 UPS 的功耗计算点）可能在短时间内被多次触发（原始点持续上报）。使用 CalcDpid 作为 Key，保证该点的所有触发消息都落入同一 Partition，Consumer Group 的单个消费者按序处理，避免并发计算同一点导致的结果覆盖问题
- **即使有 CAS 兜底也需要顺序**：虽然 `CalcProcessor` 使用 Redis CAS（`WriteDevicesWithTimestampCAS`）防止乱序写入，但顺序分区是第一道防线，可减少不必要的 CAS 失败和重试
- **级联计算天然有序**：下游级联计算点也使用自己的 CalcDpid 作为 Key，各计算点的顺序性独立维护，互不干扰

**幂等键与分区键的协同：**

```
IdempotencyKey = calc_dpid:trigger_ms:config_version
分区键         = calc_dpid

→ 同一计算点的消息顺序到达（分区保证）
→ 同一 trigger_ms 的重复消息被幂等键过滤（TTL=25h）
两者共同保证"恰好一次"语义
```

### 2.3 aidc-metric-service DLQ：继承原消息 Key

**代码位置：**
- `aidc-metric-service/pkg/calc/processor.go`

```go
func (p *CalcProcessor) sendToDLQ(ctx context.Context, msg kafka.Message, errorType string, err error) {
    // ... 构建 envelope ...
    _ = p.dlqProducer.WriteMessages(ctx, kafka.Message{
        Key:   msg.Key,  // ← 继承原消息的 Key（即 CalcDpid）
        Value: data,
    })
}
```

**设计意图：**

- **追溯一致性**：DLQ 消息使用与原消息相同的 Key，路由到相同分区编号的 DLQ Partition，便于关联排查
- **DLQ 消费顺序**：DLQ 消费者（`DLQHandler`）在补偿重试时，同一计算点的失败消息也保持顺序，避免补偿乱序

### 2.4 aidc-metric-service KafkaProducer：LeastBytes 动态均衡

**代码位置：**
- `aidc-metric-service/pkg/storage/kafka_producer.go`

```go
writer := &kafka.Writer{
    Addr:         kafka.TCP(config.Brokers...),
    Topic:        config.Topic,
    Balancer:     &kafka.LeastBytes{},  // ← 最少字节数负载均衡
    MaxAttempts:  3,
    WriteTimeout: 10 * time.Second,
    ReadTimeout:  10 * time.Second,
}
```

**`LeastBytes` 原理：**
- `kafka-go` 库提供的负载均衡器
- 每次发送时选择当前积压字节数最少的分区写入
- 动态感知各分区负载，自适应均衡，适合消息大小差异较大的场景

**适用场景：**
- 大消息自动拆分写入（`splitAndSend` 场景）：拆分后的 `_part1` 和 `_part2` 可能会路由到不同分区，因为它们 Key 不同（`key+"_part1"` / `key+"_part2"`），消费端需按 Key 后缀重组
- 一般消息写入时不设置严格的顺序要求

### 2.5 boss-jh-gateway：RoundRobin + 事务

**代码位置：**
- `boss-jh-gateway/srvs/dcim_metrics/pkg/kafka/kafka_pusher.go`

```go
config.Producer.Partitioner = sarama.NewRoundRobinPartitioner
config.Producer.Idempotent = true
config.Producer.RequiredAcks = sarama.WaitForAll
config.Producer.Transaction.ID = "dcim_metrics"

// 发送时 Key 为 nil
producer.Input() <- &sarama.ProducerMessage{
    Topic: Topic,
    Key:   nil,     // ← 无 Key，轮询分配
    Value: sarama.ByteEncoder(MsgData),
}
```

**设计意图：**

- **最大化并行吞吐**：网关承接所有设备上报数据，消息量巨大，轮询写入各分区使所有分区均匀承载写入压力
- **无顺序需求**：原始设备数据点位之间无全局顺序依赖，消费端（`aidc-process-service`）处理时以数据内的时间戳为准，而非消费顺序
- **Kafka 事务保证原子性**：结合 `BeginTxn` / `CommitTxn`，即使轮询分散到多个分区，同一批次的消息要么全部成功要么全部回滚，不会出现部分写入

## 3. 分区键选择的设计思路对比

### 3.1 有顺序需求 → 固定 Key（Hash 路由）

```
场景：同一计算点的多次触发
分区键：CalcDpid
效果：同一点的所有消息 → 同一 Partition → 单消费者顺序处理
```

### 3.2 有完整性需求 → 关联 Key（Hash 路由）

```
场景：同一请求的大数据分片
分区键：TraceID
效果：同批次所有分片 → 同一 Partition → 消费端按序拼接
```

### 3.3 纯吞吐需求 → nil Key（RoundRobin）

```
场景：网关大量原始设备数据推送
分区键：nil
效果：均匀分散到所有 Partition → 最大并行吞吐
```

### 3.4 动态负载均衡需求 → 无固定规则（LeastBytes）

```
场景：大消息拆分写入，消息大小差异大
负载均衡器：LeastBytes
效果：自适应选择积压最小的分区，避免热分区
```

## 4. 消费者与分区的对应关系

```
Topic: aidc-pretreat（K1）
  ├─ Partition 0  ──▶  aidc-process-service Consumer[0]
  ├─ Partition 1  ──▶  aidc-process-service Consumer[1]
  └─ Partition N  ──▶  aidc-process-service Consumer[N]
  （每个 Partition 由一个 Consumer 消费，同 TraceID 分片顺序到达同一 Consumer）

Topic: aidc-calc-task（K2）
  ├─ Partition 0  ──▶  aidc-metric-service CalcProcessor[0]
  ├─ Partition 1  ──▶  aidc-metric-service CalcProcessor[1]
  └─ Partition N  ──▶  aidc-metric-service CalcProcessor[N]
  （同 CalcDpid 的触发消息全部由同一 Consumer 按序处理）

Topic: aidc-calc-task-dlq（DLQ）
  （分区结构与 calc-task 一致，Key 继承原消息）
  └─ aidc-metric-service DLQHandler 消费补偿
```

## 5. 关键设计约束与注意事项

### 5.1 分区数与并发消费者数的关系

Kafka Consumer Group 中：
- **消费者数量 ≤ Partition 数量**：多余的消费者实例处于空闲状态
- **消费者数量 = Partition 数量**：最佳并发，每个分区对应一个消费者
- **消费者数量 > Partition 数量**：多余消费者空闲（`aidc-process-service` 需根据 Topic Partition 数配置合适的并发数）

### 5.2 CalcDpid 的 Key 热点风险

- 若某个高频计算点（如每秒触发多次的总功耗点）的触发消息量远超其他点，会导致该点所在 Partition 成为热点
- **现有缓解机制**：`CoalescingBuffer` 去重合并（`FlushCoalesced` 每 100ms 批量发送一次），减少单个 CalcDpid 的消息频率

### 5.3 大消息拆分的分区一致性

```go
// kafka_producer.go - splitAndSend 拆分时 Key 带后缀
p.ProduceData(key+"_part1", data[:mid])
p.ProduceData(key+"_part2", data[mid:])
```

- `_part1` 和 `_part2` 的 Key 不同，可能路由到不同 Partition
- 消费端需通过原始 `key`（去除后缀）来关联和重组数据
- 这是一个潜在的设计缺陷：若 Partition 消费进度不一致，重组会有等待延迟

### 5.4 aidc-process-service 消费端的 Partition 感知

`aidc-process-service` 的消费者（`consumer.go`）基于 `segmentio/kafka-go` 的 `ConsumerGroup` 实现：
- 每个 Topic 启动独立的消费 goroutine
- 限速器（`TopicRateLimiter`）在消费层面控制每个 Topic 的处理速率，与分区无关
- Partition 分配由 Kafka Broker 的 Consumer Group 协调协议（Range/RoundRobin）自动处理
