# AIDC 实时数据处理服务：Topic 消费设计与数据来源追踪

## 一、系统总览

AIDC 实时数据处理平台由 5 个微服务协同工作，通过 Kafka Topic 实现服务间解耦和数据流转：

```text
数据源(BMS/金华/APISIX)
       │
       ▼
┌─────────────────────┐
│ aidc-pretreat-service│  ← 数据预处理（大小分流、压缩、MinIO）
│  Producer → "raw"    │
└──────────┬──────────┘
           │ Kafka Topic: raw
           ▼
┌─────────────────────┐
│ aidc-process-service │  ← 核心消费（解析、写 Redis + VictoriaMetrics）
│  Consumer ← jinhua   │
│  Consumer ← raw      │
│  Consumer ← bms-D04  │
└──────────┬──────────┘
           │ 写入 Redis R1/R2
           ▼
┌─────────────────────┐
│ aidc-metric-service  │  ← 指标计算（公式引擎、级联触发）
│  Consumer ← aidc-calc-task     │
│  Consumer ← aidc-calc-task-dlq │
│  Producer → aidc-calc-task     │
└─────────────────────┘

┌─────────────────────┐
│ aidc-north-sync      │  ← 北向配置同步（HTTP 拉取，无 Kafka）
└─────────────────────┘

┌─────────────────────┐
│ aidc-platform-service│  ← 平台管理（API 网关，无 Kafka）
└─────────────────────┘
```

## 二、Kafka Topic 清单

| Topic 名称 | 生产者 | 消费者 | 数据内容 | 限速 |
| --- | --- | --- | --- | --- |
| `jinhua` | 外部金华系统 | process-service | APISIX 网关转发的设备实时数据 | 5 msg/s, burst=10 |
| `raw` | pretreat-service | process-service | 统一消息格式(inline/reference) | 10 msg/s, burst=20 |
| `bms-D04-rtdata-shuoyao` | 外部 BMS 系统 | process-service | BMS 设备点位实时数据 | 20 msg/s, burst=40 |
| `aidc-calc-task` | metric-service (触发器) | metric-service (处理器) | 计算任务消息 | 无限速 |
| `aidc-calc-task-dlq` | metric-service (处理器) | metric-service (DLQ Handler) | 失败任务补偿 | 无限速 |

## 三、数据类型详解（三种数据源）

### 3.1 金华数据（Topic: `jinhua`）

**数据来源**：金华系统通过 APISIX 网关转发。

**消息结构**：APISIX 日志格式，核心数据嵌套在 `request.body` 中。

```json
{
  "client_ip": "10.0.0.1",
  "upstream_latency": 5,
  "apisix_latency": 1.2,
  "response": { "status": 200, "size": 1024 },
  "start_time": 1711872000,
  "route_id": "route-001",
  "service_id": "svc-001",
  "upstream": "http://backend:8080",
  "request": {
    "method": "POST",
    "uri": "/api/v1/data",
    "body": "{\"version\":\"v1\",\"data\":{\"strategy_id\":1,\"period\":1711872000,\"devices\":[{\"guid\":\"device-001\",\"status\":1,\"points\":[{\"guid\":\"point-001\",\"value\":\"24.5\",\"timestamp\":1711872000}]}]}}",
    "size": 2048
  },
  "server": { "hostname": "apisix-node-1", "version": "3.0" }
}
```

**解析流程**：

1. JSON 反序列化为 `JinhuaMessage` 结构
2. 二次解析 `request.body` 为 `JinhuaRequestBody`
3. 提取 `devices[]` → 统一 `Device` 模型，`Source` 标记为 `"jinhua"`

**关键字段映射**：

| 原始字段 | 统一模型字段 | 说明 |
| --- | --- | --- |
| `data.devices[].guid` | `Device.GUID` | 设备唯一标识 |
| `data.devices[].status` | `Device.Status` | 设备状态 |
| `data.period` | `Device.Timestamp` | 采集周期时间戳 |
| `data.devices[].points[].guid` | `Point.GUID` | 点位唯一标识 |
| `data.devices[].points[].value` | `Point.Value` | 点位值(字符串) |
| `data.devices[].points[].timestamp` | `Point.Timestamp` | 点位时间戳 |

### 3.2 Raw 数据（Topic: `raw`）

**数据来源**：`aidc-pretreat-service` 预处理后发送。

**消息结构**：统一消息格式（`UnifiedMessage`），支持两种数据携带方式：

#### 3.2.1 Inline 消息（小数据，< 10MB）

数据直接嵌入 Kafka 消息体中。

```json
{
  "trace_id": "abc-123-def",
  "source": "raw",
  "data_type": "inline",
  "timestamp": "2024-03-31T10:00:00Z",
  "payload": "<base64编码的原始数据>"
}
```

#### 3.2.2 Reference 消息（大数据，>= 10MB）

数据存储在 MinIO，Kafka 消息仅携带引用信息。

```json
{
  "trace_id": "abc-456-ghi",
  "source": "raw",
  "data_type": "reference",
  "timestamp": "2024-03-31T10:00:00Z",
  "storage_info": {
    "bucket": "aidc-data",
    "object": "raw/2024/03/31/abc-456-ghi.zst",
    "original_size": 52428800,
    "compressed_size": 15728640,
    "compressed": true,
    "content_type": "application/zstd"
  },
  "meta_info": {
    "strategy_id": 1,
    "device_count": 500,
    "point_count": 25000
  }
}
```

**解析流程**：

1. 判断 `data_type`：`inline` 直接解析 `payload`；`reference` 从 MinIO 下载后解析
2. 解析为设备和点位列表
3. 提取 `Device` 模型，`Source` 标记为 `"raw"`

### 3.3 BMS 数据（Topic: `bms-D04-rtdata-shuoyao`）

**数据来源**：外部 BMS（楼宇管理系统）直接推送。

**消息结构**：BMS 数据项数组，每个 item 包含一个点位。

```json
[
  {
    "id": "device-001:point-001",
    "v": "24.5",
    "t": 1711872000,
    "q": 192
  },
  {
    "id": "device-001:point-002",
    "v": "60.2",
    "t": 1711872000,
    "q": 192
  }
]
```

**解析流程**：

1. JSON 反序列化为 `[]BMSItem` 数组
2. 按 `id` 中的设备前缀（`:`分隔的第一段）分组
3. 每个分组构建 `Device` 模型，`Source` 标记为 `"bms"`

**关键字段映射**：

| BMS 字段 | 统一模型字段 | 说明 |
| --- | --- | --- |
| `id` (`:` 前缀部分) | `Device.GUID` | 设备标识（从 id 解析） |
| `id` | `Point.GUID` | 完整点位标识 |
| `v` | `Point.Value` | 点位值(字符串) |
| `t` | `Point.Timestamp` | Unix 时间戳 |
| `q` | `Point.Quality` | 数据质量码(192=Good) |

## 四、aidc-pretreat-service：数据大小自动分流策略

### 4.1 核心设计

pretreat-service 作为数据入口，根据数据大小自动选择处理策略：

```text
HTTP 请求 (APISIX 转发)
       │
       ▼
  Content-Length 判断
       │
  ┌────┴────┐
  │ < 10MB  │ >= 10MB │
  ▼         ▼
inline    reference
  │         │
  │    ┌────┴────┐
  │    │ 流式压缩  │
  │    │ (zstd)   │
  │    ▼         │
  │  MinIO 上传  │
  │    │         │
  ▼    ▼         │
 Kafka Producer  │
  (raw topic)    │
```

### 4.2 配置参数

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `processor.size_threshold` | `10485760` (10MB) | 大小分流阈值 |
| `processor.source` | 环境变量配置 | 数据来源标识 |
| `processor.max_body_size` | 环境变量配置 | 最大请求体限制 |
| `compression.level` | 1-22 (zstd级别) | 压缩级别 |

### 4.3 三种处理器实现

#### 4.3.1 Processor（默认处理器）

- **小数据路径**：`io.ReadAll` → 构建 `InlineMessage` → `Producer.Send()`
- **大数据路径**：`io.ReadAll` → zstd 压缩 → MinIO 上传 → 构建 `ReferenceMessage` → `Producer.Send()`
- **特点**：全量读入内存后处理

#### 4.3.2 CompressProcessor（流式压缩处理器）

- **小数据路径**：`io.ReadAll` → 不压缩 → 构建 `InlineMessage` → `Producer.Send()`
- **大数据路径**：`io.Pipe()` 连接压缩和上传 → Goroutine 1 流式压缩 → Goroutine 2 流式上传 MinIO → 构建 `ReferenceMessage` → `Producer.Send()`
- **特点**：大数据走流式管道，**内存占用恒定**，不受数据大小影响

#### 4.3.3 MinIOProcessor（直接上传处理器）

- **小数据路径**：`io.ReadAll` → 构建 `InlineMessage` → `Producer.Send()`
- **大数据路径**：流式上传 MinIO（**不压缩**）→ 构建 `ReferenceMessage` → `Producer.Send()`
- **特点**：大数据不做压缩，适合已压缩或二进制数据

### 4.4 Kafka Producer 配置

```yaml
kafka:
  brokers: ["kafka-1:9092", "kafka-2:9092"]
  topic: "raw"
  producer_id: "aidc-pretreat-service"
  required_acks: -1          # 所有副本确认
  enable_idempotence: true   # 幂等生产者
  max_retries: 3
  retry_backoff: 100ms
  flush_frequency: 100ms
  flush_messages: 100
  max_message_bytes: 10485760
```

## 五、aidc-process-service：多 Topic 消费与解析

### 5.1 Consumer 架构设计

process-service 使用 **per-topic reader** 模式，为每个 Topic 创建独立的 `kafka.Reader`，注册独立的 `MessageHandler`：

```go
// 每个 Topic 一个 Reader + Handler
type Consumer struct {
    readers  map[string]*kafka.Reader   // topic → reader
    handlers map[string]MessageHandler   // topic → handler
    limiter  *TopicRateLimiter          // topic → rate.Limiter
}
```

### 5.2 Topic 注册与限速

```go
// 默认限速配置
TopicRateLimits: map[string]TopicRateLimit{
    "jinhua":                 {Rate: 5,  Burst: 10},
    "raw":                    {Rate: 10, Burst: 20},
    "bms-D04-rtdata-shuoyao": {Rate: 20, Burst: 40},
}
```

- 使用 `golang.org/x/time/rate.Limiter` 实现令牌桶限速
- 每个 Topic 独立限速，互不影响
- 消费前调用 `limiter.Wait(ctx)` 阻塞等待令牌

### 5.3 消息处理流程

```text
Kafka Message
    │
    ▼
TopicRateLimiter.Wait()  ← 限速控制
    │
    ▼
MessageHandler.Handle(msg)
    │
    ▼
ParserFactory.GetParser(topic, data)
    │
    ├── topic="jinhua"  → JinhuaMessage.Parse() + ExtractDevices()
    ├── topic="raw"     → RawMessage.Parse() + ExtractDevices()
    │     └── data_type="inline"    → 直接解析 payload
    │     └── data_type="reference" → MinIO 下载 → 解压 → 解析
    └── topic="bms-*"   → BMSMessage.Parse() + ExtractDevices()
    │
    ▼
[]Device (统一模型)
    │
    ├── RedisWriter.WriteDevices()     ← 写 R1 Hash
    └── VictoriaWriter.WriteDevices()  ← 写时序数据库
```

### 5.4 数据来源追踪机制

每种消息解析器在 `ExtractDevices()` 时，会在 `Device.Source` 字段标记来源：

| Topic | Source 值 | 说明 |
| --- | --- | --- |
| `jinhua` | `"jinhua"` | 金华系统数据 |
| `raw` | `"raw"` | pretreat-service 预处理数据 |
| `bms-D04-rtdata-shuoyao` | `"bms"` | BMS 系统数据 |

**Redis 写入时以 Source 为 Hash Key**：

```text
HSET "jinhua"  "point-guid-001"  '{"dp_id":"point-guid-001","dp_value":24.5,"dp_timestamp":1711872000,...}'
HSET "raw"     "point-guid-002"  '{"dp_id":"point-guid-002","dp_value":60.2,"dp_timestamp":1711872000,...}'
HSET "bms"     "point-guid-003"  '{"dp_id":"point-guid-003","dp_value":18.0,"dp_timestamp":1711872000,...}'
```

这种设计实现了：

- **多源隔离**：不同数据源写入不同 Hash Key，同名 DPID 不会冲突
- **来源可追踪**：从 Redis 中读取数据时，Hash Key 即为数据来源
- **下游计算隔离**：metric-service 计算指标时使用 Source 进行依赖隔离

### 5.5 Redis 写入的时序保证（Lua CAS）

横向扩容场景下，多个 Consumer 实例可能并发写入同一个点位。使用 Lua 脚本实现 CAS（Compare-And-Swap）：

```lua
-- 原子性时间戳比较和条件写入
local existing_data = redis.call('HGET', hash_key, hash_field)
-- ... 解析 existing_json.dp_timestamp ...
if new_timestamp > existing_timestamp then
    redis.call('HSET', hash_key, hash_field, new_data)
    return 1  -- 写入成功
else
    return 0  -- 跳过（旧数据）
end
```

- **Pipeline 批量模式**：每批 1000 个命令，使用 Redis Pipeline 减少网络往返
- **可配置**：`enableLuaScript=true` 启用 Lua CAS 模式；`false` 使用直接 HMSET（性能最高但无时序保证）

## 六、aidc-metric-service：计算指标与级联触发

### 6.1 架构概览

metric-service 是一个**流式计算引擎**，消费 `aidc-calc-task` Topic，对原始点位数据执行公式计算，并支持级联触发（计算点依赖计算点）。

```text
process-service (R1/R2 写入)
       │
       ▼
CalcTriggerEvaluator
  │ 检查依赖是否满足
  │ 构建 CalcTaskMessage
  ▼
Kafka: aidc-calc-task
       │
       ▼
CalcProcessor.Handle()
  │ 1. 幂等检查 (calc:idem:{key})
  │ 2. 分布式锁 (calc:lock:{key})
  │ 3. 加载依赖值 (R2 MGET → R1 HMGET)
  │ 4. 公式计算 (FormulaEngine)
  │ 5. 写结果到 R1/R2/VictoriaMetrics
  │ 6. 触发下游级联
  ▼
Kafka: aidc-calc-task (下游任务)
  或
Kafka: aidc-calc-task-dlq (失败补偿)
```

### 6.2 Kafka Topic 设计

#### 6.2.1 aidc-calc-task（计算任务 Topic）

**消息格式**：`CalcTaskMessage`

```json
{
  "calc_dpid": "calc:pue:site-001",
  "calc_name": "PUE",
  "label": "PUE指标",
  "cal_formula": "(A + B) / C",
  "cal_method": "formula",
  "location_id": 1,
  "location_slug": "site-001",
  "unit": "",
  "source": "jinhua",
  "values": [
    {"dpid": "point-001", "point_slug": "A", "source": "jinhua", "value": 100.5, "timestamp": 1711872000, "quality": 0},
    {"dpid": "point-002", "point_slug": "B", "source": "jinhua", "value": 50.2, "timestamp": 1711872000, "quality": 0},
    {"dpid": "point-003", "point_slug": "C", "source": "jinhua", "value": 60.0, "timestamp": 1711872000, "quality": 0}
  ],
  "trigger_ts": 1711872000,
  "trigger_type": "raw",
  "config_version": 5,
  "idempotency_key": "calc:pue:site-001:1711872000000:5",
  "hop_count": 0
}
```

**关键字段说明**：

| 字段 | 说明 |
| --- | --- |
| `calc_dpid` | 计算点唯一标识 |
| `source` | 数据源标识，与 Redis R1 Hash Key 对应 |
| `values` | 依赖点的当前值列表（可为空，轻量消息由处理器自行拉取） |
| `trigger_ts` | 触发时间戳（Unix 秒） |
| `trigger_type` | 触发类型：`"raw"`（原始点触发）或 `"calc"`（级联触发） |
| `hop_count` | 级联深度计数，最大 10 层 |
| `idempotency_key` | 幂等键：`{calc_dpid}:{trigger_ms}:{config_version}` |
| `snapshot_ref` | 大依赖（>50 个）时的 Redis 快照引用 |

#### 6.2.2 aidc-calc-task-dlq（死信队列 Topic）

**消息格式**：DLQ 信封 + 原始 CalcTaskMessage

```json
{
  "original_message": { "...CalcTaskMessage..." },
  "error_type": "formula_error",
  "error_message": "division by zero",
  "retry_count": 0,
  "failed_at": 1711872005,
  "component": "calc_processor"
}
```

**DLQ 补偿策略**：

| error_type | 补偿动作 |
| --- | --- |
| `cascade_trigger_failed` | 检查主写入是否完成，重新触发下游级联 |
| `config_missing` | 等待热重载后重新入队 |
| `snapshot_expired` | P2 告警，快照 TTL 过期 |
| `formula_error` | P1 告警，公式配置有误 |
| `max_cascade_depth_exceeded` | P1 告警，可能存在循环依赖 |
| `deserialize_error` | P2 告警，消息格式异常 |

### 6.3 双层缓存读取（R2 + R1）

metric-service 使用**双层 Redis 缓存**获取依赖值：

```text
R2 (时间点精确值)                R1 (最新值)
Key: r2:{source}:{dpid}:{ts}    Key: {source} (Hash)
TTL: 900s                        Field: {dpid}
Value: plain float               Value: JSON (CompatDataPoint)

优先读 R2 (MGET)
    │
    ├── R2 命中 → 使用精确值
    └── R2 未命中 → 降级读 R1 (HMGET)
            │
            └── 陈旧检查：age > 1h → 标记 missing
```

### 6.4 级联触发机制

计算点之间存在依赖关系（DAG），当上游计算点完成后触发下游：

```text
原始点 P1 变更
    │
    ▼
CalcTriggerEvaluator.Evaluate()
    │ 查找依赖 P1 的计算点
    ▼
CalcPoint CP1 (hop_count=0)
    │ 计算完成 → 写入 R1/R2
    │ triggerFunc() → 查找依赖 CP1 的计算点
    ▼
CalcPoint CP2 (hop_count=1)
    │ 计算完成 → 写入 R1/R2
    │ triggerFunc() → ...
    ▼
最大深度: 10 层 (MaxCascadeDepth=10)
```

**防循环保护**：

- DAG 静态校验（加载时检查）
- `hop_count` 运行时深度检查（`>= 10` 直接送 DLQ）
- `CascadeRateLimiter` 每个计算点限制级联触发频率

### 6.5 大依赖优化（SnapshotRef）

当计算点依赖数 > 50 时（`LargeDepThreshold = 50`）：

```text
小依赖（≤ 50 个）：依赖值直接嵌入 CalcTaskMessage.Values
大依赖（> 50 个）：
  1. 依赖值序列化为 JSON，存入 Redis: calc:snap:{snap_id}
  2. CalcTaskMessage 只携带 SnapshotRef{snap_id, deps_count}
  3. CalcProcessor 处理时从 Redis 加载快照
  4. 处理完毕后删除快照（fire-and-forget）
```

## 七、aidc-north-sync：北向配置同步

### 7.1 工作模式

north-sync **不使用 Kafka**，而是通过 HTTP API 定时拉取北向系统的配置数据（位置、设备、点位信息），同步到本地 PostgreSQL 数据库。

### 7.2 数据模型

同步三种节点类型：

| NodeType | 值 | 对应模型 | 说明 |
| --- | --- | --- | --- |
| Location | 1 | `northnodes.Location` | 位置/空间节点（园区/楼宇/楼层/机房） |
| Device | 2 | `northnodes.BmsDevice` | 设备节点（空调/配电柜等） |
| Point | 3 | `northnodes.BmsPoint` | 点位节点（温度/电流/功率等） |

### 7.3 同步策略

- **版本对比**：本地版本号与远端版本号对比，一致则跳过
- **分页拉取**：每页 500 条，逐页请求北向 API
- **事务写入**：每页数据在数据库事务中写入
- **定时调度**：通过 `cron` 表达式配置，支持多配置独立调度
- **数据来源标记**：所有节点写入时带 `Source` 字段（如 `northSource`），确保多来源可追踪

## 八、数据来源追踪全链路

从数据采集到最终存储，**Source 贯穿全流程**：

```text
[数据源] → [pretreat-service] → [Kafka] → [process-service] → [Redis R1] → [metric-service] → [Redis R1/R2 + VMS]
                                              Source="jinhua"/"raw"/"bms"
                                                    │
                                                    ▼
                                          Redis Hash Key = Source
                                          VMS label source = Source
                                          Calc task.Source = Source
```

### 8.1 追踪链路示例

以金华数据为例：

1. **pretreat-service**：接收 APISIX 转发的 HTTP 请求，`config.source = "jinhua"`
2. **Kafka raw topic**：`UnifiedMessage.source = "jinhua"`
3. **process-service**：解析后 `Device.Source = "jinhua"`
4. **Redis R1**：`HSET "jinhua" "point-001" '{...}'`
5. **Redis R2**：`SET "r2:jinhua:point-001:1711872000" "24.5"` (TTL 900s)
6. **VictoriaMetrics**：`device_point_value{source="jinhua", point_guid="point-001"} 24.5`
7. **metric-service**：`CalcTaskMessage.Source = "jinhua"` → 计算结果写入 `HSET "jinhua" "calc-dpid-001" '{...}'`

### 8.2 Source 隔离的意义

- **防冲突**：不同数据源下可能存在相同的 DPID，用 Source 作为 Hash Key 天然隔离
- **可审计**：任何时刻都能知道某个点位值来自哪个系统
- **计算正确性**：metric-service 依赖解析时强制 Source 一致性检查（cross-source 依赖会被拒绝）
