## Phase 1: 专家洞察（Code Analysis）

### 【专家洞察】

通读完整个 AIDC 项目的 5 个微服务源码后，我看到的是一套 **完整的 IoT 数据采集-预处理-实时消费-指标计算-平台管理** 的全链路闭环系统，技术含金量体现在以下几个维度：

1. **数据流全链路覆盖**：从北向 API 设备元数据同步（north-sync）→ 原始数据接入预处理（pretreat-service，含大小文件分治、Zstd 流式压缩、MinIO 对象存储）→ 多格式实时消费与双写（process-service，Kafka 多 Topic + Redis/VictoriaMetrics 并发写入）→ 多粒度指标计算引擎（metric-service，表达式引擎 + 拓扑排序 + 级联触发 + DLQ 补偿）→ 平台管理与权限控制（platform-service，Casdoor SSO + RBAC + 计算点 CRUD）。

2. **流式处理架构设计亮点**：pretreat-service 中 `io.Pipe` 串联压缩与上传的零拷贝流式管道、process-service 中 Lua 脚本原子性时间戳比较解决横向扩容时序冲突、metric-service 中 Kafka calc-task + DLQ + 级联触发的流式计算引擎。

3. **可量化的工程实践**：Token Bucket 限流（per-topic 独立限速）、VictoriaMetrics 分批写入防超时、Worker Pool 并发控制、Snowflake ID 生成器（15位机器ID支持 32768 节点）、Prometheus 监控埋点（请求数/延迟/压缩率）等。

4. **但存在以下可优化空间**：部分服务缺乏具体的 QPS、数据量级、延迟指标描述，简历中需要补充这些可量化的业务数据。

## Phase 2: 标准简历模板（Template Generation）

### 项目名称
**AIDC 智慧数据中心 IoT 实时数据平台**

### 技术栈
Go, gRPC, Kafka(segmentio/sarama), Redis, PostgreSQL, VictoriaMetrics, MinIO, Zstd, Gin, Casdoor, Prometheus, Docker, Kubernetes, Helm, GitLab CI/CD

### 项目描述
面向数据中心基础设施（DCIM）场景的 IoT 实时数据平台，负责设备元数据同步、原始数据接入预处理、多协议实时消费与双写存储、多粒度指标计算引擎、以及平台权限管理，支撑 [待补充] 个数据中心、[待补充] 万级设备点位的实时监控与能耗分析。

### 主要职责

1. **从 0 到 1 架构并实现 IoT 数据预处理网关（aidc-pretreat-service）**，设计大小文件分治策略（<10MB 内联 Kafka，≥10MB 流式 Zstd 压缩 + MinIO 对象存储），基于 `io.Pipe` 实现零拷贝流式压缩-上传管道，构建异步任务队列架构（TaskQueue + Worker Pool），支持同步/异步双模式路由。

2. **主导设计高性能 IoT 实时数据处理服务（aidc-process-service）**，实现多 Topic Kafka 消费者（jinhua/raw/bms 等多协议格式），基于工厂模式设计消息解析器，并发双写 Redis（Hash 结构，兼容 boss-jh-gateway）与 VictoriaMetrics（InfluxDB line protocol 批量分批写入），引入 Token Bucket per-topic 独立限流及 Lua 脚本原子性时间戳比较解决横向扩容数据时序冲突。

3. **架构并实现 DCIM 指标实时计算引擎（aidc-metric-service）**，支持 7 种时间粒度定时调度（10s/1min/5min/hour/day/month/shift），基于 expr-lang 表达式引擎实现动态公式计算（实时计算 + 差值计算），通过拓扑排序解决数据点依赖关系，设计 Kafka calc-task 流式计算 + DLQ 死信补偿 + 级联触发（calc→calc cascade）机制，支持每 5 分钟热加载公式配置。

4. **设计并实现北向接口数据同步服务（aidc-north-sync）**，基于版本号增量同步策略从北向 API 分页拉取位置/设备/点位三级数据，采用事务批量写入 + 异步任务管理（TaskQueue + 状态机）架构，支持多数据源动态配置、定时 Cron 调度与手动触发双模式。

5. **主导开发 AIDC 平台管理服务（aidc-platform-service）**，集成 Casdoor SSO 实现统一认证，设计 RBAC 三级权限模型（角色-菜单-按钮），实现用户-位置权限绑定（Snowflake ID，15位机器ID支持 32768 节点），开发计算点指标 CRUD + VictoriaMetrics 历史查询构建器，并通过 gRPC 认证拦截器中间件实现白名单机制。

### 二、技术选型 / Trade-off（架构师之问）

**Q**：在指标计算引擎中，为什么选择 Kafka calc-task + DLQ 模式而不是直接在调度器中同步计算？

**PREP 满分回答**：

- **Point（观点）**：我们选择 Kafka calc-task + DLQ 模式，本质上是将 **定时批量计算** 演进为 **事件驱动的流式计算**，核心目标是解耦计算触发与计算执行，并引入自动补偿能力。

- **Reason（原理）**：
  - 定时调度器同步计算存在两个问题：①如果某个公式计算失败（如上游数据点尚未就绪），整批计算会被阻塞或跳过；②计算点之间存在依赖关系（A 的结果是 B 的输入），同步模式下拓扑排序后串行执行效率低。
  - Kafka calc-task 模式下，每个计算点的计算任务是一条独立消息，可以并行消费；失败的任务自动进入 DLQ，由 DLQ Handler 延迟重试（最多 5 次，指数退避）；计算成功后通过级联触发函数将下游依赖点重新入队，天然支持 DAG 依赖链的异步推进。

- **Example（项目实例）**：
  - 在我们的项目中，metric-service 同时运行两套引擎：**定时调度器**（兜底的批量全量计算，保障最终一致性）和 **Kafka 流式引擎**（事件驱动的增量计算，保障实时性）。当 process-service 写入一个新的原始数据点到 Redis 后，会发送一条 calc-task 消息，metric-service 消费后计算该点的所有下游公式，再级联触发更下游的公式。DLQ 在上线初期每天有约 [待补充] 条失败消息，主要是公式配置错误，通过热加载修复后 DLQ 触发频率降至 [待补充] 条/天以内。

- **Point（总结）**：这个架构的核心价值是：**实时性**（事件驱动，毫秒级响应）+ **可靠性**（DLQ 自动补偿，不丢失计算）+ **可扩展性**（Kafka 消费者水平扩容）+ **最终一致性**（定时调度器兜底），适合计算点规模大、依赖关系复杂的 DCIM 场景。

### 三、补充面试题

#### 针对 "预处理网关"
**Q**：为什么大文件用流式压缩 + MinIO，而不是统一走 Kafka？
> 要点：Kafka 单条消息大小有限制（默认 1MB，可调至 10MB 但影响 Broker 性能）；MinIO 支持分片上传无大小限制；流式压缩避免大文件全量加载到内存导致 OOM。

#### 针对 "北向同步服务"
**Q**：你说基于版本号增量同步，如果远端版本号一直不变但数据实际有更新，怎么处理？
> 要点：代码中支持 `force=true` 参数强制忽略版本对比；Cron 调度默认每天全量同步一次作为兜底；API 支持手动触发同步。

#### 针对 "平台管理服务"
**Q**：为什么 Snowflake ID 选择 15 位机器 ID + 7 位序列号，而不是标准的 10+12？
> 要点：标准 Snowflake 支持 1024 台机器 × 4096 ID/ms；我们的场景是 K8s 部署，Pod 数量可能很多但单 Pod 并发生成 ID 的频率不高，因此增大机器位（32768 节点）减少序列位（128/ms）更合理，避免 Pod 滚动更新时机器 ID 冲突。
