## 1. 整体架构概述

整个 AIDC 系统中，Kafka 在三个服务间扮演**数据总线**角色，形成如下数据流：

```
外部设备数据
     │
     ▼
┌─────────────────────┐
│  aidc-pretreat-     │  HTTP 接收原始数据
│     service         │  (生产者)
│  [Kafka Producer]   │
└─────────────────────┘
     │  写入 Topic：raw / jinhua / bms-D04-rtdata-shuoyao 等
     ▼
┌──────────────┐
│   Kafka      │  ← 流量缓冲、削峰、异步解耦核心
│   Broker     │
└──────────────┘
     │  消费
     ▼
┌─────────────────────┐
│  aidc-process-      │  解析数据 → 写 Redis + VictoriaMetrics
│     service         │  (消费者)
│  [Kafka Consumer]   │
└─────────────────────┘
     │  写入 Redis: formula:* 算法点
     ▼
┌──────────────┐
│   Kafka      │  aidc-calc-task Topic（计算任务调度）
│   Broker     │
└──────────────┘
     │  消费
     ▼
┌─────────────────────┐
│  aidc-metric-       │  执行公式计算 → 写 Redis + VictoriaMetrics
│     service         │  (消费者 + 生产者)
│  [Kafka Consumer    │  DLQ 失败补偿
│   + DLQ Handler]   │
└─────────────────────┘
```

**核心思想：**
- `aidc-pretreat-service` 作为**唯一生产者**，将原始设备数据写入 Kafka，立即返回 HTTP 202 响应给调用方（异步解耦）。
- Kafka Broker 充当**消息缓冲队列**，在流量高峰时吸收突发写入（流量削峰）。
- `aidc-process-service` 作为**消费者**，按照自身处理能力限速消费，将数据持久化到 Redis/VictoriaMetrics。
- `aidc-metric-service` 既消费计算任务 Topic，又在计算成功后生产级联触发消息。

## 3. aidc-pretreat-service（预处理服务）

### 3.1 角色：Kafka 生产者

预处理服务是整个数据链路的**入口**，负责接收来自外部设备或数采系统的原始数据，经过预处理后写入 Kafka。**它是异步解耦的关键起点**：收到 HTTP 请求后，处理完即返回 202，调用方无需等待后续数据处理完成。

### 3.2 两种生产者实现

#### 3.2.1 同步生产者（Producer）

文件：`aidc-pretreat-service/internal/kafka/producer.go`

```go
// 使用 sarama.SyncProducer，以 TraceID 作为 Partition Key 保证消息有序
func (p *Producer) Send(msg *models.UnifiedMessage) error {
    kafkaMsg := &sarama.ProducerMessage{
        Topic:     p.topic,
        Key:       sarama.StringEncoder(msg.TraceID), // TraceID 作为 Key，同批次数据路由到同一 Partition
        Value:     sarama.ByteEncoder(data),
        Timestamp: time.Now(),
    }
    partition, offset, err := p.producer.SendMessage(kafkaMsg)
    ...
}
```

**关键配置：**

| 配置项 | 说明 |
|---|---|
| `RequiredAcks` | 可配置 Leader Ack / All Ack，控制可靠性 |
| `EnableIdempotence` | 支持幂等生产者，防止重试导致重复消息 |
| `FlushFrequency` | Batch 刷新频率 |
| `FlushMessages` | 达到多少条消息时批量发送 |
| `MaxMessageBytes` | 单条消息最大字节数 |

#### 3.2.2 企业级批量生产者（BatchProducer）

文件：`aidc-pretreat-service/internal/kafka/batch_producer.go`

`BatchProducer` 是针对**高吞吐量场景**设计的批量发送器，是流量削峰的核心实现之一：

```
消息进入 → 写入内存缓冲区 buffer
    │
    ├─ 达到 BatchSize（默认100条）→ 立即 flushLocked() 批量发送
    └─ 超过 BatchTimeout（默认100ms）→ 定时器触发 Flush()
```

**核心特性：**

```go
// 默认批量生产者配置
func DefaultBatchProducerConfig() *BatchProducerConfig {
    return &BatchProducerConfig{
        BatchSize:    100,                   // 100 条消息聚合一批
        BatchTimeout: 100 * time.Millisecond, // 最多等待 100ms 强制发送
        BufferSize:   10000,                 // 内存最大缓冲 10000 条消息
        Concurrency:  4,                     // 4 个并发发送 goroutine
        AsyncMode:    true,                  // 异步发送模式
    }
}
```

- **消息聚合**：多个单条请求在 `buffer` 中累积，通过 `sarama.SendMessages()` 一次性批量提交，极大减少网络 RTT。
- **双重触发刷新**：`BatchSize` 条数触发 + `BatchTimeout` 超时触发，兼顾吞吐量与延迟。
- **缓冲区满背压控制**：当 `len(buffer) >= BufferSize` 时强制刷新，防止内存无限增长。
- **Snappy 压缩**：`sarama.CompressionSnappy`，减少网络带宽消耗。

#### 3.2.3 流式批量发送器（StreamBatchSender）

文件：`aidc-pretreat-service/internal/kafka/batch_producer.go`

针对持续流式数据场景，`StreamBatchSender` 通过 **Channel + Worker Pool** 实现彻底异步：

```
调用方 Send() → msgChan (Channel) → Worker Goroutine × N → BatchProducer.SendBatch()
```

```go
// 非阻塞发送（背压控制）
func (sbs *StreamBatchSender) Send(key string, value []byte) error {
    select {
    case sbs.msgChan <- msg:
        return nil
    default:
        return fmt.Errorf("channel full, backpressure") // 通道满时直接返回背压错误
    }
}

// 阻塞发送（等待通道有空间）
func (sbs *StreamBatchSender) SendBlocking(key string, value []byte) error {
    select {
    case sbs.msgChan <- msg:
        return nil
    case <-sbs.ctx.Done():
        return fmt.Errorf("sender closed")
    }
}
```

#### 3.2.4 分片发送（producer_chunk.go）

文件：`aidc-pretreat-service/internal/kafka/producer_chunk.go`

当单条消息体积过大时，使用分片机制将数据切分后发送，并以 `TraceID` 作为 Key 保证同一批次分片进入同一 Partition，便于下游按序重组：

```go
// 批量发送分片，失败时自动降级逐个发送（保证数据不丢失）
func (p *Producer) SendChunkBatch(chunks []ChunkItem) error {
    err := p.producer.SendMessages(messages)
    if err != nil {
        // 回退策略：逐个发送，确保数据不丢失
        for i, chunk := range chunks {
            p.SendChunk(chunk.TraceID, chunk.Data)
        }
    }
}
```

### 3.3 大数据处理路径（MinIO + Kafka 组合）

文件：`aidc-pretreat-service/internal/processor/processor.go`

预处理服务根据数据大小采用不同处理路径，实现了**自适应的流量削峰**：

```
请求数据 (data)
    │
    ├─ data < 10MB（SizeThreshold）
    │       │
    │       └─ 直接内联写入 Kafka（InlineMessage）
    │
    └─ data >= 10MB
            │
            ├─ 1. Zstd 压缩数据
            ├─ 2. 将压缩数据存入 MinIO 对象存储
            ├─ 3. 仅发送"引用消息"（ReferenceMessage，仅含 MinIO 路径）到 Kafka
            └─ Kafka 消息体积大幅降低，削峰效果更明显
```

这样 Kafka 中流转的大数据消息仅是一个轻量级指针（MinIO 对象路径），避免 Kafka 被大消息撑爆。

### 3.4 异步任务队列（内部流量削峰）

文件：`aidc-pretreat-service/internal/async/task_queue.go`

预处理服务内部同样使用 **内存 Channel + Worker Pool** 实现 HTTP 请求的异步化处理：

```go
// 默认配置
func DefaultTaskQueueConfig() *TaskQueueConfig {
    return &TaskQueueConfig{
        QueueCapacity:   10000, // Channel 缓冲 10000 个任务
        WorkerCount:     100,   // 100 个并发 Worker goroutine
        MaxPendingTasks: 50000, // 最多 50000 个待处理任务
        RejectOnFull:    true,  // 队列满时返回 503（背压）
    }
}
```

**HTTP 请求处理流程（异步解耦）：**

```
HTTP 请求
    │
    ▼ 立即读取 Body 到内存
    │
    ▼ 构建 Task，写入 TaskQueue Channel（非阻塞）
    │
    └─► 立即返回 HTTP 202 Accepted（解耦，调用方无需等待）
              │
              Worker Pool 异步消费 Task
                   │
                   └─► 调用 Processor → 写入 Kafka
```

## 4. aidc-process-service（处理服务）

### 4.1 角色：Kafka 消费者（含限速器）

处理服务负责从 Kafka 消费设备原始数据，解析后写入 Redis 和 VictoriaMetrics。它是**流量削峰的落地点**——Kafka 在上游平滑了突发流量，处理服务可以按照自身吞吐能力稳定消费。

### 4.2 Consumer 实现

文件：`aidc-process-service/pkg/kafka/consumer.go`

```go
// 每个 Topic 启动一个独立 goroutine 并发消费
func (c *Consumer) Start(ctx context.Context) error {
    for _, topic := range c.config.Topics {
        reader := kafka.NewReader(kafka.ReaderConfig{
            Brokers:        c.config.Brokers,
            Topic:          topic,
            GroupID:        c.config.GroupID,       // Consumer Group 保证消息只被消费一次
            MinBytes:       c.config.MinBytes,
            MaxBytes:       c.config.MaxBytes,       // 10MB 最大拉取
            MaxWait:        c.config.MaxWait,        // 500ms 最长等待
            CommitInterval: c.config.BatchTimeout,  // 自动提交间隔
            StartOffset:    kafka.LastOffset,        // 从最新偏移量开始消费
        })
        go c.consumeTopic(ctx, topic, reader)
    }
}
```

**Consumer Group 机制：**
- `GroupID: "aidc-process-service"` 保证多实例部署时每条消息只被一个实例处理，实现水平扩展。
- `StartOffset: kafka.LastOffset` 服务重启后从上次 Commit 位置继续，不重复消费。

**手动提交 Offset（At-least-once 语义）：**

```go
// 先处理成功，再提交 Offset，保证消息不丢失
if err := c.handleMessage(ctx, topic, msg); err != nil {
    klog.Errorf("...")
    continue  // 处理失败不提交，下次重新消费
}
if err := reader.CommitMessages(ctx, msg); err != nil {
    klog.Errorf("...")
}
```

### 4.3 Topic 限速器（流量削峰核心）

文件：`aidc-process-service/pkg/kafka/rate_limiter.go`

这是**流量削峰**的关键机制：基于 `golang.org/x/time/rate` 令牌桶算法，对不同 Topic 设置独立限速规则。

```go
// 各 Topic 独立限速配置
RateLimits: map[string]RateLimitConfig{
    "jinhua":                 {Rate: 5,  Burst: 10},  // 金华项目：5 条/秒，最多突发 10 条
    "bms-D04-rtdata-shuoyao": {Rate: 20, Burst: 40},  // BMS 朔耀：20 条/秒，最多突发 40 条
    "raw":                    {Rate: 10, Burst: 20},  // 原始数据：10 条/秒，最多突发 20 条
}
```

**工作原理（令牌桶算法）：**

```
Kafka 中有大量积压消息（流量高峰）
    │
    ▼ Consumer 拉取消息
    │
    ▼ rateLimiter.Wait(ctx, topic) ← 等待令牌桶放行
    │    令牌按固定速率（Rate/秒）补充
    │    突发流量最多消费 Burst 个令牌
    │
    ▼ 获得令牌，调用 handleMessage()
    │
    ▼ 处理完成，提交 Offset
```

```go
// 消费循环中的限速等待
if c.rateLimiter != nil {
    if err := c.rateLimiter.Wait(ctx, topic); err != nil {
        klog.Warningf("[Kafka] rate limiter wait error for topic %s: %v", topic, err)
        continue
    }
}
```

这样无论 Kafka 中积压了多少消息（比如某次大批量数据上报），处理服务始终按照预设速率稳定消费，避免压垮下游 Redis 和 VictoriaMetrics。

### 4.4 Worker Pool 并发处理

文件：`aidc-process-service/pkg/processor/message_processor.go`

消费到的每条 Kafka 消息由 `MessageProcessor` 处理，内部通过 **Worker Pool**（容量 = `WorkerPoolSize`）控制最大并发，防止同时处理过多消息耗尽内存：

```go
func (p *MessageProcessor) Handle(ctx context.Context, msg kafka.Message) error {
    p.workerPool <- struct{}{} // 占用一个 Worker 槽位（满了则阻塞）
    defer func() { <-p.workerPool }()

    // 并行写入 Redis 和 VictoriaMetrics
    var wg sync.WaitGroup
    wg.Add(1)
    go func() { p.redisWriter.WriteDevices(ctx, topic, devices) }()
    wg.Add(1)
    go func() { p.victoriaWriter.WriteDevices(ctx, devices) }()
    wg.Wait()
}
```

## 5. aidc-metric-service（指标计算服务）

### 5.1 角色：Kafka 消费者 + 生产者（级联触发）

指标计算服务基于 Kafka 实现了**异步流式计算引擎**，包含：
1. **calc-task 消费者**：消费计算任务，执行公式计算
2. **DLQ 消费者**：消费死信队列，进行失败补偿
3. **级联触发生产者**：计算完成后，触发下游依赖计算点的新任务

### 5.2 Kafka Producer 配置

文件：`aidc-metric-service/pkg/storage/kafka_producer.go`

```go
writer := &kafka.Writer{
    Addr:         kafka.TCP(config.Brokers...),
    Topic:        config.Topic,
    Balancer:     &kafka.LeastBytes{},  // 最少字节均衡，防止某个 Partition 过热
    MaxAttempts:  3,                    // 最多重试 3 次
    WriteTimeout: 10 * time.Second,
    ReadTimeout:  10 * time.Second,
}
```

**大消息自动拆分（防止消息体积超限）：**

```go
const MaxMessageSize = 2560000 // 2.56MB

func (p *KafkaProducer) ProduceData(key string, data []byte) error {
    if len(data) <= MaxMessageSize {
        return p.sendMessage(key, data)
    }
    // 超过 2.56MB，递归对半拆分发送
    return p.splitAndSend(key, data)
}

func (p *KafkaProducer) splitAndSend(key string, data []byte) error {
    mid := len(data) / 2
    p.ProduceData(key+"_part1", data[:mid])
    p.ProduceData(key+"_part2", data[mid:])
}
```

### 5.3 calc-task 消费者（计算任务流）

文件：`aidc-metric-service/cmd/aidc-metric-service/main.go`

```go
calcReader := kafka.NewReader(kafka.ReaderConfig{

```

### 5.4 级联触发机制（calc → calc 异步解耦）

这是 `aidc-metric-service` 中**最精妙的异步解耦设计**：当某个指标点（calc_dpid）计算完成后，不直接调用下游计算，而是向 Kafka 发送新的 `calc-task` 消息触发下游——**计算节点间通过 Kafka 完全解耦，天然支持任意深度的 DAG 计算图**。

```
calc_dpid_A 计算完成
    │
    ▼ buildCascadeTriggerFunc()
    │  从 CalcDependencyIndex 查找 A 的下游依赖计算点 [B, C, D]
    │
    ├─ 向 aidc-calc-task Topic 发送 calcTask(B)
    ├─ 向 aidc-calc-task Topic 发送 calcTask(C)
    └─ 向 aidc-calc-task Topic 发送 calcTask(D)
              │
              └─► B/C/D 被其他 Consumer 实例异步消费执行
```

```go
// 级联触发：为每个下游计算点构建并发送计算任务消息
msgs := make([]kafka.Message, 0, len(downstream))
for _, cp := range downstream {
    // 从 Redis R2 缓存（精确时间点值）+ R1 缓存（最新值）读取依赖值
    // 构建 CalcTaskMessage，发送到 aidc-calc-task
    ...
}
producer.WriteMessages(ctx, msgs...)
```

**双缓存读取（R2 精确时间点 + R1 最新值回退）：**

```
R2 Key = r2:{source}:{dpid}:{trigger_ts_sec}  ← 时间点精确匹配
R1 Key = hash:{source}:{dpid}                 ← 最新值回退

先批量 MGET R2 → 命中则用精确值
未命中的依赖 → HMGET R1 → 用最新值补充
```

### 5.5 死信队列（DLQ）补偿机制

文件：`aidc-metric-service/pkg/dlq/handler.go`

当计算任务处理失败时，`CalcProcessor` 将失败消息写入 `aidc-calc-task-dlq` Topic，`DLQHandler` 异步消费并执行错误类型特定的补偿策略：

```
计算失败
    │
    ▼ 写入 aidc-calc-task-dlq（DLQ Producer）
    │
    └─► DLQHandler 消费
              │
              ├─ cascade_trigger_failed → 检查幂等键，重新触发级联
              ├─ config_missing → 等待热重载后重新入队
              ├─ snapshot_expired → 触发 P2 告警，忽略
              ├─ formula_error → 触发 P1 告警，忽略
              └─ deserialize_error → 触发 P2 告警，忽略
```

```go
// DLQ 消费者配置
dlq.NewDLQHandler(&dlq.DLQConfig{
    Brokers:       brokers,
    DLQTopic:      "aidc-calc-task-dlq",
    CalcTopic:     "aidc-calc-task",
    GroupID:       "aidc-metric-service-calc-dlq",
    MaxRetries:    5,
    RetryInterval: 2 * time.Second,
})
```

## 6. aidc-north-sync / aidc-platform-service（未使用 Kafka）

经过代码扫描：

- **`aidc-north-sync`**：无 Kafka 相关代码，该服务负责北向数据同步，直接通过 HTTP/gRPC 与上游对接，不参与 Kafka 数据流。
- **`aidc-platform-service`**：无 Kafka 相关代码，该服务负责平台管理类业务，直接操作数据库，不参与 Kafka 数据流。

## 7. 异步解耦原理说明

### 7.1 HTTP 接入层解耦（pretreat-service）

传统同步调用链：

```
调用方 ──等待──► HTTP Server ──等待──► 数据处理 ──等待──► 存储写入 ──► 返回结果
         ↑_____等待时间 = 全链路时间______↑
```

引入 Kafka 后的异步解耦链：

```
调用方 ──► HTTP Server ──► 写入 Kafka ──► 立即返回 202（毫秒级）
                                │
                                └─ 后台 Consumer 异步处理 ──► 存储写入
```

**解耦收益：**
- 调用方响应时间从"全链路时间"缩短为"写入 Kafka 时间"（通常 < 5ms）
- 任意一个下游服务故障，不影响上游接收数据（消息堆积在 Kafka，待故障恢复后继续消费）
- pretreat-service 和 process-service 可以独立扩缩容

### 7.2 计算层解耦（metric-service 级联）

公式指标点之间存在复杂的依赖关系（DAG 图），通过 Kafka 级联触发实现：
- 每个计算节点只关心自己的输入，不直接调用下游
- 下游计算通过消费 Kafka 消息异步触发
- 天然支持并行计算（多个下游点在不同 Consumer 实例上并发执行）
- 计算深度通过 `MaxCascadeDepth` 防护，避免循环依赖死锁

## 8. 流量削峰原理说明

### 8.1 Kafka 天然的削峰能力

设备数据上报往往存在周期性突发（如整点批量上报）：

```
流量曲线（无 Kafka）：
     │
高峰 │████
     │████
平均 │██████████████
     └──────────────────► 时间
       处理服务需要按峰值设计容量

流量曲线（有 Kafka）：
生产速率 ────────────────────────────── （随业务波动）
消费速率 ════════════════════════════   （稳定，按限速处理）
堆积量   ▲▲▲▲▲▲▲▼▼▼▼▼▼              （高峰时堆积，低谷时消化）
```

### 8.2 多层削峰机制

#### 层次一：pretreat-service 内部 TaskQueue（内存削峰）

```
HTTP 突发请求 → TaskQueue Channel（容量 10000）→ Worker Pool（100 个）→ Kafka
```
- Channel 吸收瞬时突发请求
- 若 `MaxPendingTasks=50000` 超限，返回 HTTP 503 背压，告知调用方限速

#### 层次二：Kafka Broker（分布式削峰）

- Kafka Broker 可无限（磁盘容量内）堆积消息
- 生产速率和消费速率完全解耦，可相差数十倍
- 消息持久化，服务重启不丢失

#### 层次三：process-service 令牌桶限速（消费侧削峰）

```
jinhua Topic：            ──5条/秒──►  处理
bms-D04-rtdata-shuoyao： ──20条/秒──► 处理
raw Topic：              ──10条/秒──► 处理
```
- 按 Topic 独立限速，高优先级 Topic 可配置更高消费速率
- 令牌桶支持 Burst，允许短时间超速（弹性）

#### 层次四：大消息 MinIO 分流（带宽削峰）

- 大于 10MB 的数据不直接进 Kafka，改走 MinIO 存储 + Kafka 只传指针
- 避免大消息占用 Kafka 网络带宽，保证小消息的低延迟

#### 层次五：BatchProducer 批量聚合（写入侧削峰）

- 100 条消息聚合为 1 次网络请求
- 将高频小写转换为低频大写，减轻 Kafka Broker 压力

## 10. 难点分析

### 10.1 级联计算的正确性保证（最难）

**难点所在**：`aidc-metric-service` 中的 calc→calc 级联触发，本质是在 Kafka 上跑一个**异步 DAG 计算图**。

核心挑战：

- **时序问题**：上游 `calc_dpid_A` 计算完成后，其值写入 R2 缓存（`r2:{source}:{dpid}:{ts}`），下游 B 依赖 A 的值。如果 B 的消费比 A 的 R2 写入更快，B 读 R2 会 miss，回退到 R1（最新值），可能拿到**旧时间点的值**，导致计算结果在时间维度上不一致。
- **循环依赖保护**：通过 `MaxCascadeDepth` 防护，但深度阈值设多少合适？设小了截断合法计算链，设大了有死循环风险。
- **幂等性**：失败重试时，同一 `calc-task` 消息可能被重复处理，必须保证重复计算不产生错误数据（目前靠 Redis 幂等键，但需确认实现是否完备）。

### 10.2 TaskQueue 背压计数 Bug

**难点所在**：`aidc-pretreat-service` 的 `async.Worker` 在 `processTask` 执行完后，**没有调用 `taskQueue.OnTaskCompleted`**，导致 `currentPending` 计数**只增不减**。

```go
// aidc-pretreat-service/internal/async/worker.go:62-97
// processTask 执行完后 ——> 缺少 taskQueue.OnTaskCompleted(task, success) 回调
```

后果：服务运行一段时间后，`currentPending` 必然超过 `maxPendingTasks=50000`，此后**所有请求都返回 503**，直到服务重启。

### 10.3 大数据流的背压控制竞态

`TaskQueue.Enqueue` 中先读 `currentPending`（原子操作）再执行 channel select，两步之间没有锁，高并发时可能同时通过检查，导致实际入队数量超过 `maxPendingTasks` 上限。

### 10.4 Kafka Consumer 限速与 Offset 提交的一致性

`process-service` 消费循环：`FetchMessage → rateLimiter.Wait → handleMessage → CommitMessages`

当 `rateLimiter.Wait` 阻塞时，消息已拉取但未提交 Offset。服务此时重启，消息会被重新消费（At-least-once 语义），下游 Redis/VictoriaMetrics 需要能接受幂等写入，否则产生重复数据。

### 10.5 BatchProducer 失败消息黑洞

`BatchProducer` 使用 `sarama.AsyncProducer`，错误通过 `Errors()` channel 异步返回。在 `StreamBatchSender` 场景下，调用方调用 `Send()` 后立即返回，**无法得知消息是否真正成功发送**。如果 Kafka Broker 临时不可用，消息在 buffer 里积压，超出 `BufferSize` 后会被静默丢弃。

## 11. 可优化的点

### 优化1：修复 `OnTaskCompleted` 未被调用（P0 Bug）

`Worker` 需要在处理完任务后回调 `TaskQueue.OnTaskCompleted`，修复 `currentPending` 只增不减的问题。

修复方案：在 `Worker` 中持有 `TaskQueue` 引用，或在 `TaskQueue.Start` 中通过装饰器包装 processor：

```go
// 在 Worker.processTask 末尾添加回调
defer func() {
    tq.OnTaskCompleted(task, err == nil)
}()
```

### 优化2：process-service 消费并发度（P2 性能）

当前每个 Topic 只有一个 goroutine 顺序拉取消息，吞吐量上限 = 单条消息处理耗时的倒数。优化方案：按 Partition 数量启动多个消费 goroutine，每个 goroutine 绑定一个 Partition：

```go
// 每个 partition 启动一个消费 goroutine，充分利用 Kafka 并发能力
for partition := 0; partition < topicPartitionCount; partition++ {
    go consumePartition(ctx, topic, partition)
}
```

### 优化3：R2 缓存时序竞争 → 级联消息携带依赖值（P1 正确性）

针对级联触发时序问题，可在 `calc-task` 消息中**直接携带上游已计算的依赖值**，下游无需再读 Redis，彻底消除时序竞争（牺牲消息体积换取时序确定性）。

### 优化4：BatchProducer 失败消息落盘（P1 正确性）

`BatchProducer` 发送失败的消息目前直接丢弃（仅记录 `errorCount`）。优化方向：

- 失败消息写入**本地 WAL 文件队列**，服务恢复后重放
- 或集成 DLQ 机制，失败消息写入 Kafka DLQ Topic，由 `DLQHandler` 统一补偿

### 优化5：metric-service 幂等键粒度（P2 正确性）

DLQ handler 中 `cascade_trigger_failed` 的幂等键建议加入时间戳维度：

- 幂等键使用 `(calc_dpid, trigger_ts_sec)` 组合，而不只是 `calc_dpid`，避免同一点不同时间戳的计算被错误去重
- 幂等键 TTL 应大于最大重试窗口（当前 `MaxRetries=5, RetryInterval=2s`，TTL 应 > 10s）

## 12. 优化优先级汇总

| 优先级 | 问题 | 影响 |
|---|---|---|
| P0（Bug）| `OnTaskCompleted` 未调用，`currentPending` 只增不减 | pretreat-service 必然退化到全量 503 |
| P1（正确性）| R2 缓存时序竞争，级联计算结果时间不一致 | 指标计算准确性 |
| P1（正确性）| BatchProducer 失败消息静默丢弃 | 数据丢失 |
| P2（性能）| 单 goroutine 消费 Topic，吞吐量受限 | 消费积压，延迟上升 |
| P2（正确性）| 幂等键粒度不足，可能跨时间戳去重 | 重复计算或漏算 |
