## 二、项目描述

AIDC 是一套面向智慧数据中心的物联网数据管理平台，采用 **Go 语言微服务架构**，覆盖从 BMS 设备数据采集、预处理、实时消费、指标计算到平台管理的完整数据链路。平台由 **5 个核心微服务** 组成，日均处理 `[待补充]` 万条设备数据，支撑 `[待补充]` 个机房、`[待补充]` 台设备、`[待补充]` 万个监控点位的全量数据管理。

**核心技术栈**：Go 1.22+ / gRPC + Protobuf / Gin / PostgreSQL + GORM / Kafka / Redis / VictoriaMetrics / MinIO / Zstd / Casdoor SSO + JWT / Docker + Kubernetes + Helm

## 四、主要职责（STAR 法则重构版）

### 职责 1：架构并研发数据预处理服务，实现多模式数据接入与智能分流

> **S（背景）**：平台需要接收来自 `[待补充]` 个机房的 BMS 设备实时数据推送，单次请求数据量从几 KB 到数百 MB 不等，Kafka 默认 1MB 消息限制无法直接承载大数据量场景，且高峰期并发请求达 `[待补充]` QPS。
>
> **T（任务）**：设计一套能够自动适配不同数据量级的预处理服务，同时保证高并发下的稳定性和吞吐量。
>
> **A（方案）**：
> - **架构**了基于 Gin 的 HTTP 数据预处理服务（aidc-pretreat-service），设计 4 种数据接入模式：流式分片、MinIO 异步上传、一次性上传、压缩上传
> - **研发**了智能数据分流引擎，以 10MB 为阈值自动选择处理策略——小数据直发 Kafka，大数据经 Zstd 压缩（压缩比 3-10 倍）后存入 MinIO 并向 Kafka 发送引用消息
> - **设计**了企业级异步任务队列架构（TaskQueue + Worker Pool），队列容量 10000、100 个并发 Worker，支持背压控制（队列满时返回 503 拒绝）和优雅关闭
> - **集成** Prometheus 监控，采集请求量、延迟分布、压缩率等核心指标
>
> **R（结果）**：HTTP 请求平均响应时间控制在 `[待补充]` ms 以内（异步模式），数据预处理吞吐量达 `[待补充]` MB/s，Zstd 压缩使网络带宽占用降低约 `[待补充]`%。

---

### 职责 2：主导设计 Kafka 多 Topic 实时消费引擎，实现设备数据双写存储

> **S（背景）**：平台需实时消费来自 `[待补充]` 个 Kafka Topic（jinhua、raw、bms-* 等）的设备数据流，日均消息量 `[待补充]` 万条，且不同 Topic 数据量差异巨大（bms 系列 Topic 数据量是 jinhua 的 `[待补充]` 倍），需要差异化流量控制以保护下游存储。
>
> **T（任务）**：构建高吞吐、高可靠的实时数据消费处理引擎，将设备数据双写到 Redis（实时查询）和 VictoriaMetrics（历史分析）。
>
> **A（方案）**：
> - **研发**了基于 segmentio/kafka-go 的多 Topic 并行消费框架，每个 Topic 独立 goroutine 消费，互不阻塞
> - **设计**了基于 Token Bucket 的差异化限流策略：为不同 Topic 配置独立限流（如 jinhua 5 msg/s、bms-* 20 msg/s），保护 Redis 和 VMS 不被打满
> - **实现**了解析器工厂模式（ParserFactory），根据 Topic 自动匹配解析器（JinhuaParser / RawParser / BmsParser），支持热插拔新数据格式
> - **优化** Redis 写入性能：采用 Hash 结构（`HSET {source} {point_guid} {json_data}`）存储实时值，配合 Pipeline 批量提交
> - **优化** VictoriaMetrics 写入：实现缓冲批量写入（10000 条 / 10s flush）+ 60s 时间窗口去重（DedupCache），避免 Kafka at-least-once 语义导致的重复写入
> - **采用**手动 offset 提交策略，处理成功后才提交，保证消息不丢失
>
> **R（结果）**：实现 `[待补充]` msg/s 的稳定消费吞吐量，Redis 实时数据查询延迟 < `[待补充]` ms，VMS 历史数据写入成功率 `[待补充]`%，服务连续运行 `[待补充]` 天无故障。

### 职责 3：研发流式指标计算引擎，支持公式热加载与拓扑排序级联计算

> **S（背景）**：数据中心有 `[待补充]` 个计算指标（如 PUE、机房功率、温湿度统计），指标之间存在复杂的依赖关系（如「楼层总功率」依赖「各机房功率」），且业务方要求新增/修改计算公式后无需重启服务即可生效。
>
> **T（任务）**：构建一套支持实时计算 + 定时调度双模式的流式计算引擎，解决指标依赖排序、公式热更新、计算失败补偿三大核心问题。
>
> **A（方案）**：
> - **架构**了双模式指标计算引擎（aidc-metric-service）：
>   - **流式模式**：消费 Kafka `aidc-calc-task` Topic，当原始数据更新时实时触发依赖链计算
>   - **定时模式**：自研 Scheduler 支持 7 种计算周期（10s / 1min / 5min / 1h / 日 / 月 / 交接班）
> - **研发**了基于 AST（抽象语法树）的公式引擎，启动时预编译公式为 AST 树，运行时递归求值，支持四则运算和聚合函数（SUM / AVG / MAX / MIN）
> - **实现**了基于 Kahn 算法的拓扑排序，将 `[待补充]` 个计算点按依赖层级分组，逐层并行计算（Worker Pool 100 个并发），确保计算顺序正确
> - **设计**了级联触发机制：计算完成后通过 CalcDependencyIndex 查询下游依赖，自动生成新的 calc-task 消息发送到 Kafka，实现「数据更新 → 逐层传播计算」
> - **实现**了 DLQ（死信队列）补偿机制：计算失败的任务进入 `aidc-calc-task-dlq`，由独立 DLQHandler 消费重试（最多 5 次，间隔 2s），不阻塞主计算流程
> - **实现**了公式热加载：MetricPointLoader 每 5 分钟从 Redis 扫描 `formula:*` 键，增量更新公式索引，无需重启服务
>
> **R（结果）**：流式计算端到端延迟 < `[待补充]` ms，定时计算 `[待补充]` 个指标全量计算耗时 < `[待补充]` s，DLQ 补偿使计算成功率从 `[待补充]`% 提升至 `[待补充]`%，公式热加载实现零停机更新。

### 职责 4：设计平台管理服务（BFF 层），实现统一认证、权限控制与数据查询

> **S（背景）**：平台需要为前端提供统一的 API 入口，涉及用户认证、RBAC 权限控制、设备数据查询、计算点管理等多个业务域，同时部署在 K8s 多 Pod 环境下，计算点初始化需要防止重复执行。
>
> **T（任务）**：构建一个高可靠的 BFF 层服务，作为前端与底层数据（PostgreSQL / Redis / VictoriaMetrics）之间的桥梁。
>
> **A（方案）**：
> - **架构**了基于 gRPC 的平台管理服务（aidc-platform-service），注册 12 个 gRPC 服务模块（Auth / User / Dict / DcomLocation / MetricPoints / DcimMetrics 等）
> - **研发**了灵活的 Casdoor SSO + JWT 认证拦截器，支持三种白名单匹配方式（gRPC 方法全路径 / HTTP 路径 / 路径后缀），在不侵入业务代码的前提下实现统一认证
> - **设计**了计算点数据初始化方案：使用 Redis 分布式锁（SETNX, TTL=5min）+ goroutine 锁续期（60s 间隔），防止多 Pod 重复初始化；分页查询（每页 1000 条）+ Redis Pipeline 批量写入，将 `[待补充]` 个计算点配置加载到 Redis
> - **封装**了 VictoriaMetrics 查询构建器（VictoriaQueryBuilder），简化 PromQL 查询逻辑，支持范围查询、聚合查询、标签过滤
> - **实现**了基于 Snowflake 算法的分布式 ID 生成器（15 位机器 ID + 7 位序列号），支持 32768 台机器
>
> **R（结果）**：API 响应平均延迟 `[待补充]` ms，计算点初始化从 `[待补充]` s 优化至 `[待补充]` s（Pipeline 批量写入），认证拦截器对请求延迟增加 < `[待补充]` ms。

### 职责 5：研发多配置动态调度的北向数据同步服务

> **S（背景）**：平台需要从 `[待补充]` 个机房的北向 API 定时拉取设备配置数据（位置 / 设备 / 点位），早期同步逻辑内嵌在 process-service 中，仅支持单一配置且修改需重启服务，无法满足多机房、多数据源的管理需求。
>
> **T（任务）**：将同步功能独立为微服务，支持多配置管理、运行时动态调度、HTTP API 管理，提升运维效率。
>
> **A（方案）**：
> - **主导**将同步模块从 process-service 中拆分为独立微服务（aidc-north-sync），采用 Gin + REST API 架构
> - **设计**了三表分离的数据模型：`north_configure`（连接信息）、`north_sync_config`（同步策略）、`sync_task`（执行记录），配置与执行完全解耦
> - **架构**了动态多配置调度器：每个同步配置拥有独立的 Cron 实例（`map[configName] → cronInstance`），支持运行时动态启停，Cron 执行时从 DB 重新读取最新配置实现热更新
> - **实现**了 ConfigService 与 Scheduler 的联动机制：配置 CRUD 操作自动触发调度器启停（创建 → 启动 Cron / 删除 → 停止 Cron / 更新 → 重载 Cron），通过 `SchedulerReloader` 接口解耦避免循环依赖
> - **优化**了同步效率：基于版本号（version）增量同步，只有远程版本变化时才执行全量拉取；分页拉取（每页 500 条）+ 事务批量写入 PostgreSQL
> - **实现**了异步任务管理器（SyncTaskManager）：channel 任务队列 + 动态创建 SyncService 实例，支持任务状态持久化和历史查询
>
> **R（结果）**：从单一配置扩展到支持 `[待补充]` 个机房的独立同步配置，版本增量同步减少 `[待补充]`% 的无效同步请求，运维人员可通过 API 实时管理同步任务，无需重启服务。

## 六、高频面试连环追问准备

### 追问链 1：Kafka 消费

1. **你提到手动提交 offset，具体在什么时机提交？如果消费成功但提交 offset 失败怎么办？**
   > PREP 答案：消费处理成功后才调用 CommitMessages。如果 offset 提交失败，下次会重新消费该消息（at-least-once 语义），通过 VictoriaMetrics 的 DedupCache（60s 窗口去重）和 Redis HSET 覆盖写来保证幂等性。

2. **Token Bucket 限流为什么不用滑动窗口？**
   > PREP 答案：Token Bucket 允许突发流量（burst），适合 IoT 场景中设备定时上报导致的流量尖刺。滑动窗口更适合严格匀速的场景。

3. **如果某个 Topic 的消费者挂了，其他 Topic 会受影响吗？**
   > PREP 答案：不会。每个 Topic 是独立的 goroutine 消费，一个 Topic 的 goroutine panic 会被 recover 捕获，不影响其他 Topic。

### 追问链 2：指标计算引擎

1. **拓扑排序如果检测到循环依赖怎么办？**
   > PREP 答案：Kahn 算法天然能检测环。如果排序结束后还有节点未被处理，说明存在循环依赖。我们的策略是跳过这些节点并记录告警日志，同时在 API 层创建计算点时做前置校验，禁止形成环形依赖。

2. **公式热加载有没有并发安全问题？**
   > PREP 答案：有。MetricPointLoader 加载新公式时使用 sync.RWMutex 保护索引：读操作用 RLock，更新操作用 Lock。更新是先构建新索引，再原子替换，不影响正在进行的计算。

3. **DLQ 最多重试 5 次之后呢？**
   > PREP 答案：超过 5 次放弃重试，记录 Error 级别日志。实际生产中会配合告警系统（如 Prometheus AlertManager）通知运维人员介入排查。

### 追问链 3：Redis 分布式锁

1. **SETNX 锁如果持有者 Pod 崩溃，锁怎么释放？**
   > PREP 答案：设置了 TTL=5min，即使持有者崩溃，锁也会在 5 分钟后自动过期释放。goroutine 续期是为了正常场景下防止任务执行时间超过 TTL。

2. **为什么不用 Redlock？**
   > PREP 答案：Redlock 需要至少 3 个独立 Redis 实例，架构更重。我们的场景是计算点初始化（低频操作，只在服务启动时执行一次），单节点 SETNX + TTL 已经足够，没有必要引入 Redlock 的复杂度。

### 追问链 4：高并发与性能优化

1. **Redis Pipeline 每批多少条？有没有做过基准测试？**
   > PREP 答案：每页 1000 条计算点，每页产生约 `[待补充]` 个 Redis 命令，统一通过 Pipeline 发送。相比逐条写入，Pipeline 将 `[待补充]` 个计算点的初始化时间从 `[待补充]` s 缩短到 `[待补充]` s。

2. **VictoriaMetrics 批量写入的 buffer 满了会怎样？**
   > PREP 答案：设置了 10000 条或 10 秒两个 flush 条件。如果写入 VMS 失败，会重试 `[待补充]` 次，最终失败则丢弃并记录日志。因为 IoT 数据是持续产生的，偶尔丢失几秒数据可接受（非金融场景）。
