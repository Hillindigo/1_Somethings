# AIDC 微服务平台 - 总体架构分析

## 一、项目概述

AIDC（AI Data Center）是一套面向**智慧数据中心**的物联网数据管理平台，采用 Go 语言微服务架构，覆盖从设备数据采集、预处理、实时消费、指标计算到平台管理的完整数据链路。

平台由 **5 个核心微服务** 组成，各服务职责清晰、松耦合，通过 Kafka 消息队列串联数据流，通过 PostgreSQL 共享业务数据。

## 三、数据流转全链路

```
1. 数据采集层
   外部设备/BMS → HTTP 推送 → aidc-pretreat-service（预处理）
                                    │
                                    ▼
                     ┌──── 小数据（<10MB）→ 直接写入 Kafka
                     │
                     └──── 大数据（≥10MB）→ Zstd 压缩 → 写入 Kafka（含压缩数据）

2. 数据同步层
   北向 API → aidc-north-sync / aidc-process-service → PostgreSQL
   （同步位置、设备、点位配置信息）

3. 数据消费处理层
   Kafka → aidc-process-service（多 Topic 消费 + 限流）
              │
              ├── 解析 JSON 格式设备数据
              ├── Redis HSET（实时值）
              └── VictoriaMetrics（时序历史数据）

4. 指标计算层
   Kafka(calc-task) → aidc-metric-service（流式计算引擎）
              │
              ├── 公式引擎（四则运算、聚合函数）
              ├── 拓扑排序（按依赖层级计算）
              ├── 定时调度（10s/1min/5min/1h/日/月/交接班）
              ├── 结果写 Redis + VictoriaMetrics
              └── DLQ 补偿机制（失败重试）

5. 平台展示层
   aidc-platform-service → gRPC API → 前端/网关
              │
              ├── 用户认证（Casdoor SSO + JWT）
              ├── 设备/位置/点位查询
              ├── 计算点 CRUD + 实时值查询
              └── VictoriaMetrics 历史数据查询
```

## 四、核心技术栈总览

| 分类 | 技术 | 说明 |
|------|------|------|
| **语言** | Go 1.22+ | 所有服务统一使用 Go |
| **RPC 框架** | gRPC + Protobuf | platform-service、process-service、metric-service |

| **时序数据库** | VictoriaMetrics | 设备历史数据存储与查询 |

| **认证** | Casdoor SSO + JWT | 统一身份认证 |
| **压缩** | Zstd | 大数据预处理压缩 |
| **定时调度** | robfig/cron | 定时同步、定时计算 |
| **日志** | klog (k8s) | 结构化日志 |
| **部署** | Docker + Kubernetes (Helm) | 容器化部署 |
| **CI/CD** | GitLab CI | 自动构建部署 |
| **监控** | Prometheus | 指标采集（pretreat-service） |
| **ID 生成** | Snowflake | 分布式 ID（platform-service） |

## 五、各服务职责与定位

| 服务 | 定位 | 协议 | 核心能力 |
|------|------|------|---------|
| **aidc-platform-service** | 平台管理服务（BFF 层） | gRPC | 用户/权限管理、设备查询、计算点管理、VMS 查询 |
| **aidc-process-service** | 数据处理服务 | gRPC | Kafka 消费 → Redis + VMS 双写、北向同步 |
| **aidc-metric-service** | 指标计算服务 | gRPC | 流式计算引擎、定时调度计算、公式引擎、DLQ |

| **aidc-north-sync** | 北向数据同步服务 | HTTP (Gin) | 多配置同步、定时+手动触发、北向配置 CRUD |

## 六、服务间协作关系

```
aidc-pretreat-service ──(Kafka)──▶ aidc-process-service ──(Kafka calc-task)──▶ aidc-metric-service
         │                                │                                          │
         │                                │                                          │
    接收外部数据                     消费 + 双写存储                           计算 + 双写存储
    压缩 + 写 Kafka                Redis / VMS                             Redis / VMS
                                          │
                                          ▼
                                  aidc-platform-service
                                  (查询 Redis/VMS/DB)

aidc-north-sync ──(PostgreSQL)──▶ aidc-process-service / aidc-platform-service
         │
    同步北向配置数据
    (位置/设备/点位)
```

**关键协作链路：**
1. **数据采集链路**：`外部设备 → pretreat-service → Kafka → process-service → Redis/VMS`
2. **计算链路**：`process-service → Kafka(calc-task) → metric-service → Redis/VMS`
3. **配置同步链路**：`北向API → north-sync/process-service → PostgreSQL`
4. **查询链路**：`前端 → platform-service → Redis/VMS/PostgreSQL`

## 七、共性设计模式

### 1. 配置管理
- 所有服务采用**环境变量**配置，统一使用 `envconfig` 或 `viper` 库
- 环境变量前缀：`AIDC_{SERVICE_NAME}_`
- 支持 `.env` 文件本地开发

### 2. 数据库访问
- 统一使用 **GORM** ORM 框架
- PostgreSQL 连接池配置：最大连接 20、空闲连接 10、最大生命周期 1h
- 支持 TLS 加密连接

### 3. Redis 客户端
- 支持**单机模式**和**集群模式**
- 统一连接池大小 2000
- 超时配置：读写 15s、连接池 15s

### 4. 优雅关闭
- 所有服务监听 `SIGINT`/`SIGTERM` 信号
- 使用 `context.WithCancel` 传播关闭信号
- `defer` 确保资源释放

### 5. 日志
- 统一使用 `klog`（Kubernetes 风格）
- 结构化日志：`klog.InfoS()`、`klog.ErrorS()`

### 6. 项目结构
```
service-name/
├── cmd/service-name/main.go   # 服务入口
├── pkg/                       # 核心业务代码
│   ├── api/                   # API 层（gRPC/HTTP Handler）
│   ├── base/                  # 基础组件初始化
│   ├── config/                # 配置定义
│   ├── middleware/            # 中间件（认证等）
│   ├── models/                # 数据模型
│   ├── service/               # 业务逻辑层
│   └── storage/               # 存储层（Redis/VMS/Kafka）
├── charts/                    # Helm Chart
├── Dockerfile                 # 容器镜像
├── Makefile                   # 构建脚本
└── go.mod                     # 依赖管理
```

## 八、面试常见问题及回答思路

### Q1: 介绍一下这个项目的整体架构？
> **回答思路**：这是一套智慧数据中心物联网平台，采用 Go 微服务架构，5 个服务覆盖「采集 → 预处理 → 消费处理 → 指标计算 → 平台展示」全链路。核心用 Kafka 做服务间数据流传输，PostgreSQL 存业务数据，Redis 存实时数据，VictoriaMetrics 存时序历史数据。

### Q2: 为什么选择微服务而不是单体？
> **回答思路**：
> - 各环节处理特性不同（IO 密集型的 process、CPU 密集型的 metric），需要独立扩缩容
> - 预处理和消费处理的吞吐量要求不同，解耦后可独立优化
> - 数据同步模块独立出来，支持多配置、多机房同步
> - 故障隔离：某个服务挂了不影响其他服务

### Q3: Kafka 在项目中的作用是什么？
> **回答思路**：Kafka 在项目中有两个核心作用：
> 1. **数据流传输**：pretreat-service 将预处理后的设备数据写入 Kafka，process-service 消费并双写到 Redis 和 VictoriaMetrics
> 2. **计算任务分发**：process-service 将需要计算的任务写入 `aidc-calc-task` topic，metric-service 消费执行公式计算，失败的任务进入 DLQ 进行补偿重试

### Q4: Redis 和 VictoriaMetrics 为什么要双写？
> **回答思路**：
> - **Redis**：存储设备数据点的**最新实时值**，供业务系统快速查询（毫秒级响应）
> - **VictoriaMetrics**：存储**历史时序数据**，用于趋势分析、报表、监控大屏
> - 两者互补：Redis 解决「现在是什么值」，VMS 解决「过去一段时间的变化趋势」

### Q5: 如何保证数据一致性？
> **回答思路**：
> - Kafka 使用 Consumer Group + 手动提交 offset：处理成功后才提交，失败不提交（at-least-once）
> - 去重机制：VictoriaMetrics 写入有时间窗口去重（DedupCache）
> - 北向同步使用版本号增量同步，避免重复处理
> - Redis 使用分布式锁（SETNX）防止多 Pod 重复初始化

### Q6: 如何处理高并发场景？
> **回答思路**：
> - pretreat-service：异步任务队列 + Worker Pool（100 个 Worker，队列容量 10000）
> - process-service：Kafka 消费者组 + 限流（token bucket 算法）+ 工作池并发处理
> - metric-service：并发 Worker Pool（100 个）+ 拓扑排序按层级并行计算
> - Redis Pipeline 批量写入、VMS 缓冲批量提交

### Q7: 项目中有哪些设计亮点？
> **回答思路**：
> 1. **流式计算引擎**（metric-service）：支持公式热加载、拓扑排序、级联触发
> 2. **DLQ 补偿机制**：计算失败的任务进入死信队列，自动重试
> 3. **大数据预处理**（pretreat-service）：根据数据大小自动选择处理策略（直写 vs 压缩）
> 4. **多配置动态调度**（north-sync）：支持运行时动态管理多个定时同步任务
> 5. **分布式锁保护**：计算点初始化使用 Redis SETNX + 锁续期

## 十、总结

AIDC 平台是一个**典型的物联网数据管理系统**，核心价值在于：

1. **全链路数据流**：从设备采集到展示的端到端数据处理能力
2. **高吞吐量**：Kafka + Worker Pool + 限流控制，支持大规模设备数据处理
3. **实时 + 历史**：Redis + VictoriaMetrics 双写，满足不同查询场景
4. **可扩展性**：微服务架构，各服务可独立扩缩容
5. **运维友好**：Docker + K8s + Helm，标准化部署流程
