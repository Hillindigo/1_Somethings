# Kafka 数据消费详解

> 本文档详细描述整个项目中点位数据的完整流动路径：从 `aidc-north-sync` 拉取原始点位，经由 `aidc-platform-service` 完成计算点转化，再到 `aidc-metric-service` 实时计算，以及数据并发安全与定时任务版本一致性分析。

---

## 1. 整体数据流架构

```
北向接口 (North API)
       │  HTTP 分页拉取
       ▼
aidc-north-sync
  - 版本比对 + 增量同步
  - 存入本地 DB (bms_point / bms_device / dcom_location)
       │  DB 落库
       ▼
aidc-platform-service
  - 用户手动触发"原始点 → 计算点"转化 (HandleConvertToMetricPoints)
  - 写入 dcim_metric_points 表
  - 同步写入 Redis Hash: formula:{source}  field={dpid}  value=JSON
       │  Redis 写入
       ▼
aidc-metric-service (MetricPointLoader)
  - 启动时从 Redis 扫描 formula:* 加载依赖图 (L1 atomic.Value)
  - 每 5 分钟周期热重载
       │  接收上游点位数据
       ▼
aidc-process-service  ──Kafka K1──▶  aidc-pretreat-service  ──Kafka K2──▶  aidc-metric-service
  消费原始点数据                          预处理/压缩                           calc-task 消费 → 公式计算
  写 Redis R1 ({source} Hash)
  写 Redis R2 (r2:{source}:{dpid}:{ts})
  查 Redis point:{source}:{type}
  触发 calc-task 消息
       │  计算结果
       ▼
  Redis R1 HSET CAS 写入
  VictoriaMetrics 时序写入
  级联触发下游计算点 (cascade trigger)
```

## 2. aidc-north-sync：点位数据拉取

### 2.1 触发方式

| 触发类型 | 说明 |
|---------|------|
| **手动触发** | 通过 HTTP API 提交同步任务 |
| **定时任务** | Scheduler 基于 Cron 表达式周期触发 |

### 2.2 拉取核心逻辑

文件：`aidc-north-sync/pkg/service/sync_service.go`

```go
// SyncAllNorthNodesWithStats 完整同步流程
func (s *SyncService) SyncAllNorthNodesWithStats(...) error {
    // 1. 获取本地配置版本
    s.getConfigVersion()

    // 2. 调用北向 HTTP 接口（分页，第1页）
    resp, _ := s.getjhconfigure(pageNo=1)

    // 3. 版本比对：remoteVersion == localVersion 且非强制同步 → 跳过
    if versionMatch && !ignoreVersion {
        return nil  // 版本未变，不拉取
    }

    // 4. 分页拉取全量数据 → 存入本地 DB
    //    表: bms_point、bms_device、dcom_location
}
```

**关键设计：**
- **版本控制**：每次拉取前与本地版本号对比，版本相同则跳过，避免重复同步
- **分页处理**：支持大量点位数据的分页拉取，防止单次请求超时
- **事务保障**：数据写入本地 DB 使用事务，保证原子性

### 2.3 定时调度器

文件：`aidc-north-sync/pkg/scheduler/scheduler.go`

```go
cronInstance.AddFunc(cronExpr, func() {
    // 每次执行时从 DB 重新读取最新配置（而非闭包缓存）
    latestConfig, _ := s.configDAO.GetByName(configName)
    s.taskManager.SubmitTask(s.ctx, latestConfig, false, "auto", "")
})
```

**关键设计：每次 Cron 触发时都从 DB 重新读取最新配置**，确保定时任务使用的是最新参数（包括新的 API 地址、Token 等）。

## 3. aidc-platform-service：计算点位转化

### 3.1 计算点是什么

- **原始点（BmsPoint）**：由 `aidc-north-sync` 从北向接口同步而来，存于 `bms_point` 表，代表设备传感器的实际采集点
- **计算点（DcimMetricPoints）**：由用户在 `aidc-platform-service` 中基于原始点定义的聚合/公式点，存于 `dcim_metric_points` 表

### 3.2 原始点 → 计算点转化流程

文件：`aidc-platform-service/pkg/service/metric_points.go:HandleConvertToMetricPoints`

```
用户选择若干 BmsPoint IDs
        │
        ▼
1. 检查是否已存在对应计算点（ExistsByBmsPointID 通过 cal_parameter LIKE 查询）
        │
        ▼
2. 查询原始点对应设备 (bms_device)
        │
        ▼
3. 查询设备位置祖先链，构建 remark（位置路径）
        │
        ▼
4. 构建 cal_parameter JSON（包含 id/type/source/device_id/dpid 等字段）
   示例：
   {
     "params": [{
       "id": 123456,          // bmsPoint.ID
       "type": "00",          // 原始点类型
       "source": "D_001",     // 数据源标识
       "dpid": "ciid_xxx",    // 原始点 CIID（dpid）
       "device_ciid": "...",
       "device_name": "UPS-01",
       "location_id": 789,
       "point_slug": "A"      // 公式变量名
     }]
   }
        │
        ▼
5. 生成雪花 ID，写入 dcim_metric_points 表
   CalIntervalType = "00"（实时）
   CalMethod       = "00"（直接取值）
   CalFormula      = "A"（直接引用变量 A）
        │
        ▼
6. 同步写入 Redis Hash
   key:   formula:{source}
   field: {dpid}（计算点 Dpid = 雪花ID字符串）
   value: JSON（DcimMetricPoints 全量数据）
```

### 3.3 Redis 数据结构（平台服务维护）

| Redis Key 模式 | 类型 | 用途 |
|---------------|------|------|
| `formula:{source}` | Hash | 存储该 source 下所有计算点配置，field=dpid，value=JSON |
| `point:{source}:{type}` | Hash | 存储原始点位映射，field=`{"dpid":"xxx","id":N}`，value=`"1"` |

**重要：`aidc-platform-service` 在每次创建/更新/删除计算点时都会同步维护 Redis。**  
服务启动时还会通过 `InitMetricPointsToRedis` 批量全量初始化（带分布式锁防止多 Pod 重复执行）。

## 4. aidc-metric-service：计算点依赖图加载

### 4.1 MetricPointLoader（L1 缓存）

文件：`aidc-metric-service/pkg/calc/loader.go`

```go
type MetricPointLoader struct {
    redisClient    *storage.RedisClient
    depKeyPattern  string        // "formula:*"
    reloadInterval time.Duration // 默认 5 分钟
    index          atomic.Value  // L1 热路径缓存，atomic 无锁读
}
```

**加载流程：**

```
1. Redis SCAN "formula:*" → 获取所有 source 的 key 列表
        │
        ▼
2. 对每个 key 执行 HGETALL → 获取该 source 下所有计算点 JSON
        │
        ▼
3. 解析 cal_parameter → 提取依赖关系（原始点 dpid / 计算点 dpid）
   过滤：仅加载 cal_method="02"（rate 模式）的计算点
        │
        ▼
4. 拓扑排序（TopologicalSort）+ DAG 环检测
        │
        ▼
5. BuildDependencyIndex → rawToCalc（原始点 → 下游计算点）
                        → calcToCalc（计算点 → 下游计算点）
        │
        ▼
6. atomic.Value.Store(newIndex)  // 无锁原子替换，不阻塞读路径
```

**重要：`atomic.Value` 保证了热重载时的无锁安全——读路径始终拿到完整的旧 index 或新 index，永远不会读到撕裂的中间状态。**

## 5. 实时计算数据流：Kafka Topic 定义与消费

### 5.1 Kafka Topic 总览

| Topic 名称 | 生产者 | 消费者 | 消息内容 |
|-----------|-------|-------|---------|
| `aidc-pretreat`（K1） | 设备数据采集端 | `aidc-pretreat-service` | 原始设备点位上报数据 |
| `aidc-calc-task`（K2） | `aidc-metric-service`（TriggerEvaluator） | `aidc-metric-service`（CalcProcessor） | 计算任务消息（CalcTaskMessage） |
| `aidc-calc-task-dlq` | `aidc-metric-service`（CalcProcessor） | `aidc-metric-service`（DLQHandler） | 死信队列，存放计算失败消息 |

> Topic 名称通过配置文件注入，`calc_topic`、`dlq_topic`、`group_id` 等均在 `aidc-metric-service/pkg/config/config.go` 中定义。

### 5.2 K2 消息结构：CalcTaskMessage

文件：`aidc-metric-service/pkg/calc/message.go`

```go
type CalcTaskMessage struct {
    CalcDpid       string            // 计算点 DPID
    CalcName       string            // 计算点名称
    CalFormula     string            // 公式表达式，如 "A+B"
    CalMethod      string            // 计算方法："" | "delta" | "rate"
    Source         string            // 数据源标识（与 R1 hash key 对应）
    Values         []DependencyValue // 依赖点当前值列表（小依赖数内嵌）
    TriggerTs      int64             // 触发时间戳（Unix 秒）
    TriggerType    string            // "raw"（原始点触发）| "calc"（级联触发）
    IdempotencyKey string            // 幂等键：calc_dpid:trigger_ms:config_version
    SnapshotRef    *SnapshotRef      // 大依赖数时使用 Redis Snapshot 传递
    HopCount       int               // 级联深度计数器，防止无限递归
    ConfigVersion  uint64            // 依赖图版本号
}
```

### 5.3 实时计算触发路径

```
设备原始点数据到达（aidc-process-service 消费 K1）
        │
        ▼ 写入 Redis
  R1: HSET {source} {dpid} {json_value_with_ts}   // 最新值
  R2: SET r2:{source}:{dpid}:{ts_sec} {float}  EX 900  // 时间点精确值
        │
        ▼ 查询 Redis point:{source}:00 → 找到关联的计算点 dpid
        │
        ▼ aidc-metric-service CalcTriggerEvaluator.Evaluate()
  1. 拿一次 index 快照（防止批次内 index 撕裂）
  2. 遍历本批次原始点 → idx.Lookup(rawDpid) → 找下游计算点列表
  3. 对每个下游计算点：
     a. fetchAndAlignDependencies（R2 优先 → R1 fallback）
     b. 时间偏斜检查（MaxSkewMs=5000ms）
     c. 构建 CalcTaskMessage → 发送到 aidc-calc-task topic
        key=CalcDpid（按计算点分区，保证顺序）
```

### 5.4 CalcProcessor 计算处理流程

文件：`aidc-metric-service/pkg/calc/processor.go`

```
消费 aidc-calc-task topic 消息
        │
        ▼
1. 反序列化 CalcTaskMessage
        │
        ▼
2. 幂等检查：Redis GET "calc:idem:{IdempotencyKey}"
   → 已存在则跳过（防止重复消费）
        │
        ▼
3. 深度检查：HopCount >= 10 → 发 DLQ，停止（防循环）
        │
        ▼
4. 分布式锁：SETNX "calc:lock:{IdempotencyKey}" EX 30s
   → 抢锁失败则跳过（同一任务并发互斥）
        │
        ▼
5. 获取依赖值（三种路径）：
   a. SnapshotRef != nil → 从 "calc:snap:{SnapID}" 读取（大依赖数 >50）
   b. Values 为空 + index 有配置 → fetchDependencyValues（R2→R1 双缓存）
   c. Values 已内嵌 → 直接使用
        │
        ▼
6. 从 L1 Index 获取预编译 AST，或按需即时编译公式
        │
        ▼
7. delta/rate 模式：从 "calc:prev:{CalcDpid}" 读取上一次值，注入 __prev_value__
        │
        ▼
8. FormulaEngine.EvaluateAST() → 得到计算结果 float64
        │
        ▼
9. 写入 Redis R1（CAS Lua 脚本，防乱序覆盖）：
   HSET {source} {calc_dpid} {json}
        │
        ▼
10. 写入 Redis R2（时间点精确值，SET EX 900）：
    SET r2:{source}:{calc_dpid}:{ts_sec} {value}
        │
        ▼
11. 写入 VictoriaMetrics（best-effort，失败不阻塞）
    写入 calc_point_completeness 指标
        │
        ▼
12. 标记幂等完成："calc:idem:{IdempotencyKey}" TTL=25h
        │
        ▼
13. 级联触发：TriggerCascade → idx.LookupCalcDownstream(calcDpid)
    → 找下游计算点 → 再次发送 CalcTaskMessage 到 aidc-calc-task
    HopCount+1, TriggerType="calc"
```

### 5.5 双缓存读取策略（R1 / R2）

| 缓存层 | Redis Key 格式 | 内容 | TTL |
|-------|---------------|------|-----|
| **R1（最新值）** | `{source}` (Hash, field={dpid}) | 设备最新值 JSON（含时间戳、quality） | 无 TTL（长期） |
| **R2（时间点精确值）** | `r2:{source}:{dpid}:{ts_sec}` | 该时间点的 float 值 | 900 秒 |

**读取优先级：R2 优先（时间点精确）→ R1 fallback（最新值，可能有时间偏斜）**

## 6. 数据并发安全分析：north-sync 重新拉取时正在计算的数据是否受影响

### 6.1 两条数据通路完全独立

| 通路 | 操作的数据层 | 相互影响 |
|-----|------------|---------|
| `aidc-north-sync` 重新拉取 | 写 DB（`bms_point`、`bms_device`、`dcom_location`） | **不直接影响 Redis R1/R2** |
| `aidc-metric-service` 计算 | 读 Redis R1/R2，写 Redis R1/R2 + VictoriaMetrics | 不读取 north-sync 的 DB 表 |

**结论：north-sync 重新拉取点位数据只更新 DB，与正在进行的实时计算（Redis + Kafka）完全解耦，正在计算的数据不会受到影响。**

### 6.2 计算点配置更新路径（潜在影响点）

north-sync 重新拉取后，如果触发了 `aidc-platform-service` 的计算点重新配置（用户手动操作），则：

```
platform-service 更新 dcim_metric_points
        │
        ▼ 同步更新
  Redis formula:{source} Hash 更新
        │
        ▼ 最快 5 分钟后
  metric-service MetricPointLoader 热重载
  atomic.Value 原子替换 Index
        │ 新到的原始点触发时
        ▼
  使用新 Index 配置进行计算（公式/依赖更新生效）
```

**在热重载期间（5 分钟窗口内），仍在进行的计算使用旧 index 快照，不会中途切换。** 这由 `CalcTriggerEvaluator.Evaluate()` 中的 `idx := e.indexFunc()` 一次性获取快照来保证。

### 6.3 幂等键防止重复计算

即使 north-sync 重新拉取导致数据重放（相同的原始点数据再次到来），`CalcProcessor` 的幂等机制也会阻止重复计算：

```go
idemKey := "calc:idem:" + task.IdempotencyKey
// IdempotencyKey = calc_dpid:trigger_ms:config_version
// 只要触发时间戳相同，就会被认定为重复任务而跳过
```

### 6.4 CAS 写入防乱序

Redis R1 写入使用 Lua CAS 脚本（`WriteDevicesWithTimestampCAS`），只有当新结果的时间戳 >= 当前存储的时间戳时才覆盖，**天然防止了延迟消息覆盖更新结果的问题**。

## 7. 定时任务计算使用的数据版本分析

### 7.1 定时任务（north-sync 侧）

文件：`aidc-north-sync/pkg/scheduler/scheduler.go`

```go
cronInstance.AddFunc(cronExpr, func() {
    // 每次执行时从 DB 重新读取最新配置
    latestConfig, _ := s.configDAO.GetByName(configName)
    s.taskManager.SubmitTask(s.ctx, latestConfig, false, "auto", "")
})
```

**结论：定时任务每次触发时都从 DB 重新读取最新配置，而不是使用闭包缓存的旧配置。** 因此若在两次定时执行之间修改了北向接口配置，下一次定时执行会自动使用新配置。

### 7.2 metric-service 定时热重载

文件：`aidc-metric-service/pkg/calc/loader.go`

`MetricPointLoader` 的 `reloadInterval`（默认 5 分钟）决定了计算点依赖图的更新频率：

```
时间线：
t=0    用户在 platform-service 修改计算点公式
t=0    Redis formula:{source} 立即更新
t=0~5m metric-service 仍使用旧 Index 进行计算
t=5m   MetricPointLoader.Reload() 触发
       → 重新扫描 Redis formula:*
       → 重新构建 Index（含新公式）
       → atomic.Value 原子替换
t>5m   所有新到的触发消息使用新 Index 计算
```

**结论：**
- **配置修改后最长 5 分钟内生效**（热重载窗口）
- 热重载期间正在处理的批次不受影响（使用 batch 开始时的 index 快照）
- **已在 Kafka 队列中待处理的 CalcTaskMessage 携带了 `ConfigVersion`**，processor 可通过版本号检测配置是否过期（当前实现以 Index 中的最新版本为准）

### 7.3 定时任务计算数据版本总结

| 组件 | 定时刷新频率 | 使用数据版本 |
|-----|------------|------------|
| `aidc-north-sync` Cron | 按配置（如每小时） | 每次执行时从 DB 读最新配置 |
| `aidc-metric-service` Index 热重载 | 每 5 分钟 | Redis `formula:*` 最新快照 |
| `aidc-metric-service` 单批计算 | 实时（毫秒级） | 批次开始时的 Index 快照（不中途切换） |
| Redis R2 时间点精确值 | 实时写入 | TTL=900s，超时后降级使用 R1 |

## 8. 关键设计亮点总结

### 8.1 三层隔离保证数据安全

```
DB 层（north-sync 写）
  ↕ 解耦
Redis 层（platform-service 维护 formula 配置）
  ↕ 解耦
计算层（metric-service 消费 Kafka + 读 Redis）
```

三层完全解耦，`north-sync` 重新拉取不会干扰正在进行的计算。

### 8.2 atomic.Value 无锁热重载

`MetricPointLoader` 使用 `sync/atomic.Value` 存储依赖图 Index，写操作（热重载）和读操作（计算触发）之间无锁竞争，避免了 `sync.RWMutex` 在高并发下的性能损耗。

### 8.3 幂等 + 分布式锁双重保障

- **幂等键（calc:idem）**：TTL=25h，防止 Kafka 重放导致重复计算
- **分布式锁（calc:lock）**：TTL=30s，防止同一时刻多个 Pod 并发计算同一任务

### 8.4 级联计算深度保护

`HopCount >= MaxCascadeDepth(10)` 时直接发 DLQ 并停止级联，从根本上防止了循环依赖导致的无限递归。

### 8.5 大依赖数 Snapshot 路径

当依赖数 > 50 时，依赖值写入 Redis Snapshot（`calc:snap:{snapID}`，TTL=24h），K2 消息只携带 `SnapshotRef`，避免 Kafka 消息体过大。

Q：计算点位是在 aidc-platform 进行转化的吗？

是的。aidc-platform-service 提供 HandleConvertToMetricPoints 接口，用户选择若干原始点（BmsPoint），系统自动生成计算点（DcimMetricPoints）：

构建 cal_parameter JSON（含原始点 ID、类型、source、公式变量名等）
写入 dcim_metric_points 表
同步写入 Redis formula:{source} Hash（这是后续计算引擎的配置来源）
Q：数据的实时计算是怎么进行的？

三个关键 Topic / 阶段：

Topic	角色
aidc-calc-task（K2）	metric-service 的 CalcTriggerEvaluator 生产，CalcProcessor 消费
R2 缓存 r2:{source}:{dpid}:{ts}	时间点精确值，TTL=900s
R1 缓存 {source} Hash	最新值，fallback 读取
计算流程：原始点到达 → 写 R1/R2 → 查 index 找下游计算点 → 发 K2 消息 → CalcProcessor 消费 → 公式计算 → 写 R1/R2 + VictoriaMetrics → 级联触发下游

Q：north-sync 重新拉取时，正在计算的数据会受影响吗？

不会。 两条通路完全解耦：

north-sync 只写 DB（bms_point 等）
实时计算只读写 Redis R1/R2
计算引擎的 index 通过 atomic.Value 无锁存储，热重载时不会撕裂正在处理的批次
幂等键（calc:idem）+ CAS 写入进一步防止重复/乱序
Q：定时任务的计算是拿更新后的数据进行计算的吗？

两个层面：

north-sync 定时调度：每次 Cron 触发都从 DB 重新读最新配置，不用缓存
metric-service Index 热重载：每 5 分钟扫描 Redis formula:* 重建依赖图，修改计算点公式后最长 5 分钟内生效
