# AIDC 数据中心智能运维平台 — 流式计算模块简历

---

## 项目概览（标准简历模板）

- **项目名称**：数据中心多源异构设备数据实时流式采集与指标计算平台
- **技术栈**：Go 1.21+、Kafka（segmentio/kafka-go、Sarama）、Redis（Hash + Lua CAS）、VictoriaMetrics、MinIO（zstd）、gRPC、Gin、Worker Pool、DAG 拓扑排序
- **项目描述**：面向数据中心动环、BMS 等多源设备的实时数据接入与指标计算系统。系统每周期处理数十万设备数据点的上报，通过三级流水线（预处理 → 数据解析双写 → 指标流式计算）实现端到端亚秒级延迟，支撑运维监控、能耗分析等核心业务场景。

## 主要职责（STAR 方法精炼版）

### 职责一：设计并实现大报文自适应分片预处理管道（aidc-pretreat-service）

**S（背景）**：
设备数据通过 APISIX 网关以 HTTP 请求方式上报，单次 Batch 报文可达数百 MB（如含 10 万+ 数据点的 BMS 批量上报）。直接投递到 Kafka 会超出 Broker 的消息大小限制（默认 1 MB），导致消息丢失或被拒绝。

**T（任务）**：
负责设计并开发 `aidc-pretreat-service` 的核心数据流处理链路，解决大报文无损分片投递问题，保证预处理吞吐不成为整体链路瓶颈。

**A（方案）**：
- 基于 **Body Size 动态分支策略**：以 10 MB 为阈值区分大小数据路径。
  - **小数据（< 10 MB）**：直接序列化为内联消息，发送至 Kafka `raw-small` Topic。
  - **大数据（≥ 10 MB）**：先使用 **zstd 压缩**（压缩率可达 6~10x），再上传至 MinIO 对象存储（`aidc-raw-data` Bucket），仅将含 `bucket/object` 路径的轻量引用索引投入 Kafka `raw-large-ref` Topic。
- 针对**超大 JSON 报文的内存问题**，使用 `jsoniter` 实现**流式 JSON 解析**（`StreamChunkProcessor`），以 800 KB 为单个 Chunk 阈值，在遍历 devices 数组时**边读边切片边发送**，从根本上消除了一次性反序列化整个大 Body 的内存峰值。
- 采用 **Worker Pool** 模型（`worker.Pool`）异步消费 HTTP 请求，支持背压控制，队列满时主动拒绝并上报告警，保证上游 HTTP 接口的低延迟响应。
- 实现**企业级 BatchProducer**（基于 Sarama `AsyncProducer`），支持批量发送（BatchSize=100，BatchTimeout=100ms）、消息缓冲（BufferSize=10000）和 4 并发发送，通过 TraceID 作为 Partition Key 确保同一批次消息的顺序投递。

**R（结果）**：
- 大报文通过 zstd 压缩后体积缩减 **80%+**，解决了 Kafka 消息大小限制问题。
- 流式分片处理将内存峰值从 O(Body大小) 降至 O(ChunkSize)，避免了大报文引起的 OOM 风险。
- HTTP 接口 P99 延迟维持在毫秒级，Worker Pool 解耦了 IO 等待与业务处理。

### 职责二：构建多 Topic 实时消费与双写存储架构（aidc-process-service）

**S（背景）**：
数据中心接入动环（jinhua）、电池管理系统（bms-*）、原始数据（raw）等多种数据源，各 Topic 数据格式、频率、优先级差异大。下游业务系统（boss-jh-gateway）需要毫秒级读取最新设备状态，同时运营分析需要完整的历史时序数据。

**T（任务）**：
设计 `aidc-process-service` 的 Kafka 多 Topic 消费框架及 Redis/VictoriaMetrics 双写存储链路，保证实时性与数据完整性并重，同时防止消费速度过快压垮下游存储。

**A（方案）**：
- 实现**多 Topic Consumer Group** 框架（`kafka.Consumer`），每个 Topic 启动独立消费 Goroutine，统一注册 `MessageHandler` 接口，使 Topic 扩展无需改动框架层代码。
- 引入**基于令牌桶的 Topic 级别独立限流器**（`TopicRateLimiter`），支持对每个 Topic 单独配置 Rate/Burst，防止突发流量击穿 Redis 或 VictoriaMetrics。
- `MessageProcessor` 收到消息后，通过 **`ParserFactory` 解析器工厂模式**按 Topic 匹配对应解析器（支持 jinhua、raw、bms 等格式），将异构数据统一转换为 `[]models.Device` 结构。
- 解析完成后，**并发双写**（`sync.WaitGroup` + 独立 Goroutine）：
  - **Redis**：`HSET {source} {point_guid} {json}` Hash 结构，与 boss-jh-gateway 格式完全兼容，单批最多写入 10000 个数据点。
  - **VictoriaMetrics**：缓冲批量提交时序数据（`device_point_value` 指标），配置去重间隔，避免重复写入。
- 使用 **Worker Pool（`workerPool chan struct{}`）** 控制最大并发度，防止消息积压时瞬间起爆大量 Goroutine。
- 消息处理失败后走**指数退避重试**（`retry.go`），超出重试次数后记录告警日志，保证处理链路不因单条坏消息阻塞 Partition。

**R（结果）**：
- 实现了动环、BMS 等多源数据接入的统一化，新增数据源只需新增 Parser，无需修改消费框架。
- 双写并发化消除了顺序写入的串行等待，Redis 与 VictoriaMetrics 写入延迟相互隔离，Redis 写入失败不影响时序数据落盘。

### 职责三：实现基于 DAG 的级联流式指标计算引擎（aidc-metric-service）

**S（背景）**：
数据中心能耗、温湿度等派生指标（如 PUE、冷却效率）依赖多个原始数据点通过公式计算得出，且指标间存在多级依赖关系（如 A 依赖原始点，B 依赖 A，C 依赖 B、A）。要求实现：任一原始数据点更新时，自动触发其所有下游派生指标的实时重算；计算结果需防乱序覆盖，并支持高频触发时的合并去抖。

**T（任务）**：
设计并开发 `aidc-metric-service` 的流式触发评估器与计算处理器，实现基于 DAG 依赖图的全自动级联计算。

**A（方案）**：

**① DAG 依赖图构建与热加载（`dag.go` + `loader.go`）**
- 从 Redis `formula:{source}` Hash 中加载所有配置的派生指标（计算点），使用 **Kahn 算法（拓扑排序）** 对依赖图进行层次化校验，自动检测循环依赖并排除跨 source 的异常配置（软排除，不中断服务）。
- 计算索引（`CalcDependencyIndex`）通过 `atomic.Value` 实现**无锁热更新**，后台每 5 分钟从 Redis 重新加载，新配置生效无需重启服务。

**② 事件驱动触发评估器（`CalcTriggerEvaluator`）**
- 每条原始数据点更新消息处理后，调用 `Evaluate()` 查询依赖索引，找出所有以该原始点为依赖的计算点，构建 `CalcTaskMessage` 投入 `calc-task` Kafka Topic。
- 针对高频数据点（如每秒上报），引入 **`CoalescingBuffer` 合并缓冲区**（100ms flush 间隔）：同一计算点的多次触发在窗口内**只保留最新时间戳**的一条，每 100ms 由后台 Goroutine 统一刷出，避免对同一指标的重复计算。
- 大依赖场景（依赖数 > 50）使用 **SnapshotRef 快照路径**：先将全量依赖值序列化写入 Redis `calc:snap:{id}`（TTL 24h），消息中仅传递 `snap_id`，解决 Kafka 单条消息大小限制问题。

**③ 幂等计算处理器（`CalcProcessor`）**
- 10 步处理流水线：反序列化 → **幂等校验**（`calc:idem:{key}`，TTL 25h）→ **Redis 分布式锁**（`calc:lock:{key}`，TTL 30s，处理完立即释放）→ SnapshotRef 加载 → 公式 AST 编译 → delta/rate 增量计算 → **公式求值** → Redis CAS 写入 → VictoriaMetrics 写入 → 触发级联下游。
- **分布式锁与幂等键分离设计**：`calc:lock` 仅用于并发互斥（短 TTL），`calc:idem` 负责持久幂等防重（长 TTL），两者职责明确，彻底消除 "并发重算但结果覆盖" 的问题。
- **Lua CAS 防乱序写入**：Redis 写入使用 `WriteDevicesWithTimestampCAS`（Lua 脚本 CAS），只有当新结果时间戳 ≥ Redis 中当前时间戳时才允许写入，防止 Kafka 消费乱序导致旧值覆盖新值。
- delta/rate 方法支持**增量状态管理**：从 Redis `calc:prev:{dpid}` 读取上一次的值与时间戳，注入 `__prev_value__` 变量参与公式求值，首次到达时初始化 prev 状态并跳过求值（但仍触发级联，保证下游计算点有机会执行）。

**④ 级联防风暴设计**
- 最大级联深度限制为 **10 层**（`MaxCascadeDepth`），超限消息投入 DLQ，配合 DAG 静态环检测双重防护，彻底杜绝无限递归。
- `CascadeRateLimiter`（令牌桶，per-calc_dpid）限制每个计算点的级联触发频率（默认 RPS=2，burst=10），防止大量上游并发完成时引发"DAG 风暴"。
- 级联 `TriggerCascade()` 使用**上游触发时间戳**（而非 `time.Now()`）传递给下游，确保多级级联下 R2 缓存 Key 时间轴一致。

**R（结果）**：
- 实现了全自动的多级派生指标实时计算，任意深度的指标依赖链路均可自动触发，无需人工干预。
- 通过 CoalescingBuffer + CascadeRateLimiter + 最大深度限制三重机制，在高频数据上报场景下有效防止了计算风暴。
- CAS 防乱序 + 幂等 Key + 分布式锁机制，保证了在多副本部署、Kafka 重平衡、消费乱序等复杂情况下结果的正确性与一致性。

## 常见面试深问与应答思路

### Q1：CoalescingBuffer 为什么用 Map 而不是队列？
**PREP 答法**：  
- **P（结论）**：Map 天然支持按 `calc_dpid` 去重，只保留最新时间戳，队列无法做到 O(1) 去重。  
- **R（原理）**：同一计算点在 100ms 内可能被触发多次（高频上报），重复计算等价，只需最后一次结果。Map 以 `dpid` 为 Key，每次 Add 只更新 `triggerMs` 更大的那条，Flush 时原子置换整个 Map，读写互斥通过 `sync.Mutex` 保护。  
- **E（项目实例）**：见 `coalescing.go`，Add 方法只在 `triggerMs > existing.triggerMs` 时更新，Flush 原子清空并返回切片，避免了 flush 期间新增被清掉的竞态。  
- **P（总结）**：Map 去重 + Mutex 保护 = 简单高效的合并窗口，比 Channel + 去重队列实现更清晰。

### Q2：分布式锁与幂等 Key 为什么要分开设计？
**PREP 答法**：  
- **P（结论）**：两者职责不同，合并会导致锁 TTL 过长或幂等保护不足。  
- **R（原理）**：`calc:lock` 只用于防止同一任务在多副本上**并发重复执行**，TTL 30s（计算超时即自动释放），计算完成后立即主动 Del，不需要长期保留。`calc:idem` 用于保证即使消息重放（Kafka Redelivery、DLQ 补偿），也不重复写入结果，TTL 25h 覆盖 Kafka 消息最长保留时间。  
- **E（项目实例）**：见 `processor.go`，lock 在 `defer` 中立即 Del，idem 在计算成功后 Set，两个 Key 生命周期完全不同。  
- **P（总结）**：短锁防并发，长幂等防重放，分离设计使两个场景的 TTL 都能精确控制。

### Q3：Lua CAS 如何防止 Kafka 乱序写入？
**PREP 答法**：  
- **P（结论）**：Lua 脚本在 Redis Server 端原子执行 "读 → 比较 → 写"，保证只有时间戳更新的值才能写入成功。  
- **R（原理）**：Kafka 同一 Partition 内消息有序，但多副本消费或重试可能导致旧消息在新消息写入后再次被消费。单纯的 `HSET` 无条件覆盖会导致旧值覆盖新值。Lua CAS 脚本逻辑：先 `HGET` 当前值取出时间戳，若新时间戳 ≥ 当前时间戳才执行 `HSET`，整个操作原子不可分割。  
- **E（项目实例）**：见 `storage.WriteDevicesWithTimestampCAS`，以 `r1HashKey`（= source）为 Hash Key，`dpid` 为 Field，JSON 为 Value，时间戳嵌入 JSON 用于比较。  
- **P（总结）**：Lua CAS 将 "比较并设置" 从应用层搬到 Redis Server 端原子执行，是防乱序的最简洁方案。
