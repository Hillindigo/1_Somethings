# 北向数据同步逻辑详解

> **涉及服务**：`aidc-north-sync`（核心）、`aidc-process-service`（旧版，已抽离）、`aidc-platform-service`（下游消费）
>
> **核心职责**：基于版本号增量同步策略，从北向 API 分页拉取 **位置（Location）/ 设备（Device）/ 点位（Point）** 三级数据，采用事务批量写入 + 异步任务管理架构，支持多数据源动态配置、定时 Cron 调度与手动触发双模式。

---

### 服务间关系

| 服务 | 与北向同步的关系 |
|------|-----------------|
| **aidc-north-sync** | **核心同步服务**。独立部署，负责从北向 API 拉取数据并写入共享数据库 |
| **aidc-process-service** | **旧版同步逻辑的原始宿主**。同步逻辑已抽离到 `aidc-north-sync`，该服务中仍保留旧版代码（单数据源、单调度器） |
| **aidc-platform-service** | **下游消费者**。通过 gRPC 暴露 `NorthSyncStatisticsService`，读取同步后的 `dcom_location`、`bms_device`、`bms_point` 表提供统计查询 |
| **aidc-pretreat-service** | 无直接北向同步逻辑 |
| **aidc-metric-service** | 无直接北向同步逻辑 |

## 二、数据库表结构

### 2.1 配置管理层（2 张表）

#### `north_configure` — 北向接口连接配置表

> 统一管理各数据源的北向 API 连接信息和版本号。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | VARCHAR(50) PK | 主键，格式 `NC` + 自增序号（如 `NC1`、`NC2`） |
| `machine_room` | VARCHAR(200) | 机房标识 |
| `north_api_url` | VARCHAR(500) NOT NULL | 北向 API 接口地址 |
| `north_token` | VARCHAR(500) NOT NULL | 北向 API 认证令牌 |
| `north_source` | VARCHAR(100) NOT NULL | 北向数据源标识（如 `aidc-north-sync`） |
| `version` | VARCHAR(200) | **增量同步版本号**（核心字段，与远程版本对比决定是否同步） |
| `topic` | VARCHAR(200) | 消息主题（预留） |
| `parsing_method` | VARCHAR(200) | 解析方式（预留） |
| `deleted_at` | TIMESTAMP | 软删除时间戳 |

#### `north_sync_config` — 同步任务配置表

> 定义同步任务的调度策略和行为参数，通过 `north_configure_id` 关联具体的北向接口配置。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INT PK | 自增主键 |
| `name` | VARCHAR(200) UNIQUE | 配置名称（唯一） |
| `sync_cron` | VARCHAR(100) | Cron 表达式，默认 `0 0 * * *`（每天零点） |
| `enable_auto_sync` | BOOL | 是否启用自动定时同步 |
| `north_configure_id` | VARCHAR(50) | **关联 `north_configure.id`**（如 `NC1`） |
| `sync_idc_info` | BOOL | 是否同步 IDC/园区类型位置数据 |
| `sync_on_startup` | BOOL | 服务启动时是否立即执行一次同步 |
| `http_timeout` | INT | HTTP 客户端超时时间（秒），默认 120 |
| `http_retry` | INT | HTTP 请求重试次数，默认 3 |

**两表关系**：`north_sync_config.north_configure_id` → `north_configure.id`（多对一）

### 2.2 任务执行层（1 张表）

#### `sync_task_history` — 同步任务执行历史表

| 字段 | 类型 | 说明 |
|------|------|------|
| `task_id` | VARCHAR(64) UNIQUE | 任务 UUID |
| `status` | VARCHAR(20) | 状态：`pending` / `running` / `completed` / `failed` |
| `triggered_by` | VARCHAR(20) | 触发方式：`auto`（定时）/ `manual`（手动）/ `api` |
| `triggered_user` | VARCHAR(255) | 手动触发时的操作人 |
| `sync_idc_info` | BOOL | 本次是否同步 IDC 信息 |
| `force_sync` | BOOL | 是否强制同步（忽略版本对比） |
| `started_at` / `completed_at` | TIMESTAMP | 开始/完成时间 |
| `duration_seconds` | INT | 执行耗时（秒） |
| `location_count` / `device_count` / `point_count` | INT | 本次同步的位置/设备/点位数量 |
| `total_pages` | INT | 本次同步拉取的总页数 |
| `version` | VARCHAR(255) | 本次同步对应的远程版本号 |
| `error_message` | TEXT | 失败时的错误信息 |

### 2.3 业务数据层（3 张表 — 同步目标）

| 表名 | 说明 | 冲突策略（Upsert） |
|------|------|-------------------|
| `dcom_location` | 位置信息（园区/楼宇/楼层/机房等） | `ON CONFLICT (source, guid) DO UPDATE name` |
| `bms_device` | 设备信息 | `ON CONFLICT (source, ciid) DO UPDATE name` |
| `bms_point` | 测点信息 | `ON CONFLICT (source, ciid, device_ciid) DO UPDATE name` |

## 三、同步触发机制（双模式）

### 3.1 定时 Cron 调度（自动触发）

**入口**：`Scheduler.InitFromDB()` → 遍历所有 `enable_auto_sync=true` 的配置 → 每个配置创建独立的 `cron.Cron` 实例

```
服务启动
  │
  ▼
Scheduler.InitFromDB(ctx)
  │
  ├─ 从 DB 加载所有 north_sync_config
  │
  ├─ 过滤 enable_auto_sync = true 的记录
  │
  └─ 对每条记录调用 StartConfig(ctx, configName)
       │
       ├─ 从 DB 读取配置的 sync_cron 表达式
       ├─ 创建独立 cron.Cron 实例
       ├─ 注册 cron 回调函数：
       │     ├─ 从 DB 重新读取最新配置（保证热更新生效）
       │     └─ 调用 taskManager.SubmitTask(ctx, config, force=false, "auto", "")
       └─ cronInstance.Start()
```

**关键设计**：
- **每个配置独立的 Cron 实例**：通过 `entries map[string]*schedulerEntry` 管理，支持按配置名动态启停
- **每次执行重新读取配置**：cron 回调中 `configDAO.GetByName(configName)` 确保配置热更新生效
- **不使用 force**：定时任务 `force=false`，依赖版本号对比决定是否实际同步

### 3.2 手动触发（API 触发）

**入口**：`POST /api/v1/sync-now`

```
HTTP 请求
  │
  ▼
Handler.SyncNow(c)
  │
  ├─ 解析请求体：{ id/name, force, user_id }
  ├─ 通过 configService.GetByKey() 查询 north_sync_config
  └─ 调用 taskManager.SubmitTask(ctx, config, force, "manual", userID)
       │
       └─ 立即返回 { task_id, config_name, status: "pending" }
           （任务在后台异步执行，前端轮询 GET /api/v1/status?task_id=xxx）
```

**另一个手动触发入口**：`POST /api/v1/sync` 的 `TriggerSyncByConfig` —— 通过配置名触发

### 3.3 动态配置管理

通过 RESTful API 实现配置的 CRUD 和调度器的动态热更新：

| 操作 | API | 调度器联动 |
|------|-----|-----------|
| 创建配置 | `POST /api/v1/create` | 自动重载调度器（若 enable_auto_sync=true 则启动 cron） |
| 更新配置 | `PUT /api/v1/update` | 自动重载调度器（停旧启新） |
| 删除配置 | `DELETE /api/v1/delete` | 停止对应的定时任务 |
| 切换自动同步 | `POST /api/v1/toggle-auto-sync` | 动态启停定时任务 |
| 查看活跃调度 | `GET /api/v1/active-schedules` | 返回当前所有运行中的 cron 任务 |

## 四、异步任务管理架构

### 4.1 SyncTaskManager 核心流程

```
SubmitTask / SubmitTaskAndWait
  │
  ▼
prepareAndEnqueue(ctx, config, force, triggeredBy, triggeredUser)
  │
  ├─ 1. 并发安全检查（RWMutex）
  │     ├─ 检查 running 状态
  │     └─ 检查 currentTask == nil（同一时刻只允许一个同步任务执行）
  │
  ├─ 2. 查询北向接口配置
  │     └─ northConfigureDAO.GetByID(config.NorthConfigureID) → 获取 API URL / Token / Source
  │
  ├─ 3. 创建任务记录
  │     ├─ 生成 UUID 作为 TaskID
  │     ├─ 初始状态 = "pending"
  │     └─ 写入 sync_task_history 表
  │
  ├─ 4. 构建 TaskRequest 并入队
  │     ├─ 包含北向接口全部配置（URL、Token、Source、Timeout、Retry）
  │     ├─ 包含 ResultChan（用于同步等待模式）
  │     └─ 写入 taskQueue (buffered chan, capacity=10)
  │
  └─ 5. 返回 taskID
```

### 4.2 任务执行流程

```
processTaskQueue (goroutine，持续从 taskQueue 读取任务)
  │
  ▼
executeTask(ctx, taskReq)
  │
  ├─ 1. 设置 panic 恢复（defer recover）
  │
  ├─ 2. 更新任务状态为 "running"
  │     └─ 设置 startedAt、写入 DB
  │
  ├─ 3. 动态创建 SyncService 实例
  │     └─ createSyncServiceFromRequest(taskReq)
  │         ├─ 使用 taskReq 中的北向配置（URL、Token、Source）
  │         ├─ 创建独立的 HTTPClient（配置的 timeout 和 retry）
  │         └─ 注入 northConfigureDAO 和 nodesDAO
  │
  ├─ 4. 执行同步
  │     └─ syncService.SyncAllNorthNodesWithStats(ctx, syncIDCInfo, force, stats)
  │         （详见第五节）
  │
  ├─ 5. 更新任务最终状态
  │     ├─ 成功 → status = "completed"，记录统计数据
  │     └─ 失败 → status = "failed"，记录 error_message
  │
  ├─ 6. 清除 currentTask（允许下一个任务执行）
  │
  └─ 7. 通过 ResultChan 通知调用方（异步模式自动消费，同步模式阻塞等待）
```

### 4.3 两种提交模式

| 模式 | 方法 | 行为 |
|------|------|------|
| **异步提交** | `SubmitTask()` | 入队后立即返回 taskID，后台 goroutine 消费 ResultChan 避免泄漏 |
| **同步等待** | `SubmitTaskAndWait()` | 入队后阻塞等待 `<-resultChan`，返回 taskID + 最终 TaskStatus + error |

## 五、同步核心逻辑详解（SyncService.SyncAllNorthNodesWithStats）

这是整个北向同步的**核心方法**，完整流程如下：

### 5.1 版本号对比（增量同步策略）

```
Step 1: 获取本地版本号
  │
  ├─ northConfigureDAO.GetVersionByID(northConfigureID)
  │   └─ 从 north_configure 表读取当前 version 字段
  │
  ▼
Step 2: 请求北向 API 第 1 页，获取远程版本号
  │
  ├─ getjhconfigure(pageNo=1)
  │   ├─ POST 请求北向 API
  │   ├─ 请求体：{ idc: null, stage: null, version: "", data: { version: "", page_no: 1, page_size: 500 } }
  │   ├─ Header：Token = northToken
  │   └─ 响应体中的 data.version = 远程版本号
  │
  ▼
Step 3: 版本对比
  │
  ├─ localVersion == remoteVersion && !ignoreVersion(force)
  │     → 版本匹配且非强制同步 → 跳过同步，直接返回 nil
  │
  └─ localVersion != remoteVersion || ignoreVersion(force)
        → 版本不匹配或强制同步 → 继续执行数据拉取
```

**关键点**：
- 版本号存储在 `north_configure.version` 字段中
- `ignoreVersion=true`（即 `force=true`）时跳过版本对比，强制全量同步
- 定时任务默认 `force=false`，依赖版本号实现增量同步
- 手动触发可选择是否 `force=true`

### 5.2 分页数据拉取

```
pageNo = 1
  │
  ▼
for 循环 {
  │
  ├─ 检查 ctx.Done()（支持取消）
  │
  ├─ getjhconfigure(pageNo)
  │     ├─ POST 北向 API
  │     ├─ page_size = 500（固定）
  │     └─ 返回 ResponseData { version, page_no, page_size, size, total, nodes[] }
  │
  ├─ 终止条件判断：
  │     ├─ resp == nil → break
  │     ├─ len(resp.Nodes) == 0 → isLastPage = true, break
  │     └─ pageNo * pageSize >= total → 所有页拉完，break（分页终止保护）
  │
  ├─ 解析 nodes（详见 5.3）
  ├─ 事务批量写入（详见 5.4）
  │
  ├─ 打印进度日志（含 DB 连接数监控）
  ├─ time.Sleep(1s)（限速保护，避免对北向 API 产生过大压力）
  └─ pageNo++
}
```

### 5.3 三级数据解析（Node → Location / Device / Point）

北向 API 返回的每个 `Node` 通过 `node_type` 字段区分类型：

| node_type | 值 | 目标表 | 目标模型 |
|-----------|---|--------|---------|
| 1 | `NodeTypeLocation` | `dcom_location` | `Location` |
| 2 | `NodeTypeDevice` | `bms_device` | `BmsDevice` |
| 3 | `NodeTypePoint` | `bms_point` | `BmsPoint` |

#### Location（位置）解析逻辑

```go
// 层级计算规则：
// path = "/"            → level = 1（园区/IDC 顶层）
// path = "xxx/yyy"      → level = 2（楼宇层）
// path = "xxx/yyy/zzz"  → level = 3（楼层/机房层）
// space_type = "FLOOR" 且 level=3 → level = 5（楼层特殊标记）
// space_type = "Building" | "楼宇" → level = 2（强制为楼宇层）

// IDC/园区 类型过滤：
// space_type = "IDC" | "园区" 且 syncIDCInfo=false → 跳过不同步
```

映射关系：

| Node 字段 | Location 字段 | 说明 |
|-----------|--------------|------|
| `tag` | `slug` | 位置标识符 |
| `name` | `name` | 位置名称 |
| `guid` | `guid` | 全局唯一标识（用于 Upsert 冲突检测） |
| `parent_guid` | `parent_guid` | 父节点 GUID |
| `path + tag` | `raw_location_path` | 原始路径（如 `/site1/building1/floor1`） |
| `space_type` | `location_type` | 空间类型 |
| 计算得出 | `level` | 层级 |
| — | `source` | 数据源标识（`northSource`） |

#### Device（设备）解析逻辑

| Node 字段 | BmsDevice 字段 | 说明 |
|-----------|---------------|------|
| `tag` | `code` | 设备编码 |
| `name` | `name` | 设备名称 |
| `guid` | `ciid` | 配置项 ID（用于 Upsert 冲突检测） |
| `device_type` | `device_type` | 设备类型 |
| `device_type` | `alias` | 设备别名 |
| `parent_guid` | `location_guid` | 所属位置的 GUID（后续通过 TrimDeviceLocations 关联为 location_id） |
| — | `source` | 数据源标识 |

#### Point（测点）解析逻辑

| Node 字段 | BmsPoint 字段 | 说明 |
|-----------|--------------|------|
| `tag` | `code` | 测点编码 |
| `name` | `name` | 测点名称 |
| `guid` | `ciid` | 配置项 ID |
| `point_type` | `ci_type` | 测点类型 |
| `path` | `alias` | 测点路径别名 |
| `parent_guid` | `device_ciid` | 所属设备的 CIID |
| `unit` | `unit` | 测点单位 |
| — | `source` | 数据源标识 |

### 5.4 事务批量写入（每页一个事务）

```
每页数据解析完成后：
  │
  ▼
dbt := db.Begin()                        // 开启事务
txDAO := nodesDAO.WithTx(dbt)            // 创建事务作用域的 DAO
  │
  ├─ txDAO.CreateLocations(newLocations)  // 批量写入位置
  │     └─ ON CONFLICT (source, guid) DO UPDATE name   （Upsert：冲突时更新 name）
  │
  ├─ txDAO.CreateBmsDevices(newDevices)   // 批量写入设备
  │     └─ ON CONFLICT (source, ciid) DO UPDATE name
  │
  ├─ txDAO.CreateBmsPoints(newPoints)     // 批量写入测点
  │     └─ ON CONFLICT (source, ciid, device_ciid) DO UPDATE name
  │
  ├─ 任一步骤失败 → dbt.Rollback()（回滚整页数据）
  └─ 全部成功 → dbt.Commit()（提交整页数据）
```

**关键设计**：
- **页级事务粒度**：每 500 条数据（一页）一个事务，在性能和数据一致性间取得平衡
- **Upsert 语义**：使用 PostgreSQL 的 `ON CONFLICT ... DO UPDATE` 实现幂等写入，重复数据只更新 `name` 字段
- **事务 DAO 隔离**：通过 `WithTx(tx)` 创建事务作用域的 DAO 实例，避免污染全局 DB 连接

### 5.5 后处理阶段（全量数据关联）

分页拉取和写入完成后，执行以下后处理：

#### 5.5.1 位置路径树构建

```
Step 1: 从 DB 加载所有 Location 记录
  │
  ▼
Step 2: 构建索引映射
  ├─ locationGUIDToID:      GUID → ID
  ├─ tagToLocationID:       Slug(Tag) → ID
  └─ floorTagToLocationID:  Slug(Tag) → ID（仅 level=5 的楼层）

Step 3: 遍历所有位置，填充关联字段
  │
  ├─ 条件：raw_location_path != "" && location_path == ""
  │
  ├─ location_path 构建：
  │     将 raw_location_path（如 "site1/building1/floor1"）
  │     按 "/" 分割 → 查找每段 tag 对应的 ID → 拼接为 ",101,9001,9002,"
  │
  ├─ parent_id：通过 parent_guid 查 locationGUIDToID 映射
  │
  ├─ floor_id：如果路径层级 > 3，取第 3 段 tag 对应的楼层 ID
  │
  ├─ location_ext：格式 "ID,Slug"
  │
  └─ 逐条 UPDATE 到数据库
```

#### 5.5.2 设备-位置关联（TrimDeviceLocations）

```sql
-- 通过 GUID 关联，将设备的 location_guid 映射为实际的 location_id 和 location_slug
UPDATE bms_device bd
SET location_id = dl.ID, location_slug = dl.slug
FROM dcom_location dl
WHERE bd.location_guid = dl.guid
  AND bd."source" = dl."source";
```

### 5.6 版本号更新

```
所有页拉取完成 && 远程 version 非空
  │
  ▼
northConfigureDAO.UpdateVersion(northConfigureID, remoteVersion)
  │
  └─ UPDATE north_configure SET version = ?, updated_at = NOW() WHERE id = ?
```

**作用**：将远程最新版本号持久化到 `north_configure.version`，下次定时同步时版本匹配则跳过，实现增量策略。

## 六、多数据源动态配置

### 6.1 架构设计

```
north_sync_config (同步策略)          north_configure (接口配置)
┌──────────────────────┐           ┌──────────────────────┐
│ id=1                 │           │ id=NC1               │
│ name="配置A"         │──────────▶│ north_api_url=...    │
│ north_configure_id=  │  关联     │ north_token=...      │
│   "NC1"              │           │ north_source="src-a" │
│ sync_cron="0 0 * * *"│           │ version="v2.1"       │
│ enable_auto_sync=true│           └──────────────────────┘
└──────────────────────┘
┌──────────────────────┐           ┌──────────────────────┐
│ id=2                 │           │ id=NC2               │
│ name="配置B"         │──────────▶│ north_api_url=...    │
│ north_configure_id=  │  关联     │ north_token=...      │
│   "NC2"              │           │ north_source="src-b" │
│ sync_cron="0 6 * * *"│           │ version="v1.5"       │
│ enable_auto_sync=true│           └──────────────────────┘
└──────────────────────┘
```

**关键设计**：
- **配置与接口分离**：`north_sync_config` 定义"怎么同步"（Cron、超时、重试），`north_configure` 定义"同步哪里"（URL、Token、Source）
- **动态创建 SyncService**：每次任务执行时，根据 `TaskRequest` 中携带的北向配置**动态创建** `SyncService` 实例，而非复用全局单例
- **每个配置独立的 Cron**：Scheduler 为每个 `north_sync_config` 维护独立的 `cron.Cron` 实例，互不干扰
- **版本号按数据源隔离**：每个 `north_configure` 记录独立维护自己的 `version`，不同数据源的增量策略互不影响

### 6.2 配置热更新流程

```
API 更新配置 → ConfigService.Update()
  │
  ├─ 更新 DB 中的配置字段
  └─ 调用 reloadScheduler(configName)
       │
       ├─ 重新从 DB 读取最新配置
       ├─ enable_auto_sync = true  → scheduler.StartConfig()（先停旧再启新）
       └─ enable_auto_sync = false → scheduler.StopConfig()
```

## 七、HTTP 客户端与重试机制

### 7.1 HTTPClient 配置

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `timeout` | 120s | 单次 HTTP 请求超时 |
| `retry` | 3 | 最大重试次数 |
| `MaxIdleConns` | 100 | 最大空闲连接 |
| `MaxIdleConnsPerHost` | 10 | 每主机最大空闲连接 |
| `IdleConnTimeout` | 90s | 空闲连接超时 |

### 7.2 重试策略

```
for attempt := 0; attempt <= retry; attempt++ {
    if attempt > 0 {
        sleep(attempt * 1s)     // 线性退避：第 1 次重试等 1s，第 2 次等 2s...
    }
    发送请求
    if 成功(2xx) → return body
    if 失败 → 记录 lastErr, continue
}
return error("failed after N attempts: lastErr")
```

### 7.3 安全措施

- **Token 脱敏**：日志中仅打印 Token 前 8 位 + `***`
- **TLS 验证**：`InsecureSkipVerify: false`（生产环境验证服务端证书）
- **限速保护**：每页拉取之间 `time.Sleep(1s)`，避免对北向 API 过载

## 九、与旧版 aidc-process-service 的差异对比

| 特性 | aidc-process-service（旧版） | aidc-north-sync（新版） |
|------|---------------------------|----------------------|
| **数据源支持** | 单数据源（配置写死在环境变量中） | 多数据源（通过 `north_configure` 表动态管理） |
| **版本号存储** | `north_configure_version` 独立表 | 合并到 `north_configure.version` 字段 |
| **调度器** | 单个全局 Cron 实例 | 每个配置独立 Cron 实例，支持按配置名动态启停 |
| **任务管理** | 直接引用全局 `SyncService` | 每次执行动态创建 `SyncService`（`createSyncServiceFromRequest`） |
| **DAO 架构** | 直接操作 `gorm.DB`，函数式调用 | 独立 DAO 层（`NorthConfigureDAO`、`NorthNodesDAO`、`TaskDAO`），支持事务注入 |
| **版本对比** | 本地 version 非空即跳过（不与远程对比） | **始终请求远程获取最新 version，与本地精确对比** |
| **Location ID** | 手动管理 ID（按 level 分段自增，易冲突） | 数据库序列自动生成，避免主键冲突 |
| **分页终止** | 仅依赖 `len(nodes)==0` | 增加 `pageNo * pageSize >= total` 的终止保护，避免无限循环 |
| **API 管理** | 无配置管理 API | 完整的 RESTful API（配置 CRUD、北向配置 CRUD、任务管理） |
| **同步等待** | 仅异步 | 支持异步（`SubmitTask`）和同步等待（`SubmitTaskAndWait`）两种模式 |

## 十、下游服务如何消费同步数据

### aidc-platform-service

通过 gRPC 暴露 `NorthSyncStatisticsService`，直接查询同步写入的三张业务表：

```protobuf
service NorthSyncStatisticsService {
  rpc GetLocationCount(Empty)  → CountResponse  // SELECT COUNT(*) FROM dcom_location
  rpc GetDeviceCount(Empty)    → CountResponse  // SELECT COUNT(*) FROM bms_device
  rpc GetPointCount(Empty)     → CountResponse  // SELECT COUNT(*) FROM bms_point
  rpc GetSourceCount(Empty)    → CountResponse  // SELECT COUNT(DISTINCT source) FROM bms_device
}
```

同时，`aidc-platform-service` 还提供丰富的数据查询 API（位置列表、设备列表、测点列表、按位置查询测点等），均基于 `dcom_location`、`bms_device`、`bms_point` 三张表进行 JOIN 查询。

## 十一、关键设计总结

| 设计点 | 实现方式 | 目的 |
|--------|---------|------|
| **增量同步** | 版本号对比（local vs remote） | 减少无效同步，节省资源 |
| **幂等写入** | PostgreSQL `ON CONFLICT ... DO UPDATE` | 重复执行不会产生脏数据 |
| **事务保护** | 每页数据一个事务（Begin/Commit/Rollback） | 保证页级数据一致性 |
| **并发控制** | `sync.RWMutex` + `currentTask` 互斥 | 同一时刻只允许一个同步任务执行 |
| **异步解耦** | buffered channel（taskQueue, capacity=10） | 请求提交与实际执行解耦 |
| **动态配置** | 数据库驱动 + 热重载调度器 | 支持运行时动态增删改数据源和调度策略 |
| **容错恢复** | panic recover + 错误状态持久化 | 异常不会导致进程崩溃，任务失败可追溯 |
| **限速保护** | 每页间隔 1s + HTTP 重试退避 | 避免对北向 API 产生过大压力 |
| **Token 安全** | 日志脱敏（仅显示前 8 位） | 防止敏感信息泄露 |
