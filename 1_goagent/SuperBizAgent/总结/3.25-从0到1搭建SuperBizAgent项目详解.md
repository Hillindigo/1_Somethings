## 一、项目定位与整体架构

### 1.1 项目是什么

SuperBizAgent 是一个**企业级智能 AI Agent 后端服务**，核心能力有三个：

| 能力 | 说明 |
|------|------|
| **智能对话（RAG 增强）** | 普通/流式对话，结合向量数据库做知识召回 |
| **知识库管理** | 上传 Markdown 文档，自动切块、向量化、入库 |
| **AI 自动化运维（AIOps）** | 自动拉取告警、查日志、查文档、生成运维报告 |

**2.3 封装 Milvus 客户端（含自动初始化）**

这是项目搭建的关键设计：`utility/client/client.go` 中的 `NewMilvusClient` 函数具备**自动建库、自动建集合、自动建索引**的能力：

```go
// utility/client/client.go 核心逻辑
func NewMilvusClient(ctx context.Context) (cli.Client, error) {
    // 1. 连接 default 数据库
    // 2. 检查 agent 数据库是否存在 → 不存在则创建
    // 3. 切换到 agent 数据库连接
    // 4. 检查 biz collection 是否存在 → 不存在则创建
    //    字段：id(VarChar/主键) + vector(BinaryVector/dim=81920) + content(VarChar) + metadata(JSON)
    //    索引：id/content 用 AUTOINDEX(L2)，vector 用 AUTOINDEX(HAMMING)
    // 5. 返回已就绪的客户端
}
```

**设计要点**：这样做的好处是不需要手动初始化数据库，第一次运行时自动完成所有建表操作，降低了部署门槛。

### 第四步：构建知识库索引 Pipeline

**`internal/ai/agent/knowledge_index_pipeline/orchestration.go`**

这是一个 **3 节点线性 DAG**：

```
START → FileLoader → MarkdownSplitter → MilvusIndexer → END
```

| 节点 | 组件 | 职责 |
|------|------|------|
| `FileLoader` | `eino-ext/document/loader/file` | 读取文件原始内容 |
| `MarkdownSplitter` | `eino-ext/splitter/markdown` | 按 `#` 标题切块，每块生成 UUID |
| `MilvusIndexer` | `eino-ext/indexer/milvus` | 向量化后写入 Milvus |

**切块策略**：
```go
// 按 Markdown 一级标题切块
config := &markdown.HeaderConfig{
    Headers:     map[string]string{"#": "title"},
    TrimHeaders: false,
    IDGenerator: func(ctx context.Context, originalID string, splitIndex int) string {
        return uuid.New().String() // 每块生成唯一 UUID
    },
}
```

**为什么按 `#` 切块**：告警处理手册每个 `#` 下是一种告警的完整处理方案，按标题切块保证每个 chunk 语义完整，不会把一个告警的处理步骤拆到两个 chunk 里。

### 第五步：构建对话 Agent（RAG + React）

**`internal/ai/agent/chat_pipeline/orchestration.go`**

这是项目最核心的 Agent，一个**并行 + 汇聚的 DAG 图**：

```
                    ┌─→ InputToRag  →  MilvusRetriever  ─┐
START ─→ (并行分流) │                                      ├─→ ChatTemplate → ReactAgent → END
                    └─→ InputToChat ─────────────────────┘
```

**完整 DAG 编排代码解析**：

```go
// 节点1: 从用户消息中提取 Query 字符串，用于 RAG 检索
g.AddLambdaNode(InputToRag, InvokableLambdaWithOption(newInputToRagLambda))

// 节点2: ChatTemplate 模板节点，组装最终 Prompt
g.AddChatTemplateNode(ChatTemplate, chatTemplateKeyOfChatTemplate)

// 节点3: React Agent（核心），带工具调用能力
g.AddLambdaNode(ReactAgent, reactAgentKeyOfLambda)

// 节点4: Milvus 向量检索，output key 设为 "documents" 匹配 Prompt 模板占位符
g.AddRetrieverNode(MilvusRetriever, milvusRetrieverKeyOfRetriever,
    compose.WithOutputKey("documents"))

// 节点5: 从用户消息中提取 content/history/date，用于 Prompt 模板
g.AddLambdaNode(InputToChat, InvokableLambdaWithOption(newInputToChatLambda))

// 边：START 同时触发 InputToRag 和 InputToChat（并行）
g.AddEdge(compose.START, InputToRag)
g.AddEdge(compose.START, InputToChat)
// 检索链路：InputToRag → MilvusRetriever → ChatTemplate
g.AddEdge(InputToRag, MilvusRetriever)
g.AddEdge(MilvusRetriever, ChatTemplate)
// 历史/内容链路：InputToChat → ChatTemplate
g.AddEdge(InputToChat, ChatTemplate)
// ChatTemplate 等待所有前驱节点完成后触发
g.AddEdge(ChatTemplate, ReactAgent)
g.AddEdge(ReactAgent, compose.END)

// AllPredecessor：所有前驱节点都完成才触发当前节点
g.Compile(ctx, compose.WithNodeTriggerMode(compose.AllPredecessor))
```

**Prompt 模板设计（`prompt.go`）**：

```
系统提示 (System):
  # 角色：对话小助手
  ## 核心能力 / 互动指南 / 输出要求 ...
  ## 上下文信息
  - 当前日期：{date}           ← 由 InputToChat Lambda 注入
  - 相关文档：{documents}      ← 由 MilvusRetriever 召回后注入

历史消息占位符 (MessagesPlaceholder):
  {history}                    ← 由 InputToChat Lambda 注入

用户消息 (User):
  {content}                    ← 由 InputToChat Lambda 注入
```

**React Agent 工具集（`flow.go`）**：

### 第六步：构建 AIOps Plan-Execute-Replan Agent

**`internal/ai/agent/plan_execute_replan/`**

AIOps 使用 Eino 的 `planexecute` 预制 Agent，分为三个角色：

#### 6.1 Planner（规划者）
```go
// 使用 DeepSeek R1（深度思考模型）做规划
func NewPlanner(ctx context.Context) (adk.Agent, error) {
    planModel, _ := models.OpenAIForDeepSeekV31Think(ctx)
    return planexecute.NewPlanner(ctx, &planexecute.PlannerConfig{
        ToolCallingChatModel: planModel,
    })
}
```
**职责**：接收用户的复杂任务描述，拆解为有序的子任务列表。

#### 6.2 Executor（执行者）
```go
// 使用 DeepSeek V3（快速模型）+ 全套工具执行
func NewExecutor(ctx context.Context) (adk.Agent, error) {
    // 工具列表：MCP日志 + Prometheus告警 + 内部文档 + 获取时间
    // MaxIterations: 999999（不限制迭代次数）
}
```
**职责**：逐一执行 Planner 拆解的子任务，调用相应工具。

#### 6.3 Replanner（重新规划者）
```go
// 同样用 DeepSeek R1
func NewRePlanAgent(ctx context.Context) (adk.Agent, error) {
    model, _ := models.OpenAIForDeepSeekV31Think(ctx)
    return planexecute.NewReplanner(ctx, &planexecute.ReplannerConfig{
        ChatModel: model,
    })
}
```
**职责**：每个子任务执行完后，根据执行结果判断是否需要调整后续计划，或直接终止输出最终结论。

#### 6.4 AIOps 任务描述（`chat_v1_ai_ops.go`）

```
固化的 Prompt 流程：
1. 调用 query_prometheus_alerts 获取所有活跃告警
2. 根据每个告警名称调用 query_internal_docs 查找处理方案
3. 完全遵循内部文档，不使用文档外信息
4. 时间参数先调用 get_current_time 获取当前时间
5. 日志查询调用 MCP 日志工具（携带地域和日志主题ID）
6. 生成结构化运维报告：
   - 活跃告警清单
   - 告警根因分析（每个告警）
   - 处理方案执行（每个告警）
   - 结论
```

## 五、核心数据流详解

### 5.1 普通对话数据流

```
用户发送 POST /api/chat {Id, Question}
  ↓
Controller 取出历史: mem.GetSimpleMemory(id).GetMessages()
  ↓
构造 UserMessage{ID, Query, History}
  ↓
BuildChatAgent() 构建 DAG Graph
  ↓
并行执行:
  分支A: Query → Milvus 向量检索 → documents
  分支B: Query+History+Date → map变量
  ↓
ChatTemplate 汇聚: 组装 [System(含documents,date), History, User(content)]
  ↓
ReactAgent (DeepSeek V3):
  如需工具调用 → 调用工具 → 继续推理（最多25步）
  ↓
返回最终 Message
  ↓
保存到内存: User消息 + Assistant消息
  ↓
返回 {answer: "..."}
```

### 5.2 知识库索引数据流

```
上传 POST /api/upload (multipart form, file=xxx.md)
  ↓
保存文件到 FileDir
  ↓
加载文件获取 _source 路径
  ↓
Milvus 删除旧数据（幂等）
  ↓
BuildKnowledgeIndexing() 构建索引 Pipeline
  ↓
FileLoader → 读取文件内容
  ↓
MarkdownSplitter → 按 # 标题切块，每块生成 UUID
  ↓
MilvusIndexer:
  对每个 chunk 调用豆包 Embedding → 得到 81920 维二进制向量
  写入 Milvus: {id=UUID, vector=..., content=chunk文本, metadata={_source=文件路径,...}}
  ↓
返回 [id1, id2, ...] 已入库的 chunk ID 列表
```

### 5.3 AIOps 数据流

```
POST /api/ai_ops
  ↓
固定 Prompt（6步运维分析任务描述）
  ↓
BuildPlanAgent():
  Planner (DeepSeek R1): 分析任务，拆解子步骤
    → 子步骤1: 查询 Prometheus 告警
    → 子步骤2: 根据告警名查内部文档
    → 子步骤3: 查询相关日志
    ...
  ↓
  循环 Executor (DeepSeek V3):
    → 调用 query_prometheus_alerts → 获取活跃告警列表
    → 调用 query_internal_docs("服务下线") → 获取处理方案
    → 调用 get_current_time → 获取当前时间戳
    → 调用 MCP日志工具(关键字, 时间范围, 日志主题ID) → 查询日志
    → 汇总分析
  ↓
  Replanner (DeepSeek R1): 检查是否还有未完成任务
    → 若有: 继续执行新计划
    → 若无: 输出最终报告
  ↓
返回: {result: 完整报告, detail: 每步详情}
```

## 九、常见坑点与注意事项

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| Milvus 向量维度不匹配 | `dim=81920` 是豆包 Embedding 特定维度 | Schema 中 vector 字段 dim 必须与 Embedding 模型一致 |
| `query_prometheus_alerts` 返回空 | 函数首行 `return PrometheusAlertsResult{}, nil` 是提前返回占位 | 需替换为真实 Prometheus 地址 |
| `GetLogMcpTool` panic | `mcp_url == ""` 时主动 panic | 需按教程申请腾讯云 CLS MCP URL |
| MySQL CRUD 工具需手动确认 | 控制台 `scanner.Scan()` 等待输入 `y` | 这是安全设计，防止 Agent 误操作 |
| 会话历史重启后丢失 | 使用内存 Map，无持久化 | 生产环境可改用 Redis 存储 |
| React Agent MaxStep=25 | 工具调用超过25步会停止 | 复杂任务可适当调大，AIOps 已设为 999999 |

## 十、架构总结

SuperBizAgent 的设计理念可以用一句话概括：

> **用 Eino DAG 图编排 AI 流程，用 Milvus 存储知识，用 GoFrame 暴露接口，用 DeepSeek 提供推理能力，用 MCP 协议接入外部工具**。

核心创新点：
1. **RAG + React Agent 融合**：对话时同时做向量检索增强上下文和工具调用，两路并行汇聚
2. **Plan-Execute-Replan 三阶段**：AIOps 用深度思考模型规划、快速模型执行、深度思考模型重规划，用正确的模型做正确的事
3. **自动初始化基础设施**：Milvus 客户端自动建库建集合建索引，降低部署门槛
4. **幂等知识库更新**：文件重复上传时先删旧数据再入库，通过 `_source` 元数据实现幂等
5. **滑动窗口内存管理**：成对丢弃历史消息，保证对话上下文连贯性
