# GopherAI 简历优化版

## 一、我对这个项目的判断

`GopherAI` 这个项目适合往 **Go 后端 + AI 应用开发 + 智能体工程化** 方向写，但它和 `SuperBizAgent` 的定位不一样。

这个项目更像一个 **面向个人用户的多模型 AI 助手平台**，核心能力集中在：

- **多模型统一接入**：OpenAI、RAG、MCP、Ollama 通过工厂模式统一管理。
- **会话式聊天能力**：支持新建会话、历史会话、普通问答、SSE 流式输出。
- **用户级 RAG 知识增强**：用户上传文件后，系统自动向量化并写入 Redis 向量索引，再结合检索结果增强回答。
- **多模态能力扩展**：包含图片识别、TTS 语音合成等能力。
- **基础平台能力**：JWT 鉴权、邮箱验证码注册、Redis、MySQL、RabbitMQ 异步落库。

从简历角度看，这个项目的优势是：

- **技术面更宽**，比单纯聊天项目更丰富。
- **工程味更足**，有模型抽象、会话管理、异步消息、鉴权、文件上传、向量检索。
- **适合体现“平台型项目”思路**，而不只是单点功能开发。

但也有几个地方要 **谨慎表述**：

- **MCP 能力目前更适合写成“完成协议接入与工具调用验证”**
  - 从代码实现看，当前是基于两阶段 Prompt + JSON 解析 + 调用本地 `MCP Server` 的方式完成工具调用。
  - 可写成“完成 MCP 工具接入闭环验证”，但不建议写成“成熟通用工具平台”。

- **RAG 目前属于轻量实现，不建议写太重**
  - 现在使用的是 **Redis 向量索引**，且每个用户默认只维护一个上传文件。
  - `IndexFile` 当前还是把整份文档作为一个文档对象写入，没有做完善的语义分块。
  - 所以更适合写“实现用户级轻量 RAG 检索增强”，不要上来就写“大规模知识库平台”。

- **图片识别和 TTS 可以作为加分项，不建议喧宾夺主**
  - 图片识别是本地 `ONNX + MobileNetV2` 推理。
  - TTS 是典型异步任务 + 查询结果模式。
  - 这两个点适合放在项目亮点里补充，但主线还是聊天平台和模型能力编排。

## 二、这个项目真实能支撑的简历亮点

结合代码实现，我认为最值得写进简历的是下面这些点：

- **多模型统一抽象与扩展机制**
  - 通过 `AIModel` 接口 + `AIModelFactory` 实现多模型统一接入。
  - 已接入 OpenAI、RAG 模型、MCP 模型、Ollama，本质上体现的是可扩展的模型编排设计。

- **用户 + 会话双维度的 AIHelper 管理机制**
  - 通过 `AIHelperManager` 维护 `userName -> sessionID -> AIHelper` 的映射关系。
  - 每个会话绑定独立消息上下文，支持多会话历史隔离。

- **流式聊天链路**
  - 支持创建新会话时先返回 `sessionId`，再通过 `SSE` 持续推送增量内容，最后发送 `[DONE]`。
  - 这类设计很适合体现实时交互和前后端协同能力。

- **消息异步持久化**
  - 消息先进入内存会话，再通过 `RabbitMQ` 异步投递，由消费者落库到 MySQL。
  - 这个点能体现你对“会话响应速度”和“持久化解耦”的工程思考。

- **用户级轻量 RAG**
  - 用户上传文档后，系统会清理旧文件和旧索引、重新建 Redis 向量索引。
  - 检索时根据当前用户目录自动定位知识文件，并将召回结果拼接进 Prompt，增强回答质量。

- **MCP 工具调用验证链路**
  - 模型先判断是否需要调用工具，再由后端解析工具请求并调用 MCP Server，最后结合工具结果生成二次回答。
  - 虽然不是完整 ReAct Agent，但非常适合写成“完成外部工具接入闭环验证”。

- **多模态补充能力**
  - 图片识别：本地 `ONNXRuntime` 推理。
  - TTS：异步创建任务、轮询查询结果。

## 三、推荐写进简历的版本

### 版本一：推荐正式写进简历的版本

**GopherAI 多模型智能问答平台**  
**项目介绍：** 面向个人用户场景，构建集多模型问答、用户级知识增强、流式会话、多模态能力于一体的 AI 助手平台。  
**技术栈：** Go、Gin、Eino、Redis、MySQL、RabbitMQ、JWT、SSE、ONNXRuntime、MCP

- **负责多模型统一接入框架设计与实现**，基于工厂模式抽象 `OpenAI`、`RAG`、`MCP`、`Ollama` 等模型能力，支持按模型类型动态创建会话助手，提升系统扩展性。
- **负责会话式聊天模块开发**，实现会话创建、历史记录查询、普通问答与 `SSE` 流式输出能力，并通过 `userName + sessionID` 维度维护独立上下文，支持多会话隔离。
- **负责消息持久化链路设计**，将会话消息先写入内存上下文，再通过 `RabbitMQ` 异步投递并落库到 `MySQL`，降低同步写库对聊天响应链路的影响。
- **负责轻量级 RAG 能力落地**，实现用户文档上传、旧索引清理、Redis 向量索引构建与检索增强问答流程，支持基于用户私有文档进行知识增强回答。
- **完成 MCP 工具调用链路验证**，实现模型工具选择、工具参数解析、MCP Server 调用及结果二次整合回答，验证外部工具接入 AI 助手的可行性。
- **扩展多模态能力**，接入基于 `ONNXRuntime` 的图片识别能力与异步 `TTS` 语音合成能力，提升平台能力边界。

### 版本二：如果你想更强调“平台设计”

**GopherAI 多模型智能助手平台**  
**项目介绍：** 基于 Go 构建多模型 AI 助手平台，支持会话式问答、私有知识增强、外部工具调用验证及图片识别、语音合成等扩展能力。  
**技术栈：** Go、Gin、Eino、Redis Vector、MySQL、RabbitMQ、JWT、SSE、MCP、ONNX

- **主导 AI 助手核心抽象设计**，通过 `AIModel` 接口、模型工厂与 `AIHelperManager` 统一管理不同模型实例和多会话上下文，形成可扩展的模型接入体系。
- **实现流式问答与会话管理机制**，支持新建会话即时返回 `sessionId`、回答增量推送、历史消息恢复，优化前端实时交互体验。
- **实现用户私有文档知识增强能力**，基于 Redis 向量索引完成文档向量化、检索与 Prompt 注入，支持用户上传资料后的个性化问答。
- **通过 RabbitMQ 解耦消息持久化流程**，将消息异步投递至消费者写库，提升聊天链路吞吐能力并降低数据库写入耦合。
- **完成 MCP 工具接入原型设计**，实现从工具意图识别、协议调用到结果二次生成的闭环，为后续扩展天气、查询类工具奠定基础。
- **扩展图片识别与 TTS 能力**，增强平台从文本助手向多模态助手的演进能力。

## 四、我建议你怎么讲这个项目

如果面试官问你：“这个项目的核心亮点是什么？”

你可以这样答：

> 这个项目我更想把它定义为一个多模型 AI 助手平台，而不是单一聊天应用。核心工作主要有三块：
> 第一块是做了统一的模型抽象，通过工厂模式接入普通大模型、RAG 模型、MCP 工具模型和 Ollama；
> 第二块是做了会话式聊天能力，包括普通问答、流式返回、会话隔离和历史恢复；
> 第三块是做了用户级知识增强和工程化支撑，比如用户上传文件后构建 Redis 向量索引、聊天消息通过 RabbitMQ 异步落库到 MySQL。
> 除此之外，我还扩展了图片识别和 TTS 这种多模态能力，让这个项目更像一个可扩展的 AI 平台，而不是简单的问答 Demo。

## 五、这个项目还可以继续加强的地方

如果你后面还要继续优化这个项目，我建议优先补下面这些：

- **补 RAG 文档切块能力**
  - 目前还是整文档入库，建议补充分块、重叠窗口、元信息过滤等机制。

- **补检索质量优化**
  - 例如：TopK 调优、重排序、按用户文件元数据过滤。

- **补 MCP 工具通用化设计**
  - 当前更偏工具调用验证，后面可以做成通用 Tool Registry 或标准 Tool Calling 流程。

- **补量化指标**
  - 例如：会话响应耗时、流式首字时间、文档检索耗时、支持文档大小、消息吞吐量。

- **补会话持久化恢复能力说明**
  - 你现在已经有“启动时从数据库回填消息初始化 AIHelperManager”的逻辑，这个点其实挺加分，可以作为工程化亮点强调。

## 六、我对这个项目的最终建议

如果让我评价，`GopherAI` 很适合写进简历，但最适合它的定位不是：

- 纯“大模型项目”
- 纯“知识库项目”
- 纯“多模态项目”

它更适合被定义成：

- **多模型 AI 助手平台**
- **带轻量 RAG 和工具调用能力的聊天系统**
- **具备一定工程化设计的 AI 应用后端项目**

所以你在简历里要重点突出：

- **模型抽象与扩展能力**
- **会话管理与流式链路**
- **异步持久化与工程化设计**
- **用户级 RAG 与外部工具接入验证**

不要把重点全压在 `MCP` 或 `RAG` 单点上，这样整体会更稳，也更符合当前项目的真实完成度。
